**Requires (только для USE IT):** платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

**Один агент мы собрали в Фазе 7. Теперь — команда.** Сложные открытые задачи плохо ложатся на одного агента: он упирается в контекст и делает всё последовательно. Фаза 8 — про координацию нескольких агентов, и начинается она с базового паттерна: оркестратор и субагенты.

> **MOTTO.** Координация важнее силы одиночки: оркестратор раздаёт роли, субагенты работают параллельно.

## PROBLEM

Посади одного агента на задачу «исследуй рынок и напиши отчёт» — он забьёт контекст сырыми фактами, перемешает разведку с письмом и пойдёт строго по очереди. Чем шире задача, тем сильнее он захлёбывается: всё в одной голове, всё в одном окне.

Сильнее не «умнее модель», а **команда**: ведущий агент (оркестратор) разбивает задачу на части и раздаёт их специализированным субагентам, у каждого свой контекст и свои инструменты. По данным Anthropic, такой мульти-агентный ресёрч обошёл одиночного агента **на 90.2%** на их внутренней оценке — но ценой токенов (об этом ниже). Соберём механику делегирования.

## CONCEPT

### Интуиция

Это как **редакция, а не один универсал**. Один человек, который и копает факты, и пишет, и редактирует, на большом материале выдыхается и путается. Редакция работает иначе: главред (оркестратор) ставит задачу, раздаёт её ресёрчеру, автору и редактору — каждый со своим столом и своими источниками, — а потом сводит результат. Выигрыш не в том, что люди умнее, а в **разделении труда и изоляции**: ресёрчер копается в сотне источников, но на стол главреду кладёт выжимку, а не весь свой бардак.

Так и субагенты: у каждого **свой контекст**, поэтому «шум» его шагов не засоряет общее окно — наверх идёт только результат.

### Как это работает

```
        задача
          │ оркестратор (lead): план + декомпозиция
   ┌──────┼──────┐
researcher  writer  reviewer     ← субагенты (свой контекст, свои инструменты)
   └──────┼──────┘
          ▼ оркестратор сводит результаты → ответ
```

Это паттерн **orchestrator-workers**: lead делегирует, субагенты выполняют, lead синтезирует.

- `register(role, fn)` — зарегистрировать субагента под роль (`researcher`, `writer`, `reviewer`).
- `delegate(role, task)` — отдать подзадачу конкретному субагенту.
- `run(plan)` — выполнить план `[(role, task)]` по порядку, собрать результаты по ролям; поздние шаги получают результаты ранних через сам `task`.

Ключевое — **изоляция контекста**: субагент `researcher` может сделать 50 шагов и прочитать гору источников, но в общий контекст вернётся только его выжимка. Без изоляции эти 50 шагов засорили бы окно оркестратора и он бы «поплыл» (context rot из 4.4).

## РАЗБОР ПО ШАГАМ

Прогоним оркестратор на цепочке ресёрч → письмо → ревью:

```
register: researcher, writer, reviewer

1. delegate("researcher", "RAG")
   → researcher работает в СВОЁМ контексте → ["факт о RAG #1", "факт о RAG #2"]
      (наверх — только 2 факта, не весь его поиск)

2. run([ ("writer", {"facts": [..2 факта..]}),
         ("reviewer", "черновик по 2 фактам") ])
   writer:   получил факты ранних шагов → "черновик по 2 фактам"
   reviewer: получил черновик          → "проверено: ..."

→ results = {writer: "...", reviewer: "..."}   (по ролям, в порядке плана)
```

Видно две вещи. **Делегирование**: каждая подзадача ушла своему субагенту по роли, неизвестная роль (`designer`) подняла бы ошибку. **Поток результатов**: `writer` работает не с нуля, а с фактами от `researcher` — поздние шаги стоят на ранних. И при этом контекст `researcher` остался при нём: оркестратор оперирует выжимками, а не сырыми шагами субагентов. Поменяй функции-роли на LLM-агентов — схема «делегировал → собрал» не изменится.

## BUILD IT

**Задание: собери оркестратор** — реестр субагентов-ролей, делегирование и сборку плана. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/8.1-orchestrator/`). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `orchestrator.py` с классом `Orchestrator`:

- **`register(role, fn)`** — сохранить субагента `fn` под именем роли.
- **`delegate(role, task)`** — вызвать субагента роли с `task`; если роли нет → `raise KeyError`.
- **`run(plan)`** — `plan` это `list[(role, task)]`; выполнить по порядку, вернуть словарь `{role: результат}`. Поздние шаги используют результаты ранних через содержимое `task`.

**Готово, когда** все тесты в `test_orchestrator.py` зелёные — они проверяют: `delegate` маршрутизирует в нужную роль; неизвестная роль → `KeyError`; `run` собирает результаты по ролям; результаты ранних шагов доходят до поздних (`researcher` → `writer`).

```bash
pytest code -q             # красное → реализуй Orchestrator → зелёное
python code/orchestrator.py # демо: researcher → writer → reviewer
```

**Подсказка.** `subagents` — словарь `{role: fn}`. `delegate` — это `self.subagents[role](task)` с проверкой ключа. `run` копит `results[role]` в цикле по плану.

## USE IT

Agent teams на практике (мульти-провайдер):

- **Claude Code subagents** — изолированные субагенты (`.claude/agents/*.md`), каждый со своим system prompt, контекстом, набором инструментов и моделью (дешёвый Haiku на разведку, Opus на сложное).
- **Claude Agent SDK** — субагенты программно, с параллельным запуском.
- Принципы Anthropic из их мульти-агентного ресёрча: **«think like your agents»** (наблюдай шаги, чтобы видеть провалы), **«scale effort to query complexity»** (простой факт — один агент с 3–10 вызовами; сложное — 10+ субагентов), **«teach the orchestrator how to delegate»** (каждому субагенту — чёткая цель, формат вывода и границы).

⚠️ Цена координации реальна: по Anthropic мульти-агент тратит **~15× токенов** против чата (одиночный агент — ~4×), а 95% разброса в качестве объясняют три фактора — объём токенов, число вызовов инструментов и выбор модели. Бери команду там, где это оправдано (FinOps, Фаза 9).

## ЧАСТЫЕ ОШИБКИ

- **Брать команду, где хватает одного агента.** Мульти-агент дороже в разы (~15× токенов). Простую задачу решает один агент; команда оправдана на широких, распараллеливаемых.
- **Не изолировать контекст субагентов.** Если сырые шаги субагента льются в окно оркестратора, тот «плывёт» (context rot). Субагент возвращает выжимку, а не весь свой процесс.
- **Размытое делегирование.** Субагенту нужны чёткая цель, формат вывода, инструменты и границы. «Поисследуй тему» без рамок — и субагенты дублируют работу или уходят не туда.
- **Делать всё последовательно, когда можно параллельно.** Сила команды — в параллельной работе субагентов с разными контекстами. Линейная цепочка теряет главное преимущество.
- **Считать, что больше агентов = всегда лучше.** Эффорт масштабируют под сложность: лишние субагенты жгут токены без выигрыша. Сложность задачи определяет размер команды.
