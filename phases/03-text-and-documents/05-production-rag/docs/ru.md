**Это венец Фазы 3: базовый RAG доводим до продакшн-уровня.** Один ретривер из 3.3–3.4 в реальности промахивается, а «стало ли лучше» — неизвестно. Прод-RAG добавляет три вещи: **гибрид** двух ретриверов (семантика + точные слова), **переранжирование** слиянием рангов (RRF) и **измеримую оценку** (RAGAS). Собираем гибрид сами, чтобы понять, как сводятся сигналы.

> **MOTTO.** Прод-RAG — это не «один ретривер», а гибрид сигналов, reranking и измеримое качество.

## PROBLEM

Чистый vector-RAG (3.3–3.4) в проде проседает: семантика промахивается на **точных** терминах, кодах, именах (`numpy`, артикул, фамилия), потому что «по смыслу» они близки много к чему. А чистый keyword-поиск промахивается на синонимах и перефразировках. Вдобавок без оценки непонятно, помогла ли правка пайплайна. Прод-RAG объединяет оба сигнала, переранжирует и **меряет** качество ретрива и ответа.

## CONCEPT

### Интуиция

Это как спросить **двух экспертов** с разными слепыми зонами: один судит по смыслу (vector), другой — по точным словам (keyword). Документ, который оба ставят высоко, почти наверняка релевантен. Гибрид берёт лучшее из обоих, а оценка заменяет «мне кажется, лучше» на число.

### Как это работает

```
запрос ─┬─ vector ranking (семантика, TF-IDF/эмбеддинги)
        └─ keyword ranking (точные слова)
                │ Reciprocal Rank Fusion (слияние рангов)
                ▼
        rerank → top-k → ответ
                │
        eval: faithfulness · context precision/recall (RAGAS)
```

- **Два сигнала.** `vector_ranking` — по косинусной близости (из 3.3), ловит смысл и синонимы. `keyword_ranking` — по числу точных совпадений слов, ловит термины и коды, которые семантика «размывает».
- **RRF (Reciprocal Rank Fusion).** Как слить два списка, если их «оценки» несравнимы (косинус 0..1 против числа совпадений)? RRF смотрит только на **ранги**: каждый документ получает `score = Σ 1/(k + rank + 1)` по всем спискам (обычно `k=60`). Документ, стоящий высоко **в обоих** списках, набирает больше всех — консенсус побеждает, и шкалы сигналов сравнивать не нужно.
- **Reranking → top-k.** Берём верх слитого списка. (В проде сюда же ставят cross-encoder-реранкер — отдельную модель, которая переоценивает пары «запрос-документ».)
- **Оценка (RAGAS).** Меряем пайплайн без ручной разметки и **отдельно** по частям: `context precision/recall` — качество **ретрива** (нашли ли нужное и не лишнее), `faithfulness` / `answer relevancy` — качество **ответа** (заземлён ли он на контекст и по делу ли). Так видно, чинить ретрив или генерацию.

## РАЗБОР ПО ШАГАМ

Главное здесь — **как RRF сводит два ранжирования**. Пусть на запрос vector выдал порядок документов `[2, 0, 1]`, а keyword — `[2, 1, 0]` (списки расходятся). Считаем RRF-скор каждого документа, `score = Σ 1/(60 + rank + 1)`:

```
doc2: vector #1 (1/61) + keyword #1 (1/61) = 0.0328   ← топ в обоих → максимум
doc0: vector #2 (1/62) + keyword #3 (1/63) = 0.0320
doc1: vector #3 (1/63) + keyword #2 (1/62) = 0.0320
```

Слитый порядок → `[2, 0, 1]`: **`doc2` первым**, потому что он на вершине обоих списков. А вот `doc0` и `doc1` набрали **поровну** (0.0320) — их взаимный порядок решает тай-брейк: наш код обрабатывает vector-список первым, поэтому при равенстве очков выше встаёт тот, кто раньше попал в счётчик (`doc0`). Ключевая идея: RRF использует только позиции, поэтому честно сводит сигналы с разными шкалами.

И зачем нужен keyword рядом с vector: на запросе вроде `numpy матрицы` keyword **гарантированно** поднимает документ с точным словом `numpy`, даже если чистая семантика поставила бы его ниже. Гибрид через RRF забирает оба попадания — и точный термин, и смысловое.

Реализуй (TF-IDF `build_index` — как в уроке 3.3):

- **`vector_ranking(query, index)`** — верни индексы документов по убыванию косинусной близости запроса к их tf-idf векторам (семантика).
- **`keyword_ranking(query, index)`** — верни индексы по убыванию числа точных совпадений слов запроса со словами документа (лексика).
- **`reciprocal_rank_fusion(rankings, k=60)`** — слей несколько ранжирований: для каждого списка и каждой позиции `rank` (с 0) прибавь документу `1/(k + rank + 1)`; верни индексы по убыванию суммарного score.
- **`hybrid_search(query, index, top_k=3)`** — слей `vector_ranking` и `keyword_ranking` через RRF и верни `top_k` как `[{"doc": ...}]`.

## ЧАСТЫЕ ОШИБКИ

- **Полагаться только на vector.** На точных терминах, кодах и именах семантика промахивается — добавь keyword-сигнал и слей через RRF.
- **Складывать несравнимые скоры.** Косинус и число совпадений в разных шкалах; усреднять их «в лоб» неверно. RRF сводит по рангам, а не по сырым значениям.
- **Оценивать RAG «на глаз» и целиком.** Меряй метриками (RAGAS) и **раздельно**: плохой ответ может быть из-за ретрива (не нашли) или генерации (не использовал контекст) — лечится по-разному.
- **Усложнять раньше времени.** GraphRAG и реранкеры стоят денег и задержки; включай их, когда базовый гибрид измеримо не дотягивает, а не «на всякий случай».
