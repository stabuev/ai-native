# Артефакт: пайплайн обработки длинных документов

Рецепт «как обработать документ, который не влезает (или дорого влезает) в окно». Код — в [`code/chunking.py`](../code/chunking.py).

## Развилка: какой путь выбрать

```
документ помещается в окно с запасом?
  ├─ да, и он небольшой      → одним промптом
  ├─ да, но крупный (≤1M)    → длинный контекст напрямую (данные вверх, вопрос в конец)
  └─ нет / дорого / их много → chunking + map-reduce  ИЛИ  RAG (урок 3.3)
```

## Map-reduce суммаризация

```python
from chunking import chunk_text, map_reduce
chunks = chunk_text(doc, size=800, overlap=100)
summary = map_reduce(chunks,
    map_fn=lambda c: summarize_with_model(c),       # сжать кусок
    reduce_fn=lambda parts: summarize_with_model("\n".join(parts)))  # свести
```

## Параметры под тип документа

| Документ | size (слов) | overlap | Стратегия |
|---|---|---|---|
| Статья/письмо | — | — | один промпт |
| Отчёт (десятки стр.) | 600–1000 | 80–120 | map-reduce |
| Книга/множество файлов | 400–800 | 50–100 | RAG (retrieval, урок 3.3) |
| Таблицы/структура | по секциям | по строкам-заголовкам | structure-aware split |

## Правила

- **Overlap обязателен** — иначе теряется смысл на стыках кусков.
- **Считай бюджет** заранее (`prompt-token-budgeter`, урок 1.1): вход + ответ ≤ окно.
- **Длинное окно ≠ бесплатно** — за токены платишь, «середина» хуже вспоминается; данные клади вверх, вопрос — в конец.
- **Много документов и вопросы по ним** → не суммаризация, а RAG (уроки 3.3–3.4).
