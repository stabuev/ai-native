# Урок 10.1 · Интеграция в рабочие пакеты

**Фаза 10 — Production и интеграция** · **Результат фазы:** Встроить ИИ в рабочий инструмент с мониторингом токенов и стоимости.
<!-- exercise -->

**Requires (только для USE IT):** лицензия Copilot/Gemini или API-ключ. BUILD IT работает офлайн, без ключей и без сети.

**Это вход в Фазу 10 — прод.** Мы умеем строить агентов, считать их стоимость, оркестрировать. Но ИИ полезен, только когда встроен в реальную работу — и встроен **безопасно**, чтобы сбой модели не положил рабочий инструмент.

> **MOTTO.** ИИ полезен, когда встроен в работу: фича-флаг + fallback делают встраивание безопасным.

## PROBLEM

Демо в чате — это не прод. В демо модель «обычно отвечает», и если разок выдала чушь — не страшно. В рабочем процессе (CRM, поддержка, документы) всё иначе: если AI-шаг упал по таймауту или выдал мусор, **весь процесс не должен встать**. Менеджер не может «подождать, пока модель оживёт» — ему нужно закрыть тикет сейчас.

Значит, встраивая ИИ, закладывай две вещи сразу: **фича-флаг** (мгновенно выключить ИИ, не выкатывая релиз) и **graceful fallback** (модель упала или выключена → процесс деградирует на старый детерминированный путь, а не падает). ИИ должен быть **заменяемой деталью**, а не точкой отказа. Соберём такую безопасную обёртку.

## CONCEPT

### Интуиция

Думай про AI-шаг как про **раздвижную дверь с ручным режимом**. В норме она открывается автоматически (ИИ работает). Отключили питание или заклинило — она не запирает людей внутри, а переходит в ручной режим (fallback). И есть рубильник (фича-флаг), которым охрана мгновенно переводит её в ручной, если что-то не так. Дверь полезна автоматикой, но безопасна тем, что **всегда есть запасной путь**.

ИИ в проде — такая дверь: ценность от автоматики, безопасность от запасного пути и рубильника.

### Как это работает

```
существующий процесс
   └─ [ AI-шаг ]
        enabled? ──нет──► fallback (старый детерминированный путь)
        try AI ──ошибка──► fallback
        ok ──► AI-результат
```

- `AIFeature(fn, fallback, enabled)` — обёртка: `fn` (вызов модели), `fallback` (детерминированный запасной путь), `enabled` (фича-флаг).
- `run(x)` — если фича выключена → сразу `fallback`; иначе `try fn(x)`, при **любом** исключении → `fallback`. Прод не падает из-за ИИ.
- `run_pipeline(steps, x)` — ИИ как **один из шагов** существующего пайплайна, а не его центр.

Две независимые защиты: фича-флаг (управляемое отключение — для инцидента или раскатки) и try/except (защита от непредвиденного сбоя). Точки встраивания ищут там, где ИИ даёт ценность (черновик, классификация, саммари), но **всегда** с запасным путём — это делает откат безрисковым.

## РАЗБОР ПО ШАГАМ

Прогоним `AIFeature.run` в трёх состояниях (ИИ-саммари падает на тексте со словом «ошибка», fallback — наивная выжимка):

```
1. enabled=True,  вход "длинный текст для пересказа"
   fn не падает → "AI-саммари: длинный текст для ..."          (ИИ сработал)

2. enabled=True,  вход "текст с ошибкой внутри"
   fn кидает RuntimeError → except → fallback → "первые слова: текст с ошибкой"  (мягкая деградация)

3. enabled=False, вход "любой текст"
   фича выключена → сразу fallback → "первые слова: любой текст"  (рубильник)
```

Все три исхода дали **результат**, ни один не уронил процесс. И в пайплайне `[strip, feat.run, upper]` на входе `"  hi "`: `strip` → `"hi"`, `feat.run` → `"AI:hi"`, `upper` → `"AI:HI"` — AI-шаг встроен в цепочку как обычный шаг, и если он деградирует, цепочка всё равно дойдёт до конца. Это и есть «ИИ — заменяемая деталь»: поменяй `fn` на реальный вызов модели — контракт безопасности не изменится.

## BUILD IT

**Задание: собери безопасную обёртку AI-шага** — фича-флаг + graceful fallback. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/10.1-integration/`), а файлы урока клади в подпапку `code/` (по соглашению курса). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `integration.py`:

- **`AIFeature(fn, fallback, enabled=True)`** — класс-обёртка.
  - **`run(x)`** — если `not enabled` → `fallback(x)`; иначе `try: return fn(x)` / `except Exception: return fallback(x)`.
- **`run_pipeline(steps, x)`** — прогнать `x` через список `steps` (каждый шаг — функция; AI-шаг — один из них) и вернуть результат.

**Готово, когда** все тесты в `test_integration.py` зелёные — они проверяют: при `enabled` используется `fn`; при выключенной фиче — `fallback`; при исключении в `fn` — `fallback` (прод не падает); `run_pipeline` встраивает AI-шаг в цепочку.

```bash
pytest code -q           # красное → реализуй AIFeature → зелёное
python code/integration.py # демо: ИИ работает / упал → fallback / выключен → fallback
```

**Подсказка.** Лови **широкий** `except Exception` намеренно: в проде AI-шаг не должен ронять процесс никаким исключением. Порядок в `run`: сначала проверь флаг, потом try/except.

## USE IT

Встраивание через рабочие пакеты (мульти-провайдер):

- **Microsoft 365 Copilot** — ИИ в Word / Excel / Outlook / Teams поверх Microsoft Graph внутри Microsoft 365 service boundary: по доке Microsoft, «Copilot can't access data that the user doesn't have permission to access» — доступ всегда ограничен правами вошедшего пользователя, плюс соблюдаются Conditional Access и MFA, а данные не покидают периметр тенанта.
- **Gemini for Google Workspace** — ИИ в Gmail / Docs / Sheets; данные организации не уходят на обучение и не покидают периметр.
- **Свой инструмент через API** — AI-шаг за фича-флагом и fallback (как в Build It), с мониторингом (10.2) и телеметрией (10.3).

Везде принцип один: ИИ **рядом с данными процесса** + контроль доступа (наследует права пользователя) + откат. Встраивают ИИ туда, где он даёт ценность, не ломая существующий контроль доступа.

> **Врезка (2026): self-hosted агент-рантайм и многоканальность.** Агента можно деплоить не только за корпоративный API, но и на локальную платформу — **OpenClaw / Hermes** (MIT): скиллы, память, мульти-модельная оркестрация и подключение к мессенджер-каналам (Telegram, Slack, Signal). Удобно для персонального агента с многоканальным доступом (подробно — урок 4.5). Те же гарантии — фича-флаг, fallback, контроль доступа — остаются обязательными.

## ЧАСТЫЕ ОШИБКИ

- **Встраивать ИИ без fallback.** Модель упадёт по таймауту или лимиту — и весь процесс встанет. Запасной детерминированный путь обязателен: ИИ деградирует, процесс работает.
- **Нет фича-флага.** Если выключить ИИ можно только релизом, инцидент тянется часами. Флаг гасит фичу мгновенно — и для аварии, и для постепенной раскатки.
- **Ловить только «ожидаемые» ошибки.** В проде AI-шаг падает непредсказуемо (сеть, лимиты, кривой ответ). Лови широкий `except` вокруг вызова модели — процесс важнее «чистоты» обработки.
- **Делать ИИ центром процесса, а не шагом.** Если всё завязано на модель, нет куда деградировать. ИИ — один из шагов пайплайна с запасным путём, а не его ядро.
- **Игнорировать контроль доступа при встраивании.** ИИ рядом с данными процесса должен **наследовать** права пользователя, а не обходить их. Встраивание не должно открывать модели больше, чем видит сам пользователь.
