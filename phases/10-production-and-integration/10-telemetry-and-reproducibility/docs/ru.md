# Урок 10.3 · Телеметрия и воспроизводимость

**Фаза 10 — Production и интеграция** · **Результат фазы:** Встроить ИИ в рабочий инструмент с мониторингом токенов и стоимости.
<!-- **Requires:** observability/биллинг провайдера — только для блока USE IT -->

> **MOTTO.** Считай токены и стоимость на процесс и фиксируй конфиг прогона — чтобы повторить и сравнить.

## PROBLEM

Мониторинг (10.2) говорит «здорово ли», телеметрия — «сколько стоит и воспроизводимо ли». Без учёта токенов/стоимости в разрезе процессов не понять, что дорого; без фиксации конфига (модель, параметры, версия промпта) нельзя воспроизвести результат и честно сравнить «до/после» (мост к FinOps 9.4 и Capstone 12).

## CONCEPT

```
телеметрия:   record(process, tokens_in, tokens_out, cost) → by_process()
воспроизводимость: RunConfig{model, temperature, prompt_version, seed}
                       └─ fingerprint() : одинаковый конфиг → одинаковый хэш
```

Телеметрия привязывает расходы к процессам; fingerprint конфига делает прогон воспроизводимым и сравнимым.

## BUILD IT

Учёт токенов/стоимости + отпечаток конфига: [`code/telemetry.py`](../code/telemetry.py).

- `Telemetry.record(...)` / `by_process()` / `total_cost()` — расходы по процессам;
- `RunConfig.fingerprint()` — стабильный хэш конфига (воспроизводимость).

```bash
python code/telemetry.py
pytest code -q
```

Тесты: агрегация токенов/стоимости по процессам; одинаковый конфиг → одинаковый отпечаток, смена параметра/версии промпта → другой. Это основа честного «до/после».

## USE IT

Дашборд расходов и качества (мульти-провайдер):

- **Токены/стоимость на процесс** — из usage в ответе API (`input_tokens`/`output_tokens`) × карта цен (9.1).
- **Качество** — LLM-as-judge / eval-наборы (урок 2.5) рядом со стоимостью на дашборде.
- **Воспроизводимость** — логируй `RunConfig` (модель, temperature, версия промпта, seed) с каждым прогоном; OTel GenAI стандартизует поля токенов.
