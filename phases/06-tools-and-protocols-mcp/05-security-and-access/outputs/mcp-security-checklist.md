# Артефакт: чек-лист безопасности MCP

Перед выкладкой MCP-сервера. Слой доступа — [`code/mcp_security.py`](../code/mcp_security.py).

## Права и доступ

- [ ] **Read-only по умолчанию**; write/delete — только явным principal'ам (`can_write`).
- [ ] **Least privilege**: каждый инструмент разрешён только тем, кому реально нужен.
- [ ] **Scope токенов минимален** — не `db:*`/`admin:*`, а точечные права (RAR, RFC 9396).
- [ ] **Авторизация перед действием** (`check_access`/`guarded_call`), а не «исполнить и потом думать».

## Изоляция

- [ ] Код-исполнение (Code Interpreter, урок 5.2) — в контейнере/seccomp, **без сети**.
- [ ] Файловые инструменты — ограничены `allowed_root`, проверка **path traversal** (`is_safe_path`).
- [ ] Опасные операции (`delete`, массовые правки) — за human-in-the-loop (Фаза 7).

## Секреты и данные

- [ ] **Секреты не попадают** в ответы инструментов, схемы и логи (модель их «запомнит»).
- [ ] STDIO-транспорт → секреты из окружения (`.env`, урок 0.2); не хардкодить.
- [ ] HTTP-транспорт → **OAuth 2.1**: сервер как Resource Server, проверка **audience** токена.
- [ ] Токены: HTTPS, срок жизни и ротация; валидация redirect URI.

## Входы и инъекции

- [ ] **Санитизация и валидация** всех аргументов (типы по схеме, границы значений).
- [ ] Данные из инструментов — **недоверенные**: могут содержать prompt-инъекции (Фаза 11).
- [ ] Прокси-серверы — защита от **confused deputy** (per-client consent).

## Наблюдаемость

- [ ] **Аудит** каждого вызова (principal, tool, allowed, причина).
- [ ] Алерты на отказы доступа и аномальные паттерны вызовов.
- [ ] Сервер прогнан инспектором (урок 6.4) до подключения к клиенту.
