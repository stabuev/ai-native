# Артефакт: карта MCP под свой стек

Что вынести в свои MCP-серверы, что подключить готовым, и к каким host'ам. Модель архитектуры — [`code/mcp_architecture.py`](../code/mcp_architecture.py).

## Шаблон карты

| Возможность | Примитив | Свой сервер / готовый | Host(ы) | Доступ |
|---|---|---|---|---|
| Поиск по корпусу | tool + resource | свой (doc-assistant, Фаза 3) | Claude Desktop, агент | read-only |
| Запрос в БД | tool | свой (NL→SQL, Фаза 5) | агент | read-only |
| Файлы проекта | resource | готовый (filesystem) | IDE | ограничить root |
| GitHub | tool | готовый (github) | Claude/Cursor | scope-токен |
| Метрики | tool | свой (семслой 5.3) | дашборд-бот | read-only |

## Как решать «свой vs готовый»

- **Готовый коннектор** — если есть проверенная референс-реализация (`modelcontextprotocol/servers`): файлы, git, поиск.
- **Свой сервер** — для внутренних данных/логики (твоя БД, метрики, бизнес-правила).

## Три примитива — что куда

- **tools** — действия с побочным эффектом или вычислением (как POST).
- **resources** — данные для контекста (как GET): файлы, схемы, записи.
- **prompts** — переиспользуемые шаблоны рабочих процессов.

## Правила

- Один сервер — одна доменная зона (не «всё в одном»).
- Каждый сервер проходит инспектор перед подключением (урок 6.4).
- Права и изоляция — до выкладки (урок 6.5): read-only по умолчанию.
