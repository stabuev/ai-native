**Этот урок — про то, как получить от модели не прозу, а данные, которые можно скормить коду.** Когда ответ идёт в БД, API или UI, нужен не «текст про тикет», а строгий объект `{title, priority, urgent}`. Системный промпт задаёт формат, провайдерский **structured output** заставляет модель его держать, а собственный **валидатор** проверяет, что пришедшее реально соответствует схеме.

> **MOTTO.** Если ответ парсит программа — он должен быть схемой, а не прозой.

## PROBLEM

Когда вывод модели идёт в код, свободный текст — катастрофа: то лишние пояснения вокруг JSON, то поле пропало, то число пришло строкой `"3"`, то `true` вместо `1`. И просто «верни JSON» в промпте **не гарантирует** ни валидный JSON, ни нужную структуру — модель может обернуть его в Markdown-ограждение (тройные обратные кавычки), добавить «Вот ваш ответ:» или перепутать тип. Нужны два рычага: **системный промпт** + provider structured output, чтобы формат держался, и **валидатор** на нашей стороне, чтобы поймать то, что всё же протекло.

## CONCEPT

### Интуиция

Если ответ читает программа, то промпт — это **контракт API**, а не просьба «напиши красиво». У контракта есть схема: какие поля, каких типов, какие обязательны. Текстом такой контракт не задать надёжно — нужна машинно-проверяемая форма (JSON Schema) и проверка на приёме. Принцип — **«доверяй, но проверяй»**: даже когда провайдер обещает структуру, на своей стороне сверяешь.

### Как это работает

```
system prompt (роль + строгий формат)
        │
        ▼
ответ модели ──extract_json──► объект ──validate_schema──► ✓ / список ошибок
   (часто в ```json … ```)        (типы, обязательные поля)
```

Два уровня надёжности:

- **(1) Provider structured output / function calling** — заставляет модель держать схему на стороне API. У OpenAI это Structured Outputs: ответ **гарантированно соответствует** переданной JSON-схеме (это сильнее «JSON mode», который обещает лишь валидный JSON, но не твою структуру). У Google — `response_schema`, у Anthropic — через tool use (описываешь `input_schema` инструмента).
- **(2) Свой валидатор** — страховка на приёме: распарсить ответ и проверить, что поля и типы на месте. Дёшево и ловит то, что молча ломает прод.

**JSON Schema** — стандарт описания структуры: `type` (string/number/integer/boolean/array/object), `required` (обязательные поля), `additionalProperties: false` (запретить лишние). На нём говорят все провайдеры.

Тонкость, на которой все спотыкаются: в Python `bool` — **подкласс** `int`, поэтому `isinstance(True, int)` это `True`. Наивный валидатор пропустит `priority: true` как «целое». Поэтому для `integer`/`number` boolean отсекают **явно** — иначе `urgent: true`, случайно попавший в числовое поле, проскочит.

## РАЗБОР ПО ШАГАМ

Прогоним конвейер руками. Модель часто возвращает JSON не голым, а в Markdown-ограждении из тройных обратных кавычек (с пометкой `json`), иногда ещё с пояснением вокруг.

**Шаг 1 — `extract_json`** снимает это ограждение и парсит текст в объект:

```
{"title": "Не грузит отчёт", "priority": 2, "urgent": true}
```

**Шаг 2 — `validate_schema`** по схеме `title: string (req), priority: integer (req), urgent: boolean`:

- `title` — строка, обязательное, на месте → ок;
- `priority` — целое `2` → ок;
- `urgent` — boolean → ок.

```
validate_schema(obj, SCHEMA) → []      # пустой список = всё валидно
```

А теперь ловим типичные поломки:

```
{"priority": 1}                  → ["title: отсутствует обязательное поле"]
{"title": "X", "priority": "high"} → ["priority: ожидался integer"]
{"title": "X", "priority": true}   → ["priority: ожидался integer, получен boolean"]
```

Последний случай — та самая ловушка `bool`/`int`: без явной проверки `true` проскочил бы как число. Валидатор превращает «вроде JSON пришёл» в **проверяемую гарантию**.

## BUILD IT

**Задание: собери извлечение JSON и валидатор схемы** — стандартная библиотека (`json`), без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/2.4-structured-output/`). Нужен только **Python 3**, всё офлайн (для теста ещё `pytest`).

Данные бери готовыми (карта типов и пример схемы/системного промпта):

```python
_TYPES = {"string": str, "number": (int, float), "integer": int,
          "boolean": bool, "array": list, "object": dict}
SCHEMA = {
    "title":    {"type": "string",  "required": True},
    "priority": {"type": "integer", "required": True},
    "urgent":   {"type": "boolean", "required": False},
}
```

Реализуй две функции:

- **`extract_json(text)`** — достань JSON из ответа: если текст начинается с тройных обратных кавычек, сними это ограждение (и слово `json` после него) и закрывающие кавычки, затем `json.loads`. Обычный JSON без ограждения тоже должен парситься.
- **`validate_schema(obj, schema)`** — верни список ошибок (пустой = валидно). Если `obj` не `dict` → `["корень: ожидался объект (dict)"]`. Для каждого поля схемы: нет поля и оно `required` → «отсутствует обязательное поле»; есть, но тип не совпал с `_TYPES[...]` → «ожидался {тип}». **Важно:** для `integer`/`number` отдельно отсекай `bool` («получен boolean»), потому что в Python `bool` — подкласс `int`.

**Готово, когда** тест `test_structured_output.py` зелёный — он проверяет извлечение из ограждённого и обычного JSON, прохождение валидного объекта, отлов пропущенного обязательного поля, неверного типа, boolean вместо integer и не-объекта в корне.

```bash
pytest code -q                    # красное → реализуй функции → зелёное
python code/structured_output.py  # демо: парс + валидация хорошего и плохого объекта
```

**Подсказка.** В `validate_schema` обходи `schema.items()`; для проверки типа держи порядок: сперва спецслучай `bool` для числовых полей, потом общий `isinstance(val, _TYPES[spec["type"]])`.

## USE IT

Проси у провайдера гарантированную структуру (мульти-провайдер):

```python
schema = {"type": "object", "properties": {
    "title": {"type": "string"}, "priority": {"type": "integer"}},
    "required": ["title", "priority"], "additionalProperties": False}

# OpenAI — Structured Outputs (text.format = json_schema)
OpenAI().responses.create(model="gpt-5.x", input="...",
    text={"format": {"type": "json_schema", "name": "ticket", "schema": schema}})
# Google — responseSchema в config
genai.Client().models.generate_content(model="gemini-2.x-flash", contents="...",
    config={"response_mime_type": "application/json", "response_schema": schema})
# Anthropic — через tool use (function calling): описываешь input_schema инструмента
```

Даже с гарантией провайдера прогоняй ответ через свой `validate_schema` — это дешёвая страховка (доверяй, но проверяй).

## ЧАСТЫЕ ОШИБКИ

- **«Верни JSON» без схемы и проверки.** Это не гарантия: модель добавит пояснение, обернёт в Markdown-ограждение или перепутает тип. Нужен structured output на стороне провайдера плюс свой валидатор.
- **Путать JSON mode и Structured Outputs.** JSON mode обещает лишь валидный JSON, а не твою структуру. Соответствие именно схеме даёт structured output / function calling.
- **Ловушка `bool` vs `integer`.** В Python `True` — это `int`, и наивная проверка пропустит `priority: true`. Отсекай boolean для числовых полей явно.
- **Доверять провайдеру без проверки.** Гарантия не отменяет валидатор на приёме — внешние ответы всегда сверяй перед записью в БД/API.
