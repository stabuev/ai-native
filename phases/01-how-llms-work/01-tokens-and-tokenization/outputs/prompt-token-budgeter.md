# Артефакт: prompt-token-budgeter

Заполни этот шаблон для одного реального запроса из личной карты задач 0.1. Он отделяет измеренный вход от планируемого выхода и не маскирует приблизительный подсчёт под точный.

## 1. Запрос и модель

- **Задача:** `<какой рабочий результат нужен>`
- **Точный model ID:** `<значение, которое передаётся в API>`
- **Состав запроса:** `<system + сообщения + файлы + tools или только текст>`
- **Данные безопасны для выбранного сервиса:** `да / нет / требуется обезличивание`

## 2. Подсчёт токенов

- **Метод:** `<tiktoken encoding_for_model / Anthropic count_tokens / Gemini count_tokens / другое>`
- **Статус:** `exact for text / provider estimate / approximate`
- **Что именно посчитано:** `<полный структурированный запрос или текстовый фрагмент>`
- **Входных токенов:** `<N>`
- **Дата подсчёта:** `<YYYY-MM-DD>`

Если использована чужая кодировка или грубая эвристика, объясни, почему результат всё же полезен и для какого решения его точности достаточно.

- `exact for text` — текст локально закодирован токенизатором указанной модели, но служебная оболочка полного запроса могла не учитываться;
- `provider estimate` — предварительный счётчик провайдера получил структуру будущего запроса;
- `approximate` — использована чужая кодировка или эвристика, поэтому число годится только для грубой оценки.

## 3. План ответа и тариф

- **Планируемых выходных токенов:** `<N>`
- **Почему такой объём реалистичен:** `<формат ответа, прошлый usage или верхний лимит>`
- **Цена входа, USD / 1M токенов:** `<число>`
- **Цена выхода, USD / 1M токенов:** `<число>`
- **Официальный источник цены:** `<прямая ссылка>`
- **Дата проверки цены:** `<YYYY-MM-DD>`

Не копируй цену из старого примера курса: тариф и выбранный model ID должны совпадать на дату расчёта.

## 4. Расчёт

```text
input_cost  = input_tokens / 1_000_000 × input_price
output_cost = planned_output_tokens / 1_000_000 × output_price
estimate    = input_cost + output_cost
```

- **Оценка выхода:** `$<...>`
- **Оценка запроса:** `$<...>`

Рядом сохрани вывод личного `token_budgeter.py`, чтобы арифметику можно было повторить.

## 5. Решение до запуска

- **Влезает ли запрос в доступный бюджет:** `да / нет / неизвестно`
- **Что изменить, если не влезает:** `<сократить контекст / ограничить ответ / выбрать другую модель / не запускать>`
- **Какую величину проверить после ответа:** `фактические input/output usage и итоговый биллинг`

## Критерии приёмки

- model ID, метод подсчёта и тариф относятся к одной выбранной модели;
- точность подсчёта названа явно;
- входные и выходные токены не смешаны;
- цены имеют единицы, источник и дату;
- расчёт воспроизводится локальной функцией;
- оценка не названа фактической стоимостью до получения usage;
- по результату принято конкретное решение, а не просто заполнена таблица.
