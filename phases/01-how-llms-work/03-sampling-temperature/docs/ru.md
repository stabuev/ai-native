## Откуда берутся разные продолжения

Представь, что на очередном шаге модель оценила четыре возможных токена так:

| Токен | Вероятность |
|---|---:|
| `данные` | 0.64 |
| `результаты` | 0.24 |
| `цифры` | 0.09 |
| `банан` | 0.03 |

Сумма вероятностей равна `1.00`: это полный набор вариантов для одного учебного шага.

Сначала сделай два прогноза:

1. Какой токен будет выбран при greedy?
2. Может ли при другом правиле быть выбран токен `результаты`, хотя он не лидер?

При **greedy** ответ на первый вопрос всегда один: `данные`, потому что `0.64` — максимум. Но распределение содержит больше информации, чем только победителя. Можно выбирать токен случайно так, чтобы `данные` выпадали чаще, `результаты` — реже, а `банан` — совсем редко.

Правило, которое превращает оценки модели в выбранный следующий токен, называют **стратегией декодирования**. Здесь «декодирование» не означает обратную токенизацию: речь именно о выборе продолжения на каждом шаге генерации.

## Greedy и взвешенное сэмплирование

Есть два базовых варианта выбора.

- **Greedy** — взять позицию с самым большим значением. Операцию «найти позицию максимума» часто называют **argmax**. Для таблицы выше argmax — позиция токена `данные`.
- **Сэмплирование** — разыграть случайный выбор с указанными весами. Токен с вероятностью `0.64` не обязан выпасть всегда, но на длинной серии одинаковых шагов будет выпадать заметно чаще токена с вероятностью `0.03`.

Сэмплирование можно представить как отрезок от `0` до `1`:

```text
0.00 ├──────────── данные ────────────┤ 0.64
0.64 ├──── результаты ────┤ 0.88
0.88 ├── цифры ──┤ 0.97
0.97 ├ банан ┤ 1.00
```

Генератор случайных чисел выдаёт число `r` в диапазоне от `0` включительно до `1` не включительно. Если `r = 0.80`, оно попадает в интервал `результаты`; если `r = 0.02` — в интервал `данные`.

В Python `random.Random(...)` — **псевдослучайный** генератор: его последовательность полностью определяется начальным числом, которое называют **seed**. Одинаковые seed и одинаковая последовательность вызовов дают одинаковую последовательность бросков.

```python
rng = random.Random(7)       # создать один генератор
first = rng.random()         # первый бросок
second = rng.random()        # второй бросок того же генератора
```

Важно не перепутать это с постоянным перезапуском генератора:

```python
random.Random(7).random()    # каждый раз снова первый бросок
random.Random(7).random()    # снова первый бросок, а не второй
```

Seed делает локальную псевдослучайную последовательность повторяемой. Он не превращает один вариант в «правильный» и сам по себе ничего не обещает о поведении внешнего сервиса.

## От оценок модели к вероятностям

В 1.2 распределения были уже готовы. Реальная модель перед выбором токена обычно выдаёт для всего словаря числовые оценки — **логиты**. Логит может быть положительным, нулевым или отрицательным; сам по себе он ещё не является вероятностью.

Например, для четырёх позиций модель могла выдать:

```text
logits = [3, 2, 1, 0]
```

Чтобы получить положительные числа с суммой `1`, применяют функцию **softmax**:

```text
            exp(logit_i / T)
p_i =  ─────────────────────────
        Σ_j exp(logit_j / T)
```

Здесь:

- `i` — позиция одного кандидата;
- `Σ` означает «сложить значения для всех кандидатов»;
- `exp(x)` — экспонента: она превращает любое число в положительный вес и усиливает разницу между большими и маленькими оценками;
- деление каждого веса на общую сумму называется **нормализацией** — после неё вероятности складываются в `1`;
- `T` — температура.

В коде перед `exp` из всех логитов вычитают максимальный логит. Это не меняет итоговые доли: все веса уменьшаются в одинаковое относительное число раз. Зато вычисления не переполняются на больших значениях.

## Что именно меняет temperature

**Temperature** меняет разницу между логитами перед softmax.

- `T = 1` — используем исходные различия.
- `0 < T < 1` — делим на маленькое число; различия увеличиваются, распределение становится острее, лидер получает больше массы.
- `T > 1` — делим на большое число; различия уменьшаются, распределение становится ровнее, у остальных кандидатов растут шансы.

Для логитов `[3, 2, 1, 0]` получаются такие распределения:

| Температура | Вероятности после softmax | Что видно |
|---:|---|---|
| `0.5` | `[0.865, 0.117, 0.016, 0.002]` | лидер почти забрал всю массу |
| `1.0` | `[0.644, 0.237, 0.087, 0.032]` | исходный баланс |
| `2.0` | `[0.455, 0.276, 0.167, 0.102]` | распределение стало ровнее |

До следующего абзаца попробуй ответить: при какой строке четвёртый токен имеет наибольший шанс выпасть? Ответ — при `T = 2.0`: его шанс вырос с `0.2%` при `T = 0.5` до `10.2%`.

### Особый случай `temperature = 0`

Подставить `T = 0` в формулу нельзя: деление на ноль не определено. Поэтому в нашем учебном сэмплере `temperature = 0` — это отдельное соглашение:

```text
temperature = 0 → не вызывать softmax(T) и случайный генератор → вернуть argmax
```

Так локальная функция становится детерминированной: одинаковые логиты приводят к одной и той же позиции. Это свойство нашего явно заданного алгоритма, а не универсальная гарантия любого API с параметром, который тоже называется `temperature`.

## Как top-k и top-p обрезают хвост

Temperature перераспределяет вероятность, но не удаляет кандидатов. **Top-k** и **top-p** решают другую задачу: исключают часть маловероятного хвоста перед броском.

После удаления кандидатов оставшиеся вероятности снова делят на их сумму. Это называется **перенормировкой**: итоговый набор снова должен иметь сумму `1`.

### Top-k: фиксированное число кандидатов

`top_k = 2` означает «оставить две самые вероятные позиции»:

```text
[0.644, 0.237, 0.087, 0.032]
          top-k = 2
                ↓
[0.731, 0.269, 0.000, 0.000]
```

Первые два значения поделили на их сумму `0.881`. Теперь третий и четвёртый токены не могут выпасть вообще.

### Top-p: достаточная суммарная масса

`top_p = 0.8` означает «сортировать кандидатов от самого вероятного и оставить минимальный набор, который набрал не меньше `0.8` общей массы».

Для уверенного распределения:

```text
[0.64, 0.24, 0.09, 0.03]
  0.64        < 0.8  → нужен ещё один токен
  0.64 + 0.24 = 0.88 ≥ 0.8 → остановиться
```

Остаются два кандидата. Но для более ровного распределения `[0.35, 0.30, 0.20, 0.15]` первые два дают только `0.65`, поэтому при том же `top_p = 0.8` понадобятся уже три кандидата: `0.35 + 0.30 + 0.20 = 0.85`.

В этом различие:

- top-k всегда задаёт фиксированное **количество**;
- top-p задаёт фиксированную **долю вероятностной массы**, поэтому количество меняется от шага к шагу.

## Полный учебный конвейер одного шага

В BUILD IT мы явно зафиксируем такой порядок:

```text
непустые logits
      │
      ├─ temperature = 0 ───────────────────────────► argmax
      │
      └─ temperature > 0
             │
             ▼
       softmax(logits / T)
             │
             ▼
       top-k и перенормировка
             │
             ▼
       top-p и перенормировка
             │
             ▼
       один бросок rng
             │
             ▼
       индекс выбранного токена
```

Этот порядок — точный контракт нашего кода. Провайдер может использовать другую реализацию, скрытые ограничения или вообще не предоставлять эти параметры. Поэтому учебный сэмплер объясняет причинный механизм, но не притворяется копией закрытого сервиса.

## Детерминизм и воспроизводимость — не одно обещание

**Детерминированный алгоритм** при одинаковом входе всегда возвращает одинаковый результат. **Воспроизводимый эксперимент** удаётся повторить при зафиксированных условиях. Эти понятия связаны, но условия воспроизводимости шире одного параметра.

| Ситуация | Что можно утверждать |
|---|---|
| Наш `sample(..., temperature=0)` | Возвращает argmax и не использует RNG |
| Наш `sample(...)` с одним seed | Повторяет последовательность, если совпали код, входы и порядок вызовов RNG |
| Запрос к внешнему API | Повторяемость нужно измерять; параметр `0` или seed не является гарантией сервиса |

У внешнего API могут измениться точный model ID, версия модели, серверная реализация, значения параметров по умолчанию и другие скрытые условия. Поэтому для честного сравнения записывают не только prompt, но и модель, дату, доступные параметры, число повторов и наблюдаемый результат.

## Чего эти параметры не делают

- Низкая температура не делает модель умнее.
- `temperature = 0` не превращает правдоподобное утверждение в проверенный факт.
- Seed не исправляет ошибочный prompt и не валидирует ответ.
- Большее разнообразие не означает автоматически лучшее качество.
- Одинаковый текст в двух запусках ещё не доказывает, что API гарантирует детерминизм.

Главная практическая граница урока: параметры выбора меняют **какой вероятный вариант будет продолжен**, но не проверяют **истинен ли этот вариант**. Проверка доверия к содержанию начнётся в уроке 1.4.

## BUILD IT

**Задание:** собери офлайн-сэмплер одного токена на стандартных модулях `math` и `random`.

Работай в личной папке `course-work/1.3/`. В блоке [«Исходники урока»](#lesson-files) скопируй туда только `code/test_sampling.py`; опубликованный `code/sampling.py` пока не копируй. Первый запуск должен завершиться ошибкой импорта — это ожидаемая красная точка.

Создай `course-work/1.3/sampling.py` и реализуй функции.

### 1. `softmax(logits, temperature=1.0)`

- принимает непустую последовательность логитов и положительную температуру;
- перед `exp` вычитает максимальный логит;
- возвращает список неотрицательных вероятностей с суммой `1`;
- при пустых логитах или `temperature <= 0` выбрасывает `ValueError`.

### 2. `top_k_filter(probs, k)`

- принимает непустую последовательность неотрицательных весов;
- при `k is None` или `k >= len(probs)` только нормализует копию;
- иначе оставляет ровно `k` самых больших позиций, обнуляет остальные и перенормирует;
- при `k < 1`, нецелом `k` или нулевой сумме весов выбрасывает `ValueError`.

### 3. `top_p_filter(probs, p)`

- при `p is None` или `p == 1` только нормализует копию;
- при `0 < p < 1` оставляет минимальное вероятностное ядро и перенормирует;
- при значении вне интервала `(0, 1]`, отрицательных весах или нулевой сумме выбрасывает `ValueError`.

### 4. `sample(logits, temperature=1.0, top_k=None, top_p=None, rng=None)`

- при отрицательной температуре выбрасывает `ValueError`;
- при `temperature == 0` сразу возвращает argmax и не вызывает RNG;
- при положительной температуре выполняет `softmax → top-k → top-p → бросок`;
- использует переданный объект `rng` с методом `.random()`, а без него — модуль `random`;
- возвращает индекс выбранной позиции.

Перед реализацией письменно предскажи:

- какие позиции останутся после `top_k=2` для логитов `[3, 2, 1, 0]`;
- сколько позиций оставит `top_p=0.8` для распределений `[0.64, 0.24, 0.09, 0.03]` и `[0.35, 0.30, 0.20, 0.15]`;
- будет ли один генератор `Random(7)` выдавать один и тот же бросок на каждом вызове.

**Готово, когда** тесты зелёные, а ты можешь объяснить, почему они проверяют не только запуск:

- softmax нормализует значения и реагирует на температуру;
- top-k сохраняет точные позиции, а top-p меняет размер ядра вместе с формой распределения;
- `temperature=0` обходит случайный выбор;
- отфильтрованный токен никогда не выбирается;
- два отдельных генератора с одним seed воспроизводят одинаковые последовательности;
- ошибочные границы параметров завершаются явно.

## Реалистичные ошибки и диагностика

- **«`temperature=0` можно подставить в softmax».** Нет: в формуле получится деление на ноль. В нашем коде это отдельная ветка argmax.
- **«Одинаковый seed означает одинаковый токен на каждом шаге».** Один генератор выдаёт последовательность разных бросков; одинаковый seed позволяет повторить всю последовательность при одинаковом порядке вызовов.
- **«Top-p оставляет одинаковое число токенов».** Это свойство top-k. Размер top-p-ядра зависит от формы текущего распределения.
- **«Низкая температура делает факты надёжными».** Она сужает случайность выбора, но не добавляет источники и не проверяет утверждения.
- **«Параметр есть в SDK, значит его принимает любая модель».** Поддержка определяется точным model ID и текущим контрактом API.
- **«Я изменил temperature и top-p и увидел эффект каждого».** При двух изменениях причина неразличима. Для учебного эксперимента меняй одну переменную.

## Дополнительное чтение

Не читай список подряд. Для необязательного USE IT открой контракт только своего
провайдера. Чтобы закрепить механизм, выбери один практический разбор. Исследовательские
материалы нужны лишь для углубления в top-p и причины недетерминизма.

### 1. Для эксперимента с API — выбери своего провайдера

- [Anthropic — Messages API](https://platform.claude.com/docs/en/api/messages) — проверь параметры запроса и ограничения точного model ID; наличие поля в SDK ещё не гарантирует его поддержку выбранной моделью.
- [OpenAI — Models](https://developers.openai.com/api/docs/models) — открой карточку точного model ID и проверь поддерживаемые параметры: возможности соседней модели нельзя переносить автоматически.
- [Gemini — Text generation](https://ai.google.dev/gemini-api/docs/text-generation) — найди `generation_config`, затем сверь доступность параметра для выбранной модели и используемого API.

### 2. Чтобы закрепить механизм — выбери один материал

- [Hugging Face — How to generate text](https://huggingface.co/blog/how-to-generate) — практический разбор greedy, top-k, top-p и temperature с кодом; сопоставь порядок операций со своим `sampling.py`.
- [Maxime Labonne — Decoding Strategies in LLMs](https://huggingface.co/blog/mlabonne/decoding-strategies) — более подробное продолжение с реализациями стратегий декодирования; открывай, если одного учебного шага уже недостаточно.

### 3. Для глубокого погружения

- [Holtzman et al., 2019 — Nucleus Sampling](https://arxiv.org/abs/1904.09751) — первоисточник top-p; начни с abstract и иллюстрации вырождения текста при максимизации вероятности.
- [Thinking Machines — Defeating Nondeterminism in LLM Inference](https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/) — техническое объяснение, почему даже `temperature=0` не всегда обеспечивает одинаковый удалённый результат; для завершения урока детали вычислительных ядер не нужны.
