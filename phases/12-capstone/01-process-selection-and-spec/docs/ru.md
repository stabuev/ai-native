# Урок 12.1 · Выбор процесса и спецификация

**Фаза 12 — Capstone** · **Результат фазы:** Автоматизировать реальный процесс end-to-end и защитить решение с метриками эффекта.
<!-- exercise -->

**Весь курс вёл к этому: собрать своё, работающее, и доказать, что оно лучше.** Capstone — не «покажи, что умеешь звать API», а «возьми реальный процесс и измеримо его улучши». И первый шаг тут не код, а спецификация: без неё нечего защищать и нечем мерить «до/после».

> **MOTTO.** Capstone начинается не с кода, а со спецификации: цель, измеримые метрики, ограничения, бюджет.

## PROBLEM

«Автоматизирую что-нибудь с ИИ» — путь в никуда. Через две недели ты не сможешь ответить на простой вопрос защиты: «стало лучше — насколько?». Без зафиксированной цели и **измеримых** метрик нет базовой линии («до»), нет цели («после»), нет доказательства эффекта — только ощущение «вроде работает».

Нужно выбрать один реальный процесс и записать спецификацию так, чтобы успех был **проверяемым числом**, а не мнением. Anthropic формулирует это прямо: хорошие критерии успеха — конкретные и измеримые; вместо «хорошая работа модели» пиши «F1 ≥ 0.85 на отложенной выборке из 10 000 постов». Соберём валидатор, который не пропустит спеку без измеримых метрик и бюджета.

## CONCEPT

### Интуиция

Спецификация Capstone — это как **техзадание на ремонт с приёмкой по акту**, а не «сделайте красиво». «Красиво» нельзя принять — каждый поймёт по-своему. А «плитка уложена, шов ≤ 2 мм, срок 5 дней, бюджет 50 т.р.» — можно проверить и предъявить. Метрика без базовой линии («стало быстрее») бесполезна так же, как «стало красивее»: быстрее чего? на сколько?

Ключевая мысль: **измеримость — это пара чисел, а не одно.** Нужен baseline («как сейчас, руками») и target («куда целимся»). Одно число без другого не даёт эффекта: target без baseline не с чем сравнить, baseline без target — это не цель, а констатация.

### Как это работает

```
CapstoneSpec
  goal        — что автоматизируем (один конкретный процесс)
  metrics[]   — каждая: baseline («до») + target (цель)   ← без пары чисел нет защиты
  constraints — приватность (11.1), human-in-the-loop (7.3), безопасность (6.5)
  budget_usd  — потолок расходов (FinOps, Фаза 9)
        │ validate_spec → чего не хватает до сборки
```

- `Metric` / `CapstoneSpec` — структура спеки (dataclass): у метрики есть `baseline`, `target`, `unit`.
- `is_measurable(metric)` — метрика измерима, если заданы **и** baseline, **и** target.
- `validate_spec(spec)` — вернуть список проблем: нет цели / нет метрик / ни одна не измерима / не задан бюджет (пусто = готова к сборке).

Хороший выбор процесса: **частый · болезненный · с понятной метрикой эффекта** (время / стоимость / качество) · не критичный для жизни на старте. Именно такой процесс легко измерить и не страшно доверить ИИ на первом заходе.

## РАЗБОР ПО ШАГАМ

Возьмём сырую идею и доведём до спеки, которую примет валидатор.

Плохая формулировка (в терминах Anthropic — не Specific, не Measurable):

```
goal="сделать поддержку лучше", metrics=[Metric("качество")], budget=None
validate_spec → ["ни одна метрика не измерима (нужны baseline и target)", "не задан бюджет"]
   (защищать нечего: «лучше» — это не число)
```

Доводим до измеримой (Specific + Measurable + Achievable + Relevant):

```
CapstoneSpec(
  goal="Автоматизировать разбор входящих обращений поддержки",
  metrics=[
    Metric("время на обращение",     baseline=10.0, target=3.0,  unit="мин"),
    Metric("стоимость на обращение", baseline=0.50, target=0.10, unit="$"),
  ],
  constraints=["PII не во внешние модели", "human review на эскалациях"],
  budget_usd=200.0,
)
validate_spec → []          (готова: есть цель, измеримые метрики и бюджет)
```

Теперь защита (12.3) считается сама: 10→3 мин это −70% времени, 0.50→0.10 это −80% стоимости. Пара чисел на каждую метрику превратила «вроде лучше» в проверяемый эффект.

Но вот **честная граница валидатора**. Он проверяет, что метрика **измерима** (два числа заданы), но не что цель **реалистична и уместна**:

- **Achievable** — target=«качество 100%» пройдёт валидатор, но недостижим для фронтир-модели. Реалистичность цели проверяет человек, опираясь на бенчмарки и прошлые замеры.
- **Relevant** — можно набрать измеримых, но неважных метрик (например, «длина ответа»), а не тех, что бьют по реальной боли. Уместность метрики — тоже человек.

Поэтому `validate_spec` — это **ворота структуры**, а не гарантия хорошей спеки: он не пустит очевидно неполную, но осмысленность цели и метрик всё равно за тобой.

## BUILD IT

**Задание: собери спецификацию Capstone и валидатор измеримости.** Только стандартная библиотека (`dataclasses`), без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/12.1-capstone-spec/`), а файлы урока клади в подпапку `code/` (по соглашению курса). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `spec.py` с двумя dataclass'ами и тремя функциями:

- **`Metric`** — поля `name`, `baseline=None`, `target=None`, `unit=""`.
- **`CapstoneSpec`** — поля `goal=""`, `metrics=[]`, `constraints=[]`, `budget_usd=None`.
- **`is_measurable(metric)`** — `True`, только если заданы и `baseline`, и `target` (не `None`).
- **`validate_spec(spec)`** — вернуть список проблем: `"нет цели"` (пустая `goal`), `"нет метрик"` (пустой список), `"ни одна метрика не измерима (нужны baseline и target)"` (метрики есть, но ни одна не measurable), `"не задан бюджет"` (`budget_usd is None`). Пустой список = спека готова.

**Готово, когда** все тесты в `test_spec.py` зелёные — они проверяют: полная спека проходит; пустая цель / нет бюджета / неизмеримая метрика ловятся; `is_measurable` верно различает пару чисел и одно.

```bash
pytest code -q        # красное → реализуй spec.py → зелёное
python code/spec.py   # демо: валидная спека → "нет — готова к сборке"
```

**Подсказка.** Списки в dataclass задавай через `field(default_factory=list)`, иначе Python ругнётся на mutable default. В `validate_spec` для метрик используй `any(is_measurable(m) for m in spec.metrics)`.

## USE IT

Согласование спеки с реальной задачей (мульти-провайдер, применимо к любой модели):

- **Возьми свой процесс** — разбор тикетов, подготовка отчётов, ответы клиентам. Зафиксируй **baseline руками**: сколько времени/денег он занимает сейчас (без этого «после» не с чем сравнить).
- **Критерии успеха по Anthropic (SMART)** — Specific, Measurable, Achievable, Relevant. Даже «размытое» (этика, безопасность) квантуется: не «безопасные ответы», а «< 0.1% из 10 000 прогонов помечены фильтром токсичности».
- **Проектируй eval заранее** — как именно померишь качество (exact match / string match / LLM-судья), а не «на глаз потом». Автоматизируй грейдинг, где можно (это урок 2.5 про eval-harness).
- **Согласуй ограничения** — приватность (11.1), human-in-the-loop на критичном (7.3), бюджет и алерты (Фаза 9). Спека без ограничений и потолка расходов — риск в проде.

## SHIP IT

**Артефакт:** Проектная спецификация → [`outputs/capstone-spec.md`](../outputs/capstone-spec.md)

Заполненная спека твоего процесса: цель · метрики baseline/target · ограничения · бюджет. Прогони её через `validate_spec` — если список проблем пуст, спека готова. Это вход в сборку решения (12.2) и фундамент защиты с метриками «до/после» (12.3).

## ЧАСТЫЕ ОШИБКИ

- **Метрика без пары чисел.** «Стало быстрее» без baseline и target — не эффект, а ощущение. Измеримость = baseline + target, иначе `validate_spec` заблокирует.
- **Измеримо, но недостижимо.** target=«100% качества» пройдёт валидатор, но нереалистичен. Achievable проверяет человек по бенчмаркам, не код.
- **Измеримо, но неважно.** Набрать удобных метрик (длина ответа) вместо тех, что бьют по боли. Relevance — тоже за человеком.
- **Нет ограничений и бюджета.** Спека без приватности/HITL/потолка расходов = мина в проде. Ограничения — часть ТЗ, а не «потом допишем».
- **Выбрать критичный или редкий процесс.** Первый Capstone — частый, болезненный, некритичный для жизни: его легко мерить и не страшно доверить ИИ на старте.
