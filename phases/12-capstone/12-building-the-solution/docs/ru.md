## PROBLEM

За курс собрано много кирпичей: промпты, RAG, агенты, MCP, cost-aware роутинг, память, мониторинг. Но по отдельности это набор демок. Реальная задача Capstone — соединить их в **один end-to-end процесс**, который решает конкретную задачу из спеки (12.1): принял вход, выбрал модель по сложности, проверил guardrail'ом, вызвал инструмент, записал состояние, посчитал стоимость, вернул результат.

Сложность не в каждом блоке (они уже собраны), а в **интеграции**: чтобы блоки передавали друг другу данные и вместе давали рабочий результат, а не пять несвязанных кусков. Соберём скелет такого процесса — с правильными стыками, на детерминированных стабах вместо реальной модели.

## CONCEPT

### Интуиция

Собранный Capstone — это **конвейер, а не куча деталей на верстаке**. Двигатель, коробка, колёса по отдельности не едут — едет машина, где они соединены в нужном порядке. Так и здесь: роутинг без инструмента бесполезен, инструмент без телеметрии неучётен, память без guardrail'а небезопасна. Ценность рождается на **стыках**, когда выход одного блока становится входом следующего.

Ключевая мысль: Capstone не изобретает механизмы — он **оркестрирует** уже изученные. Каждый блок ты собирал в своей фазе; здесь важен только порядок и передача данных между ними.

### Как это работает

```
вход → classify (Ф9): по сложности → выбор модели (haiku/opus)
     → guardrail (Ф7): пустое/опасное → блок
     → инструмент (Ф6, в проде через MCP) → действие
     → память (Ф4/7): записать состояние процесса
     → телеметрия (Ф10): модель, уровень, стоимость → учёт
     → результат
```

- `classify(task)` — роутинг по сложности (Фаза 9): по стем-триггерам («анализ», «код», «сложн», «стратег») → `high`, иначе `low`.
- `Capstone.run(task)` — прогон по конвейеру: guardrail → выбор модели → инструмент → память → телеметрия; вернуть `{result, model, level, cost}`.
- `total_cost()` — суммарные расходы процесса из телеметрии (FinOps, Фаза 9).

Стоимость считается как в Фазе 9: по числу токенов (здесь прокси — слова×3) на цену модели за 1M. Haiku 4.5 — $1/$5, Opus 4.8 — $5/$25 (вход/выход), поэтому простая задача через Haiku стоит кратно дешевле сложной через Opus.

## РАЗБОР ПО ШАГАМ

Прогоним два задания через один и тот же `Capstone`:

```
run("сделай саммари письма")
   classify → "low"   (нет стем-триггеров сложности)
   модель  → haiku-4.5 ($1/$5)
   n_in = 4 слова × 3 = 12 токенов, n_out = 20
   cost = (12×1 + 20×5) / 1e6 = 112/1e6 ≈ $0.000112
   → {result: "результат: ...", model: "haiku-4.5", level: "low", cost: 0.000112}

run("проведи анализ архитектуры кода")
   classify → "high"  ("анализ" и "код" — триггеры)
   модель  → opus-4.8 ($5/$25)
   n_in = 4×3 = 12, n_out = 20
   cost = (12×5 + 20×25) / 1e6 = 560/1e6 ≈ $0.00056
   → {..., model: "opus-4.8", level: "high", cost: 0.00056}

в памяти: 2 задачи | total_cost() ≈ $0.000672
```

Один конвейер сам направил дешёвую задачу на дешёвую модель, а дорогую — на топовую, и обе записал в память и телеметрию. Это и есть интеграция: роутинг (Ф9), инструмент (Ф6), память (Ф4/7) и учёт (Ф10) сработали вместе за один вызов.

Но вот **честная граница скелета**:

- **Стабы вместо реальных блоков.** `_tool` возвращает строку, модель не вызывается — здесь важен **контракт и проводка**, а не интеллект. В проде `_tool` идёт через MCP-сервер, а `run` дергает реальную модель.
- **Один проход, не цикл.** Настоящий агент петляет reason→act→observe (Фаза 7), пока не решит задачу; скелет показывает единичный проход — каркас, на который навешивается цикл.
- **Стоимость — оценка.** `cost` из прокси-токенов; фактические берутся из телеметрии реальных вызовов (Фаза 10). Скелет учит **куда** класть учёт, а не даёт точную цифру.

Поменяй стабы на реальные блоки курса — контракт «вход → роутинг → guardrail → инструмент → память → телеметрия → результат» не изменится.

## BUILD IT

**Задание: собери end-to-end процесс из кирпичей курса.** Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/12.2-solution/`). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `solution.py`: словарь `ROUTE` (уровень → модель и цены), функцию `classify` и класс `Capstone`:

- **`classify(task)`** — вернуть `"high"`, если в задаче есть стем сложности («анализ», «код», «сложн», «стратег»), иначе `"low"`.
- **`Capstone.__init__`** — завести `self.memory = {}` и `self.telemetry = []`.
- **`Capstone.run(task)`** — guardrail (пустая задача → `raise ValueError`); `classify` → выбрать модель и цены из `ROUTE`; вызвать инструмент; посчитать `cost` по токенам-прокси; записать в `memory` и `telemetry`; вернуть `{result, model, level, cost}`.
- **`Capstone.total_cost()`** — сумма `cost` по телеметрии.

**Готово, когда** все тесты в `test_solution.py` зелёные — они проверяют: `classify` различает простое/сложное; простая задача → `haiku-4.5`/`low`, сложная → `opus-4.8`; память и телеметрия пополняются, `total_cost > 0`; пустая задача → `ValueError` (guardrail).

```bash
pytest code -q          # красное → реализуй solution.py → зелёное
python code/solution.py # демо: простая→haiku, сложная→opus, итог по памяти и $
```

**Подсказка.** Стем-триггеры ищи в `task.lower()` через `any(w in low for w in (...))` (как в Фазе 9 — морфология ломает точное совпадение, поэтому стемы). Цены держи в `ROUTE` рядом с моделью, чтобы `run` брал их одним обращением.

## USE IT

Интеграция в рабочий инструмент (мульти-провайдер): замени стабы на реальные блоки курса.

- **Агент** — цикл reason→act→observe (Фаза 7) на Claude Agent SDK или OpenAI Agents SDK вместо единичного прохода.
- **Инструменты и данные** — через MCP-сервер (Фаза 6, актуальная спека 2025-11-25): доступ к БД/файлам с контролем прав и allowlist (6.5, 11.1, 11.5).
- **Cost-aware** — роутер/каскад (Фаза 9), кэш и batch: `classify` тут — упрощённый роутер, в проде добавь quality-gate и кэш префикса.
- **Память** — модуль из 4.2 (релевантность + свежесть) вместо словаря; **наблюдаемость** — телеметрия и трейсинг (Фаза 10) вместо списка.
- **Встрой безопасно** — за фича-флагом с fallback на старый путь (10.1), чтобы откат был мгновенным, а не выкатывай «в лоб».

## ЧАСТЫЕ ОШИБКИ

- **Собирать новый кубик вместо интеграции.** Capstone — оркестрация изученного, а не изобретение механизма. Ценность на стыках блоков, а не в новом коде.
- **Пропустить учёт и память.** Без телеметрии не посчитать стоимость для защиты (12.3), без памяти теряется состояние процесса. Это не опционально — это часть конвейера.
- **Один проход вместо цикла в проде.** Скелет линеен; реальный агент петляет reason→act→observe (Ф7). Не приняв это, получишь «однократный» процесс там, где нужен итеративный.
- **Дать инструменту прямой доступ без MCP-гейта.** В проде инструмент идёт через MCP с контролем прав (6.5, 11.5), а не дергает БД напрямую из `run`.
- **Выкатывать без фича-флага и fallback.** Интеграция в живой инструмент без флага и отката (10.1) — риск: нечем быстро выключить, если пойдёт не так.
