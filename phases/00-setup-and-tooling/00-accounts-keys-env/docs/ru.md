## PROBLEM

Первый запрос к API спотыкается не о модель, а о быт. Где взять ключ? Куда его положить, чтобы код его видел, а git — нет? Почему `ANTHROPIC_API_KEY` «не виден» в соседнем терминале? И как не закоммитить секрет в репозиторий?

Захардкоженный ключ утекает в историю git — и оттуда его уже не вычистить. «Работает у меня» ломается на другой машине, где не тот Python и не те пакеты. Пока окружение не зафиксировано, ты тратишь время не на ИИ, а на ловлю 401 и «module not found».

## CONCEPT

### Что такое API-ключ и почему он секрет

Когда ты обращаешься к модели не в веб-чате, а кодом (через API), провайдер должен понять, **кто** делает запрос: чей это аккаунт, с чьего баланса списывать токены, какие применять лимиты. «Удостоверением» служит **API-ключ** — длинная строка, которую ты создаёшь в консоли своего аккаунта: **Anthropic Console**, **OpenAI Platform** или **Google AI Studio**.

Ключ — это **секрет**, и относиться к нему надо как к паролю. Документация Google формулирует прямо: «Treat your Gemini API key like a password. If compromised, others can consume your project's quota, incur unexpected billing charges, and access private resources» — у кого ключ, тот шлёт запросы от твоего имени и тратит твои деньги, отдельного логина не нужно. Отсюда два железных правила из тех же доков: **не хардкодь ключ в коде** и **никогда не коммить его в git** («Never check API keys into source control»). Если ключ всё же утёк — заходишь в консоль, отзываешь старый и выпускаешь новый.

А подавать ключ в код удобнее всего через окружение: SDK провайдера **сам читает его из переменной окружения** (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GOOGLE_API_KEY`) — в квикстарте Anthropic так и сказано: «The SDK reads `ANTHROPIC_API_KEY` automatically». Значит, ключу не место в исходнике; как сложить это в аккуратное и воспроизводимое окружение — ниже.

### Как это работает

Здоровое окружение — это три вещи, которые работают вместе:

```
venv (изоляция)  +  .env (секреты)  +  .env.example (контракт)
      │                  │                     │
 свои пакеты       ключи вне кода        список ключей без
 не ломают          и вне git            значений — что
 систему           (он в .gitignore)     нужно завести
```

- **venv** — изолированный Python: пакеты урока не конфликтуют с системными и ставятся предсказуемо.
- **`.env`** — файл `KEY=VALUE` с секретами. Он **в `.gitignore`** и в репозиторий не попадает. Загрузить его в окружение помогает `load_dotenv()` из **python-dotenv**, после чего SDK берёт ключ из окружения, а не из исходника.
- **`.env.example`** — те же ключи без значений. Коммитится как контракт: «вот что нужно завести, чтобы запустить проект».

Правило одно: **секрет живёт в `.env`, а не в коде**. Тогда ключ не утечёт в git, а проект соберётся на любой машине по `.env.example`.

## РАЗБОР ПО ШАГАМ

Прежде чем тратить вызов API на «401 Unauthorized», `.env` стоит проверить офлайн. Сделаем это руками на типичном «сыром» файле:

```
# my keys
export ANTHROPIC_API_KEY="sk-ant-demo123"   # рабочий ключ
OPENAI_API_KEY=your-key-here
```

**Шаг 1 — разобрать (`parse_env`).** Идём по строкам. `# my keys` — комментарий, пропускаем. У второй строки снимаем `export `, режем кавычки и inline-комментарий ` # рабочий ключ`. Третью берём как есть. Получаем словарь:

```
{ "ANTHROPIC_API_KEY": "sk-ant-demo123", "OPENAI_API_KEY": "your-key-here" }
```

**Шаг 2 — проверить (`validate`)** по требуемым ключам `ANTHROPIC_API_KEY, OPENAI_API_KEY, GOOGLE_API_KEY`:

- `ANTHROPIC_API_KEY` — есть, начинается с `sk-ant-` → ок;
- `OPENAI_API_KEY` — значение `your-key-here` из списка плейсхолдеров → **«похоже на плейсхолдер»**;
- `GOOGLE_API_KEY` — в файле нет → **«отсутствует»**.

Две проблемы пойманы за секунды и офлайн — до единого запроса к API. Ровно это делает валидатор из Build It.

## BUILD IT

**Задание: собери офлайн-валидатор окружения** — стандартная библиотека, без сети. Он ловит типовые ошибки `.env` до первого запроса.

> **Перед запуском.** Работай в своей папке курса из урока 0.1 — заведи вложенную `ai-native/0.2-accounts-keys-env/`. Для BUILD IT нужен только **Python 3**, всё офлайн и без ключей (для теста ещё `pytest`). Настоящие ключи и venv заведём ниже, в USE IT.

Создай `envcheck.py`. Константы бери готовыми (это просто данные):

```python
# значения, которыми выдаёт себя незаполненный .env
PLACEHOLDERS = {"", "your-key-here", "changeme", "xxx", "...", "todo"}
# ожидаемый префикс ключа по имени (пустой — формат не проверяем)
KEY_HINTS = {"ANTHROPIC_API_KEY": "sk-ant-", "OPENAI_API_KEY": "sk-", "GOOGLE_API_KEY": ""}
```

Реализуй две функции:

- **`parse_env(text)`** — текст `.env` превращает в dict `{KEY: VALUE}`. Пропускай пустые строки и комментарии (`#`); снимай ведущий `export `; у значения убирай кавычки и inline-комментарий (` # ...`).
- **`validate(env, required)`** — возвращает список проблем по требуемым ключам (пустой список = всё ок): ключа нет → `"{key}: отсутствует"`; значение из `PLACEHOLDERS` → `"{key}: похоже на плейсхолдер"`; значение не начинается с префикса из `KEY_HINTS` → `"{key}: ожидался префикс ..."`.

**Готово, когда** тест `test_envcheck.py` зелёный — он проверяет разбор (комментарии, `export`, кавычки) и все три типа проблем валидации.

```bash
pytest code -q             # сейчас красное → реализуй две функции → стало зелёным
python code/envcheck.py    # демо: распознанные ключи и найденные проблемы
```

**Подсказка.** В `parse_env` иди построчно: `strip`, отсекай пустые и `#`, срезай `export `, дели по первому `=`. Значение чисти отдельно: если в кавычках — бери содержимое, иначе режь хвост ` #комментарий`.

Внизу, в [«Исходниках урока»](#lesson-files), — три способа пройти упражнение (собрать самому · подсмотреть эталон · делегировать ИИ) и тесты-ТЗ.

## USE IT

Теперь заведём **настоящее окружение** и сделаем первый реальный запрос. Та самая папка `ai-native/` из урока 0.1 становится venv-проектом:

```bash
cd ai-native
python -m venv .venv && source .venv/bin/activate   # изолированный Python проекта (Windows: .venv\Scripts\activate)
pip install anthropic openai google-genai python-dotenv   # SDK + загрузчик .env
```

Заведи ключи в консолях провайдеров и положи их в `.env`. Файлы `.env.example` и `check_env.py` бери из артефакта Ship It (`env-starter/`):

- **Anthropic Console** → `ANTHROPIC_API_KEY` (`sk-ant-…`)
- **OpenAI Platform** → `OPENAI_API_KEY` (`sk-…`)
- **Google AI Studio** → `GOOGLE_API_KEY`

```bash
cp .env.example .env       # 1) создать локальный .env
# 2) вписать в .env реальные ключи
echo ".env" >> .gitignore  # 3) .env НИКОГДА не коммитится
python check_env.py        # 4) офлайн-проверка (тот же валидатор) — exit 0 = ок
```

Только после зелёной проверки — первый запрос. Форма почти одинакова у всех, а ключ берётся из окружения, не из кода:

```python
from dotenv import load_dotenv
load_dotenv()   # положить ключи из .env в окружение — дальше SDK берёт их сам

# Anthropic
from anthropic import Anthropic
print(Anthropic().messages.create(model="claude-haiku-4-5", max_tokens=50,
    messages=[{"role": "user", "content": "Привет одним словом"}]).content[0].text)

# OpenAI
from openai import OpenAI
print(OpenAI().responses.create(model="gpt-5.x", input="Привет одним словом").output_text)

# Google
from google import genai
print(genai.Client().models.generate_content(
    model="gemini-2.x-flash", contents="Привет одним словом").text)
```

Для проверки бери **самую дешёвую** модель (Haiku / Flash) — первый запрос не должен стоить дорого (правило из урока 0.1).

## SHIP IT

**Артефакт:** Готовое окружение + проверочный скрипт → [`outputs/env-starter/`](../outputs/env-starter/)

Drop-in набор для любого проекта курса: `.env.example` (контракт ключей), `check_env.py` (тот же валидатор, что ты собрал, но самодостаточный — без зависимостей и сети) и строка `.env` для `.gitignore`. Копируешь в новый проект — и окружение проверяемо с первой минуты.

## ЧАСТЫЕ ОШИБКИ

- **Ключ прямо в коде.** Захардкоженный секрет утекает в git и в историю (даже если потом удалить). Ключи — только в `.env`, а `.env` — в `.gitignore`.
- **Закоммитить `.env`.** Сразу добавь `.env` в `.gitignore`. `.env.example` (без значений) коммить, `.env` — никогда.
- **Проверять ключ запросом к API.** 401 стоит времени и иногда денег. Сначала офлайн-валидатор (`check_env.py`), потом запрос.
- **Работать без venv.** Глобальная установка пакетов рано или поздно ломает другой проект, а «работает у меня» не воспроизводится. Заводи venv на проект.
