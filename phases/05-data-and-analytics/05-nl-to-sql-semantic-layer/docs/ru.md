# Урок 5.3 · NL→SQL и семантический слой

**Фаза 5 — Данные и аналитика** · **Результат фазы:** Анализировать данные с ИИ, делать NL→SQL и автоотчёты с валидацией результатов.
<!-- **Requires:** платный API-ключ — только для блока USE IT -->

> **MOTTO.** NL→SQL без семантического слоя — лотерея; метрики надо определить один раз и для всех.

## PROBLEM

«Спроси базу на человеческом языке» ломается о неоднозначность: что такое «выручка» — сумма `amount` или с вычетом возвратов? Без **семантического слоя** (единых определений метрик и измерений) LLM каждый раз гадает, и числа расходятся. Соберём NL→SQL поверх семантического слоя, чтобы смысл был один.

## CONCEPT

```
вопрос → [семантический слой: метрики + измерения] → SQL → результат
            выручка = SUM(amount)        регион = region
            заказы  = COUNT(*)           месяц  = month
```

Семантический слой — контракт смысла: «выручка» всегда `SUM(amount)`. LLM лишь сопоставляет вопрос с метрикой/разрезом, а SQL собирается из проверенных определений — отсюда консистентность.

## BUILD IT

NL→SQL поверх семантического слоя + исполнение на sqlite: [`code/nl_to_sql.py`](../code/nl_to_sql.py).

- `SEMANTIC` — метрики (стем-триггеры → SQL) и измерения;
- `nl_to_sql(question)` — собрать SQL (метрика + опц. GROUP BY);
- `run_sql(sql, rows)` — выполнить на in-memory sqlite (stdlib).

```bash
python code/nl_to_sql.py
pytest code -q
```

Тест прогоняет «выручка по регионам» → `SELECT region, SUM(amount) ... GROUP BY region` и реально считает на sqlite. Матч наивный (по словам) — настоящий парсинг делает LLM, но **семантический слой остаётся**.

## USE IT

Запросы к БД на естественном языке (мульти-провайдер): LLM генерирует SQL по схеме + семантике, ты исполняешь и проверяешь.

```python
schema = "Таблица sales(region, month, amount). Метрики: выручка=SUM(amount)."
sql = Anthropic().messages.create(model="claude-sonnet-4-6", max_tokens=200,
    messages=[{"role": "user", "content": f"{schema}\nSQL для: выручка по месяцам"}]).content[0].text
# OpenAI / Gemini — аналогично; затем выполнить sql в своей БД и сверить
```

Лучше — готовые слои: **dbt Semantic Layer** (метрики раз и навсегда), а инструменты (Genie/Spotter) дают NL→SQL поверх них. ⚠️ Всегда показывай и проверяй сгенерированный SQL.
