**Это финал Фазы 5 — и страховка для всего, что было до неё.** ИИ-аналитик (5.1), Code Interpreter (5.2), NL→SQL (5.3) и BI-агент (5.4) уверенно выдают числа. Но «уверенно» ≠ «верно». Прежде чем цифра уйдёт в отчёт, её надо **проверить** автоматически.

> **MOTTO.** Цифру от ИИ нельзя показывать непроверенной: сначала валидация и поиск аномалий, потом отчёт.

## PROBLEM

ИИ-аналитик выдаёт число с одинаковой уверенностью, права она или нет: мог взять не ту метрику, потерять строки при джойне, не заметить выброс из-за битой записи. Цифра выглядит правдоподобно — и именно поэтому опасна: уйдёт в отчёт руководству, и решение примут по мусору.

Нужен **контроль качества данных** — автоматический, перед показом: проверки на вменяемость (бизнес-инварианты), детекция аномалий (что выбивается из ряда) и сборка отчёта с флагами. Не «посмотреть глазами, если будет время», а встроить в пайплайн.

## CONCEPT

### Интуиция

Это **фактчекинг для цифр**, как редактор проверяет статью перед публикацией. Две независимые проверки:

- **Валидация по правилам** — отвечает на «может ли это вообще быть правдой?». Выручка не бывает отрицательной, заказов не бывает ноль в рабочий день, доля не больше 100%. Это твои **бизнес-инварианты** — то, что обязано выполняться всегда.
- **Поиск аномалий** — отвечает на «не выбивается ли что-то из ряда?». Даже если значение проходит правила, резкий скачок относительно соседей подозрителен и достоин флага.

Прошло обе проверки — собираем отчёт. Не прошло — отчёт всё равно собираем, но **с флагами и алертом**, чтобы человек посмотрел до того, как цифра разойдётся.

### Как это работает

```
результат анализа
   │ validate_result (правила-предикаты: revenue ≥ 0, orders > 0, нет пропусков)
   │ find_anomalies  (z-score: |value − mean| / std > порога)
   ▼
build_report → отчёт с метриками и флагами → (по расписанию + алерт)
```

- `validate_result(metrics, rules)` — прогнать каждую метрику через предикат-правило; вернуть список нарушений (и пометить отсутствующие метрики).
- `find_anomalies(series, z)` — посчитать z-score каждого значения (на сколько стандартных отклонений оно отстоит от среднего) и вернуть те, где `|z|` больше порога.
- `build_report(...)` — собрать markdown-отчёт из метрик и найденных аномалий.

Автоотчёт — это те же три шага, запускаемые **регулярно** (по cron/в оркестраторе): данные → анализ → валидация → аномалии → отчёт, с уведомлением при флагах.

## РАЗБОР ПО ШАГАМ

Поищем аномалию в ряду `[100, 110, 95, 105, 100, 500]` (порог `z = 2`). z-score = `(value − mean) / std`:

```
1. mean = (100+110+95+105+100+500) / 6 = 168.33
2. std (pstdev) ≈ 148.4
3. z каждого значения:
     100 → (100−168.33)/148.4 = −0.46     |z| < 2  ок
     110 → −0.39                            ок
      95 → −0.49                            ок
     105 → −0.43                            ок
     100 → −0.46                            ок
     500 → (500−168.33)/148.4 = +2.23      |z| > 2  ← АНОМАЛИЯ
```

Флаг получает только `500` (индекс 5, z = 2.23). Дальше `build_report` соберёт отчёт, где в разделе «Аномалии» будет строка про индекс 5 — и пайплайн поднимет алерт. Тонкость: один большой выброс **сам раздувает** mean и std, маскируя себя и сдвигая порог — поэтому z-score на маленькой выборке грубоват; в проде берут устойчивые методы (медиана/IQR, см. Материалы). Но как первый дешёвый фильтр он ловит явное.

## BUILD IT

**Задание: собери контроль качества** — валидацию по правилам, детекцию аномалий и сборку отчёта. Только стандартная библиотека (`statistics`), без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/5.5-validation/`), а файлы урока клади в подпапку `code/` (как в 0.1). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `validation.py` с тремя функциями:

- **`validate_result(metrics, rules)`** — `metrics` это `{ключ: значение}`, `rules` это `{ключ: предикат(значение)->bool}`. Для каждого правила: если ключа нет в метриках → добавь нарушение «отсутствует»; если предикат вернул `False` → добавь «не прошёл проверку». Верни список нарушений (пустой = всё ок).
- **`find_anomalies(series, z=2.0)`** — если в ряду меньше 2 значений или `std == 0` → верни `[]`. Иначе посчитай `mean` и `pstdev`, верни список `{"index", "value", "z": round(zscore, 2)}` для значений с `|z| > z`.
- **`build_report(title, metrics, anomalies)`** — собери markdown: заголовок `# {title}`, раздел `## Метрики` (по строке на метрику), раздел `## Аномалии` (строки `индекс N: value (z=...)` или «не обнаружено»).

**Готово, когда** все тесты в `test_validation.py` зелёные — они проверяют: валидация ловит провал правила и отсутствующую метрику, пропускает хорошие; `find_anomalies` находит выброс 500 и молчит на ровном ряду; отчёт содержит метрики и аномалии.

```bash
pytest code -q          # красное → реализуй три функции → зелёное
python code/validation.py # демо: ловит выброс 500, собирает отчёт
```

**Подсказка.** z-score: `(v - mu) / sigma`, где `mu = mean(series)`, `sigma = pstdev(series)`. Обязательно проверь `sigma == 0` (ровный ряд) и `len(series) < 2` до деления. Правила — это лямбды: `{"revenue": lambda x: x >= 0}`.

## USE IT

Автоотчёт по расписанию — это пайплайн из наших трёх шагов, запускаемый регулярно (мульти-провайдер):

```python
metrics   = run_daily_analysis()              # NL→SQL (5.3) / BI-агент (5.4)
problems  = validate_result(metrics, RULES)
anomalies = find_anomalies(series)
report    = build_report("Дневной отчёт", metrics, anomalies)
if problems or anomalies:
    notify(report)                            # алерт вместо «тихого» отчёта
```

- Источник данных в проде подключают через **MCP-коннектор к БД** (Фаза 6) — агент сам ходит за свежими данными.
- Расписание — планировщик (cron / dbt jobs / Genie).
- Главное правило: при флагах — **алерт человеку**, а не молчаливый отчёт «всё хорошо».

## ЧАСТЫЕ ОШИБКИ

- **Показывать цифру от ИИ без проверки.** Уверенный тон ≠ правильность. Валидация и аномалии — обязательный шаг перед отчётом, а не «если будет время».
- **Только аномалии без бизнес-правил (или наоборот).** Это две разные проверки: правила ловят невозможное (выручка < 0), аномалии — подозрительное (скачок). Нужны обе.
- **Доверять z-score на маленькой выборке.** Один выброс раздувает mean и std и может замаскировать сам себя. Для прода — устойчивые методы (медиана/IQR, isolation forest).
- **«Тихий» автоотчёт.** Если отчёт уходит всегда одинаково, флаги теряются. При нарушениях/аномалиях — отдельный алерт, который нельзя пролистать.
- **Правила «в коде навсегда».** Бизнес-инварианты меняются (новый продукт, сезон). Держи `rules` конфигурируемыми и пересматривай, иначе валидация устареет и пропустит ошибку.
