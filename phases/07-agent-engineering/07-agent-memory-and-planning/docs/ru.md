**Голый цикл из 7.1 умеет действовать, но не умеет вести длинную задачу.** Добавим ему две вещи, которые делают агента надёжным: **план** (разбить цель на шаги) и **память** (нести состояние между шагами). Обе детали мы уже видели по отдельности — теперь соединим в агенте.

> **MOTTO.** Надёжный агент не импровизирует на каждом шаге: он строит план и несёт состояние в памяти.

## PROBLEM

Цикл reason→act→observe (7.1) хорош для простого. Но на многошаговой задаче чистая импровизация «решаю на ходу, что делать дальше» спотыкается: агент забывает промежуточные результаты, теряет нить, повторяет уже сделанное.

Не хватает ровно двух вещей, и обе знакомы:
- **План** — заранее разбить цель на упорядоченные шаги (это prompt chaining из 2.3, но внутри агента).
- **Память** — носить состояние между шагами, чтобы шаг N видел результат шага N−1 (это рабочая память из 4.2).

Соединим их поверх цикла из 7.1 — и хаотичная импровизация превратится в управляемый процесс.

## CONCEPT

### Интуиция

Сравни двух исполнителей. Первый делает задачу «по наитию»: сделал шаг, забыл, что делал, думает заново — и легко сбивается на длинной цепочке. Второй сначала **набросал план** на бумаге, а по ходу **ведёт записи** (что уже посчитано). Второй надёжнее именно потому, что не держит всё в голове: план задаёт маршрут, записи хранят состояние.

Агент с памятью и планом — это второй исполнитель. `plan` рисует маршрут из цели, `Memory` — это блокнот, куда каждый шаг пишет результат и откуда следующий его читает.

### Как это работает

```
цель → plan() → [шаг1, шаг2, шаг3]
                   │ каждый шаг:
                   │   recall(value)  ← читаем прошлое значение из памяти
                   │   tool(...)      ← действуем
                   │   remember(value)→ пишем новое значение в память
                   ▼
              итог + память (все промежуточные шаги) + трейс
```

- `Memory` — `remember(key, value)` / `recall(key)`: рабочая память «ключ → значение».
- `plan(goal)` — декомпозиция: цель `{start, ops}` → список упорядоченных шагов.
- `run(goal)` — исполнить план, **неся состояние через память**: положить `value = start`, на каждом шаге прочитать `value`, применить инструмент, записать новый `value` (и сохранить `step_i` для истории). Вернуть `(итог, память, трейс)`.

Ключевая разница с 7.1: там состояние жило неявно в `history`, здесь оно **явно** в памяти под именами (`value`, `step_0`, …). Это делает процесс прозрачным и переиспользуемым — и масштабируется на настоящую долговременную память (4.2).

## РАЗБОР ПО ШАГАМ

Прогоним `run` на цели `(2 + 3) * 4 - 1` = `{start: 2, ops: [(add,3), (mul,4), (sub,1)]}`:

```
plan(goal) → [ {op:add, operand:3}, {op:mul, operand:4}, {op:sub, operand:1} ]
remember("value", 2)

шаг 0: recall("value")=2 → add(2,3)=5  → remember value=5,  step_0=5
шаг 1: recall("value")=5 → mul(5,4)=20 → remember value=20, step_1=20
шаг 2: recall("value")=20→ sub(20,1)=19→ remember value=19, step_2=19

итог = recall("value") = 19
память = {value: 19, step_0: 5, step_1: 20, step_2: 19}
```

Видно две роли. **План** задал, что и в каком порядке делать (три шага из `ops`). **Память** связала шаги: каждый читает `value` предыдущего и пишет свой. В отличие от 7.1, в памяти остаются **все** промежуточные результаты (`step_0..step_2`) — это трейс для отладки и основа для возврата к прошлому шагу. Поменяй `plan` на LLM-планировщик — каркас «recall → tool → remember» не изменится.

Создай файл `planner_agent.py`:

- **`Memory`** — класс с `store = {}`, методами `remember(key, value)` и `recall(key, default=None)`.
- **`TOOLS`** — `{"add", "sub", "mul"}` (функции двух аргументов).
- **`plan(goal)`** — вернуть список шагов `{"op", "operand"}` из `goal["ops"]`.
- **`run(goal, memory=None)`** — создать память (если не передана), положить `value = goal["start"]`; для каждого шага плана: `recall("value")` → применить инструмент → `remember("value", result)` и `remember(f"step_{i}", result)`; копить трейс. Вернуть `(итог, memory, trace)`.

**Готово, когда** все тесты в `test_planner_agent.py` зелёные — они проверяют: `plan` даёт 3 шага; `run` считает 19 через память (трейс из 3 шагов, итоговое `value` = 19); в памяти лежат промежуточные `step_0` = 5, `step_1` = 20; `Memory.remember`/`recall` работают (с дефолтом).

```bash
pytest code -q             # красное → реализуй Memory/plan/run → зелёное
python code/planner_agent.py # демо: (2+3)*4-1 = 19, видны шаги и память
```

**Подсказка.** Индекс шага бери из `enumerate(plan(goal))`. «Текущее значение» всегда живёт под ключом `"value"`: прочитал → посчитал → записал обратно. `step_i` — для истории, не для вычисления.

## USE IT

В реальности план строит LLM, а память — платформенная или своя (мульти-провайдер):

- **Claude Agent SDK** — agent loop + memory tool (`/memories`) или managed memory stores; план — через промптинг или субагентов.
- **OpenAI Agents SDK** — sessions хранят историю диалога; планирование — через инструкции агенту.
- Свой вариант: `MemoryStore` из урока 4.2 как долговременная память + LLM-планировщик поверх `run`.

⚠️ Память, в которую пишет агент, — это поверхность для prompt-инъекций: справочные (reference) данные держи read-only, отделяй их от того, что агент записал сам (6.5, 11.5).

## ЧАСТЫЕ ОШИБКИ

- **Импровизировать вместо плана на длинной задаче.** Без декомпозиции агент теряет нить и повторяет шаги. План задаёт маршрут — особенно важно, когда шагов много.
- **Не нести состояние через память.** Если шаг не пишет результат в память, следующий его не увидит — цепочка рвётся. Контракт «recall → tool → remember» обязателен.
- **Хранить только итог, теряя промежуточное.** Сохраняй `step_i`: это трейс для отладки и возможность вернуться к прошлому шагу. Один итог не даёт понять, где сломалось.
- **Смешивать память агента и доверенные данные.** То, что агент записал сам, и справочные данные — разные уровни доверия. Reference держи read-only, иначе инъекция через память (6.5/11.5).
- **План «намертво» в коде.** Здесь `plan` детерминирован для наглядности; в проде его строит LLM. Не хардкодь шаги, если задачи разные — планируй под цель.
