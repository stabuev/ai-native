**Requires (только для USE IT):** платный API-ключ Realtime/Live (OpenAI/Gemini). BUILD IT работает офлайн, без ключей и без сети.

**Это финал Фазы 7 — агент, который говорит.** Текстовый агент отвечает «целиком», голосовой устроен иначе: он стримит ответ и его можно перебить. Разберём, что меняется в цикле, когда агент выходит в реальное время.

> **MOTTO.** Голосовой агент — это поток и право перебить: ответ идёт по чанкам, пользователь может вмешаться.

## PROBLEM

В чате агент думает, потом выдаёт ответ блоком — задержка в пару секунд незаметна. В голосе так нельзя: пауза в две секунды звучит как «агент завис», а ответ, который нельзя перебить, ощущается как глухой автоответчик, не дающий вставить слово.

Голосовой агент меняет две вещи в цикле: ответ идёт **потоком** (по аудио-чанкам, чтобы начать говорить как можно раньше — метрика **TTFT**, время до первого токена), и пользователь может **перебить** (barge-in) — тогда текущий ответ надо **оборвать** и слушать. Смоделируем этот турновый цикл, чтобы понять управление «кто сейчас говорит».

## CONCEPT

### Интуиция

Разница как между **письмом и живым разговором**. В переписке ты дочитываешь сообщение целиком. В разговоре собеседник начинает отвечать сразу (не ждёт, пока сформулирует всё), и ты можешь его перебить на полуслове — он замолкает и слушает. Голосовой агент должен вести себя как живой собеседник: говорить сразу и уметь замолчать, когда заговорили с ним.

Отсюда две новые механики поверх цикла 7.1: **стриминг вывода** (начать отвечать рано) и **прерываемость** (бросить текущий ответ на barge-in). Сердце realtime — управление турами, кто держит слово.

### Как это работает

```
аудио пользователя ──► VAD (детект речи) ──► агент
                                              │ стримит ответ по токенам/чанкам
       barge-in (пользователь заговорил) ──► отменить текущий ответ
                                              │
                              TTFT — ключевая метрика (время до 1-го токена)
```

- `StreamingAgent.respond(tokens, interrupt_after)` — «произносит» ответ потоком, токен за токеном; если `interrupt_after=N` (пользователь перебил после N токенов) — обрывает турн, ставит состояние `interrupted` и возвращает только сказанную часть.
- состояния турна: `idle` (молчит) → `speaking` (говорит) → `idle` (договорил) или `interrupted` (перебили).
- `first_token_latency(timeline)` — TTFT, время до первого токена: в голосе важнее средней скорости, потому что определяет, как быстро агент «отзовётся».

В проде роль «пользователь заговорил» играет **VAD** (voice activity detection), а стрим идёт по WebSocket/WebRTC. Но логика турна — та же, что в нашей модели.

## РАЗБОР ПО ШАГАМ

Прогоним два турна `StreamingAgent`:

```
1. respond(["при","вет","как","дела"])              без перебивания
   state: speaking → токены идут по одному → дошли до конца
   → said = ["при","вет","как","дела"];  state = idle
   transcript += {said:[...], interrupted: False}

2. respond(["я","сейчас","расскажу","длинно"], interrupt_after=2)   barge-in
   i=0 "я", i=1 "сейчас", i=2 → i ≥ 2 → ПЕРЕБИЛИ: бросаем остаток
   → said = ["я","сейчас"];  state = interrupted
   transcript += {said:["я","сейчас"], interrupted: True}
```

И TTFT: `first_token_latency([0.18, 0.05, 0.05]) = 0.18` — первый токен пришёл через 0.18с, дальше быстрее. Видно главное отличие от чат-цикла: ответ не атомарен. Второй турн оборвался на полуслове, а в transcript честно записано, что реплика **прервана** — это и есть управление турами. Поменяй список токенов на аудио-чанки из Realtime API, а `interrupt_after` на сигнал VAD — структура не изменится.

Создай файл `streaming_loop.py`:

- **`StreamingAgent`** — `state` (`"idle"` | `"speaking"` | `"interrupted"`) и `transcript` (список реплик).
  - **`respond(tokens, interrupt_after=None)`** — поставить `state="speaking"`; идти по `tokens`: если `interrupt_after` задан и индекс `>= interrupt_after` → `state="interrupted"`, записать в `transcript` `{said, interrupted: True}` и вернуть сказанное; иначе добавить токен. Дошли до конца → `state="idle"`, записать `{said, interrupted: False}`, вернуть всё.
- **`first_token_latency(timeline)`** — вернуть первый элемент `timeline` (TTFT) или `None` для пустого.

## USE IT

Готовые realtime/voice API делают это с настоящим аудио (мульти-провайдер):

- **OpenAI Realtime API** — speech-to-speech одной моделью (`gpt-realtime-2`, низкая задержка), транспорты WebRTC / WebSocket / SIP, встроенный VAD и автоматическое прерывание, tool use; три типа сессий — voice-agent, перевод, транскрипция. Для браузера/мобильного — ephemeral-кредитал через `POST /v1/realtime/client_secrets`, а не основной ключ.
- **Gemini Live API** — двунаправленный WebSocket, аудио/видео/текст, barge-in, 70+ языков, live-перевод, function calling.
- Паттерн: VAD ловит речь → стрим ответа → barge-in отменяет текущий ответ (ровно как наш `interrupt_after`); клиенту выдают ephemeral tokens, а не основной ключ (безопасность).

## ЧАСТЫЕ ОШИБКИ

- **Отвечать блоком, как в чате.** В голосе пауза = «завис». Стримь ответ по чанкам и оптимизируй TTFT (время до первого токена), а не только общую скорость.
- **Не давать перебить (нет barge-in).** Агент, которого нельзя остановить, звучит как глухой автоответчик. Заговорил пользователь — текущий ответ обрывается.
- **Мерить среднюю скорость вместо TTFT.** В realtime решает, как быстро агент **начал** отвечать. Высокий средний throughput при медленном первом токене всё равно ощущается тормозным.
- **Терять состояние турна.** Нужно явно знать, кто говорит (`idle`/`speaking`/`interrupted`), и записывать, что реплика прервана. Иначе агент путается, чья очередь.
- **Голосовые действия без guardrails.** Голосом легко «случайно» подтвердить опасное. Необратимое всё равно через подтверждение и валидацию (7.3) — голос не отменяет заслоны.
