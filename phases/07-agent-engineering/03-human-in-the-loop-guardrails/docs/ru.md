**Агент из 7.1–7.2 действует сам. Теперь научим его останавливаться.** Чем выше автономия (режим Agency из 7.1), тем выше цена ошибки. Этот урок — про заслоны между решением агента и необратимым действием.

> **MOTTO.** Автономность без заслонов опасна: критичные действия — через подтверждение и валидацию.

## PROBLEM

Агент, который сам жмёт «удалить», «перевести деньги», «задеплоить», однажды сделает это **не вовремя**: по галлюцинации, по кривому аргументу или по prompt-инъекции из данных. Разница между полезным помощником и миной — в том, есть ли между «агент решил» и «действие выполнено» заслон.

Нужны две вещи: **валидация** (проверить аргументы до исполнения — лимиты, типы, разрешённые операции) и **human-in-the-loop** (для необратимых действий — пауза и подтверждение человека). Причём безопасные действия должны идти без трения, иначе агент бесполезен. Соберём такой заслон.

## CONCEPT

### Интуиция

Это как **банковский перевод с двумя барьерами**. Сначала система проверяет форму: сумма в пределах лимита, счёт существует, валюта верная (валидация — машинная, мгновенная). Потом, если операция крупная или необратимая, приходит SMS «подтвердите перевод» (human-in-the-loop — человек в петле). Мелкий безобидный платёж проходит без подтверждения; перевод миллиона — только после явного «да».

Агент должен работать так же: рутину делает сам, на необратимом — тормозит и спрашивает.

### Как это работает

```
действие агента {tool, args}
   │ 1) валидаторы     (типы, лимиты, разрешённые операции)  → не прошло → блок
   │ 2) опасное?       → пауза → подтверждение человека       → reject → блок
   │ 3) известный tool?                                        → нет → блок
   ▼ только тогда — исполнение
```

- `requires_confirmation(tool)` — опасное ли действие (множество `DANGEROUS`: `delete`, `send_money`, `deploy`, `drop_table`…).
- `run_action(action, tools, confirm, validators)` — три заслона **по порядку**: прогнать валидаторы → если действие опасное, спросить `confirm` → проверить, что инструмент известен. Только потом исполнить.
- `amount_limit(max)` — пример валидатора: запретить перевод сверх лимита.

Порядок не случаен: **валидаторы идут первыми** — нет смысла отвлекать человека подтверждением заведомо невалидного действия (перевод на сумму сверх лимита отсекается до вопроса). Подтверждение — только для того, что прошло проверку и при этом необратимо.

## РАЗБОР ПО ШАГАМ

Прогоним `run_action` на трёх действиях (лимит перевода — 100):

```
1. {tool: "read"}                              безопасное
   валидаторов нет → не опасное → tool известен → ИСПОЛНЕНО → "данные"

2. {tool: "send_money", args:{amount: 50, to: "Иван"}}   confirm=да
   amount_limit(100): 50 ≤ 100 → ok → опасное → confirm()=да → известен → ИСПОЛНЕНО → "перевод 50→Иван"

3. {tool: "send_money", args:{amount: 500, to: "X"}}     confirm=да
   amount_limit(100): 500 > 100 → GuardrailError ДО подтверждения  → БЛОК
```

Видно три исхода: рутина (`read`) проходит без трения; валидный опасный перевод — после явного «да»; невалидный (сверх лимита) режется **до** того, как человека о чём-то спросят. Если бы на шаге 2 `confirm` вернул `False` (человек отклонил) — тоже блок. Так автономность не превращается в «удалил прод по галлюцинации»: необратимое всегда проходит через валидацию и человека.

## BUILD IT

**Задание: собери заслоны агента** — валидацию, подтверждение и проверку инструмента. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/7.3-guardrails/`), а файлы урока клади в подпапку `code/` (по соглашению курса). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `guardrails.py`: множество `DANGEROUS` (необратимые действия), исключение `GuardrailError` и:

- **`requires_confirmation(tool)`** — `True`, если `tool` в `DANGEROUS`.
- **`run_action(action, tools, confirm=None, validators=None)`** — `action` это `{tool, args}`. По порядку: (1) прогнать каждый валидатор `validate(action) -> (ok, reason)`, при `not ok` → `raise GuardrailError`; (2) если `requires_confirmation(tool)` и (`confirm is None` или `confirm(action)` ложно) → `raise GuardrailError`; (3) если `tool not in tools` → `raise GuardrailError`; иначе исполнить `tools[tool](**args)`.
- **`amount_limit(max_amount)`** — вернуть валидатор: если `args["amount"] > max_amount` → `(False, reason)`, иначе `(True, "ok")`.

**Готово, когда** все тесты в `test_guardrails.py` зелёные — они проверяют: безопасное действие исполняется без подтверждения; опасное без `confirm` → блок, с `confirm=да` → исполняется; `confirm=нет` → блок; перевод сверх лимита → блок валидатором; неизвестный инструмент → блок.

```bash
pytest code -q          # красное → реализуй заслоны → зелёное
python code/guardrails.py # демо: read проходит; перевод 50 — ок; 500 — блок
```

**Подсказка.** Порядок проверок важен: валидаторы → подтверждение → известность инструмента. `amount_limit` — это функция, возвращающая функцию-валидатор (замыкание на `max_amount`).

## USE IT

Контроль агента на критичных шагах — готовыми механизмами (мульти-провайдер):

- **OpenAI Agents SDK** — `guardrails` (input / output / tool tripwire) + human-in-the-loop: на инструменте флаг `needs_approval` (`True` или async-функция для условной логики); на опасном вызове прогон встаёт на паузу и кладёт `ToolApprovalItem` в `RunResult.interruptions`. Состояние сериализуется через `result.to_state()` (`RunState`), решение — `state.approve(...)` / `state.reject(...)`, продолжение — `Runner.run(agent, state)`.
- **Anthropic** — tool use с подтверждением на стороне приложения; разрешения в Claude Code / Agent SDK (что агенту можно без спроса, что — только с одобрения).
- Паттерн: дешёвая guardrail-модель отсекает плохое **до** дорогого вызова; необратимое — только после approve человека.

## ЧАСТЫЕ ОШИБКИ

- **Давать агенту необратимые действия без подтверждения.** delete/send_money/deploy по галлюцинации или инъекции — катастрофа. Необратимое — только через human-in-the-loop.
- **Подтверждать всё подряд.** Если агент спрашивает на каждый шаг, им невозможно пользоваться. Трение — только на опасном; рутина идёт сама.
- **Подтверждение раньше валидации.** Сначала отсей заведомо плохое (лимит, типы), потом спрашивай человека. Незачем подтверждать то, что не пройдёт проверку.
- **Считать confirm заменой валидации (или наоборот).** Это разные заслоны: валидатор ловит «плохие аргументы», подтверждение — «необратимый эффект». Нужны оба.
- **Доверять аргументам от модели.** Опасное действие могло прийти из prompt-инъекции в данных (11.5). Валидаторы и подтверждение — именно та граница, что ловит это до исполнения.
