# Урок 11.4 · Корпоративные политики ИИ

**Фаза 11 — Этика, безопасность, governance** · **Результат фазы:** Оценить риски, настроить приватность и написать политику использования ИИ.

> **MOTTO.** Политика ИИ — это не бюрократия, а правила, которые делают использование ИИ безопасным и проверяемым.

## PROBLEM

Без политики каждый сотрудник решает сам: что отдавать модели, какие инструменты использовать, кто отвечает за ошибку. Это хаос и риски (11.1–11.3). Нужна **политика использования ИИ**: какие данные допустимы, какие инструменты одобрены, где обязателен человек, кто отвечает — и проверка конкретных кейсов против неё.

## CONCEPT

```
Политика ИИ (обязательные разделы):
  Допустимые данные · Запрещённые данные · Одобренные инструменты ·
  Проверка человеком · Ответственность
        │ validate_policy → чего не хватает
        │ check_usecase → можно ли этот кейс
   (PII во внешнюю модель? авто-решение о людях? → блок)
```

Политика опирается на рамки: **NIST AI RMF** (Govern), **ISO/IEC 42001** (сертифицируемая система управления ИИ), **EU AI Act** (обязанности по уровню риска).

## BUILD IT

Валидатор полноты политики + проверка кейса: [`code/policy.py`](../code/policy.py).

- `validate_policy(text)` — какие обязательные разделы отсутствуют;
- `check_usecase(sends_pii, external, automated_decision)` — разрешён ли кейс.

```bash
python code/policy.py
pytest code -q
```

Демо: PII во внешнюю модель — заблокировано; автоскоринг людей без человека — заблокировано; саммари публичных данных — разрешено. Это политика как исполняемые правила.

## USE IT

Согласование политики с реальными процессами (мульти-фреймворк):

- **NIST AI RMF** — функция Govern: роли, ответственность, процедуры на весь жизненный цикл.
- **ISO/IEC 42001** — Plan-Do-Check-Act для AIMS; единственный сертифицируемый стандарт.
- **EU AI Act** — обязанности по уровню риска (прозрачность, оценка, логирование); сроки вступления.
- Практика: политику пишут с участием security/legal, проверяют на реальных кейсах, ревизуют регулярно.
