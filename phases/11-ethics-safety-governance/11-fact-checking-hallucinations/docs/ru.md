> **MOTTO.** Доверяй, но заземляй: каждое фактическое утверждение должно опираться на источник.

## PROBLEM

Модель уверенно генерирует факты, которых нет в источниках (урок 1.4). В governance-контексте это риск: неверная цифра в отчёте, выдуманная ссылка в письме клиенту. Нужен **пайплайн фактчекинга**: разложить ответ на утверждения и проверить каждое на заземлённость (grounding) в доверенных источниках.

## CONCEPT

```
ответ → extract_claims (утверждения)
           │ для каждого:
   verify_claim(claim, sources): есть ли заземление в источниках?
           │
   supported / unsupported → отчёт (что проверять руками)
```

Фактчекинг — это не «верю/не верю», а **проверка опоры**: незаземлённое утверждение — кандидат в галлюцинации и в ручную проверку.

## BUILD IT

Пайплайн проверки утверждений на заземление: [`code/fact_check.py`](../code/fact_check.py).

- `extract_claims(text)` — разбить на утверждения;
- `verify_claim(claim, sources)` — `supported` / `unsupported` по совпадению с источниками;
- `fact_check(text, sources)` — отчёт по всем утверждениям.

```bash
python code/fact_check.py
pytest code -q
```

Демо: «возврат в течение 14 дней» — supported (есть в источниках), «гарантия 100 лет» — unsupported. Грубое заземление по совпадению; в проде — эмбеддинги/NLI и реальные источники.

## USE IT

Перекрёстная проверка моделями и источниками (мульти-провайдер):

- **Grounding/цитаты** — проси модель цитировать источник для каждого факта (Anthropic «reduce hallucinations»).
- **Перекрёстная проверка** — тот же вопрос на другой модели; расхождения = красный флаг.
- **RAG как источник истины** — отвечать только по найденным документам (Фаза 3) и помечать незаземлённое.
- **Право на «не знаю»** — снижает выдумывание (урок 1.4).
