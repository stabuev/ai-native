> **MOTTO.** Не одна модель на всё, а маршрут по сложности: простое — дёшево, сложное — топ.

## PROBLEM

Гонять всё через Opus — дорого; всё через Flash — качество проседает на сложном. Нужен **роутер**: оценить сложность запроса и направить на подходящую модель. Это главный рычаг экономии (40–70%) без потери качества.

## CONCEPT

```
запрос → classify(сложность) → route(модель)
   low   (короткое, классификация/перевод)   → Gemini Flash
   medium(обычная задача)                     → Sonnet 4.6
   high  (код, анализ, рассуждение, длинное)  → Opus 4.8
```

Сложность оцениваем по признакам: длина, ключевые слова (код/анализ/докажи), нужно ли рассуждение. Это дешёвый «привратник» перед дорогим вызовом.

## BUILD IT

Классификатор сложности + маршрутизация: [`code/complexity_router.py`](../code/complexity_router.py).

- `classify(request)` — `low` / `medium` / `high` по признакам;
- `route(request)` — модель под сложность.

```bash
python code/complexity_router.py
pytest code -q
```

Демо: «переведи фразу» → Flash, «саммари» → Sonnet, «проанализируй архитектуру и оптимизируй код» → Opus. Матч по стемам (учитывает морфологию, см. грабли 5.3).

## USE IT

Маршрут «дешёвые → средние → топ» в проде (мульти-провайдер):

- **Anthropic** — Haiku для классификации/роутинга/извлечения, Sonnet как дефолт, Opus только на сложное (см. «Choosing the right model»).
- **Кросс-провайдер** — Gemini Flash на массовое, GPT/Sonnet на универсальное, Opus на топ.
- Роутер можно делать дешёвой моделью или эвристикой (как наш `classify`) — главное, чтобы он сам был дешёвым.
