**Requires (только для USE IT):** платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

**В 9.1 мы выяснили: выбор модели — рычаг №1 стоимости (разброс в десятки раз).** Теперь применим этот рычаг: не гонять всё через топ, а направлять каждый запрос на модель **под его сложность**.

> **MOTTO.** Не одна модель на всё, а маршрут по сложности: простое — дёшево, сложное — топ.

## PROBLEM

Две крайности одинаково плохи. Гонять **всё через Opus** — платишь топ-цену за «переведи фразу», счёт раздут на ровном месте. Гонять **всё через Flash** — экономишь, но на коде, анализе и рассуждении качество проседает, и дешёвый ответ оказывается бесполезным (а переделка — дороже).

Нужен **роутер**: дешёвый «привратник», который смотрит на запрос, оценивает его сложность и направляет на подходящую модель. Это и есть главный рычаг экономии Фазы 9 — срезать расходы, не теряя качества на сложном. Соберём классификатор сложности.

## CONCEPT

### Интуиция

Это **сортировка обращений в поддержке**. Не сажают доктора наук на каждый звонок: типовой вопрос («как сбросить пароль») закрывает оператор первой линии, сложный кейс эскалируют эксперту. Сортировщик на входе дёшев и быстр — он не решает задачу, он лишь **определяет, кому её отдать**.

Роутер моделей работает так же: оценивает сложность по дешёвым признакам и отправляет простое на «эконом», сложное — на «эксперта». Главное свойство привратника — он сам должен быть **дешёвым** (эвристика или мелкая модель), иначе съест экономию.

### Как это работает

```
запрос → classify(сложность) → route(модель)
   low   (короткое, перевод/классификация)     → Gemini Flash   (дёшево)
   medium(обычная задача)                       → Sonnet 4.6     (дефолт)
   high  (код, анализ, рассуждение, длинное)    → Opus 4.8       (топ)
```

- `classify(request)` — вернуть `low` / `medium` / `high` по признакам: ключевые слова-стемы (`код`, `проанализир`, `докаж`…), длина запроса, нужно ли рассуждение.
- `route(request)` — отдать модель под класс сложности из таблицы `ROUTES`.

Признаки-стемы, а не целые слова, — чтобы ловить морфологию (`проанализируй`/`анализ`/`проанализир`); та же грабля, что в 5.3. И дефолт — `medium` (Sonnet): если непонятно, берём универсальную середину, а не самую дорогую. Это совпадает с рекомендацией Anthropic: Haiku — на простое, Sonnet — на большинство прод-задач, Opus — только на самое сложное рассуждение.

## РАЗБОР ПО ШАГАМ

Прогоним `classify` по четырём запросам (порядок проверок: сначала «high», потом «low», иначе «medium»):

```
1. "Переведи фразу на английский"        длина ~28
   HARD-стемы? нет.  EASY ("переведи") И длина<200? ДА → low  → gemini-flash

2. "Сделай саммари встречи в 5 пунктов"
   HARD? нет.  EASY? нет ("саммари" не в списке) → medium → sonnet-4.6  (дефолт)

3. "Проанализируй архитектуру и оптимизируй код"
   HARD-стемы: "проанализир", "архитектур", "оптимизир", "код" → high → opus-4.8

4. строка длиной 700 символов
   HARD? нет, но длина>600 → high → opus-4.8
```

Видно логику привратника. Простое и короткое (`перевод`) уходит на дёшево; явно сложное (`анализ`+`код`) — на топ; длинное эскалируется в `high` само (большой вход = дороже и обычно сложнее); всё неопределённое садится на безопасный дефолт `medium`. Привратник не решает задачу — он за один проход по словам и длине выбирает исполнителя. Поменяй эвристику на мелкую модель-классификатор — схема не изменится.

## BUILD IT

**Задание: собери классификатор сложности и роутер** — `low`/`medium`/`high` → модель. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/9.2-router/`), а файлы урока клади в подпапку `code/` (по соглашению курса). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `complexity_router.py`:

- **`HARD_HINTS`** / **`EASY_HINTS`** — кортежи стемов-триггеров (`код`, `проанализир`, `докаж`… / `переведи`, `классифиц`…).
- **`ROUTES`** — `{"low": "gemini-flash", "medium": "sonnet-4.6", "high": "opus-4.8"}`.
- **`classify(request)`** — вернуть сложность: если есть HARD-стем **или** длина > 600 → `high`; иначе если есть EASY-стем **и** длина < 200 → `low`; иначе `medium`.
- **`route(request)`** — `ROUTES[classify(request)]`.

**Готово, когда** все тесты в `test_complexity_router.py` зелёные — они проверяют: простой запрос → `low` → gemini-flash; сложный (анализ/код, доказательство) → `high` → opus; обычный → `medium` → sonnet; длинный запрос (700 символов) эскалируется в `high`.

```bash
pytest code -q                  # красное → реализуй classify → зелёное
python code/complexity_router.py # демо: перевод→Flash, саммари→Sonnet, анализ→Opus
```

**Подсказка.** Приведи запрос к нижнему регистру. Порядок важен: сначала проверяй `high` (HARD-стем или длина), потом `low` (EASY-стем и короткое), иначе `medium`. Стемы как подстроки (`h in text`) ловят все словоформы.

Внизу, в [«Исходниках урока»](#lesson-files), — три способа пройти упражнение (собрать самому · подсмотреть эталон · делегировать ИИ) и тесты-ТЗ.

## USE IT

Маршрут «дешёвые → средние → топ» в проде (мульти-провайдер):

- **Anthropic** — по их же гайду: **Haiku** на классификацию/роутинг/извлечение, **Sonnet** как дефолт для большинства прод-нагрузок, **Opus** только на самое сложное рассуждение.
- **Кросс-провайдер** — Gemini Flash на массовое дешёвое, GPT/Sonnet на универсальное, Opus на топ.
- **Open-weight тир** — DeepSeek V4 (дёшево + reasoning через thinking-mode); подходит, когда нужен self-host ради приватности/суверенитета данных (Фаза 11).
- Роутер делают эвристикой (как наш `classify`) или **мелкой** моделью — главное, чтобы привратник сам был дешёвым, иначе он съест экономию.

Маршрутизация — самый большой единичный рычаг; вместе с кэшем и batch (9.1) и каскадом (9.3) она и даёт целевые **50–85%** экономии фазы.

## ЧАСТЫЕ ОШИБКИ

- **Одна модель на всё.** Топ на простое — переплата, дешёвая на сложное — брак и переделка. Маршрут по сложности снимает обе крайности.
- **Дорогой привратник.** Если классификатор сам тяжёлая модель, он съест экономию. Роутер — эвристика или мелкая модель, дешёвая и быстрая.
- **Матч по целым словам, не стемам.** `проанализируй` ≠ `анализ`. Морфология ломает точный матч — триггеры держи стемами (как в 5.3).
- **Агрессивный дефолт в дешёвое.** Если непонятно — безопаснее `medium`, а не `low`: дешёвый брак на сложном дороже, чем чуть переплатить. Дефолт — универсальная середина.
- **Роутинг без проверки качества.** Дешёвая модель иногда не тянет. Сам роутер не ловит брак — для этого нужен quality-gate и каскад (9.3).
