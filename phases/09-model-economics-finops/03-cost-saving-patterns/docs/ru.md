> **MOTTO.** Сначала дёшево, эскалация по необходимости: каскад + кэш + batch дают кратную экономию.

## PROBLEM

Роутер (9.2) решает наперёд: «это сложно → сразу на Opus». Но он судит по поверхностным признакам и иногда промахивается — отправит простое на топ (переплата) или сложное на дешёвое (брак).

Надёжнее судить **по результату**: запустить дешёвую модель, проверить качество ответа, и эскалировать на сильную **только если** дешёвая не дотянула. Это **каскад**. Сверху — инфраструктурные скидки (prompt caching, batch API), которые стэкаются с каскадом. Вместе — кратная экономия. Соберём каскад с quality-gate и честно посмотрим, когда он выигрывает, а когда нет.

## CONCEPT

### Интуиция

Каскад — это **«сначала спроси стажёра, не справился — эскалируй к старшему»**. Большинство задач закрывает дешёвый исполнитель; дорогого эксперта подключаешь, только когда дешёвый явно не вытянул.

Каскад — из работы **FrugalGPT**, и там это лишь **одна из трёх** стратегий экономии, которые стэкаются: **prompt adaptation** (сжать вход — короче промпт, меньше токенов), **LLM approximation** (кэш ответов и дообучение мелкой модели) и **LLM cascade** (наша лестница моделей со scoring-функцией). Результат у FrugalGPT двойной: либо **то же качество, что у лучшей модели (GPT-4), дешевле до 98%**, либо **+4% к точности за ту же цену**. То есть экономия — это не только «срезать счёт», но и «за те же деньги получить больше».

Но у каскада есть цена: за **проваленную** дешёвую попытку ты всё равно заплатил. Поэтому каскад выгоден, когда дешёвые модели проходят **часто**; если почти всё сложное — он, наоборот, дороже, чем сразу звать топ.

### Как это работает

```
запрос → cheap → quality >= gate? ──да──► ответ (заплатили мало)
                     │ нет
                     ▼
                  mid → quality >= gate? ──да──► ответ
                     │ нет
                     ▼
                  top → ответ (платим за топ только когда нужно)
   + prompt caching (чтение 0.1× входа)  + batch (−50%)  — стэкаются
```

- `cascade(request, answer_fn, quality_fn, gate)` — идти по лестнице моделей (дешёвая→дорогая), накапливая стоимость; на каждой проверять `quality_fn(request, answer)`; вернуть первый ответ с качеством ≥ `gate` (или лучший доступный, если до топа никто не прошёл).
- `quality_fn` — оценка качества (здесь стаб; в проде — дешёвая **модель-судья** из 2.5, чтобы не эскалировать зря).
- `always_top_cost()` — сколько стоило бы «всё через топ», для сравнения экономии.

В терминах FrugalGPT каскад держится на двух функциях: **generation function** задаёт порядок моделей (у нас — фиксированная лестница `CASCADE` дёшево→дорого), а **scoring function** решает, принять ответ или эскалировать (у нас — `quality_fn` + порог `gate`). Ключ — **quality-gate**: порог, ниже которого ответ считается недостаточным и запускает следующую модель.

## РАЗБОР ПО ШАГАМ

Лестница с условной ценой за вызов: `flash=1, sonnet=10, opus=30`, порог `gate=0.7`. Стаб качества: простое тянет любая модель (0.9), сложное — только Opus.

```
"простой вопрос":
  flash  → quality 0.9 ≥ 0.7  → СТОП.  cost = 1

"сложный вопрос":
  flash  → 0.4 < 0.7 → эскалация   (потратили 1)
  sonnet → 0.4 < 0.7 → эскалация   (потратили 1+10=11)
  opus   → 0.9 ≥ 0.7 → СТОП.        cost = 1+10+30 = 41
```

Теперь смесь — 3 простых + 1 сложный:

```
каскад:     1 + 1 + 1 + 41 = 44
всё на топ: 30 × 4         = 120        → каскад дешевле на 63%
```

Каскад выиграл — но обрати внимание на честную деталь: **сам сложный запрос в каскаде стоил 41 против 30 «сразу топ»** — мы переплатили за провалившиеся flash и sonnet. Экономию дали три простых по цене 1 вместо 30. Вывод: каскад окупается **долей простого**. Если бы нагрузка была почти вся сложная, каскад проиграл бы прямому вызову топа — это его граница.

**Задание: собери каскадный роутер с quality-gate** — дёшево, эскалация по факту. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/9.3-cascade/`), а файлы урока клади в подпапку `code/` (по соглашению курса). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `cost_router.py`:

- **`CASCADE`** — лестница `[(model, price), ...]` от дешёвой к дорогой (`flash=1, sonnet=10, opus=30`).
- **`cascade(request, answer_fn, quality_fn, gate=0.7, ladder=CASCADE)`** — идти по `ladder`: прибавлять `price` к `spent`, получить `answer = answer_fn(model, request)`, оценить `q = quality_fn(request, answer)`, запомнить `{model, answer, quality, cost: spent}`; если `q >= gate` → вернуть. Дошли до конца → вернуть последний (лучший доступный).
- **`always_top_cost(ladder=CASCADE)`** — цена топ-модели (`ladder[-1][1]`).

**Готово, когда** все тесты в `test_cost_router.py` зелёные — они проверяют: простой запрос останавливается на самой дешёвой (cost 1); сложный эскалирует до Opus (quality ≥ gate); на смешанной нагрузке каскад дешевле «всегда топ»; если никто не прошёл gate — возвращается верхняя модель.

```bash
pytest code -q          # красное → реализуй cascade → зелёное
python code/cost_router.py # демо: простой → flash, сложный → opus
```

**Подсказка.** Копи `spent` ДО вызова модели (вызов уже стоит денег). Возвращай первый ответ с `q >= gate`; если цикл закончился — отдавай последний `last`. `quality_fn` — заменяемая деталь (стаб ↔ LLM-judge).

## USE IT

Паттерны экономии в проде стэкаются (мульти-провайдер). Ниже — best practices каждого, а не только «включи и сэкономишь».

**Каскад (best practices):**

- **Дешёвый scorer.** Если scoring-функция — тяжёлая модель, она съест экономию. Гейт делают мелкой моделью-судьёй (2.5) или эвристикой; он должен быть в разы дешевле эскалации, которую предотвращает.
- **Калибруй gate на eval-наборе.** Порог — это компромисс «деньги vs брак»: высокий гонит всё на топ (нет экономии), низкий пропускает плохие ответы. Подбирай его на размеченном наборе (2.5/7.5), а не на глаз.
- **Порядок дёшево→дорого и немного ступеней.** 2–3 модели обычно достаточно; лишние ступени добавляют цену провалов без выигрыша.
- **Каскад + роутер вместе, а не вместо.** Роутер (9.2) сразу отсекает явно сложное, чтобы не тратить дешёвую попытку впустую; каскад страхует на неопределённом. Связка экономит больше, чем любой по отдельности.

**Prompt caching (best practices — кэш работает по префиксу!):**

- **Статика в начало, динамика в конец.** Порядок `tools → system → messages`: стабильное (инструменты, system-промпт, документы) — в начало, изменчивое (таймстамп, ввод пользователя) — в конец. `cache_control` ставь на **последний неизменный** блок. Частая ошибка: поставить breakpoint после блока с таймстампом — тогда префикс каждый раз разный, кэш **никогда** не попадает, и ты платишь только за записи.
- **Следи за минимумом токенов.** Короче порога кэш молча не создаётся (на 2026-06: Opus 4.8 и Sonnet 4.6 — ~1024 токена, Haiku 4.5 — ~4096). Проверяй поля `cache_read_input_tokens` / `cache_creation_input_tokens` в ответе: оба 0 → кэш не сработал.
- **Выбирай TTL под частоту переиспользования.** 5-минутный кэш (запись 1.25×) обновляется бесплатно при каждом использовании — для частых запросов; 1-часовой (запись 2×) — когда тот же контекст зовут реже, но в пределах часа.
- **Помни про инвалидацию.** Изменение на уровне иерархии бьёт этот уровень и все ниже: правка определения инструмента инвалидирует и system, и messages. Стабильный префикс — залог попаданий.

**Batch API:** асинхронные нетерминальные задачи (классификация, eval, массовая генерация) — −50% на вход и выход; стэкается с кэшем.

Маршрутизация + каскад + кэш + batch вместе и дают целевые **50–85%** экономии фазы — но каждый паттерн со своей границей применимости, и применять их надо по best practices выше, а не «в лоб».

## ЧАСТЫЕ ОШИБКИ

- **Считать каскад бесплатным на сложном.** За проваленную дешёвую попытку платишь. Каскад окупается долей простого; на почти-всё-сложной нагрузке прямой вызов топа дешевле.
- **Дорогой quality-gate.** Если судья — тяжёлая модель, он съест экономию каскада. Гейт — дешёвая модель-судья или эвристика (2.5).
- **Слишком высокий или низкий gate.** Высокий порог гонит всё на топ (нет экономии); низкий пропускает брак. Порог калибруют на eval-наборе (2.5/7.5).
- **Не стэкать скидки.** Caching и batch комбинируются и между собой, и с каскадом. Использовать только один паттерн — оставить деньги на столе.
- **Каскад вместо роутера, а не вместе.** Роутер отсекает явно сложное сразу (не тратя дешёвую попытку), каскад страхует на неопределённом. Лучшая экономия — связка, а не одно из двух.
- **Динамика перед кэш-breakpoint.** Кэш работает по префиксу: если перед точкой кэширования стоит изменчивый блок (таймстамп, ввод), префикс каждый раз новый и кэш **никогда** не попадает — платишь только за записи. Статика в начало, `cache_control` на последний неизменный блок.
