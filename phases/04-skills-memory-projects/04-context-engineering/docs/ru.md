> **MOTTO.** Контекст — конечный бюджет; инженерия в том, ЧТО и в каком порядке в него класть.

## PROBLEM

Скиллы, память, найденные RAG-куски, история диалога, описания инструментов — всё претендует на место в окне. А окно конечно (урок 1.2) и платное (урок 1.1). Запихнуть всё нельзя физически, и — что менее очевидно — **вредно**: чем больше токенов, тем хуже модель вспоминает важное. Anthropic называет это так: у модели ограниченный «бюджет внимания», каждый новый токен его расходует, а с ростом окна точность припоминания падает (context rot).

Значит, нужна не «напихать побольше», а **политика**: собрать контекст по приоритету, а когда не влезает — осознанно отбросить наименее важное. Цель — по формулировке Anthropic — найти наименьший набор высокосигнальных токенов, дающий нужный результат. Соберём такую политику сами.

## CONCEPT

### Интуиция

Контекстное окно — это **чемодан фиксированного объёма** перед поездкой. Вещей больше, чем влезает. Разумный сбор: сначала кладёшь то, без чего никак (документы, лекарства = system prompt, релевантные факты), потом — что полезно (память), и в последнюю очередь — «приятно бы взять» (вся история диалога). Когда чемодан полон — низкоприоритетное остаётся дома.

Контекст-инжиниринг — это и есть упаковка чемодана: не «какие слова подобрать» (это prompt engineering), а **какая конфигурация контекста** даёт нужное поведение при ограниченном объёме.

### Как это работает

```
слои контекста (по приоритету, больше = важнее):
  system (4)  >  RAG (3)  >  память (2)  >  история (1)
        │ assemble(layers, budget)
        ▼
  сортируем по убыванию приоритета и берём слой за слоем,
  пока влезает в бюджет; не влезло → в dropped (компакция)
```

- `estimate_tokens(text)` — оценка размера слоя (учебно ≈ число слов; точнее — токенайзером из 1.1).
- `assemble(layers, budget)` — жадная сборка: идём от самого приоритетного слоя, кладём, пока хватает бюджета; что не поместилось — в `dropped`.
- `compact_history(history, max_items)` — компакция истории: оставляем последние `max_items` сообщений, старое «забываем».

Ключевая идея — **приоритет, а не порядок прихода**. System prompt и свежий RAG важнее старой истории, поэтому при нехватке места жертвуем именно историей, а не ролью.

## РАЗБОР ПО ШАГАМ

Соберём контекст с бюджетом `budget = 60` токенов. Четыре слоя (размер в «словах»):

```
system   20 токенов  приоритет 4
rag      30 токенов  приоритет 3
memory   20 токенов  приоритет 2
history  40 токенов  приоритет 1
```

`assemble` сортирует по убыванию приоритета и идёт жадно, накапливая `used`:

```
system : used 0 + 20 = 20 ≤ 60  → берём  (used = 20)
rag    : used 20 + 30 = 50 ≤ 60  → берём  (used = 50)
memory : used 50 + 20 = 70 > 60  → DROPPED
history: used 50 + 40 = 90 > 60  → DROPPED
```

Итог: `kept = [system, rag]`, `tokens = 50`, `dropped = [memory, history]`. Важное (роль + релевантные знания) уцелело, низкоприоритетное отброшено — окно не переполнено. Будь бюджет всего 20 — остался бы только `system`. А `compact_history(["m1","m2","m3","m4"], 2)` отдаст `["m3","m4"]`: история ужата до двух свежих сообщений, чтобы влезть. Заметь: сборка **жадная по приоритету**, а не оптимальная упаковка — это намеренно просто и предсказуемо.

## BUILD IT

**Задание: собери движок контекста** — оценка размера, сборка под бюджет, компакция истории. Только стандартная библиотека, без сети.

> **Перед запуском.** Работай в своей папке курса (`ai-native/4.4-context/`), а файлы урока клади в подпапку `code/` (как в 0.1). Нужен только **Python 3** (для теста ещё `pytest`).

Создай файл `context_engine.py` с тремя функциями:

- **`estimate_tokens(text)`** — грубая оценка размера: число слов (`len(text.split())`). В реальном коде здесь был бы токенайзер из 1.1.
- **`assemble(layers, budget)`** — собрать контекст. Вход: `layers` — список `{name, content, priority}` (больше priority = важнее) и `budget` в токенах. Отсортируй по убыванию `priority`, иди по списку: если `used + estimate_tokens(content) <= budget` — клади слой в `kept` и увеличивай `used`, иначе — в `dropped`. Верни `{"kept", "dropped", "tokens": used}`.
- **`compact_history(history, max_items)`** — верни последние `max_items` элементов списка (если `max_items < 0` — верни всё без изменений).

**Готово, когда** все тесты в `test_context_engine.py` зелёные — они проверяют: подсчёт слов; при большом бюджете влезает всё (`dropped` пуст); при тесном — остаются `system` и `rag`, а `history` отброшена и `tokens <= budget`; при бюджете на один слой остаётся только `system`; `compact_history` оставляет свежие сообщения.

```bash
pytest code -q                 # красное → реализуй три функции → зелёное
python code/context_engine.py  # демо: что влезло в бюджет 60, что отброшено
```

**Подсказка.** Сортировка по приоритету: `sorted(layers, key=lambda l: l["priority"], reverse=True)`. Жадно копи `used` и сравнивай с `budget` перед добавлением. Для `compact_history` хватит среза `history[-max_items:]` (но отдельно обработай отрицательный `max_items`).

## USE IT

Контекст-инжиниринг связывает в одно RAG, память и стоимость (мульти-провайдер):

- **RAG вместо «всего корпуса»** — кладём топ-`k` релевантных кусков (Фаза 3), а не весь документ.
- **Память — выборочно** — подмешиваем релевантные записи (4.2), а не всю переписку.
- **История — компакция** — старые сообщения режем или резюмируем (3.1), оставляя свежие и итоги.
- **Инструменты тоже едят контекст** — держи 3–5 ключевых всегда, остальное подгружай по запросу (как скиллы в 4.1).
- **Стоимость** — каждый слой считаем `prompt-token-budgeter` (1.1): меньше токенов → дешевле и быстрее (FinOps, Фаза 9).

## ЧАСТЫЕ ОШИБКИ

- **«Больше контекста = лучше».** Наоборот: лишние токены тратят бюджет внимания и роняют точность припоминания (context rot). Цель — наименьший набор высокосигнальных токенов.
- **Собирать по порядку прихода, а не по приоритету.** Тогда старая история может вытеснить system prompt или свежий RAG. Сортируй по важности, режь снизу.
- **Путать prompt engineering и context engineering.** Первый — какие слова в одном запросе; второй — какая конфигурация контекста (слои, бюджет, компакция) держится через всю сессию.
- **Не сжимать историю.** Диалог растёт бесконечно; без `compact_history` (или резюме) окно переполнится и подорожает. Оставляй свежее и итоги.
- **Считать `estimate_tokens` = словам в проде.** Это учебная оценка; для денег и лимитов считай реальным токенайзером модели (1.1).
