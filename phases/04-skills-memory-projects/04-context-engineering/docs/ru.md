# Урок 4.4 · Контекст-инжиниринг

**Фаза 4 — Скиллы, память и проекты** · **Результат фазы:** Упаковывать способности в SKILL.md, хранить контекст в памяти и собирать рабочие проекты.

> **MOTTO.** Контекст — конечный бюджет; инженерия в том, ЧТО и в каком порядке в него класть.

## PROBLEM

Скиллы, память, RAG, история диалога — всё хочет места в контекстном окне, а оно конечно (урок 1.2) и платное (урок 1.1). Запихнуть всё нельзя, да и вредно: лишнее снижает внимание модели («context rot»). Нужна **политика сборки и компакции** контекста: по приоритету собрать, лишнее — отбросить.

## CONCEPT

```
слои контекста (по приоритету):
  system (4) > RAG (3) > память (2) > история (1)
        │ assemble(layers, budget)
        ▼
  берём по убыванию приоритета, пока влезает в бюджет
  не влезло → компакция: отбрасываем/режем низкоприоритетное
```

Это «context engineering»: не «какие слова», а «какая конфигурация контекста даёт нужное поведение» при ограниченном окне.

## BUILD IT

Сборка контекста под бюджет + компакция, без зависимостей: [`code/context_engine.py`](../code/context_engine.py).

- `estimate_tokens(text)` — оценка размера (≈ слова; точнее — токенайзером из 1.1);
- `assemble(layers, budget)` — собрать по приоритету, что не влезло — в `dropped`;
- `compact_history(history, max_items)` — оставить последние сообщения.

```bash
python code/context_engine.py
pytest code -q
```

Тест показывает политику в действии: при тесном бюджете остаются `system` и `RAG`, а `history` отбрасывается; всё влезает — берём всё.

## USE IT

Контекст-инжиниринг связывает RAG и стоимость (мульти-провайдер):

- **RAG вместо «всего корпуса»** — кладём только топ-k релевантных кусков (Фаза 3), а не весь документ.
- **Память — выборочно** — подмешиваем релевантные записи (4.2), а не всю историю.
- **Стоимость** — каждый слой считаем `prompt-token-budgeter` (1.1); меньше токенов → дешевле и быстрее (Фаза 9).
- Инструменты тоже занимают контекст: держи 3–5 ключевых всегда, остальное — по запросу (см. Anthropic context engineering).
