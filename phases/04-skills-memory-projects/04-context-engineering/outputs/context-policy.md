# Артефакт: политика управления контекстом

Что и в каком порядке класть в контекстное окно при ограниченном бюджете. Движок — [`code/context_engine.py`](../code/context_engine.py).

## Приоритеты слоёв (сверху — важнее)

| Приоритет | Слой | Что входит | Как компактим |
|---|---|---|---|
| 4 | **system** | роль, правила, формат | держим всегда; коротко |
| 3 | **RAG** | top-k релевантных кусков | меньше k, выше порог релевантности |
| 2 | **память** | релевантные записи (4.2) | только по теме, не всю историю |
| 1 | **история** | прошлые сообщения | `compact_history`: последние N |

## Алгоритм сборки

```python
from context_engine import assemble, compact_history
layers = [
  {"name": "system",  "content": SYSTEM,           "priority": 4},
  {"name": "rag",     "content": retrieved,        "priority": 3},
  {"name": "memory",  "content": mem_snippets,     "priority": 2},
  {"name": "history", "content": "\n".join(compact_history(hist, 6)), "priority": 1},
]
res = assemble(layers, budget=8000)   # влезшее — res["kept"], отброшенное — res["dropped"]
```

## Правила

- **Бюджет считай заранее** (`prompt-token-budgeter`, урок 1.1): сумма слоёв + ответ ≤ окно.
- **Больше ≠ лучше:** лишний контекст снижает внимание модели («context rot»).
- **RAG вместо всего корпуса; память — выборочно.** Не дублируй знания в system prompt.
- **Инструменты тоже занимают контекст** — держи 3–5 ключевых, остальное по запросу.
- **Режь с низа приоритета**: первой жертвуется старая история, не system prompt.
- Меньше токенов → дешевле и быстрее (мост к FinOps, Фаза 9).
