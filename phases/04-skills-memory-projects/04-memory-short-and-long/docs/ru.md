# Урок 4.2 · Память: короткая и длинная

**Фаза 4 — Скиллы, память и проекты** · **Результат фазы:** Упаковывать способности в SKILL.md, хранить контекст в памяти и собирать рабочие проекты.

> **MOTTO.** Без памяти агент каждый раз начинает с чистого листа; память — это контекст, переживающий сессию.

## PROBLEM

Модель забывает всё между запросами: предпочтения, факты о пользователе, решения прошлых задач. Каждый диалог — заново. Нужна **память**: короткая (в пределах сессии) и длинная (на диске, между сессиями) с поиском по релевантности и учётом свежести.

## CONCEPT

```
add(text, ts)  → запись с меткой времени
                     │
search(query)  → релевантность (совпадение) + свежесть (затухание во времени)
                     │
              топ-k записей → в контекст следующего запроса
save / load    → переживает перезапуск (длинная память)
```

Темпоральность важна: при равной релевантности свежее обычно важнее. Это ровно тот принцип, по которому работает и моя межсессионная память.

## BUILD IT

Файловое хранилище памяти с поиском и темпоральностью: [`code/memory_store.py`](../code/memory_store.py).

- `add(text, tags, ts)` — запись с меткой времени;
- `search(query, k, now)` — релевантность + свежесть (полураспад `half_life`);
- `recent(n)` — последние записи; `save/load` — персистентность.

```bash
python code/memory_store.py
pytest code -q
```

Тест доказывает темпоральность: две записи про «кофе», но свежая выходит первой; а после save/load память переживает перезапуск.

## USE IT

Платформенная память делает то же самое под капотом (мульти-провайдер):

- **ChatGPT Memory** — «заметки» отдельно от истории чатов: помнит факты о тебе между диалогами, управляется в Settings → Personalization.
- **Claude** — постоянный контекст через Projects (knowledge) и память сессий.
- **Gemini** — персонализация и Gems с заданным контекстом.

Принцип один: хранить вне промпта и подмешивать релевантное; разница — в том, что и как долго хранится и как этим управлять (приватность — Фаза 11).
