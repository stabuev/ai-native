Урок 2.1 · Анатомия промпта

Фаза 2 — Промпт-инжиниринг · Результат фазы: Писать воспроизводимые промпты и измерять их качество через eval-harness.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Это первый урок про промптинг — и он про структуру, а не про «магические слова». Промпт — это интерфейс к модели, и у хорошего интерфейса есть анатомия: роль, контекст, задача, формат, примеры. Разложишь промпт на эти блоки — и сможешь управлять ответом осознанно: менять по одной части, переиспользовать находки и (со следующих уроков) измерять, что стало лучше.

MOTTO. Промпт — это интерфейс к интеллекту; у хорошего интерфейса есть структура.

PROBLEM

«Напиши покороче» работает случайно: сегодня ок, завтра мимо. Без структуры промпт невоспроизводим — нельзя понять, что именно повлияло на ответ, нельзя переиспользовать удачную формулировку и нельзя сравнить две версии. Получается не инженерия, а гадание. Нужна анатомия: разложить промпт на части, каждой из которых можно управлять отдельно.

CONCEPT

Интуиция

Думай о промпте как о техзадании для очень способного, но буквального исполнителя. Он сделает близко к тому, что ты написал — поэтому неоднозначность превращается в случайный результат. Официальные техники Anthropic (ясность и прямота, роль, примеры, XML-структура, рассуждение, цепочки) — это по сути способы убрать неоднозначность из ТЗ. Анатомия промпта собирает их в пять управляемых блоков.

Как это работает

[ Роль ]      кто отвечает (эксперт, редактор, парсер)
[ Контекст ]  что нужно знать для задачи
[ Задача ]    что именно сделать  ← единственный обязательный блок
[ Формат ]    как должен выглядеть ответ
[ Примеры ]   few-shot образцы вход→выход (урок 2.2)

Разберём каждый блок — что он даёт и почему:

  • Роль (role prompting). Задаёт, «кем» модель отвечает: эксперт, редактор, строгий парсер. Меняет тон, глубину и словарь ответа. Технически роль часто кладут в системный промпт (см. USE IT — у провайдеров это разные поля).
  • Контекст. Факты и данные, нужные для задачи: фрагмент документа, прошлые решения, ограничения. Это продолжение урока 1.4: явный источник сужает пространство допустимого ответа и даёт опору для проверки, но не гарантирует, что модель истолкует его без ошибок.
  • Задача. Единственный обязательный блок. Главный приём Anthropic здесь — «be clear and direct»: одна чёткая, конкретная инструкция, без «сделай красиво». Размытая задача = размытый ответ.
  • Формат. Как должен выглядеть ответ: список, таблица, JSON, длина, язык. Чем строже формат, тем предсказуемее вывод и тем легче его потом проверять и парсить (мост к уроку 2.4 — structured output).
  • Примеры (multishot / few-shot). Образцы «вход → выход» показывают желаемое поведение нагляднее любого описания. Это отдельная большая тема урока 2.2.

Чем явнее блоки, тем стабильнее ответ и тем легче его измерять. Anthropic советует начинать промпт-инжиниринг с определения критериев успеха и тестов — поэтому анатомию мы здесь делаем проверяемой (линтер блоков), а системно мерить версии будем в уроке 2.5.

Врезка: цикл Description ↔ Discernment (AI Fluency). Хороший промпт редко выходит с первого раза: ты описываешь задачу (Description), оцениваешь ответ (Discernment) и уточняешь — и так по кругу. Этот урок — про первую половину (анатомия описания); измерять вторую системно будем в уроке 2.5 (eval-harness).

РАЗБОР ПО ШАГАМ

Соберём промпт из блоков руками и посмотрим, как линтер делает «полноту» проверяемой.

Шаг 1 — голый промпт (только задача):

# Задача
Суммаризируй отзыв в 3 пунктах

lint_prompt идёт по ожидаемым заголовкам и возвращает, каких блоков нет:

lint_prompt(голый) → ["role", "context", "output_format", "examples"]

Задача на месте, всё остальное — нет. Такой промпт сработает «как повезёт»: модель сама придумает роль, формат и объём.

Шаг 2 — добавим блоки роль, контекст, формат и пример:

# Роль
Ты редактор поддержки

# Контекст
Отзывы клиентов из чата поддержки

# Задача
Суммаризируй отзыв в 3 пунктах

# Формат ответа
Маркированный список из 3 пунктов

# Примеры
Вход: Долго грузит
Выход: - проблема: производительность
lint_prompt(полный) → []      # пустой список = все блоки на месте

Теперь ответ предсказуем: задана роль (тон), контекст (на что опираться), формат (как оформить) и пример (эталон). А lint_prompt превратил расплывчатое «вроде нормальный промпт» в проверяемый критерий — на этом и строится воспроизводимость.

BUILD IT

Задание: собери конструктор промпта и линтер — стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/2.1-prompt-anatomy/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3, всё офлайн (для теста ещё pytest).

Заголовки блоков бери готовыми (это данные):

HEADERS = {
    "role": "# Роль",
    "context": "# Контекст",
    "task": "# Задача",
    "output_format": "# Формат ответа",
    "examples": "# Примеры",
}

Реализуй две функции:

  • build_prompt(task, role=None, context=None, output_format=None, examples=None) — собери промпт из блоков в порядке роль → контекст → задача → формат → примеры. Обязателен только task; каждый заданный блок выводится под своим заголовком из HEADERS. examples — список пар (вход, выход), каждую оформи как Вход: …\nВыход: …. Блоки разделяй пустой строкой.
  • lint_prompt(text) — верни список имён блоков (ключей HEADERS), чьих заголовков нет в тексте. Пустой список = промпт структурно полон.

Готово, когда тест test_prompt_template.py зелёный — он проверяет, что задача присутствует всегда, опциональные блоки рендерятся при передаче, а линтер ловит недостающие блоки и пропускает полный промпт.

pytest code -q                 # красное → реализуй функции → зелёное
python code/prompt_template.py # демо: голый промпт vs полный + что не хватает

Подсказка. В build_prompt собирай список кусков f"{HEADERS[...]}\n{значение}" (задача — всегда, остальное — по условию) и склей через "\n\n". lint_prompt — это list comprehension: для каждого (name, header) проверь header not in text.

Внизу, в «Исходниках урока», — три способа пройти упражнение (собрать самому · подсмотреть эталон · делегировать ИИ) и тесты-ТЗ.

USE IT

Один и тот же структурированный промпт — на трёх платформах; сравни, как меняется ответ:

role = "Ты редактор поддержки"
prompt = build_prompt(task="Суммаризируй отзыв в 3 пунктах",
                      output_format="Маркированный список")   # роль НЕ в тексте — уйдёт в system
# Anthropic — роль в поле system
Anthropic().messages.create(model="claude-sonnet-4-6", max_tokens=300,
    system=role, messages=[{"role": "user", "content": prompt}])
# OpenAI — роль в поле instructions
OpenAI().responses.create(model="gpt-5.x", instructions=role, input=prompt)
# Google — отдельного system-поля нет, роль кладём в текст промпта
genai.Client().models.generate_content(model="gemini-2.x-flash",
    contents=build_prompt(task="Суммаризируй отзыв в 3 пунктах", role=role,
                          output_format="Маркированный список"))

Заметь: «роль» у провайдеров живёт в разных местах (system / instructions / в тексте) — анатомия одна, синтаксис разный. Меняй по одному блоку и смотри, что именно сдвинуло ответ.

ЧАСТЫЕ ОШИБКИ

  • Размытая задача. «Сделай покороче/получше» — это не ТЗ. Будь конкретен (be clear and direct): сколько пунктов, в каком тоне, на каком языке.
  • Менять всё сразу. Поправил роль, формат и задачу одним махом — и не понять, что помогло. Меняй по одному блоку, как в эксперименте.
  • Не задавать формат, а потом удивляться разнобою. Если ответ нужно парсить или сравнивать — фиксируй формат (список/JSON/длина) заранее.
  • Путать роль и задачу. «Ты редактор» (роль) не говорит, что сделать; нужна ещё явная задача. Роль настраивает, задача поручает.

from prompt_template import build_prompt, lint_prompt, HEADERS


def test_task_is_always_present():
    p = build_prompt("сделай X")
    assert HEADERS["task"] in p and "сделай X" in p


def test_optional_blocks_render_when_given():
    p = build_prompt("сделай X", role="ты аналитик", output_format="JSON",
                     examples=[("вход", "выход")])
    for key in ("role", "task", "output_format", "examples"):
        assert HEADERS[key] in p
    assert "вход" in p and "выход" in p


def test_lint_flags_missing_blocks():
    bare = build_prompt("только задача")
    missing = lint_prompt(bare)
    assert "role" in missing and "context" in missing and "examples" in missing
    assert "task" not in missing


def test_lint_passes_full_prompt():
    full = build_prompt("задача", role="r", context="c", output_format="f",
                        examples=[("i", "o")])
    assert lint_prompt(full) == []

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/prompt_template.pyПоказать решение
"""Анатомия промпта: структурный шаблон + линтер — Build It для урока 2.1.

Без зависимостей. Промпт — это интерфейс к модели, и у него есть анатомия:
роль, контекст, задача, формат ответа, примеры. Здесь собираем промпт из этих
блоков и проверяем линтером, что ключевые части на месте (воспроизводимость).
"""

# Заголовки секций промпта (порядок = рекомендуемая структура).
HEADERS = {
    "role": "# Роль",
    "context": "# Контекст",
    "task": "# Задача",
    "output_format": "# Формат ответа",
    "examples": "# Примеры",
}


def build_prompt(task, role=None, context=None, output_format=None, examples=None):
    """Собрать промпт из блоков. Обязателен только `task`, остальное опционально.

    `examples` — список пар (вход, выход).
    """
    parts = []
    if role:
        parts.append(f"{HEADERS['role']}\n{role}")
    if context:
        parts.append(f"{HEADERS['context']}\n{context}")
    parts.append(f"{HEADERS['task']}\n{task}")
    if output_format:
        parts.append(f"{HEADERS['output_format']}\n{output_format}")
    if examples:
        ex = "\n\n".join(f"Вход: {i}\nВыход: {o}" for i, o in examples)
        parts.append(f"{HEADERS['examples']}\n{ex}")
    return "\n\n".join(parts)


def lint_prompt(text):
    """Каких анатомических блоков не хватает в промпте (эвристика по заголовкам)."""
    return [name for name, header in HEADERS.items() if header not in text]


if __name__ == "__main__":
    bare = build_prompt("Суммаризируй текст в 3 пунктах")
    print("--- голый промпт ---", bare, sep="\n")
    print("\nНе хватает:", lint_prompt(bare))

    full = build_prompt(
        task="Суммаризируй отзыв в 3 пунктах",
        role="Ты редактор продуктовой команды",
        context="Отзывы клиентов из поддержки",
        output_format="Маркированный список из 3 пунктов",
        examples=[("Долго грузит", "- проблема: производительность")],
    )
    print("\n--- полный промпт ---", full, sep="\n")
    print("\nНе хватает:", lint_prompt(full) or "ничего")

# Артефакт: базовый шаблон промпта

Каркас «роль · контекст · задача · формат · примеры» с подстановками. Сборка и проверка — в [`code/prompt_template.py`](../code/prompt_template.py).

## Шаблон

```
# Роль
{Кто отвечает: эксперт/редактор/парсер и с каким опытом}

# Контекст
{Что нужно знать для задачи: источник данных, ограничения, аудитория}

# Задача
{Что именно сделать — одно понятное действие}     ← обязательный блок

# Формат ответа
{Как выглядит ответ: список из N пунктов / JSON по схеме / таблица}

# Примеры
Вход: {пример входа}
Выход: {эталонный выход}
```

## Чек-лист перед отправкой

- [ ] **Задача** сформулирована как одно действие (не «сделай всё»).
- [ ] **Роль** задаёт нужную экспертизу и тон.
- [ ] **Формат** однозначен (если ответ парсит код — это JSON-схема, см. урок 2.4).
- [ ] **Примеры** добавлены, если важен формат/класс (урок 2.2).
- [ ] Прогнал `lint_prompt()` — нет случайно забытых блоков.

## Как переиспользовать

```python
from prompt_template import build_prompt, lint_prompt
p = build_prompt(task="...", role="...", output_format="...", examples=[("вход","выход")])
assert lint_prompt(p) == []     # все нужные блоки на месте
```

Где «роль» у провайдеров: Anthropic — `system`, OpenAI — `instructions`, Google — в начале `contents`. Измерять версии шаблона — урок 2.5.
