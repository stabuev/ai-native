Урок 1.1 · Токены и токенизация

Фаза 1 — Как работают LLM · Результат фазы: объяснить токены, контекст и inference; собрать токен-бюджетер; отличать надёжный ответ от галлюцинации.

Результат урока: объяснить, как текст превращается в пронумерованные куски на входе модели, самостоятельно собрать этот механизм в уменьшенном виде с точным обратным преобразованием и оценить бюджет запроса, не выдавая приблизительный подсчёт за точный.

Опора: личная папка course-work/1.1/, красная стартовая проверка и выбранный путь выполнения из 0.4. · Дальше: в 1.2 число токенов станет ограничением контекстного окна и единицей одного шага генерации.

Requires: Python 3 и pytest. Build It и расчёт стоимости работают локально. Для ветки OpenAI понадобится установка tiktoken; установка пакета требует сети. Для точного подсчёта через Anthropic или Gemini нужны пакет для работы с API (SDK), API-ключ и сеть.

Пять коротких определений перед стартом

  • Токен — кусок текста или его байтового представления, который токенизатор считает одной единицей.
  • Token ID — целое число, которым этот кусок обозначен для модели.
  • Токенизация — преобразование текста в последовательность токенов и их ID.
  • Токенизатор — правила и словарь, выполняющие это преобразование и обратную сборку текста.
  • BPE — один из алгоритмов обучения таких правил: он многократно склеивает частые соседние пары.

Пока этих определений достаточно. Ниже каждый объект станет наблюдаемым на конкретном примере.

MOTTO. Сначала узнай, какой токенизатор считает запрос. Только потом считай окно и деньги.

Зачем это понимать

Допустим, в документе 10 000 символов. Этого числа недостаточно, чтобы ответить на два рабочих вопроса:

  1. поместятся ли документ, инструкция и будущий ответ в контекстное окно модели;
  2. сколько примерно будет стоить запрос через API.

Причина не в сложной формуле. Причина в единице измерения: модель получает не слова и не символы, а последовательность token IDs. Один и тот же текст разные токенизаторы могут превратить в разное число ID. Значит, правило вроде «один токен — это четыре символа» годится только для очень грубой прикидки, но не для проверки лимита и не для сметы массового запуска.

В этом уроке мы сначала разберём, что находится между строкой текста и числами на входе модели. Затем соберём маленький BPE-токенизатор, чтобы увидеть обучение правил своими глазами. После этого посчитаем токены рабочим способом и отделим известную часть стоимости от оценки будущего ответа.

Текст и символ

Текст — строка, которую видит человек. Символ — отдельный знак вроде a, я или 👋. В Python строка "low" содержит три символа.

Unicode — общий стандарт, в котором собраны символы разных языков, знаки и эмодзи. Но компьютер хранит строку как байты. Байт — маленькое целое число от 0 до 255. Способ преобразовать Unicode-символы в байты называется кодировкой. Сегодня почти везде используется UTF-8:

  • английская a занимает один байт;
  • кириллическая я — два байта;
  • многие эмодзи — четыре байта или больше с учётом составных последовательностей.

Проверить это можно без библиотек:

for text in ["a", "я", "👋"]:
    print(text, list(text.encode("utf-8")))

Эта разница ещё не определяет итоговое число токенов, но объясняет, почему byte-level токенизатору изначально приходится работать с разным количеством элементов.

Токен

Токен — единица, которую токенизатор выделил в тексте. Токеном может оказаться целое слово, часть слова, пробел вместе со словом, знак пунктуации или несколько байтов одного символа. Универсального разбиения нет: оно зависит от конкретного токенизатора и его выученных правил.

В объяснениях токен часто показывают как читаемый кусок — например low или ing. Это удобно человеку, но внутри API модель получает не надпись low, а номер.

Token ID и словарь

Token ID — целое число, назначенное токену. Словарь токенизатора — соответствие между допустимыми кусками и их ID:

кусок b"l"   → ID 75
кусок b"low" → ID 2417

Числа здесь условные. У другого токенизатора тот же кусок может иметь другой ID или вообще не существовать как один токен. Поэтому token IDs нельзя без преобразования переносить между моделями с разными токенизаторами.

Токенизатор

Токенизатор — алгоритм и данные, которые выполняют два преобразования:

encode: текст → token IDs
decode: token IDs → текст

Хороший токенизатор на байтах делает это без потери информации: после decode(encode(text)) получается тот же текст, включая пробелы и любые поддерживаемые UTF-8 символы. Такая проверка «туда и обратно» называется round-trip.

Откуда токенизатор берёт удобные куски

Если каждому отдельному байту уже можно назначить ID, зачем вообще склеивать их в более длинные токены?

Представь стенографию. Частое выражение стенографист обозначает одним знаком, а редкое имя выписывает подробнее. В результате обычный текст занимает меньше знаков. Токенизатор решает похожую задачу: часто встречающиеся последовательности байтов превращаются в крупные куски, а незнакомый текст всё равно можно представить базовыми байтами.

Чтобы узнать, какие куски частые, нужен корпус — набор текстов, на которых учат правила токенизатора. Корпус — это не запрос пользователя и не память модели, а обучающая выборка для отдельного этапа подготовки токенизатора.

Распределение корпуса влияет на результат. Если определённый язык, формат кода или терминология часто встречались в корпусе, токенизатор может выучить для них удобные длинные куски. Если встречались редко, тот же текст может распадаться мельче. Это тенденция, а не правило «любой русский текст всегда дороже любого английского»: сравнение нужно измерять конкретным токенизатором на конкретных строках.

BPE по шагам

BPE — byte-pair encoding, кодирование с помощью пар байтов. В учебной версии алгоритм начинается с отдельных UTF-8-байтов и несколько раз выполняет одно действие:

найти самую частую соседнюю пару текущих токенов и заменить каждое её непересекающееся вхождение одним новым токеном.

Есть два разных процесса:

  1. обучение токенизатора — один раз на корпусе находим и сохраняем merge-правила;
  2. кодирование нового текста — применяем уже сохранённые правила в том же порядке, ничего заново не обучая.

Merge-правило — указание слить два соседних токена. Например, (b"a", b"b") создаёт кусок b"ab". Порядок правил важен: правило (b"ab", b"c") сработает только после того, как появится b"ab".

Шаг 0. Начальное представление

Возьмём корпус из двух строк:

"abab"
"ab"

Латинские a и b входят в базовый набор ASCII и кодируются одним UTF-8-байтом, поэтому здесь один символ равен одному байту:

[a, b, a, b]
[a, b]

Пока каждый байт — отдельный токен. Первая строка занимает четыре токена, вторая — два.

Шаг 1. Посчитать соседние пары

Соседняя пара — два токена, стоящие рядом. Считаем пары во всём корпусе:

ПараГде встретиласьЧастота
(a, b)дважды в abab, один раз в ab3
(b, a)один раз в abab1

Самая частая пара — (a, b).

Шаг 2. Слить все непересекающиеся вхождения

Добавляем правило (a, b) → ab и применяем его слева направо:

[a, b, a, b] → [ab, ab]
[a, b]       → [ab]

Первая строка теперь занимает два токена вместо четырёх. Исходные байты не исчезли: токен ab хранит их последовательность.

Шаг 3. Повторить на новом состоянии

После слияния считаются уже новые пары. В первой строке появилась пара (ab, ab). Если она окажется самой частой и мы добавим второе правило, получится токен abab.

Каждое принятое правило добавляет один возможный крупный кусок. Но merge-правила — не весь словарь: в byte-level BPE словарь также содержит базовые байты, назначенные им ID и обычно специальные токены вроде маркеров начала или конца сообщения.

Шаг 4. Закодировать новый текст

Обучение закончено, правила заморожены. Для нового "abc" сначала получаем:

[a, b, c]

Правило (a, b) → ab даёт:

[ab, c]

Мы не ищем самую частую пару внутри abc: на этапе encode только воспроизводятся правила, выученные на корпусе.

Шаг 5. Декодировать без потерь

Каждый токен хранит исходные байты. Чтобы выполнить decode, достаточно соединить байтовые куски и прочитать результат как UTF-8:

[b"ab", b"c"] → b"abc" → "abc"

Поэтому byte-level подход способен представить даже текст, которого не было в корпусе: в крайнем случае останутся отдельные байты.

Сначала предскажи

До запуска кода ответь себе:

  • сколько токенов будет у abab без правил;
  • сколько останется после правила (a, b) → ab;
  • изменится ли исходный текст после decode.

Ожидаемый ход рассуждения: 4 → 2, а decode вернёт ровно abab. Если ответ получился только после просмотра программы, вернись к шагам 1–2 и проиграй слияние руками.

Где наша модель упрощает production

Наш mini-BPE передаёт центральный механизм, но не копирует целый токенизатор провайдера:

  • мы начинаем с UTF-8-байтов и разрешаем слияния в любом месте строки;
  • реальный токенизатор провайдера может сначала предварительно разделить текст (pre-tokenization), отдельно обрабатывать пробелы и запрещать некоторые слияния;
  • у него есть большой готовый словарь, token IDs и специальные токены протокола;
  • не все токенизаторы используют именно BPE: встречаются WordPiece, Unigram и другие варианты;
  • токенизатор подготавливают отдельно; в этом уроке мы не обучаем языковую модель.

Это граница модели, а не её недостаток. Мы строим ровно столько, сколько нужно, чтобы увидеть: частые соседние куски сжимаются, правила применяются последовательно, а конкретное число токенов зависит от конкретного токенизатора.

Работай в подготовленной папке course-work/1.1/. После 0.4 там должен лежать только test_tokenizer.py; готовый tokenizer.py со страницы урока заранее не копируй.

Контракт функций

Реализуй три публичные функции:

  1. train_bpe(corpus, num_merges)
  • corpus — список строк;
  • num_merges — максимальное число правил;
  • каждую строку преврати в список однобайтовых объектов bytes;
  • на каждой итерации посчитай соседние пары, выбери самую частую, сохрани её и слей во всех строках;
  • если пар больше нет, остановись раньше;
  • отрицательное num_merges отклони через ValueError;
  • верни правила в порядке обучения.
  1. encode(text, merges)
  • преврати строку в отдельные UTF-8-байты;
  • примени каждое правило из merges по порядку;
  • верни список объектов bytes.
  1. decode(tokens)
  • соедини byte-токены;
  • декодируй итоговые байты как UTF-8;
  • верни исходную строку без нормализации пробелов.

Минимальные операции Python

raw = "я".encode("utf-8")       # одна последовательность bytes
tokens = [bytes([value]) for value in raw]  # отдельный bytes на каждый байт
text = b"".join(tokens).decode("utf-8")

Для слияния иди слева направо с индексом i:

если tokens[i], tokens[i + 1] совпали с парой:
    добавить их сумму в результат
    сдвинуть i на 2
иначе:
    перенести текущий токен
    сдвинуть i на 1

Так вхождения не пересекаются. Например, пара (a, a) в aaa сольётся в [aa, a], а не в два пересекающихся aa.

Если несколько пар имеют одинаковую частоту, выбери детерминированное правило — например, лексикографически меньшую пару. Детерминизм означает, что одинаковый корпус всегда создаёт одинаковый список merge-правил.

Что именно проверяют тесты

Тесты должны доказать не просто факт запуска:

  • первой учится действительно самая частая пара;
  • merge применяется ко всем непересекающимся вхождениям;
  • правила применяются в сохранённом порядке;
  • Unicode, переводы строк и повторные пробелы переживают точный round-trip;
  • выученные правила строго сокращают знакомый фрагмент;
  • невозможное отрицательное число правил отвергается.

Запусти цикл red → green:

python -m pytest course-work/1.1/test_tokenizer.py -q
python course-work/1.1/tokenizer.py

Демо должно показать правила, токены до и после BPE и успешный decode хотя бы для двух новых строк. Если делегировал реализацию ИИ, перед приёмкой объясни своими словами, почему тест на порядок правил не пройдёт, если encode применять их в обратном порядке.

Внизу, в «Исходниках урока», доступны тест-ТЗ, опубликованный эталон и три пути выполнения. Эталон нужен для разбора после собственной попытки, а не вместо личной зелёной проверки.

USE IT

Практика: посчитай токены реальным способом.

Mini-BPE объясняет механизм, но его число токенов нельзя подставлять в лимиты и цены коммерческой модели. Выбери одну доступную ветку. Не нужно устанавливать SDK всех провайдеров.

Сначала подготовь четыре коротких примера: английский текст, русский текст, фрагмент кода и строку с эмодзи. До подсчёта запиши прогноз: какой пример даст больше токенов и почему. Затем сравни прогноз с измерением.

Ветка OpenAI: локальный tiktoken

tiktoken — BPE-токенизатор для моделей OpenAI. Установка пакета обычно требует сети; после установки сам текст можно кодировать локально без отправки в API.

pip install tiktoken
import os
import tiktoken

model = os.environ.get("OPENAI_MODEL", "gpt-4o")
encoding = tiktoken.encoding_for_model(model)
text = "Проверь этот отчёт и перечисли три риска."
token_ids = encoding.encode(text)

print("model:", model)
print("token IDs:", token_ids)
print("input tokens:", len(token_ids))
print("round-trip:", encoding.decode(token_ids) == text)

encoding_for_model важнее случайно выбранного get_encoding: он связывает подсчёт с указанной моделью. При этом пример считает текстовый фрагмент. Полный API-запрос может включать system-инструкцию, историю, описания инструментов и служебное оформление; фактический usage после вызова остаётся контрольной точкой.

Ветка Anthropic: Token Counting API

Anthropic предоставляет messages.count_tokens. Передавай тот же точный model ID и ту же структуру сообщения, которую собираешься отправить:

import os
from anthropic import Anthropic

model = os.environ["ANTHROPIC_MODEL"]
text = "Проверь этот отчёт и перечисли три риска."
result = Anthropic().messages.count_tokens(
    model=model,
    messages=[{"role": "user", "content": text}],
)
print("input tokens:", result.input_tokens)

Это запрос к сервису: нужны ключ и сеть, а содержимое уходит провайдеру. Не используй секретные или запрещённые корпоративной политикой данные. Сам провайдер отмечает, что предварительный count может немного отличаться от фактического usage.

Ветка Gemini: count_tokens

Gemini предоставляет метод models.count_tokens:

import os
from google import genai

model = os.environ["GEMINI_MODEL"]
text = "Проверь этот отчёт и перечисли три риска."
result = genai.Client().models.count_tokens(model=model, contents=text)
print("input tokens:", result.total_tokens)

Здесь тоже нужны ключ и сеть, а считать следует именно для выбранного model ID.

Если точного счётчика нет

Можно измерить текст доступным токенизатором как учебный эксперимент, но в отчёте явно напиши approximate и название использованной кодировки. Нельзя посчитать Claude через o200k_base, назвать число точным и на его основе обещать лимит или стоимость.

Переносимый результат: собери prompt-token-budgeter.

Теперь преврати подсчёт в переносимый рабочий результат. У бюджетера два слоя:

  1. провайдер-специфичный счётчик получает число входных токенов;
  2. локальная функция применяет актуальные цены и оценивает стоимость.

Мы намеренно не зашиваем таблицу моделей и цен в код: названия моделей и тарифы меняются. Перед расчётом возьми цены с официальной страницы выбранного провайдера и запиши дату проверки.

В блоке исходников урока скопируй в course-work/1.1/ только test_token_budgeter.py. Затем самостоятельно создай token_budgeter.py и реализуй:

estimate_request_cost(
    input_tokens,
    planned_output_tokens,
    input_usd_per_million,
    output_usd_per_million,
)

Формула для каждой части одинакова:

стоимость = число_токенов / 1_000_000 × цена_за_миллион

Например, если иллюстративные, а не текущие тарифы равны $3 за миллион входных и $15 за миллион выходных токенов:

2 000 входных  → 2 000 / 1 000 000 × 3  = $0.006
500 выходных   →   500 / 1 000 000 × 15 = $0.0075
оценка итого                                  $0.0135

Число входных токенов уже посчитано. Число выходных до генерации неизвестно, поэтому planned_output_tokens — план или консервативная верхняя оценка, а не факт. Окончательную стоимость сверяют с фактическим usage ответа и действующим биллингом.

Функция должна вернуть как минимум:

{
    "input_tokens": 2000,
    "planned_output_tokens": 500,
    "input_cost_usd": 0.006,
    "planned_output_cost_usd": 0.0075,
    "estimated_total_usd": 0.0135,
}

Отрицательные количества, отрицательные цены и бесконечные значения должны завершаться понятным ValueError, а не создавать бессмысленный бюджет.

Реалистичные ошибки и как их заметить

  • «Токен — это слово». Проверка: посмотри token IDs для слова с окончанием, кода и эмодзи. Границы не обязаны совпасть со словами.
  • «BPE каждый раз учится на моём prompt». Нет: merge-правила уже обучены и заморожены; encode только применяет их.
  • «Merge-правила можно применять в любом порядке». Нельзя: позднее правило может ссылаться на кусок, созданный ранним.
  • «Наш mini-BPE даёт лимит модели». Нет: у него другой корпус, другой набор правил и нет протокольных токенов.
  • «Одна кодировка точно считает все модели». Нет: счётчик должен соответствовать model ID, а приближение должно быть подписано как приближение.
  • «pip install работает без сети». Обычно установка скачивает пакет. Локальным является дальнейший запуск библиотеки, а не обязательно её установка.
  • «Стоимость известна заранее до цента». Вход можно посчитать, но длина ответа планируется; фактические usage и тариф проверяются после вызова.
  • «Больше токенов означает хуже текст». Нет: это свойство представления и словаря, а не оценка смысла или качества.

"""Локальная арифметика бюджета запроса без привязки к провайдеру.

Точное или оценочное число входных токенов нужно получить токенизатором
выбранной модели. Актуальные цены студент также передаёт явно, поэтому код не
превращается в устаревающий каталог моделей.
"""
from math import isfinite

TOKENS_PER_MILLION = 1_000_000


def _validate_token_count(name: str, value: int) -> None:
    if isinstance(value, bool) or not isinstance(value, int) or value < 0:
        raise ValueError(f"{name} must be a non-negative integer")


def _validate_price(name: str, value: float) -> None:
    if isinstance(value, bool) or not isinstance(value, (int, float)):
        raise ValueError(f"{name} must be a non-negative finite number")
    if value < 0 or not isfinite(value):
        raise ValueError(f"{name} must be a non-negative finite number")


def estimate_request_cost(
    input_tokens: int,
    planned_output_tokens: int,
    input_usd_per_million: float,
    output_usd_per_million: float,
) -> dict[str, int | float]:
    """Оценить стоимость по числу токенов и ценам за миллион токенов."""
    _validate_token_count("input_tokens", input_tokens)
    _validate_token_count("planned_output_tokens", planned_output_tokens)
    _validate_price("input_usd_per_million", input_usd_per_million)
    _validate_price("output_usd_per_million", output_usd_per_million)

    input_cost = input_tokens / TOKENS_PER_MILLION * input_usd_per_million
    output_cost = (
        planned_output_tokens / TOKENS_PER_MILLION * output_usd_per_million
    )
    return {
        "input_tokens": input_tokens,
        "planned_output_tokens": planned_output_tokens,
        "input_cost_usd": round(input_cost, 8),
        "planned_output_cost_usd": round(output_cost, 8),
        "estimated_total_usd": round(input_cost + output_cost, 8),
    }


if __name__ == "__main__":
    print(
        estimate_request_cost(
            input_tokens=2_000,
            planned_output_tokens=500,
            input_usd_per_million=3.0,
            output_usd_per_million=15.0,
        )
    )
code/tokenizer.pyПоказать решение
"""Учебный byte-level BPE для урока 1.1.

Реализация намеренно невелика: UTF-8 текст превращается в отдельные байты,
частые соседние пары байтов сливаются, а выученные правила затем применяются
к новому тексту. Здесь нет pre-tokenization и специальных токенов production-
токенизатора, зато основной механизм BPE и точный round-trip наблюдаемы.
"""
from collections import Counter

Token = bytes
Merge = tuple[Token, Token]


def _byte_tokens(text: str) -> list[Token]:
    """UTF-8 текст -> список однобайтовых токенов."""
    return [bytes([value]) for value in text.encode("utf-8")]


def _merge(tokens: list[Token], pair: Merge) -> list[Token]:
    """Слить все непересекающиеся вхождения пары слева направо."""
    a, b = pair
    out, i = [], 0
    while i < len(tokens):
        if i < len(tokens) - 1 and tokens[i] == a and tokens[i + 1] == b:
            out.append(a + b)
            i += 2
        else:
            out.append(tokens[i])
            i += 1
    return out


def train_bpe(corpus: list[str], num_merges: int) -> list[Merge]:
    """Выучить до ``num_merges`` правил по частоте соседних byte-токенов."""
    if num_merges < 0:
        raise ValueError("num_merges must be non-negative")

    sequences = [_byte_tokens(text) for text in corpus]
    merges: list[Merge] = []
    for _ in range(num_merges):
        pair_counts: Counter[Merge] = Counter()
        for tokens in sequences:
            pair_counts.update(zip(tokens, tokens[1:]))

        if not pair_counts:
            break

        # При равной частоте выбираем лексикографически меньшую пару: один и тот
        # же корпус всегда даёт один и тот же порядок правил.
        best = min(pair_counts, key=lambda pair: (-pair_counts[pair], pair))
        merges.append(best)
        sequences = [_merge(tokens, best) for tokens in sequences]
    return merges


def encode(text: str, merges: list[Merge]) -> list[Token]:
    """Текст -> byte-токены с применением правил в порядке обучения."""
    tokens = _byte_tokens(text)
    for pair in merges:
        tokens = _merge(tokens, pair)
    return tokens


def decode(tokens: list[Token]) -> str:
    """Byte-токены -> исходный UTF-8 текст без потери пробелов и Unicode."""
    return b"".join(tokens).decode("utf-8")


if __name__ == "__main__":
    corpus = [
        "low lower lowest",
        "newer wider newest",
        "low low low newer newer wider",
    ]
    merges = train_bpe(corpus, num_merges=10)
    print("Выучено merge-правил:", len(merges))
    print("Первые правила:", merges[:5])
    for text in ["low newer", "lowest"]:
        toks = encode(text, merges)
        print(f"\n{text!r}")
        print("  токены:", toks, f"({len(toks)} шт.)")
        print("  без BPE:", len(encode(text, [])), "шт.")
        print("  decode :", repr(decode(toks)))
