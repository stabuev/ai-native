Урок 11.4 · Корпоративные политики ИИ

Фаза 11 — Этика, безопасность, governance · Результат фазы: Оценить риски, настроить приватность и написать политику использования ИИ.

Приватность (11.1), фактчекинг (11.2) и аудит вывода (11.3) — это три отдельных инструмента. Без общего документа каждый сотрудник применяет их (или не применяет) по-своему: кто-то шлёт PII во внешнюю модель, кто-то доверяет ИИ финальное решение о найме. Нужен один документ, который сводит это в правила — и код, который эти правила проверяет.

MOTTO. Политика ИИ — не бюрократия для галочки, а исполняемые правила: что можно, что нельзя, кто отвечает.

PROBLEM

Без политики каждый сотрудник решает сам: какие данные можно отдать модели, какие инструменты одобрены, где обязательно участие человека, кто отвечает, если ИИ ошибся. Это не гипотетический риск — это то, что уже видели в 11.1–11.3: PII утекает во внешний сервис, галлюцинация уходит клиенту, дискриминирующий вывод публикуется. Каждый раз кто-то должен был сверить действие с правилом — но правила нигде не записаны и не проверяемы.

Устная договорённость не масштабируется и не проходит аудит. Нужна политика использования ИИ: документ с обязательными разделами (какие данные допустимы, какие инструменты одобрены, где нужен человек, кто отвечает) — и функция, которая проверяет конкретный кейс против неё, а не полагается на память сотрудника.

CONCEPT

Интуиция

Политика ИИ — это как правила дорожного движения для конкретного перекрёстка, а не общий свод «водите аккуратно». Абстрактное «будь осторожен с ИИ» никого не останавливает в моменте. А конкретное «PII запрещено отправлять во внешнюю модель» — это правило, которое можно проверить кодом до того, как запрос уйдёт, а не после инцидента.

Два уровня работы с политикой: валидация документа (все ли обязательные разделы на месте — иначе это не политика, а декларация) и валидация кейса (конкретное действие сотрудника — разрешено правилами или нет). Оба уровня нужны: полная политика без проверки кейсов — мёртвый текст; проверка кейсов без полной политики — набор частных патчей без системы.

Как это работает

Документ политики → validate_policy(text)
   ищет 5 обязательных разделов
   → [] (всё на месте) или ["Раздел X", ...] (не хватает)

Конкретный кейс → check_usecase(sends_pii, external, automated_decision)
   PII + внешняя модель        → BLOCK (нельзя отдавать наружу)
   автоматическое решение      → BLOCK (нужен человек)
   иначе                       → ALLOW
  • validate_policy(text) — проверить документ на обязательные разделы: допустимые данные, запрещённые данные, одобренные инструменты, проверка человеком, ответственность. Вернуть список отсутствующих (пусто = политика полная).
  • check_usecase(sends_pii, external, automated_decision) — прогнать конкретный сценарий через базовые правила и вернуть решение с причиной.

Порядок правил в check_usecase важен: PII-во-внешнюю проверяется первым, потому что это самый острый риск (юридический + репутационный одновременно), а не потому что остальные менее значимы.

РАЗБОР ПО ШАГАМ

Возьмём черновик политики без двух разделов и три кейса:

text = "Допустимые данные\nОтветственность"
validate_policy(text) → ["Запрещённые данные", "Одобренные инструменты", "Проверка человеком"]
   (документ неполный — 3 раздела не хватает, публиковать рано)

check_usecase(sends_pii=True,  external=True,  automated_decision=False)
   → {allowed: False, reason: "PII нельзя отдавать во внешнюю модель"}

check_usecase(sends_pii=False, external=False, automated_decision=True)
   → {allowed: False, reason: "решения о людях требуют проверки человеком"}

check_usecase(sends_pii=False, external=True,  automated_decision=False)
   → {allowed: True, reason: "разрешено политикой"}

Один и тот же кейс — «отправить данные во внешнюю модель» — разрешён без PII и заблокирован с PII. Разница в одном булевом флаге решает исход: это и есть суть исполняемой политики — не «в целом будьте аккуратны», а конкретная функция от конкретных условий.

Но у обеих функций есть честная граница. validate_policy проверяет наличие названия раздела, а не качество содержимого — политика с разделом «Ответственность», где написано одно слово «TBD», пройдёт проверку. И check_usecase знает только два жёстких правила: PII-во-внешнюю и автоматическое решение о людях. Всё, что не описано явным флагом (например, «использовать неодобренный инструмент» или «отправить финансовый прогноз без фактчекинга») эта функция не увидит — правила нужно расширять по мере появления новых кейсов, а полноту содержания разделов всё равно проверяет человек (security/legal), не код.

BUILD IT

Задание: собери валидатор политики ИИ и проверку кейса. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/11.4-ai-usage-policy/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл policy.py: кортеж REQUIRED_SECTIONS (5 обязательных разделов политики) и:

  • validate_policy(text) — вернуть список названий разделов из REQUIRED_SECTIONS, которых нет в text (пустой список = политика полная, все разделы присутствуют).
  • check_usecase(sends_pii=False, external=False, automated_decision=False) — вернуть {"allowed": bool, "reason": str}: если sends_pii and external → False с причиной про PII; иначе если automated_decision → False с причиной про проверку человеком; иначе → True с причиной «разрешено политикой».

Готово, когда все тесты в test_policy.py зелёные — они проверяют: полная политика проходит без замечаний; неполная — называет конкретные отсутствующие разделы; PII+внешняя модель блокируется; автоматическое решение о человеке блокируется; безопасный кейс (PII без внешней модели, или внешняя модель без PII) разрешён.

pytest code -q          # красное → реализуй validate_policy/check_usecase → зелёное
python code/policy.py   # демо: неполная политика + 3 кейса (блок/блок/разрешено)

Подсказка. validate_policy — однострочный список: [s for s in REQUIRED_SECTIONS if s not in text]. В check_usecase порядок проверок важен — сначала PII+external, потом automated_decision, иначе можно перепутать причину блокировки.

USE IT

Согласование политики с реальными фреймворками (мульти-провайдер, применимо к любой модели):

  • NIST AI RMF, функция Govern — роли, ответственность и процедуры на весь жизненный цикл ИИ-системы: кто утверждает использование новой модели, кто расследует инцидент, как часто ревизуется политика.
  • ISO/IEC 42001 — единственный сертифицируемый стандарт системы управления ИИ (AIMS), опубликован в декабре 2023. Построен на цикле Plan-Do-Check-Act, как ISO 9001/27001, но под специфику ИИ-рисков. Сертификация по нему подтверждает наличие системы управления, но не равна автоматическому соответствию EU AI Act — это разные по природе документы (система управления vs юридические обязанности).
  • EU AI Act — не одна дата, а поэтапное вступление в силу: через 6 месяцев после вступления в силу — запрещённые практики; через 12 месяцев — обязанности для general-purpose AI; через 24 месяца — требования к высокорисковым системам из Annex III; через 36 месяцев — требования Annex I. Политика должна учитывать, какие обязанности уже действуют, а какие ещё впереди.
  • Практика согласования: политику пишут с участием security и legal (не только инженеров), проверяют на реальных кейсах из 11.1–11.3 (а не абстрактно), ревизуют регулярно и при инцидентах — не «написали и забыли».

ЧАСТЫЕ ОШИБКИ

  • Политика без проверки кейсов. Документ с правильными разделами, который никто не сверяет с реальными действиями — это декларация, а не политика. Нужна функция вроде check_usecase, встроенная в процесс.
  • Путать наличие раздела с качеством содержимого. validate_policy ловит только отсутствие названия раздела; раздел из одного слова «TBD» пройдёт проверку — содержание всё равно проверяют security/legal.
  • Считать ISO 42001 автоматическим соответствием EU AI Act. Это система управления (PDCA), а не юридический чек-лист по регуляции — сертификат не заменяет оценку по Annex III/I.
  • Игнорировать поэтапность EU AI Act. «Закон вступит в силу — тогда и подготовимся» не работает: запрещённые практики действуют уже через 6 месяцев, а не одномоментно со всем остальным.
  • Писать политику одними инженерами. Без security и legal документ не покрывает реальные юридические и репутационные риски — только техническую часть.

from policy import validate_policy, check_usecase, REQUIRED_SECTIONS


def test_full_policy_passes():
    text = "\n".join(REQUIRED_SECTIONS)
    assert validate_policy(text) == []


def test_missing_sections_flagged():
    text = "Допустимые данные\nОтветственность"
    missing = validate_policy(text)
    assert "Запрещённые данные" in missing
    assert "Проверка человеком" in missing


def test_pii_to_external_blocked():
    res = check_usecase(sends_pii=True, external=True)
    assert res["allowed"] is False
    assert "PII" in res["reason"]


def test_automated_decision_blocked():
    assert check_usecase(automated_decision=True)["allowed"] is False


def test_safe_usecase_allowed():
    assert check_usecase(sends_pii=False, external=True)["allowed"] is True
    assert check_usecase(sends_pii=True, external=False)["allowed"] is True

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/policy.pyПоказать решение
"""Корпоративные политики ИИ: шаблон + проверки — Build It для урока 11.4.

Без зависимостей. Политика использования ИИ должна покрывать ключевые разделы, а
конкретный кейс — проверяться против правил (PII во внешнюю модель, автоматические
решения о людях и т.п.). Здесь валидируем полноту политики и решаем по кейсу.
"""

REQUIRED_SECTIONS = (
    "Допустимые данные",
    "Запрещённые данные",
    "Одобренные инструменты",
    "Проверка человеком",
    "Ответственность",
)


def validate_policy(text):
    """Вернуть список отсутствующих обязательных разделов (пустой = политика полная)."""
    return [s for s in REQUIRED_SECTIONS if s not in text]


def check_usecase(sends_pii=False, external=False, automated_decision=False):
    """Решение по кейсу против базовых правил. → {allowed, reason}."""
    if sends_pii and external:
        return {"allowed": False, "reason": "PII нельзя отдавать во внешнюю модель"}
    if automated_decision:
        return {"allowed": False, "reason": "решения о людях требуют проверки человеком"}
    return {"allowed": True, "reason": "разрешено политикой"}


if __name__ == "__main__":
    policy = ("# Политика ИИ\n## Допустимые данные\n...\n## Запрещённые данные\n...\n"
              "## Одобренные инструменты\n...\n## Проверка человеком\n...")
    print("Не хватает разделов:", validate_policy(policy) or "нет")
    print("PII во внешнюю:", check_usecase(sends_pii=True, external=True))
    print("Автоскоринг людей:", check_usecase(automated_decision=True))
    print("Саммари публичных данных:", check_usecase())

# Артефакт: чек-лист политики использования ИИ

Минимальная рабочая политика ИИ для команды. Валидатор/проверка кейса — [`code/policy.py`](../code/policy.py).

## Обязательные разделы политики

- [ ] **Допустимые данные** — что можно отдавать модели (уровни, урок 11.1).
- [ ] **Запрещённые данные** — секреты, restricted, PII во внешние модели.
- [ ] **Одобренные инструменты** — какие модели/сервисы и в каком режиме (enterprise/no-train).
- [ ] **Проверка человеком** — где обязателен human-in-the-loop (решения о людях, необратимое — урок 7.3).
- [ ] **Ответственность** — кто отвечает за вывод ИИ, как сообщать об инцидентах.

## Базовые правила (исполняемые)

- [ ] PII во внешнюю модель — **запрещено** (redaction/enterprise-канал).
- [ ] Автоматические решения о людях — **только с проверкой человеком**.
- [ ] Фактические ответы наружу — **с фактчекингом** (урок 11.2).
- [ ] Каждый ИИ-процесс — **логируется и тегируется** (уроки 10.2, 9.4).

## Рамки и согласование

- **NIST AI RMF** (Govern) — роли и процедуры на весь жизненный цикл.
- **ISO/IEC 42001** — сертифицируемая система управления ИИ (Plan-Do-Check-Act); не равна автоматическому соответствию EU AI Act.
- **EU AI Act** — обязанности по уровню риска; поэтапное вступление в силу (6 мес. — запрещённые практики, 12 — GPAI, 24/36 — высокорисковые системы).
- Пишется с **security/legal**, проверяется на реальных кейсах, ревизуется регулярно.
