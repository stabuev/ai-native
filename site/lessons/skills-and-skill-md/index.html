Урок 4.1 · Skills и SKILL.md

Фаза 4 — Скиллы, память и проекты · Результат фазы: Упаковывать способности в SKILL.md, хранить контекст в памяти и собирать рабочие проекты.

Это вход в Фазу 4. До сих пор всё, что агент «умел», жило в одном промпте на один разговор. Теперь учимся упаковывать умение в файл, который агент сам подхватит, когда оно понадобится, — и не будет таскать его в контексте, пока не понадобится.

MOTTO. Скилл — это навык в виде папки: агент сам подгружает нужный по описанию.

PROBLEM

Один и тот же приём ты объясняешь агенту заново в каждом разговоре: «оформляй отчёт вот так», «считай юнит-экономику по этой формуле», «пиши коммиты в таком стиле». Промпт живёт ровно один диалог — завтра всё сначала.

Хочется наоборот: написать инструкцию один раз, положить в файл, и чтобы агент сам понял, когда она применима, и подтянул её. Но если просто склеить все свои инструкции в один гигантский системный промпт, контекст распухнет, станет дороже и грязнее — модель будет читать инструкцию по вёрстке Excel, даже когда ты спрашиваешь про SQL. Нужен механизм, который держит много умений «под рукой», но грузит в контекст только релевантное.

CONCEPT

Интуиция

Скилл устроен как онбординг-гайд для нового сотрудника. У гайда есть обложка с одной строкой — «Обработка PDF: извлечь текст и таблицы, заполнить формы» — и есть толстое тело с пошаговыми инструкциями внутри. Новичок не зачитывает весь гайд наизусть: он держит в голове только обложки всех разделов, а открывает конкретный раздел, лишь когда упёрся в соответствующую задачу.

Агент делает ровно это. На старте он подгружает только обложки (frontmatter name + description) всех установленных скиллов — это дёшево, ~100 токенов на скилл. Тело инструкции читается с диска, только когда запрос совпал с описанием. Этот приём называется progressive disclosure (постепенное раскрытие): по формулировке доки Anthropic, Claude грузит информацию поэтапно, по мере необходимости, а не загружает весь контекст заранее.

Как это работает

Скилл — это папка с обязательным файлом SKILL.md. В нём YAML-frontmatter (метаданные) и тело (инструкция):

.claude/skills/<name>/SKILL.md
   ---
   name: <kebab-case, как зовут скилл>
   description: <ЧТО делает + КОГДА применять + триггеры>   ← по этому агент выбирает скилл
   ---
   <тело: инструкция, грузится в контекст только если скилл выбран>

Загрузка идёт тремя уровнями — это и есть progressive disclosure:

Уровень 1  Метаданные (name + description)   всегда, на старте     ~100 токенов/скилл
Уровень 2  Тело SKILL.md (инструкция)         когда скилл сработал   < 5k токенов
Уровень 3  Файлы рядом (scripts, REFERENCE)   когда тело сослалось   по сути без лимита

Ключевой вывод: главное поле скилла — description. Именно его (и только его) агент видит на старте и по нему решает, грузить ли тело. Плохой description = скилл не сработает, даже если внутри идеальная инструкция. Поэтому в описании важны и что делает, и когда применять, и разные формулировки-триггеры — агент матчит запрос на описание семантически.

Формальные требования к полям (из доки Anthropic): name — до 64 символов, только строчные буквы/цифры/дефисы, без слов claude/anthropic; description — непустой, до 1024 символов.

РАЗБОР ПО ШАГАМ

Посмотрим руками, как агент выбирает скилл по описанию. Пусть установлены два скилла (видны только их description):

editor : "Редактирует текст: сокращает, чистит, меняет тон письма"
calc   : "Считает арифметику, числа и проценты"

Приходит запрос: «отредактируй текст письма». Наш учебный матчер берёт пересечение слов запроса и описания:

слова запроса:  {отредактируй, текст, письма}
∩ editor desc:  {текст, письма}    → score 2   ← выбран
∩ calc desc:    {}                 → score 0

Выигрывает editor — у него есть общие слова текст и письма. На запрос «посчитай проценты» победит calc (общее слово проценты). А на «погода завтра» оба дадут score 0 → ни один скилл не сработает.

Здесь же видно, почему description пишут с запасом формулировок. Запрос «сократи имейл» по словам не пересечётся с editor (сократи≠сокращает, имейл≠письмо) — и наш простой матчер промахнётся. Настоящий агент матчит семантически и поймёт, но и он работает тем точнее, чем больше разных слов-триггеров в описании. Отсюда правило: в description перечисляй синонимы и типичные фразы запроса («сократи», «ужми», «резюмируй», «письмо», «имейл», «отчёт»).

BUILD IT

Задание: собери загрузчик скиллов — парсер SKILL.md + валидатор + матчер. Только стандартная библиотека Python, без PyYAML и без сети.

Перед запуском. Работай в своей папке курса (ai-native/4.1-skills/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл skill_loader.py и реализуй три функции:

  • parse_skill(text) — разобрать SKILL.md. Вход: содержимое файла строкой. Если в начале есть frontmatter между двумя строками ---, вытащи из него name и description (парси построчно по key: value, без PyYAML), остальное — тело. Выход: словарь {"name", "description", "body"}; если frontmatter нет — name/description равны None, а весь текст идёт в body.
  • validate_skill(parsed) — проверить обязательные поля. Вход: результат parse_skill. Выход: список строк-проблем (пустой список = всё ок). Минимум: ругайся, если нет name, нет description, или description короче ~20 символов (значит, нет триггеров).
  • match_skill(query, skills) — выбрать скилл под запрос. Вход: строка query и список parsed-скиллов. Посчитай для каждого пересечение множеств слов запроса и его description, верни скилл с максимальным пересечением; если у всех ноль совпадений — верни None.

Готово, когда все тесты в test_skill_loader.py зелёные — это твоя приёмка, и там на примерах видно ожидаемое поведение (парс с frontmatter и без, валидатор ловит пустые/короткие поля, матчер выбирает релевантный скилл и возвращает None без совпадений).

Подсказка. Frontmatter удобно отделить регуляркой r"\s*---\s*\n(.*?)\n---\s*\n?(.*)" с флагом re.DOTALL. Для слов запроса/описания заведи хелпер set(re.findall(r"\w+", text.lower())). Матчер намеренно простой (пересечение слов) — он показывает суть выбора; настоящий агент матчит семантически, поэтому в реальном description так важны разные формулировки.

USE IT

В реальной жизни матчер писать не нужно — кладёшь SKILL.md в нужную папку, и агент подхватывает его сам по описанию. Один и тот же файл кросс-платформенный:

  • Claude Code — скиллы живут на файловой системе: .claude/skills/<name>/SKILL.md (проектные, едут с репозиторием) или ~/.claude/skills/<name>/SKILL.md (личные, во всех проектах). Никакой загрузки — агент находит их сам.
  • claude.ai — залить свой скилл zip-архивом через Settings → Features (на планах с code execution).
  • Claude API — загрузить через Skills API (/v1/skills) и сослаться на skill_id в container.

Проверить, что видно в Claude Code:

ls .claude/skills/*/SKILL.md     # проектные скиллы
ls ~/.claude/skills/*/SKILL.md   # личные скиллы

Важная оговорка: скиллы не синхронизируются между поверхностями — то, что залито в claude.ai, не появится в API и наоборот; файловые скиллы Claude Code отдельны от обоих. В этом курсе так сделаны рабочие скиллы editor, doc-assistant, prompt-eval и онбординг-скиллы — открой любой из них и посмотри на его description.

ЧАСТЫЕ ОШИБКИ

  • Слабый description. Самая частая. Агент видит на старте только описание — если в нём нет «когда применять» и слов-триггеров, скилл не сработает, какой бы умной ни была инструкция внутри. Пиши «что + когда + синонимы».
  • Раздувать тело скилла. Длинная инструкция = дорогой Уровень 2 при каждом срабатывании. Держи тело компактным (< ~5k токенов), редкие детали выноси в отдельные файлы (Уровень 3), которые грузятся по ссылке.
  • Думать, что скилл = системный промпт. Промпт всегда в контексте; скилл подгружается по требованию (progressive disclosure). В этом вся экономия — иначе проще было бы склеить всё в один промпт.
  • Ждать, что скилл переедет сам. Поверхности не синхронизируются: залил в claude.ai — в API его нет. Под каждую поверхность ставь скилл отдельно.
  • Нарушать формат name. Только строчные буквы/цифры/дефисы, до 64 символов, без claude/anthropic — иначе скилл не зарегистрируется.

def test_parse_extracts_frontmatter_and_body():
    p = parse_skill(SAMPLE)
    assert p["name"] == "editor"
    assert "Редактирует текст" in p["description"]
    assert p["body"].startswith("# editor")


def test_parse_without_frontmatter():
    p = parse_skill("просто текст без frontmatter")
    assert p["name"] is None
    assert p["body"] == "просто текст без frontmatter"


def test_validate_flags_missing_and_short():
    assert validate_skill({"name": None, "description": None})
    short = validate_skill({"name": "x", "description": "коротко"})
    assert any("короткий" in p for p in short)
    assert validate_skill(parse_skill(SAMPLE)) == []


def test_match_picks_relevant_skill():

        parse_skill(SAMPLE),
        {"name": "calc", "description": "считает арифметику числа и проценты"},

    assert match_skill("посчитай проценты", skills)["name"] == "calc"

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/skill_loader.pyПоказать решение
"""Skills и SKILL.md: парсер + матчер — Build It для урока 4.1.

Без зависимостей. Скилл — это папка с SKILL.md (frontmatter `name`/`description`
+ инструкция). Здесь парсим frontmatter без PyYAML, валидируем обязательные поля
и выбираем подходящий скилл под запрос по триггерам. Так работает progressive
disclosure: агент грузит инструкцию только релевантного скилла.
"""
import re


def parse_skill(text):
    """SKILL.md -> {name, description, body}. Frontmatter между `---` ... `---`."""
    m = re.match(r"\s*---\s*\n(.*?)\n---\s*\n?(.*)", text, re.DOTALL)
    if not m:
        return {"name": None, "description": None, "body": text.strip()}
    front, body = m.group(1), m.group(2)
    meta = {}
    for line in front.splitlines():
        if ":" in line:
            key, _, val = line.partition(":")
            meta[key.strip()] = val.strip().strip('"').strip("'")
    return {"name": meta.get("name"), "description": meta.get("description"),
            "body": body.strip()}


def validate_skill(parsed):
    """Список проблем по обязательным полям skill (пустой = ок)."""
    problems = []
    if not parsed.get("name"):
        problems.append("нет поля name во frontmatter")
    desc = parsed.get("description")
    if not desc:
        problems.append("нет поля description во frontmatter")
    elif len(desc) < 20:
        problems.append("description слишком короткий — добавь триггеры")
    return problems


def _words(text):
    return set(re.findall(r"\w+", (text or "").lower()))


def match_skill(query, skills):
    """Выбрать скилл с наибольшим пересечением слов запроса и description.

    skills: list[parsed]. Возвращает лучший parsed или None (нет совпадений).
    """
    q = _words(query)
    best, best_score = None, 0
    for s in skills:
        score = len(q & _words(s.get("description")))
        if score > best_score:
            best, best_score = s, score
    return best


if __name__ == "__main__":
    sample = (
        "---\n"
        "name: editor\n"
        "description: Редактирует текст: сокращает, чистит, меняет тон письма\n"
        "---\n"
        "# editor\nИнструкция по редактуре...\n"
    )
    parsed = parse_skill(sample)
    print("Распознано:", parsed["name"], "|", parsed["description"][:30])
    print("Проблемы:", validate_skill(parsed) or "нет")
    skills = [parsed, {"name": "calc", "description": "Считает арифметику и числа"}]
    hit = match_skill("отредактируй текст", skills)        # 'текст' есть в описании editor
    print("Под запрос 'отредактируй текст' →", hit["name"] if hit else "нет совпадений")

## Как переиспользовать (инструкция к шаблону — удалить в реальном скилле)

1. Скопируй папку в `.claude/skills/<name>/` (проектные) или `~/.claude/skills/<name>/` (личные).
2. Заполни `name` (kebab-case) и **`description` с триггерами** — это главное: по нему агент выбирает скилл.
3. Тело (`body`) грузится только когда скилл выбран (progressive disclosure) — не раздувай.
4. Проверь: `ls .claude/skills/*/SKILL.md`. Кросс-платформенно: Claude Code / Cursor / Codex.
5. Хороший `description`: глагол действия + объект + 3–5 разных формулировок-триггеров.
