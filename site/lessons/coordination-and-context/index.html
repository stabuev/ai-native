Урок 8.3 · Координация и контекст

Фаза 8 — Мульти-агенты и оркестрация · Результат фазы: Построить мульти-агентный процесс с оркестратором и передачей контекста.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Оркестратор раздал роли (8.1), агенты нашли друг друга (8.2). Но они работают над общим результатом — и тут начинаются конфликты. У одиночного агента такой проблемы нет; у команды — есть. Этот урок про общее состояние и предотвращение конфликтов.

MOTTO. Несколько агентов над одним состоянием — это про передачу контекста и предотвращение конфликтов.

PROBLEM

Когда несколько субагентов пишут в общий результат, появляются две беды, которых нет у одиночки:

  • конфликты — двое записали один и тот же ключ по-разному (researcher положил факты, rogue-агент их перезаписал своими);
  • потеря контекста — агент не видит, что уже сделал другой, и дублирует или противоречит.

Без координации мульти-агентный процесс выдаёт несогласованный результат: каждый работал «правильно», а вместе вышел разнобой. Нужно общее состояние, видимое всем, и правило, которое замечает конфликтную запись. Соберём такой общий контекст.

CONCEPT

Интуиция

Это общая доска (blackboard), на которой работает команда. Каждый пишет на доску свой результат и читает, что написали другие, — поэтому writer видит факты researcher и строит на них. Но если двое пишут в одну клетку разное, доску надо не молча затирать, а зафиксировать спор: кто, что, поверх чего. Тогда ведущий (lead) увидит конфликт и разрешит его, а не отдаст наружу случайную из двух версий.

Anthropic в своём мульти-агентном ресёрче сводит результаты субагентов именно через общее хранилище (краткие выжимки), а не пересылкой длинных чат-ответов, и реконсилирует их у lead-агента.

Как это работает

        SharedContext (blackboard)
   agent1 ─ write(facts, ...) ┐
   agent2 ─ read(facts) ──────┤  состояние видно всем
   agent3 ─ write(facts, ...) ┘  ← конфликт, если ДРУГОЙ агент пишет ДРУГОЕ значение
  • write(key, value, agent) — записать в общее состояние с автором; если ключ уже занят другим агентом и значение другое → зафиксировать запись в conflicts (но всё равно записать — детекция, не блокировка).
  • read(key) — прочитать общее состояние (так контекст передаётся между агентами).
  • run_process(steps) — прогнать агентов [(name, fn)] над одним контекстом.

Важно, что считать конфликтом: тот же агент переписывает свой ключ — норма (он уточняет себя); другой агент, но то же значение — тоже норма (согласие). Конфликт — только когда другой автор кладёт другое значение: это и есть несогласованность, которую нельзя терять.

Как разрешают конфликт

Разделяй два уровня. Детекция — это механизм: дёшево, детерминированно, доска просто замечает спор (наш SharedContext). Разрешение — это политика: зависит от задачи, и единственно верной нет. Поэтому доска фиксирует конфликт, но не решает его сама — решает резолвер (обычно lead). Типовые стратегии:

  • Last-write-wins — побеждает последняя запись. Так делает наш data по умолчанию: просто, но молча теряет работу первого автора. Годится для идемпотентных/неважных ключей, опасно для содержательных.
  • Приоритет роли — у авторов ранг: запись researcher бьёт запись случайного rogue. Детерминированно и дёшево, если иерархия ролей ясна.
  • Кворум/голосование — если значение независимо подтвердили ≥N агентов, берём его. Гасит одиночный сбойный голос, но дороже (нужно несколько мнений).
  • Эскалация к lead — lead-агент (LLM) смотрит обе версии и решает или сливает их (["A","B"] + ["X"] → ["A","B","X"]). Самый гибкий и самый дорогой; для этого и нужна полная запись спора (old/new/by/prev_by).

Выбор стратегии — часть дизайна процесса, как и выбор guardrail в 7.3. Дешёвые правила (приоритет роли) — для рутины; эскалация к lead — там, где цена ошибки высока.

РАЗБОР ПО ШАГАМ

Прогоним процесс из трёх агентов над общим контекстом:

1. researcher: write("facts", ["A","B"], "researcher")
   data = {facts: ["A","B"]}   writers = {facts: researcher}

2. writer: read("facts") → ["A","B"] (видит чужой результат!)
            write("draft", "черновик по 2 фактам", "writer")
   data = {facts:["A","B"], draft:"черновик по 2 фактам"}

3. rogue: write("facts", ["X"], "rogue")
   ключ facts занят (researcher), значение другое (["A","B"] ≠ ["X"]), автор другой
   → КОНФЛИКТ: {key:facts, old:["A","B"], new:["X"], by:rogue, prev_by:researcher}
   data = {facts:["X"], ...}   (значение перезаписано, но спор зафиксирован)

Видно обе механики. Передача контекста: writer не работает с нуля — он прочитал факты researcher и построил на них. Детекция конфликта: rogue перезаписал чужой ключ другим значением — это попало в conflicts с полной историей (кто, что, поверх чего), чтобы lead мог разрешить. А если бы rogue записал те же ["A","B"] или это был сам researcher — конфликта бы не было.

Теперь приложим к этому спору (["A","B"] от researcher против ["X"] от rogue) разные стратегии разрешения — и увидим, что они дают разный итог:

last-write-wins  → ["X"]            (что и лежит в data; работа researcher потеряна)
приоритет роли   → ["A","B"]        (researcher старше rogue → его версия)
эскалация к lead → ["A","B","X"]    (lead решил слить обе)

Это и есть граница урока: доска детектит одинаково, а итог зависит от выбранной политики. Наш SharedContext по умолчанию делает last-write-wins для data, но сохраняет запись спора — поэтому резолвер (lead) может переопределить результат любой из стратегий выше.

Создай файл coordination.py:

  • SharedContext — поля data (значения), writers (автор по ключу), conflicts (список).
  • write(key, value, agent) — если ключ уже в data, автор был другой и значение другое → добавить в conflicts запись {key, old, new, by, prev_by}. В любом случае записать data[key]=value, writers[key]=agent.
  • read(key, default=None) — вернуть значение или default.
  • run_process(steps, ctx=None) — steps это [(name, fn)]; создать контекст (если не передан), вызвать каждый fn(ctx, name), вернуть контекст.

USE IT

Прогон мульти-агентного процесса в проде = оркестратор (8.1) + discovery (8.2) + общий контекст (мульти-провайдер):

  • Shared memory store — субагенты пишут краткие выжимки в общее хранилище, а lead их сводит (как у Anthropic), вместо пересылки длинных ответов между агентами.
  • Изолированные контексты субагентов (Claude Code) + явная передача только нужного — меньше context rot (4.4): на доску кладут результат, а не весь свой процесс.
  • Реконсиляция у lead — единая точка сборки и разрешения конфликтов: доска фиксирует спор, lead решает, чья версия верна.

ЧАСТЫЕ ОШИБКИ

  • Молча затирать чужую запись. Если rogue перезаписал факты и никто не заметил, наружу уйдёт случайная версия. Конфликт надо фиксировать (кто, что, поверх чего), а не терять.
  • Считать любую перезапись конфликтом. Тот же агент, уточняющий свой ключ, или другой агент с тем же значением — это норма, не спор. Конфликт — только другой автор + другое значение.
  • Пересылать между агентами длинные ответы. Это раздувает контекст. Субагенты кладут на доску краткие выжимки, lead сводит — так дешевле и чище (как у Anthropic).
  • Нет единой точки реконсиляции. Без lead, который разрешает конфликты, несогласованность дойдёт до пользователя. Доска детектит, lead решает — это разные роли.
  • Блокировать запись вместо детекции. Жёсткий лок усложняет процесс и плодит дедлоки. Проще зафиксировать конфликт и разрешить его на сборке у lead.
  • Брать last-write-wins для содержательных ключей. Удобный дефолт молча теряет работу первого автора. Под важные данные выбирай стратегию осознанно — приоритет роли, кворум или эскалацию к lead (как guardrail в 7.3).

from coordination import SharedContext, run_process


def test_write_read():
    ctx = SharedContext()
    ctx.write("k", 1, "a")
    assert ctx.read("k") == 1
    assert ctx.read("missing", "def") == "def"


def test_conflict_detected_on_cross_agent_overwrite():
    ctx = SharedContext()
    ctx.write("facts", ["A"], "agent1")
    ctx.write("facts", ["X"], "agent2")          # другой агент, другое значение
    assert len(ctx.conflicts) == 1
    assert ctx.conflicts[0]["by"] == "agent2" and ctx.conflicts[0]["prev_by"] == "agent1"


def test_no_conflict_same_agent_or_same_value():
    ctx = SharedContext()
    ctx.write("k", 1, "a")
    ctx.write("k", 2, "a")                        # тот же агент → не конфликт
    ctx.write("k", 2, "b")                        # другой агент, но то же значение → не конфликт
    assert ctx.conflicts == []


def test_process_passes_context_between_agents():
    def producer(ctx, me):
        ctx.write("facts", ["A", "B"], me)

    def consumer(ctx, me):
        ctx.write("draft", len(ctx.read("facts")), me)

    ctx = run_process([("producer", producer), ("consumer", consumer)])
    assert ctx.read("draft") == 2                 # consumer увидел контекст producer'а
    assert ctx.conflicts == []

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/coordination.pyПоказать решение
"""Координация и контекст — Build It для урока 8.3.

Без зависимостей. Несколько агентов работают над общим состоянием (blackboard):
читают и пишут разделяемый контекст. Главная опасность мульти-агентности —
конфликты записи; здесь они детектируются. Так оркестрация (8.1) и A2A (8.2)
превращаются в связный процесс с передачей состояния.
"""


class SharedContext:
    """Общий контекст (blackboard) с отслеживанием авторов и конфликтов записи."""

    def __init__(self):
        self.data = {}
        self.writers = {}
        self.conflicts = []

    def write(self, key, value, agent):
        # конфликт: другой агент уже записал в этот ключ ДРУГОЕ значение
        if key in self.data and self.writers[key] != agent and self.data[key] != value:
            self.conflicts.append({
                "key": key, "old": self.data[key], "new": value,
                "by": agent, "prev_by": self.writers[key],
            })
        self.data[key] = value
        self.writers[key] = agent

    def read(self, key, default=None):
        return self.data.get(key, default)


def run_process(steps, ctx=None):
    """steps: list[(agent_name, fn(ctx))]. Прогоняет агентов над общим контекстом."""
    ctx = ctx or SharedContext()
    for name, fn in steps:
        fn(ctx, name)
    return ctx


if __name__ == "__main__":
    def researcher(ctx, me):
        ctx.write("facts", ["A", "B"], me)

    def writer(ctx, me):
        ctx.write("draft", f"черновик по {len(ctx.read('facts'))} фактам", me)

    def rogue(ctx, me):
        ctx.write("facts", ["X"], me)            # перезапишет facts — конфликт

    ctx = run_process([("researcher", researcher), ("writer", writer), ("rogue", rogue)])
    print("data:", ctx.data)
    print("conflicts:", ctx.conflicts)

Артефакт

outputs/multi_agent_process.py
"""multi_agent_process.py — артефакт урока 8.3: сквозной мульти-агентный процесс.

Самодостаточно, без зависимостей. Объединяет Фазу 8: оркестратор раздаёт роли
субагентам, субагенты пишут в общий контекст (с детекцией конфликтов), lead сводит
результат. В проде субагенты — это LLM-агенты (Claude subagents / Agent SDK).

Запуск:  python multi_agent_process.py
"""


class SharedContext:
    """Общий blackboard с авторами записей и детекцией конфликтов."""

    def __init__(self):
        self.data, self.writers, self.conflicts = {}, {}, []

    def write(self, key, value, agent):
        if key in self.data and self.writers[key] != agent and self.data[key] != value:
            self.conflicts.append({"key": key, "by": agent, "prev_by": self.writers[key]})
        self.data[key] = value
        self.writers[key] = agent

    def read(self, key, default=None):
        return self.data.get(key, default)


# --- субагенты-роли (в проде — LLM-агенты со своим контекстом) ---
def researcher(ctx, me):
    ctx.write("facts", ["возврат 14 дней", "гарантия 12 мес"], me)


def writer(ctx, me):
    facts = ctx.read("facts", [])
    ctx.write("draft", f"Памятка по {len(facts)} пунктам: " + "; ".join(facts), me)


def reviewer(ctx, me):
    draft = ctx.read("draft", "")
    ctx.write("review", "ок" if "14 дней" in draft else "проверить факты", me)


def orchestrate(steps):
    """Lead: прогоняет субагентов над общим контекстом и сводит результат."""
    ctx = SharedContext()
    for name, fn in steps:
        fn(ctx, name)
    return {"result": ctx.read("draft"), "review": ctx.read("review"),
            "conflicts": ctx.conflicts}


if __name__ == "__main__":
    out = orchestrate([("researcher", researcher), ("writer", writer), ("reviewer", reviewer)])
    print("Результат :", out["result"])
    print("Ревью     :", out["review"])
    print("Конфликты :", out["conflicts"] or "нет")
