Урок 3.2 · Длинный контекст и работа с файлами

Фаза 3 — Текст и документы · Результат фазы: Строить пайплайны резюмирования и редактуры; собрать базовый RAG над своим корпусом.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Этот урок — про то, как обрабатывать документы, которые не влезают в один запрос. Отчёт на 200 страниц не поместится в контекстное окно (урок 1.2), а если и поместится (1M токенов) — это дорого и качество в середине проседает. Решение — chunking + map-reduce: разрезать документ на куски, сжать каждый и свести результаты. Это же режет данные для RAG (урок 3.3).

MOTTO. Документ не влезает в окно — режь на части, сжимай каждую, своди вместе.

PROBLEM

Длинный документ не помещается в контекстное окно, а класть «всё целиком» в окно на 1M токенов — дорого и ненадёжно: модель хуже видит то, что в середине длинного контекста. Значит, нужна стратегия: разбить документ на управляемые куски, обработать каждый по отдельности и собрать общий результат — масштабируемо и предсказуемо по цене.

CONCEPT

Интуиция

Это как читать толстую книгу: целиком в голове не удержать, поэтому читаешь по главам, делаешь заметку по каждой, а потом пишешь резюме из заметок. Машинная версия того же — chunking (нарезать на куски) и map-reduce (сжать каждый кусок → свести вместе).

Как это работает

длинный документ
   │ chunking (перекрывающиеся куски)
   ▼
[chunk] [chunk] [chunk] ...
   │ map: сжать каждый кусок
   ▼
[summary] [summary] [summary]
   │ reduce: свести в один результат
   ▼
итоговое резюме
  • Chunking. Режем текст на куски по size слов. Ключевая деталь — перекрытие (overlap): соседние чанки делят несколько слов на стыке, чтобы предложение или мысль, разорванные границей, не потерялись. Шаг между началами чанков = size − overlap. Стратегии бывают разные (по словам, по токенам, по структуре документа — см. сплиттеры LangChain); мы делаем простейшую, по словам.
  • Map-reduce. map применяет операцию к каждому чанку (например, суммаризацию), reduce сводит результаты в один. Так обрабатывается документ любой длины: добавилось страниц — добавилось чанков, алгоритм тот же.

Альтернатива chunking'у — длинное окно напрямую (загрузить файл целиком в 1M токенов у Claude/Gemini). Подходит для умеренных объёмов, но помни про позицию: по подсказкам Anthropic для длинного контекста данные кладут вверх, а вопрос — в конец (модель хуже работает с серединой длинного ввода). Когда документ реально большой или вызовов много — chunking + map-reduce дешевле и надёжнее.

РАЗБОР ПО ШАГАМ

Нарежем 25 слов (w1…w25) с size=10, overlap=3. Шаг между чанками: step = size − overlap = 10 − 3 = 7. Начинаем с позиций 0, 7, 14, 21:

chunk0:  w1  w2  w3  w4  w5  w6  w7  w8  w9  w10
chunk1:  w8  w9  w10 w11 w12 w13 w14 w15 w16 w17
chunk2:  w15 w16 w17 w18 w19 w20 w21 w22 w23 w24
chunk3:  w22 w23 w24 w25

Две вещи, которые проверяют тесты:

  • Перекрытие. Последние 3 слова chunk0 (w8 w9 w10) — это первые 3 слова chunk1. Именно overlap не даёт «разрезать мысль» на границе.
  • Ничего не потеряно. Объединение всех чанков покрывает w1…w25 целиком; первый чанк начинается с w1, последний кончается на w25.

Теперь map_reduce. Возьмём map_fn = длина чанка в словах, reduce_fn = sum. На нарезке без перекрытия (overlap=0) сумма длин = ровно 25 — слова не дублируются. С перекрытием сумма была бы больше: перекрытые слова считаются дважды (это плата за связность). В проде map_fn — это суммаризация куска моделью, а reduce_fn — сведение саммари в одно.

BUILD IT

Задание: собери chunking с перекрытием и map-reduce — стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/3.2-long-context/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3, всё офлайн (для теста ещё pytest).

Реализуй две функции:

  • chunk_text(text, size=100, overlap=20) — режь текст на чанки по словам: size слов в чанке, overlap слов внахлёст. Проверь параметры: size > 0 и overlap < size, иначе raise ValueError. Шаг = size − overlap; иди по словам с этим шагом, на каждой позиции бери срез size слов; останавливайся, когда чанк дотянулся до конца текста. Пустой текст → [].
  • map_reduce(chunks, map_fn, reduce_fn) — примени map_fn к каждому чанку, затем reduce_fn к списку результатов, верни итог.

Готово, когда тест test_chunking.py зелёный — он проверяет, что чанки не больше size, что покрыты все слова (первое w1, последнее w25), что соседние чанки перекрываются ровно на overlap, что неверные параметры дают ValueError, что map_reduce композирует и пустой текст даёт [].

Подсказка. Шаг цикла — range(0, len(words), size - overlap); на каждой итерации бери words[start:start + size] и прерывайся, когда start + size >= len(words).

USE IT

В проде map_fn — суммаризация куска моделью, reduce_fn — сведение саммари; либо грузишь файл целиком в длинное окно:

chunks = chunk_text(big_doc, size=800, overlap=100)
ask = lambda p: Anthropic().messages.create(model="claude-haiku-4-5", max_tokens=300,
    messages=[{"role": "user", "content": p}]).content[0].text
summarize = lambda c: ask(f"Сожми кусок:\n{c}")                       # map: сжать каждый кусок
reduce_ = lambda parts: ask("Сведи саммари в одно:\n" + "\n".join(parts))  # reduce: свести вместе
final = map_reduce(chunks, summarize, reduce_)

Альтернатива для умеренных объёмов — длинный контекст напрямую: загрузка PDF/таблиц и окно до 1M токенов (Claude/Gemini), с подсказками из long context tips: данные — вверх, вопрос — в конец.

ЧАСТЫЕ ОШИБКИ

  • Резать без перекрытия. На стыке чанков легко разорвать предложение или мысль, и обе половины потеряют смысл. Небольшой overlap сохраняет связность (плата — повтор перекрытых слов).
  • «Просто положу всё в окно на 1M». Дорого и середина теряется. Для больших документов или массовых прогонов chunking + map-reduce надёжнее и дешевле.
  • Игнорировать позицию в длинном контексте. Если грузишь целиком — данные вверх, вопрос в конец; иначе модель хуже находит нужное в середине.
  • Слишком мелкие или крупные чанки. Мелкие рвут контекст и плодят вызовы; крупные снова упираются в окно. Подбирай size/overlap под документ.

code/test_chunking.py
import pytest
from chunking import chunk_text, map_reduce

TEXT = " ".join(f"w{i}" for i in range(1, 26))   # w1..w25, 25 слов


def test_chunks_respect_size():
    chunks = chunk_text(TEXT, size=10, overlap=3)
    assert all(len(c.split()) <= 10 for c in chunks)


def test_chunks_cover_all_words():
    chunks = chunk_text(TEXT, size=10, overlap=3)
    seen = set()
    for c in chunks:
        seen.update(c.split())
    assert seen == set(TEXT.split())          # ни одно слово не потеряно
    assert chunks[0].split()[0] == "w1"
    assert chunks[-1].split()[-1] == "w25"


def test_overlap_between_consecutive_chunks():
    chunks = chunk_text(TEXT, size=10, overlap=3)
    first, second = chunks[0].split(), chunks[1].split()
    assert first[-3:] == second[:3]           # последние 3 = первые 3 следующего


def test_invalid_params_raise():
    with pytest.raises(ValueError):
        chunk_text(TEXT, size=0)
    with pytest.raises(ValueError):
        chunk_text(TEXT, size=5, overlap=5)


def test_map_reduce_composes():
    chunks = chunk_text(TEXT, size=10, overlap=0)
    total = map_reduce(chunks, lambda c: len(c.split()), sum)
    assert total == 25                        # без перекрытия слова не дублируются


def test_empty_text():
    assert chunk_text("", size=10, overlap=2) == []

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/chunking.pyПоказать решение
"""Длинный контекст: chunking + map-reduce — Build It для урока 3.2.

Без зависимостей. Длинный документ не влезает в контекстное окно (урок 1.2):
режем его на перекрывающиеся чанки, обрабатываем каждый (map) и сводим
результаты (reduce). Перекрытие сохраняет связность на стыках.
"""


def chunk_text(text, size=100, overlap=20):
    """Резать текст на чанки по словам: `size` слов, `overlap` слов внахлёст."""
    if size <= 0:
        raise ValueError("size должен быть > 0")
    if overlap >= size:
        raise ValueError("overlap должен быть < size")
    words = text.split()
    if not words:
        return []
    step = size - overlap
    chunks = []
    for start in range(0, len(words), step):
        chunks.append(" ".join(words[start:start + size]))
        if start + size >= len(words):
            break
    return chunks


def map_reduce(chunks, map_fn, reduce_fn):
    """Применить map_fn к каждому чанку, затем reduce_fn к списку результатов.

    Это паттерн map-reduce суммаризации: сжать части, потом свести их вместе.
    """
    mapped = [map_fn(c) for c in chunks]
    return reduce_fn(mapped)


if __name__ == "__main__":
    text = " ".join(f"слово{i}" for i in range(1, 26))   # 25 слов
    chunks = chunk_text(text, size=10, overlap=3)
    print(f"{len(chunks)} чанк(ов):")
    for c in chunks:
        print("  ", c)
    # map: длина чанка; reduce: суммарно слов (с учётом перекрытий)
    total = map_reduce(chunks, lambda c: len(c.split()), sum)
    print("Слов суммарно по чанкам (с перекрытием):", total)

# Артефакт: пайплайн обработки длинных документов

Рецепт «как обработать документ, который не влезает (или дорого влезает) в окно». Код — в [`code/chunking.py`](../code/chunking.py).

## Развилка: какой путь выбрать

```
документ помещается в окно с запасом?
  ├─ да, и он небольшой      → одним промптом
  ├─ да, но крупный (≤1M)    → длинный контекст напрямую (данные вверх, вопрос в конец)
  └─ нет / дорого / их много → chunking + map-reduce  ИЛИ  RAG (урок 3.3)
```

## Map-reduce суммаризация

```python
from chunking import chunk_text, map_reduce
chunks = chunk_text(doc, size=800, overlap=100)
summary = map_reduce(chunks,
    map_fn=lambda c: summarize_with_model(c),       # сжать кусок
    reduce_fn=lambda parts: summarize_with_model("\n".join(parts)))  # свести
```

## Параметры под тип документа

| Документ | size (слов) | overlap | Стратегия |
|---|---|---|---|
| Статья/письмо | — | — | один промпт |
| Отчёт (десятки стр.) | 600–1000 | 80–120 | map-reduce |
| Книга/множество файлов | 400–800 | 50–100 | RAG (retrieval, урок 3.3) |
| Таблицы/структура | по секциям | по строкам-заголовкам | structure-aware split |

## Правила

- **Overlap обязателен** — иначе теряется смысл на стыках кусков.
- **Считай бюджет** заранее (`prompt-token-budgeter`, урок 1.1): вход + ответ ≤ окно.
- **Длинное окно ≠ бесплатно** — за токены платишь, «середина» хуже вспоминается; данные клади вверх, вопрос — в конец.
- **Много документов и вопросы по ним** → не суммаризация, а RAG (уроки 3.3–3.4).
