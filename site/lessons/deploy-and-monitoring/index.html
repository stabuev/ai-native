Урок 10.2 · Деплой и мониторинг

Фаза 10 — Production и интеграция · Результат фазы: Встроить ИИ в рабочий инструмент с мониторингом токенов и стоимости.

Requires (только для USE IT): прод-окружение / observability-бэкенд. BUILD IT работает офлайн, без ключей и без сети.

В 10.1 мы встроили ИИ безопасно. Теперь — увидеть, как он живёт в проде. Выложить мало: без логов и метрик «у пользователя не работает» превращается в гадание. А ИИ ещё и недетерминирован, так что наблюдаемость тут критична вдвойне.

MOTTO. Задеплоил — значит измеряешь: без логов prompt/response и метрик здоровья прод вслепую.

PROBLEM

Процесс выложен, вроде работает. Но: не растёт ли доля ошибок после вчерашнего релиза? Не деградировала ли латентность? Что именно сломалось у того клиента, который жалуется? Без структурного лога каждого вызова и метрик здоровья ответов нет — есть догадки.

И есть ловушка: среднее лжёт. Средняя латентность может выглядеть отлично, пока каждый двадцатый запрос висит 5 секунд — и именно эти пользователи уходят. Нужны метрики, которые ловят хвост (перцентили), а не сглаживают его. Соберём логгер вызовов и метрики мониторинга.

CONCEPT

Интуиция

Мониторинг — это приборная панель самолёта, а не «вроде летим». Пилот не смотрит на «среднюю высоту за полёт» — он видит текущие показатели и тревожные пики: давление, температуру, отказ датчика. Так и прод: на панели — доля ошибок, латентность по перцентилям, последние сбои. Среднее скрывает аварию (один отказавший двигатель на фоне трёх рабочих усредняется), перцентиль — показывает.

Каждый вызов модели — это запись в бортовой журнал; метрики поверх журнала — приборы; пороги на приборах — алерты.

Как это работает

каждый вызов → лог {model, latency, status, error}
                     │
   метрики:  error_rate · p50/p95 latency · recent_errors
                     │
              алерты при превышении порогов
  • log(model, latency_ms, ok, error) — структурная запись вызова (не «строка в файл», а поля для агрегации).
  • error_rate() — доля неуспешных вызовов (здоровье №1).
  • latency_percentile(p) — латентность на перцентиле p: p95 = «95% запросов быстрее этого», то есть граница худших 5%.
  • recent_errors(n) — последние сбои для разбора.

Почему p95, а не среднее: среднее сглаживает редкие, но болезненные зависания, а p95/p99 их показывают — именно по хвосту латентности уходят пользователи. Логируют то, что позволит и отладить, и оценить здоровье: модель, латентность, статус, ошибку (а контент — только с согласия и с редактированием PII).

РАЗБОР ПО ШАГАМ

Залогируем 5 вызовов (латентность в мс, последний — таймаут) и снимем метрики:

вызовы: 120✓  300✓  90✓  1500✗(timeout)  200✓

error_rate:  1 ошибка / 5 = 0.2
p95 latency: сортируем [90,120,200,300,1500]; ранг 95% → последний → 1500
recent_errors: [{latency:1500, error:"timeout"}]

А теперь сравни p95 со средним: (120+300+90+1500+200)/5 = 442ms. Среднее говорит «всё ок, ~0.4с», а p95 = 1500ms кричит, что у худших 5% запросов — полуторасекундные зависания. Это и есть смысл перцентилей: один таймаут утонул в среднем, но всплыл в p95 — и именно он станет алертом. error_rate 0.2 — второй сигнал: каждый пятый вызов упал, повод бить тревогу. Метрики превращают «вроде работает» в числа, по которым видно, что чинить.

BUILD IT

Задание: собери логгер вызовов и метрики здоровья — error_rate, перцентиль латентности, последние ошибки. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/10.2-monitoring/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл monitoring.py с классом CallLogger:

  • log(model, latency_ms, ok=True, error=None) — добавить запись вызова в self.calls.
  • error_rate() — доля записей с ok=False (пустой лог → 0.0; округли до 3).
  • latency_percentile(p=95) — nearest-rank перцентиль: отсортировать латентности, индекс k = round(p/100 * n) - 1 (с защитой границ 0..n-1); вернуть значение (пустой лог → 0.0).
  • recent_errors(n=5) — последние n записей с ok=False.

Готово, когда все тесты в test_monitoring.py зелёные — они проверяют: error_rate = 0.2 (1 из 5); latency_percentile(95) = 1500 (ловит хвост); recent_errors собирает таймаут; пустой логгер безопасен (0.0).

pytest code -q          # красное → реализуй CallLogger → зелёное
python code/monitoring.py # демо: error_rate 0.2, p95 1500, последние ошибки

Подсказка. Перцентиль: values = sorted(...), k = max(0, min(len(values)-1, round(p/100*len(values)) - 1)). Это nearest-rank — простой и достаточный для мониторинга; точные перцентили считают t-digest, но для алертов хватает.

USE IT

Запуск процесса в проде с observability (мульти-провайдер):

  • OpenTelemetry GenAI — семантические конвенции gen_ai.* (модель, токены, латентность, tool-calls): единый формат вместо самописных логов, читаемый любым observability-бэкендом без привязки к вендору. Это развивающийся стандарт (конвенции в статусе development и активно меняются — сверяйся с актуальной спекой), но даже сырой он лучше самописных полей: меньше лок-ина.
  • Langfuse / аналоги — трейсинг вызовов, ошибок и токенов; LLM-as-judge для оценки качества прямо на трафике (online-evals из 7.5).
  • Алерты — на error_rate, p95 latency и всплеск стоимости (FinOps из 9.4); редактируй PII до логирования контента.

Принцип: логируй в стандартном формате (gen_ai.*), меряй по перцентилям, алертай по порогам — тогда прод виден, а не угадывается.

ЧАСТЫЕ ОШИБКИ

  • Смотреть на среднюю латентность. Среднее прячет хвост: при average 0.4с каждый двадцатый запрос может висеть секунды. Меряй p95/p99 — по хвосту уходят пользователи.
  • Логировать неструктурно. «Строка в файл» не агрегируется. Пиши поля (модель, латентность, статус, ошибка), чтобы считать метрики и резать по разрезам.
  • Логировать контент без редактирования PII. Промпт/ответ могут содержать персданные. Контент — только с согласия и после маскирования PII (приватность, Фаза 11).
  • Пороги без алертов. Метрика, на которую никто не смотрит, бесполезна. Ставь алерты на error_rate, p95, всплеск стоимости — узнавать о проблеме до жалобы пользователя.
  • Самописный формат вместо стандарта. Свои поля привязывают к одному бэкенду. gen_ai.* (OpenTelemetry) читается любым инструментом — меньше лок-ина.

Тесты — это твоё ТЗ

code/test_monitoring.py
from monitoring import CallLogger


def _log():
    log = CallLogger()
    for ms, ok in [(120, True), (300, True), (90, True), (1500, False), (200, True)]:
        log.log("sonnet-4.6", ms, ok=ok, error=None if ok else "timeout")
    return log


def test_error_rate():
    assert _log().error_rate() == 0.2          # 1 из 5


def test_latency_percentile_returns_high_value():
    p95 = _log().latency_percentile(95)
    assert p95 == 1500                          # худшая латентность в хвосте


def test_recent_errors_captured():
    errs = _log().recent_errors()
    assert len(errs) == 1
    assert errs[0]["error"] == "timeout"


def test_empty_logger_safe():
    log = CallLogger()
    assert log.error_rate() == 0.0
    assert log.latency_percentile() == 0.0

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/monitoring.pyПоказать решение
"""Деплой и мониторинг: логирование вызовов — Build It для урока 10.2.

Без зависимостей. В проде каждый вызов модели логируется структурно (модель,
латентность, статус, ошибка), а поверх лога считаются метрики здоровья: доля
ошибок и перцентили латентности. Без этого «работает/не работает» — вслепую.
"""


class CallLogger:
    """Структурный лог вызовов + базовые метрики мониторинга."""

    def __init__(self):
        self.calls = []

    def log(self, model, latency_ms, ok=True, error=None):
        self.calls.append({"model": model, "latency_ms": latency_ms,
                           "ok": ok, "error": error})

    def error_rate(self):
        if not self.calls:
            return 0.0
        return round(sum(1 for c in self.calls if not c["ok"]) / len(self.calls), 3)

    def latency_percentile(self, p=95):
        """Приблизительный перцентиль латентности (nearest-rank)."""
        if not self.calls:
            return 0.0
        values = sorted(c["latency_ms"] for c in self.calls)
        k = max(0, min(len(values) - 1, round(p / 100 * len(values)) - 1))
        return values[k]

    def recent_errors(self, n=5):
        return [c for c in self.calls if not c["ok"]][-n:]


if __name__ == "__main__":
    log = CallLogger()
    for ms, ok in [(120, True), (300, True), (90, True), (1500, False), (200, True)]:
        log.log("sonnet-4.6", ms, ok=ok, error=None if ok else "timeout")
    print("error_rate:", log.error_rate())
    print("p95 latency:", log.latency_percentile(95))
    print("errors:", log.recent_errors())

# Артефакт: мониторинг-конфиг для ИИ-процесса

Что логировать и какие алерты ставить после деплоя. Код логгера — [`code/monitoring.py`](../code/monitoring.py).

## Что логировать на каждый вызов

```json
{
  "ts": "...", "process": "summarize", "model": "sonnet-4.6",
  "latency_ms": 320, "status": "ok", "error": null,
  "input_tokens": 1200, "output_tokens": 180, "cost": 0.0063,
  "prompt_version": "v3", "cache_hit": true
}
```

⚠️ Содержимое prompt/response логируй только с согласия и с **редактированием PII** (Фаза 11).

## Метрики и пороги алертов

| Метрика | Источник | Порог алерта |
|---|---|---|
| error_rate | `error_rate()` | > 2% (5 мин) |
| p95 latency | `latency_percentile(95)` | > 3000 ms |
| стоимость/час | телеметрия (10.3) | > бюджета (9.4) |
| cache-hit rate | лог | < ожидаемого (упустили экономию) |
| дрейф качества | LLM-judge (2.5) | падение score |

## Стандарты и инструменты

- **OpenTelemetry GenAI** (`gen_ai.*`) — единый формат спанов/метрик, без вендор-лока.
- **Langfuse / аналоги** — трейсинг, ошибки, токены, оценка качества.
- **Алерты** — в существующий on-call (Slack/PagerDuty).

## Правила

- Логируй **структурно** (JSON), не строками — иначе не построишь метрики.
- Считай `error_rate` и `p95`, а не только среднее (хвост важнее).
- Свяжи мониторинг (здоровье) с телеметрией (стоимость, 10.3) и FinOps (9.4).
