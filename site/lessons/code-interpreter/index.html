Урок 5.2 · Code Interpreter

Фаза 5 — Данные и аналитика · Результат фазы: Анализировать данные с ИИ, делать NL→SQL и автоотчёты с валидацией результатов.

Requires (только для USE IT): платный API-ключ или ChatGPT/Claude с исполнением кода. BUILD IT работает офлайн, без ключей и без сети.

В 5.1 ИИ строил план аналитики. Теперь — кто его исполняет. Code Interpreter — это среда, где модель пишет код, а изолированная песочница его запускает и возвращает результат. Разберём механизм и, главное, где проходит граница безопасности.

MOTTO. Code Interpreter = модель пишет код, песочница его исполняет и возвращает результат.

PROBLEM

LLM плохо считает «в уме»: арифметика на больших числах, статистика, точные агрегаты — всё это она угадывает и ошибается. Зато она отлично пишет код, который посчитает точно.

Code Interpreter замыкает цикл: модель генерирует Python → среда его исполняет → результат (или ошибка) возвращается модели, и она при необходимости себя исправляет. Но исполнять код, написанный моделью, в своём обычном процессе — опасно: один import os — и код ходит по твоей файловой системе. Поэтому ключевой вопрос не «как запустить», а «как запустить изолированно». Соберём учебную песочницу и честно увидим, где её границы.

CONCEPT

Интуиция

Представь стажёра, который считает быстро и без ошибок, но которому ты не доверяешь ключи от офиса. Ты сажаешь его в переговорку без интернета и доступа к серверам, даёшь задачу и калькулятор, забираешь результат — и не пускаешь к общим файлам. Песочница (sandbox) — это такая переговорка для кода модели: исполнить дай, но к системе, сети и секретам не подпускай.

Настоящий Code Interpreter крутится в одноразовом контейнере без сети, который уничтожается после сессии. Наша учебная версия слабее, но показывает тот же принцип: что коду можно, а что нельзя.

Как это работает

запрос → модель пишет код → [ sandbox: исполнить ] → результат/ошибка → модель
                                  │ только разрешённые builtins
                                  │ нет import / сети / файлов
                                  ▼ ошибка → модель сама себя чинит (self-correction)

Две линии защиты в нашей песочнице:

  • Статическая проверка (unsafe_tokens) — до запуска ищем в коде запрещённые конструкции (import, os., open(, __, eval( …) и отклоняем такой код вовсе.
  • Ограниченное окружение (run_code) — исполняем с урезанным списком builtins (только len, sum, sorted…), без доступа к опасным функциям. Результат код кладёт в переменную result.

⚠️ Честная оговорка: это не настоящая изоляция. Ограниченные builtins и поиск подстрок обходятся изобретательным кодом. Это демонстрация идеи; для прода нужен реальный sandbox — контейнер, seccomp, отдельный процесс с лимитами CPU/памяти (безопасность исполнения — урок 6.5).

РАЗБОР ПО ШАГАМ

Прогоним run_code на двух кусках кода — «хорошем» и «плохом».

Код 1 (считает среднее): result = sum(data) / len(data), data = [10, 20, 30].

1. unsafe_tokens(code) → []            запрещённого нет → пускаем дальше
2. exec в окружении {builtins: SAFE, data: [10,20,30]}
3. в local появилась переменная result = 60 / 3 = 20.0
4. return 20.0

Код 2 (лезет в систему): import os; result = os.listdir('.').

1. unsafe_tokens(code) → ["import", "os."]   найдены запрещённые конструкции
2. raise SandboxError — код НЕ исполняется вовсе

Видно главное: «хороший» код доходит до exec и возвращает точное число, а «плохой» отсекается до исполнения статической проверкой. А если код «чистый», но забыл присвоить result (например x = 5), песочница тоже ругается — контракт «результат кладётся в result» обязателен.

BUILD IT

Задание: собери учебную песочницу — статическую проверку + ограниченный исполнитель. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/5.2-code-interpreter/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3 (для теста ещё pytest).

Создай файл sandbox.py: исключение SandboxError, словарь SAFE_BUILTINS (безопасные функции вроде len, sum, min, max, sorted, range…), кортеж FORBIDDEN (запрещённые подстроки: import, __, open(, exec(, eval(, os., sys., subprocess…) и две функции:

  • unsafe_tokens(code) — вернуть список запрещённых подстрок из FORBIDDEN, найденных в code (пустой список = код чист).
  • run_code(code, data=None) — если unsafe_tokens непуст → raise SandboxError. Иначе исполнить exec(code, {"__builtins__": SAFE_BUILTINS}, local), где в local положен data (если передан). После исполнения: если переменной result нет в local → raise SandboxError; иначе вернуть local["result"].

Готово, когда все тесты в test_sandbox.py зелёные — они проверяют: разрешённый код считает (sum(data) = 6); import блокируется; __/open( ловятся; отсутствие result → ошибка; чистый код не содержит запрещённых токенов.

pytest code -q        # красное → реализуй sandbox → зелёное
python code/sandbox.py # демо: считает среднее; блокирует import os

Подсказка. unsafe_tokens — это [t for t in FORBIDDEN if t in code]. В run_code передавай __builtins__ как SAFE_BUILTINS, чтобы отрезать доступ к опасным функциям. Контракт результата: код обязан присвоить result.

ЧАСТЫЕ ОШИБКИ

  • Считать учебную песочницу безопасной. Ограниченные builtins и поиск подстрок обходятся. Реальная изоляция — контейнер/seccomp/процесс с лимитами (6.5); наша версия только иллюстрирует идею.
  • Исполнять код модели в своём процессе. Без изоляции один import os или open(...) даёт коду доступ к файлам и секретам. Код модели — всегда в песочнице.
  • Просить результат без кода. Если модель выдала число, но не показала код — проверить нельзя. Всегда требуй показать исполняемый код (и желательно данные).
  • Полагаться, что модель посчитает «в уме». Для арифметики/статистики пусть пишет и исполняет код — иначе получишь правдоподобную, но неверную цифру.
  • Забыть про сеть и побочные эффекты. Прод-песочницы режут сеть специально; код, который «сходит в API» внутри интерпретатора, в изоляции упадёт — это норма, а не баг.

Тесты — это твоё ТЗ

code/test_sandbox.py
import pytest
from sandbox import run_code, unsafe_tokens, SandboxError


def test_runs_allowed_code():
    assert run_code("result = sum(data)", data=[1, 2, 3]) == 6
    assert run_code("result = max(data) - min(data)", data=[5, 1, 9]) == 8


def test_blocks_import():
    with pytest.raises(SandboxError):
        run_code("import os\nresult = 1")


def test_blocks_dunder_and_open():
    assert unsafe_tokens("__import__('os')")
    with pytest.raises(SandboxError):
        run_code("result = open('/etc/passwd').read()")


def test_requires_result_variable():
    with pytest.raises(SandboxError):
        run_code("x = 5")        # result не задан


def test_clean_code_has_no_unsafe_tokens():
    assert unsafe_tokens("result = sorted(data)") == []

code/sandbox.pyПоказать решение
"""Code Interpreter: упрощённый sandbox-исполнитель — Build It для урока 5.2.

Без зависимостей. Модель пишет код — sandbox его исполняет и возвращает результат
(так работает Code Interpreter / Advanced Data Analysis). Здесь УЧЕБНЫЙ песок:
ограниченные builtins + статическая проверка опасных конструкций.

ВНИМАНИЕ: это не настоящая изоляция, а демонстрация идеи. Для прода нужен реальный
sandbox (контейнер, seccomp, отдельный процесс с лимитами) — см. урок 6.5.
"""

SAFE_BUILTINS = {
    "len": len, "sum": sum, "min": min, "max": max, "sorted": sorted, "range": range,
    "abs": abs, "round": round, "map": map, "filter": filter, "list": list,
    "dict": dict, "set": set, "tuple": tuple, "enumerate": enumerate, "zip": zip,
    "float": float, "int": int, "str": str, "bool": bool,
}

FORBIDDEN = ("import", "__", "open(", "exec(", "eval(", "compile(", "input(",
             "os.", "sys.", "subprocess", "globals(", "locals(", "getattr(")


class SandboxError(Exception):
    pass


def unsafe_tokens(code):
    """Список запрещённых конструкций, найденных в коде (пустой = чисто)."""
    return [t for t in FORBIDDEN if t in code]


def run_code(code, data=None):
    """Исполнить код в ограниченном окружении. Результат — переменная `result`."""
    bad = unsafe_tokens(code)
    if bad:
        raise SandboxError(f"запрещённые конструкции: {bad}")
    env = {"__builtins__": SAFE_BUILTINS}
    local = {"data": data} if data is not None else {}
    exec(code, env, local)            # noqa: S102 — учебный sandbox, см. оговорку выше
    if "result" not in local:
        raise SandboxError("код не присвоил переменную result")
    return local["result"]


if __name__ == "__main__":
    print(run_code("result = sum(data) / len(data)", data=[10, 20, 30]))   # среднее
    try:
        run_code("import os; result = os.listdir('.')")
    except SandboxError as e:
        print("Заблокировано:", e)

# Артефакт: набор аналитических промптов для Code Interpreter

Промпты под типовые задачи. Везде проси **показать код** — это аудируемость и воспроизводимость. Учебная песочница — [`code/sandbox.py`](../code/sandbox.py).

## Профилирование данных

```
Загрузи файл. Покажи: размер, типы колонок, доля пропусков по колонкам,
5 примеров строк. Отметь подозрительные значения. Покажи код.
```

## Агрегации

```
Посчитай выручку (SUM по amount) по region, топ-10, и долю каждого.
Проверь, что сумма по группам = общей. Покажи таблицу и код.
```

## Графики

```
Построй линейный график выручки по месяцам и bar-chart топ-5 продуктов.
Подпиши оси, добавь заголовки. Дай 3 вывода по графикам.
```

## Проверка гипотезы

```
Гипотеза: продажи в выходные выше будней. Проверь статистически
(сравни средние, отметь значимость), без переусложнения. Покажи код и вывод.
```

## Поиск аномалий

```
Найди выбросы в ряду daily_revenue (z-score или IQR). Перечисли даты-аномалии
со значениями и отклонением. Покажи код.
```

## Правила

- **Показывай код** (View Analysis / code execution) — без этого результат непроверяем.
- **Один вопрос — один промпт** (декомпозиция, урок 2.3); сложное — цепочкой.
- **Проверяй выводы** (урок 5.5): ИИ не знает происхождения данных.
- Низкая температура для расчётов (урок 1.3) — нужна воспроизводимость.
