Урок 2.3 · Декомпозиция и итеративное уточнение

Фаза 2 — Промпт-инжиниринг · Результат фазы: Писать воспроизводимые промпты и измерять их качество через eval-harness.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Этот урок — про то, как браться за задачи, которые не решаются одним промптом. Вместо мегапромпта «сделай всё сразу» сложный процесс разбивают на цепочку шагов, где выход одного — вход следующего. Каждый шаг прост, проверяем и отлаживаем по отдельности. Это и приём промптинга, и прямой мост к агентам (Фаза 7).

MOTTO. Сложную задачу не «промптят» одним мегапромптом, а разбивают на цепочку шагов.

PROBLEM

Один огромный промпт «сделай всё сразу» ненадёжен: модель роняет шаги, путает порядок, смешивает форматы — а отладить «почему плохо» невозможно, потому что всё слиплось в один ответ. Сложные процессы (исследование → черновик → правка → формат) так не делаются. Нужна декомпозиция: разбить задачу на цепочку промптов, каждый — с одной чёткой целью.

CONCEPT

Интуиция

Это та же идея, что и в коде: не пишут одну гигантскую функцию на 300 строк, а раскладывают на маленькие функции с одной ответственностью каждая. С промптами так же — один шаг делает одно дело, его легко сформулировать, проверить и переиспользовать. А если что-то сломалось, видно в каком именно шаге, а не «где-то в большом ответе».

Как это работает

вход → [шаг 1] → [шаг 2] → [шаг 3] → итог
          │ каждый шаг = отдельный промпт с одной целью
          ▼ выход шага виден и проверяем (трейс)
   типовые цепочки: Extract → Transform → Analyze → Format
                    Generate → Review → Improve  (self-correction)

Что даёт декомпозиция (по гайду Anthropic «Chain complex prompts»):

  • Точность. Каждый шаг в фокусе одной подзадачи — модель не распыляется и реже роняет требования.
  • Ясность. Простую инструкцию проще написать однозначно, чем мегапромпт на полстраницы.
  • Трассируемость. Выход каждого шага виден (трейс/история), поэтому понятно, где именно сломалось — и чинишь один шаг, а не весь промпт.

Две частые формы цепочек:

  • Конвейер Extract → Transform → Analyze → Format — данные последовательно преобразуются.
  • Self-correction Generate → Review → Improve — модель сначала генерирует, потом сама критикует свой ответ и улучшает его. Это и есть «итеративное уточнение»: тот же цикл Description ↔ Discernment из урока 2.1, но автоматизированный шагами.

Независимые шаги можно выполнять параллельно; последовательность нужна только там, где шаг зависит от выхода предыдущего. Когда между шагами появляются решения и инструменты — цепочка превращается в агента (цикл reason → act → observe, урок 7.1).

РАЗБОР ПО ШАГАМ

Возьмём задачу «посчитай сумму чисел из текста и оформи» — её один шаг не решает, а цепочка из трёх решает. Прогоним run_chain руками на входе "купил 2 яблока, 3 груши и 5 слив":

вход:    "купил 2 яблока, 3 груши и 5 слив"
extract: вытащить числа        → [2, 3, 5]
sum:     сложить               → 10
format:  оформить              → "Итого: 10"

run_chain на каждом переходе кладёт в history запись {step, in, out} — это и есть трейс: видно вход и выход каждого шага.

history = [
  {step: "extract", in: "купил 2 …",     out: [2, 3, 5]},
  {step: "sum",     in: [2, 3, 5],       out: 10},
  {step: "format",  in: 10,              out: "Итого: 10"},
]

А вот почему нужна вся цепочка: запусти только первый шаг — получишь [2, 3, 5], но не финальную строку. До результата "Итого: 10" доводит лишь композиция всех трёх шагов. И если бы, скажем, sum ошибся, по history сразу видно, что вход к нему был верный [2, 3, 5] — значит, чинить надо именно sum.

BUILD IT

Задание: собери раннер цепочки промптов с трейсом — стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/2.3-decomposition/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3, всё офлайн (для теста ещё pytest).

Демо-шаги бери готовыми (это пример цепочки; в проде вместо них — вызовы модели):

def _extract_numbers(text): return [int(w) for w in text.split() if w.lstrip("-").isdigit()]
def _sum(nums):             return sum(nums)
def _format(total):         return f"Итого: {total}"
DEMO_CHAIN = [ChainStep("extract", _extract_numbers), ChainStep("sum", _sum), ChainStep("format", _format)]

Реализуй:

  • ChainStep — шаг цепочки: name (строка) и fn (callable value -> value). Удобно как @dataclass.
  • run_chain(steps, initial, trace=False) — прогони initial через шаги по порядку: на каждом шаге out = step.fn(value), запиши в историю {"step": имя, "in": value, "out": out}, затем value = out. При trace=True печатай переход. Верни кортеж (итоговое значение, история).

Готово, когда тест test_prompt_chain.py зелёный — он проверяет, что цепочка композирует шаги (extract → sum → format даёт "Итого: …"), что история пишет вход/выход, что один шаг не решает то, что решает вся цепочка, и что работает произвольная цепочка.

pytest code -q                # красное → реализуй ChainStep и run_chain → зелёное
python code/prompt_chain.py   # демо: трейс extract → sum → format

Подсказка. run_chain — это цикл по steps: держи текущее value, на каждом шаге зови step.fn(value), дописывай запись в history и обновляй value. Верни (value, history).

Внизу, в «Исходниках урока», — три способа пройти упражнение (собрать самому · подсмотреть эталон · делегировать ИИ) и тесты-ТЗ.

USE IT

В проде каждый ChainStep.fn — отдельный вызов модели; выход подаётся в следующий промпт:

def step_llm(instruction):
    def fn(value):
        prompt = f"{instruction}\n\nВход:\n{value}"
        return Anthropic().messages.create(model="claude-sonnet-4-6", max_tokens=500,
            messages=[{"role": "user", "content": prompt}]).content[0].text
        # OpenAI: OpenAI().responses.create(model="gpt-5.x", input=prompt).output_text
        # Google: genai.Client().models.generate_content(model="gemini-2.x-flash", contents=prompt).text
    return fn

chain = [ChainStep("outline", step_llm("Составь план")),
         ChainStep("draft",   step_llm("Напиши черновик по плану")),
         ChainStep("edit",    step_llm("Отредактируй: короче и яснее"))]

Независимые шаги можно гонять параллельно; последовательно — только когда выход одного нужен следующему.

ЧАСТЫЕ ОШИБКИ

  • Мегапромпт вместо цепочки. «Сделай всё сразу» роняет шаги и не отлаживается. Если задача из нескольких этапов — дроби на шаги с одной целью каждый.
  • Дробить тривиальное. Простой одношаговый запрос не нужно разбивать на цепочку — это лишние вызовы, токены и задержка. Декомпозиция — для многошаговых задач.
  • Терять трейс. Без истории вход/выход непонятно, какой шаг сломался. Сохраняй промежуточные результаты — в этом половина пользы.
  • Гнать независимые шаги последовательно. Если шаг не зависит от предыдущего, его можно выполнить параллельно — последовательность нужна только при зависимости.

from prompt_chain import ChainStep, run_chain, DEMO_CHAIN


def test_chain_composes_steps():
    answer, history = run_chain(DEMO_CHAIN, "взял 2 и 3 и 5")
    assert answer == "Итого: 10"
    assert len(history) == 3
    assert [h["step"] for h in history] == ["extract", "sum", "format"]


def test_history_records_in_and_out():
    _, history = run_chain(DEMO_CHAIN, "1 и 4")
    assert history[0]["in"] == "1 и 4"
    assert history[-1]["out"] == "Итого: 5"


def test_single_step_cannot_solve_what_chain_solves():
    # один «extract» не доводит до финальной строки — нужна вся цепочка
    extract_only, _ = run_chain([DEMO_CHAIN[0]], "2 и 3")
    assert extract_only == [2, 3]
    full, _ = run_chain(DEMO_CHAIN, "2 и 3")
    assert full == "Итого: 5"


def test_custom_chain():
    steps = [ChainStep("upper", str.upper), ChainStep("bang", lambda s: s + "!")]
    out, _ = run_chain(steps, "ok")
    assert out == "OK!"

"""Декомпозиция: цепочка промптов — Build It для урока 2.3.

Без зависимостей. Сложную задачу разбиваем на шаги; выход каждого шага — вход
следующего. Здесь шаги — детерминированные функции (вместо вызова LLM), чтобы
урок шёл офлайн; в USE IT каждый шаг становится отдельным промптом к модели.
"""
from dataclasses import dataclass
from typing import Callable


@dataclass
class ChainStep:
    name: str
    fn: Callable          # callable(value) -> value


def run_chain(steps, initial, trace=False):
    """Прогнать значение через цепочку шагов. Возвращает (итог, история)."""
    value = initial
    history = []
    for step in steps:
        out = step.fn(value)
        history.append({"step": step.name, "in": value, "out": out})
        if trace:
            print(f"{step.name}: {value!r} -> {out!r}")
        value = out
    return value, history


# --- демонстрация декомпозиции: «посчитай сумму чисел из текста и оформи» ---
def _extract_numbers(text):
    return [int(w) for w in text.split() if w.lstrip("-").isdigit()]


def _sum(nums):
    return sum(nums)


def _format(total):
    return f"Итого: {total}"


DEMO_CHAIN = [
    ChainStep("extract", _extract_numbers),
    ChainStep("sum", _sum),
    ChainStep("format", _format),
]


if __name__ == "__main__":
    answer, history = run_chain(DEMO_CHAIN, "купил 2 яблока, 3 груши и 5 слив", trace=True)
    print("\nИтог:", answer, f"({len(history)} шага)")

# Артефакт: промпт-цепочка под рабочий процесс

Шаблон декомпозиции сложной задачи на цепочку шагов. Раннер — в [`code/prompt_chain.py`](../code/prompt_chain.py).

## Как описать свою цепочку

| # | Шаг (одна цель) | Вход | Выход | Проверка |
|---|---|---|---|---|
| 1 | extract / outline | сырой ввод | структура/план | непусто, формат ок |
| 2 | transform / draft | выход шага 1 | черновик | покрыты все пункты |
| 3 | review / edit | выход шага 2 | правки | критерии качества |
| 4 | format | выход шага 3 | финал | валидна схема (урок 2.4) |

## Типовые цепочки

- **Контент:** Research → Outline → Draft → Edit → Format
- **Данные:** Extract → Transform → Analyze → Format
- **Решение:** Gather → Options → Analyze → Recommend
- **Self-correction:** Generate → Review (по критериям) → Improve

## Правила

- Один шаг — **одна цель**; так модель не роняет подзадачи.
- Между шагами передавай **только нужное** (экономия контекста, урок 1.1).
- **Логируй трейс** каждого шага — это отладка и точка для метрик (урок 2.5).
- Независимые шаги — **параллель**; последовательно только при зависимости.
- Каждый шаг можно оценивать отдельным eval-набором.

## Как переиспользовать

```python
from prompt_chain import ChainStep, run_chain
chain = [ChainStep("outline", fn1), ChainStep("draft", fn2), ChainStep("edit", fn3)]
result, trace = run_chain(chain, initial=raw_input, trace=True)
```

Где `fnN` — вызов модели с промптом шага. Дальше цепочка превращается в агента (reason→act→observe, урок 7.1).
