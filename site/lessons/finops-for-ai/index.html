Урок 9.4 · FinOps для ИИ

Фаза 9 — Экономика и оркестрация моделей (FinOps) · Результат фазы: Спроектировать маршрутизацию моделей и посчитать бюджет «до/после».

Requires (только для USE IT): платный API-ключ / биллинг провайдера. BUILD IT работает офлайн, без ключей и без сети.

Это финал Фазы 9 — учёт. Маршрутизация и каскад (9.2–9.3) экономят, но как узнать, сработало ли и где именно уходят деньги? Без измерения оптимизация слепа. Этот урок — про FinOps-контур: тегирование, агрегация, бюджеты и ROI.

MOTTO. Нельзя оптимизировать то, что не измеряешь: тегируй запросы, считай по разрезам, меряй ROI.

PROBLEM

Ты включил роутер и кэш — счёт вроде упал. Но на сколько? Какая команда жжёт больше всех? Окупилась ли автоматизация процесса вообще? Без учёта это ощущения, а не цифры — и в следующем месяце расход неожиданно вырастет, а ты не поймёшь почему.

Нужен FinOps-контур: каждый запрос логируется с тегами (команда, модель, cache-hit, стоимость), расходы агрегируются по разрезам, на бюджеты ставятся алерты, а у автоматизации считается ROI «до/после». Это превращает «вроде дешевле» в управляемые числа. Соберём такой учёт.

CONCEPT

Интуиция

FinOps для ИИ — это бухгалтерия с аналитикой по статьям, а не одна строчка «потрачено $N». Бизнес не спрашивает «сколько всего ушло» — он спрашивает «кто потратил, на что, и окупилось ли». Поэтому каждый расход помечают тегами (как чек с категорией), а потом режут по разрезам: по команде, по модели, по фиче.

Три функции FinOps: видимость (куда уходят деньги), ответственность (кому отнести расход) и оптимизация (что улучшить). И метрика зрелости — не «cost per inference» (стоимость вызова), а cost per outcome (стоимость результата): дешёвый вызов, который не решил задачу, — это не экономия.

Как это работает

запрос → лог с тегами {team, model, cache_hit, cost}
              │ агрегация
   по команде / по модели / cache-hit rate
              │
   бюджет + алерт при превышении
   ROI = (экономия + выгода) / затраты   ← «до/после»
  • Ledger.record(cost, **tags) — записать расход с произвольными тегами.
  • by_tag(key) — суммарная стоимость в разрезе значения тега (по команде, по модели…).
  • cache_hit_rate() — доля запросов с попаданием в кэш (здоровье кэширования).
  • over_budget(limit) — флаг превышения бюджета (основа алерта).
  • roi(before, after, value) — окупаемость автоматизации: (сэкономлено + выгода) / затраты.

Ключ — теги: без них есть общая сумма, но нет attribution (кто и на что), а значит, нечего оптимизировать адресно.

РАЗБОР ПО ШАГАМ

Залогируем три запроса и снимем метрики:

record 0.02,  team=analytics, model=sonnet-4.6, cache_hit=False
record 0.002, team=analytics, model=haiku-4.5,  cache_hit=True
record 0.05,  team=support,   model=opus-4.8,   cache_hit=False

Агрегации:

total()              → 0.02 + 0.002 + 0.05 = 0.072
by_tag("team")       → {analytics: 0.022, support: 0.05}   ← кто тратит
by_tag("model")      → {sonnet: 0.02, haiku: 0.002, opus: 0.05}  ← на чём
cache_hit_rate()     → 1 из 3 с кэшем = 0.333
over_budget(0.05)    → 0.072 > 0.05 → True  ← алерт

И ROI автоматизации: процесс стоил 100, после ИИ стал стоить 20, плюс выгода 50:

roi(100, 20, 50): сэкономлено 80, выгода 80+50=130, /20 → 6.5

Видно три функции FinOps разом. Видимость: by_tag("model") показывает, что Opus у support — главная статья. Ответственность: by_tag("team") относит расход на команду. Оптимизация: низкий cache_hit_rate (0.333) — сигнал включить кэш шире, а over_budget — повод для алерта. ROI 6.5 переводит «дешевле» в «окупается в 6.5 раза». Без тегов всё это схлопнулось бы в одну цифру 0.072 — и оптимизировать было бы нечего.

ЧАСТЫЕ ОШИБКИ

  • Логировать без тегов. Общая сумма есть, attribution нет — нечего оптимизировать адресно. Тегируй командой/моделью/фичей/cache-hit с первого запроса.
  • Мерить cost per inference вместо cost per outcome. Дешёвый вызов, не решивший задачу, — не экономия. Метрика зрелости — стоимость результата, а не вызова.
  • Бюджет без алерта. Потолок, о превышении которого узнаёшь из месячного счёта, бесполезен. Алерт — при приближении, а не постфактум.
  • Считать расходы ИИ линейными. Они растут нелинейно с нагрузкой (длиннее контексты, больше шагов агентов). Форкастируй, а не экстраполируй прошлый месяц.
  • Экономить без ROI. Срезать расход в ущерб результату — ложная экономия. ROI «до/после» показывает, окупается ли автоматизация вообще, а не только «стало дешевле».

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/finops.pyПоказать решение
"""FinOps для ИИ: тегирование, учёт, ROI — Build It для урока 9.4.

Без зависимостей. Каждый запрос логируется с тегами (команда, модель, cache-hit) и
стоимостью; затем агрегируем по тегам, считаем ROI автоматизации и проверяем бюджет.
Нельзя оптимизировать то, что не измеряешь.
"""


class Ledger:
    """Журнал запросов с тегами и стоимостью."""

    def __init__(self):
        self.records = []

    def record(self, cost, **tags):
        self.records.append({"cost": cost, "tags": tags})

    def total(self):
        return round(sum(r["cost"] for r in self.records), 6)

    def by_tag(self, key):
        """Суммарная стоимость в разрезе значения тега key."""
        out = {}
        for r in self.records:
            val = r["tags"].get(key, "—")
            out[val] = round(out.get(val, 0.0) + r["cost"], 6)
        return out

    def cache_hit_rate(self):
        if not self.records:
            return 0.0
        hits = sum(1 for r in self.records if r["tags"].get("cache_hit"))
        return round(hits / len(self.records), 3)

    def over_budget(self, limit):
        return self.total() > limit


def roi(before_cost, after_cost, value_gained):
    """ROI автоматизации: (выгода − затраты) / затраты. before/after — стоимость процесса."""
    saved = before_cost - after_cost
    benefit = saved + value_gained
    return round(benefit / after_cost, 2) if after_cost else float("inf")


if __name__ == "__main__":
    led = Ledger()
    led.record(0.02, team="analytics", model="sonnet-4.6", cache_hit=False)
    led.record(0.002, team="analytics", model="haiku-4.5", cache_hit=True)
    led.record(0.05, team="support", model="opus-4.8", cache_hit=False)
    print("Итого:", led.total())
    print("По командам:", led.by_tag("team"))
    print("По моделям:", led.by_tag("model"))
    print("Cache hit rate:", led.cache_hit_rate())
    print("Над бюджетом 0.05?", led.over_budget(0.05))
    print("ROI (было 100, стало 20, выгода 50):", roi(100, 20, 50))

# Артефакт: бюджет-дашборд + расчёт ROI

Шаблон FinOps-контура для ИИ. Код учёта — [`code/finops.py`](../code/finops.py).

## Что тегировать в каждом запросе

```
{team, feature, model, cache_hit, input_tokens, output_tokens, cost}
```

Без тегов нет attribution — невозможно понять, куда уходят деньги.

## Разрезы дашборда

| Разрез | Метрика | Зачем |
|---|---|---|
| По команде/фиче | сумма $, доля | кому отнести расход |
| По модели | сумма $, доля вызовов | где гонять дешевле (роутинг 9.2) |
| Cache-hit rate | % попаданий в кэш | потенциал prompt caching (9.3) |
| Cost per outcome | $ / результат | главная метрика зрелости (не cost per inference) |
| Тренд/форкаст | $/день, прогноз | расходы ИИ нелинейны — нужен форкаст |

## Лимиты и алерты

- Потолок бюджета на команду/фичу → алерт при приближении (`over_budget`).
- Алерт на скачок (новая фича/модель может удвоить расход за ночь).

## Расчёт ROI «до/после»

```python
from finops import roi
# было: процесс стоил 100; стало с ИИ: 20; доп. выгода (скорость/качество): 50
print(roi(before_cost=100, after_cost=20, value_gained=50))   # 6.5
```

ROI = (экономия + выгода) / затраты. Это финальная метрика capstone (Фаза 12).

## Правила

- **Измеряй до оптимизации** — иначе не докажешь эффект.
- **Тегируй на входе**, агрегируй на выходе (`Ledger.by_tag`).
- **Cost per outcome > cost per inference** — считай стоимость результата, а не вызова.
- Связывай с маршрутизацией (9.2), каскадом и кэшем (9.3): дашборд показывает, сработали ли они.
