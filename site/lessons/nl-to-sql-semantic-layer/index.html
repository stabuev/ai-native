Урок 5.3 · NL→SQL и семантический слой

Фаза 5 — Данные и аналитика · Результат фазы: Анализировать данные с ИИ, делать NL→SQL и автоотчёты с валидацией результатов.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети (sqlite входит в Python, ставить не нужно).

В 5.1–5.2 данные жили в файле. Реальные данные живут в БД. «Спроси базу на человеческом языке» (NL→SQL) звучит мечтой — но без одной важной детали превращается в лотерею с цифрами. Эта деталь — семантический слой.

MOTTO. NL→SQL без семантического слоя — лотерея; метрики надо определить один раз и для всех.

PROBLEM

Просишь модель: «покажи выручку по регионам» — она пишет SQL. Завтра коллега просит то же — модель пишет другой SQL: один взял SUM(amount), другой вычел возвраты, третий отфильтровал тестовые заказы. Три «выручки» — три разных числа. Кому верить?

Проблема не в модели, а в неоднозначности: что такое «выручка»? «активный пользователь»? «месяц» — календарный или скользящий? Пока определения метрик живут в голове у каждого, NL→SQL гадает заново на каждый вопрос. Лекарство — зафиксировать смысл один раз и для всех в семантическом слое, а модель пусть лишь сопоставляет вопрос с готовыми определениями.

CONCEPT

Интуиция

Семантический слой — это глоссарий компании, превращённый в код. В нём раз и навсегда записано: «выручка» = SUM(amount), «заказы» = COUNT(*), «регион» = колонка region. Дальше никто не спорит, что считать выручкой — определение одно, проверенное, общее.

Роль модели при этом сужается и становится надёжной: она не изобретает SQL с нуля, а лишь узнаёт в вопросе, какую метрику и какой разрез ты имеешь в виду, и подставляет их из слоя. Меньше свободы у модели — меньше разнобоя в цифрах.

Как это работает

вопрос → [семантический слой: метрики + измерения] → SQL → результат
            выручка = SUM(amount)        регион = region
            заказы  = COUNT(*)           месяц  = month
                         │
   1. найти МЕТРИКУ по триггеру в вопросе  → её SQL-выражение
   2. найти ИЗМЕРЕНИЕ (разрез) → колонка для GROUP BY (опционально)
   3. собрать SELECT из проверенных кусков → исполнить в БД
  • SEMANTIC — словарь смысла: метрики (триггер-слово → SQL) и измерения (триггер → колонка).
  • nl_to_sql(question) — найти в вопросе метрику и (опционально) измерение, собрать SELECT (+GROUP BY, если есть разрез).
  • run_sql(sql, rows) — исполнить запрос на in-memory sqlite (стандартная библиотека).

Важная деталь: триггеры метрик — это стемы (выручк, заказ), а не целые слова. Иначе «выручка», «выручки», «выручке» не совпадут — морфология русского ломает наивный матч по точному слову. И порядок метрик важен: более специфичную (средний чек → AVG) проверяем до общей (выручк → SUM), иначе она перехватит совпадение.

РАЗБОР ПО ШАГАМ

Соберём SQL для вопроса «выручка по регионам». Семантический слой:

metrics:    "средний чек" → AVG(amount)   "выручк" → SUM(amount)   "заказ" → COUNT(*)
dimensions: "регион" → region             "месяц" → month
table:      sales

Шаги nl_to_sql:

1. метрика: "средний чек" в вопросе? нет → "выручк"? ДА → SUM(amount)
2. измерение: "регион" в вопросе? ДА → колонка region
3. собрать: SELECT region, SUM(amount) FROM sales GROUP BY region

Исполняем на данных и получаем по регионам: Москва → 150, Питер → 100, Казань → 20. Заметь: вопрос можно перефразировать («сколько выручки в разрезе регионов») — пока в нём есть стем выручк и регион, SQL соберётся тот же. Это и есть консистентность: смысл «выручки» задан слоем, а не настроением формулировки. А «какая погода завтра» не содержит ни одной метрики → nl_to_sql честно поднимет ошибку, а не сочинит SQL.

BUILD IT

Задание: собери NL→SQL поверх семантического слоя + исполнение на sqlite. Только стандартная библиотека (sqlite3), без сети.

Перед запуском. Работай в своей папке курса (ai-native/5.3-nl-to-sql/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3 (для теста ещё pytest).

Создай файл nl_to_sql.py:

  • SEMANTIC — словарь: table (имя таблицы), metrics (триггер-стем → SQL-выражение; более специфичные выше: средний чек→AVG(amount), выручк→SUM(amount), заказ→COUNT(*)), dimensions (триггер → колонка: регион→region, месяц→month).
  • nl_to_sql(question, semantic=SEMANTIC) — привести вопрос к нижнему регистру; найти первую метрику, чей триггер есть в вопросе (нет — raise ValueError); найти измерение (опционально). Если есть разрез — вернуть SELECT {col}, {metric} FROM {table} GROUP BY {col}, иначе SELECT {metric} FROM {table}.
  • run_sql(sql, rows) — создать in-memory sqlite с таблицей sales(region TEXT, month TEXT, amount REAL), вставить rows, выполнить sql, вернуть результат.

Готово, когда все тесты в test_nl_to_sql.py зелёные — они проверяют: метрика + разрез дают GROUP BY; метрика без разреза — без него; средний чек → AVG; неизвестная метрика → ValueError; run_sql реально считает (Москва = 150, Питер = 100).

pytest code -q          # красное → реализуй три части → зелёное
python code/nl_to_sql.py # демо: выручка по регионам, сколько заказов, средний чек

Подсказка. Поиск метрики: next((sql for name, sql in semantic["metrics"].items() if name in q), None) — порядок в словаре сохраняется, потому специфичные ставь выше. Триггеры держи стемами (выручк), чтобы ловить все формы слова.

ЧАСТЫЕ ОШИБКИ

  • NL→SQL без семантического слоя. Тогда «выручка» каждый раз новая, и цифры расходятся. Зафиксируй определения метрик один раз — это главное лекарство.
  • Триггеры целыми словами вместо стемов. выручка ≠ выручки ≠ выручке. Морфология ломает точный матч — храни триггеры стемами (выручк).
  • Неправильный порядок метрик. Общий триггер (выручк) перехватит специфичный (средний чек), если стоит выше. Специфичные — раньше.
  • Гнать сгенерированный SQL на прод без проверки. Модель может подставить не ту колонку или потерять фильтр. Показывай SQL и сверяй результат (5.5).
  • Считать NL→SQL решённой задачей. Это сложно (бенчмарк Spider): сложные джойны, вложенность, диалекты. Семантический слой снимает неоднозначность смысла, но не отменяет проверку.

import pytest
from nl_to_sql import nl_to_sql, run_sql, DEMO_ROWS


def test_metric_with_dimension_makes_group_by():
    assert nl_to_sql("выручка по регионам") == \
        "SELECT region, SUM(amount) FROM sales GROUP BY region"


def test_metric_without_dimension():
    assert nl_to_sql("сколько всего заказов") == "SELECT COUNT(*) FROM sales"


def test_avg_metric():
    assert nl_to_sql("какой средний чек по месяцам") == \
        "SELECT month, AVG(amount) FROM sales GROUP BY month"


def test_unknown_metric_raises():
    with pytest.raises(ValueError):
        nl_to_sql("какая погода завтра")


def test_run_sql_executes_aggregation():
    sql = nl_to_sql("выручка по регионам")
    result = dict(run_sql(sql, DEMO_ROWS))
    assert result["Москва"] == 150
    assert result["Питер"] == 100

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/nl_to_sql.pyПоказать решение
"""NL→SQL + семантический слой — Build It для урока 5.3.

Без внешних зависимостей (sqlite3 из stdlib). Семантический слой описывает метрики
и измерения в терминах бизнеса; наивный NL→SQL по ключевым словам собирает запрос
и исполняет его на sqlite. В USE IT разбор естественного языка делает LLM, но
семантический слой (определения метрик) остаётся — он гарантирует единый смысл.
"""
import sqlite3

SEMANTIC = {
    "table": "sales",
    "metrics": {                        # ключ — стем-триггер (подстрока в вопросе)
        "средний чек": "AVG(amount)",   # порядок важен: более специфичные — выше
        "выручк": "SUM(amount)",
        "заказ": "COUNT(*)",
    },
    "dimensions": {"регион": "region", "месяц": "month"},
}


def nl_to_sql(question, semantic=SEMANTIC):
    """Собрать SQL по вопросу: найти метрику и (опц.) измерение для GROUP BY."""
    q = question.lower()
    metric_sql = next((sql for name, sql in semantic["metrics"].items() if name in q), None)
    if metric_sql is None:
        raise ValueError("не нашёл метрику в вопросе")
    dim_col = next((col for name, col in semantic["dimensions"].items() if name in q), None)
    table = semantic["table"]
    if dim_col:
        return f"SELECT {dim_col}, {metric_sql} FROM {table} GROUP BY {dim_col}"
    return f"SELECT {metric_sql} FROM {table}"


def run_sql(sql, rows):
    """Создать in-memory таблицу sales(region, month, amount) и выполнить SQL."""
    con = sqlite3.connect(":memory:")
    con.execute("CREATE TABLE sales (region TEXT, month TEXT, amount REAL)")
    con.executemany("INSERT INTO sales VALUES (?,?,?)", rows)
    result = con.execute(sql).fetchall()
    con.close()
    return result


DEMO_ROWS = [
    ("Москва", "jan", 100), ("Москва", "feb", 50),
    ("Питер", "jan", 30), ("Питер", "feb", 70), ("Казань", "jan", 20),
]

if __name__ == "__main__":
    for q in ["выручка по регионам", "сколько заказов", "средний чек"]:
        sql = nl_to_sql(q)
        print(f"{q!r}\n  SQL: {sql}\n  ->  {run_sql(sql, DEMO_ROWS)}")

# Артефакт: семантический слой проекта

Единые определения метрик и измерений — чтобы «выручка» везде значила одно. Прототип NL→SQL — [`code/nl_to_sql.py`](../code/nl_to_sql.py).

## Таблица(ы)

```
sales(region TEXT, month TEXT, product TEXT, amount REAL, returned REAL)
```

## Метрики

| Метрика | Определение (SQL) | Синонимы-триггеры |
|---|---|---|
| Выручка | `SUM(amount)` | выручк, оборот, продаж |
| Чистая выручка | `SUM(amount - returned)` | чистая выручк, net |
| Заказы | `COUNT(*)` | заказ, число продаж |
| Средний чек | `AVG(amount)` | средний чек, avg |

## Измерения

| Измерение | Колонка | Синонимы |
|---|---|---|
| Регион | `region` | регион, город |
| Месяц | `month` | месяц, period |
| Продукт | `product` | продукт, товар, sku |

## Как использовать

```python
from nl_to_sql import nl_to_sql, run_sql
sql = nl_to_sql("чистая выручка по регионам")   # → SELECT region, SUM(amount-returned) ...
rows = run_sql(sql, data)
```

## Правила

- **Определи метрику один раз** — все инструменты (NL→SQL, BI-агенты, дашборды) берут отсюда.
- **Спорные метрики уточняй** (выручка с возвратами или без?) — здесь это видно явно.
- **Синонимы-триггеры** покрывают разные формулировки вопроса (учти морфологию — бери стемы).
- В проде это `dbt Semantic Layer` / Cube; принцип тот же — единый контракт смысла.
- Сгенерированный SQL **всегда показывай и проверяй** (урок 5.5).
