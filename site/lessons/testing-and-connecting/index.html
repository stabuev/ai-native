Урок 6.4 · Тест и подключение

Фаза 6 — Инструменты и протоколы (MCP) · Результат фазы: Объяснить tool use изнутри и поднять собственный MCP-сервер с контролем доступа.

Requires (только для USE IT): Node/npx (MCP Inspector) и клиент (Claude Desktop / Cursor). BUILD IT работает офлайн, без зависимостей и без сети.

Сервер из 6.3 написан — но работает ли он по протоколу? Прежде чем подключать его к Claude или Cursor, сервер тестируют изолированно. Иначе при ошибке непонятно, что виновато: сам сервер, транспорт или конфиг клиента.

MOTTO. Сначала прогони сервер инспектором, потом подключай к клиенту — не наоборот.

PROBLEM

Классика: «подключил свой сервер к Claude Desktop — инструменты не появились». Где сломалось? Сервер не отвечает на initialize? Транспорт не настроен? Опечатка в конфиге клиента? Когда переменных три, отладка превращается в гадание.

Решение — разделить проверки. Сначала прогнать сервер по протоколу в изоляции (это делает MCP Inspector): отвечает ли он на initialize, отдаёт ли tools/list со схемами, корректно ли возвращает ошибки. Зелёный отчёт → проблема не в сервере, и можно спокойно подключать к клиенту. Соберём мини-инспектор и поймём, что именно он проверяет.

CONCEPT

Интуиция

Инспектор — это стендовая проверка перед установкой, как прозвонить плату до того, как ставить её в корпус. Ты не подключаешь непроверенный сервер к «живому» клиенту и не гадаешь по симптомам — ты гоняешь сервер по чек-листу протокола в стерильных условиях и получаешь отчёт «конформен / вот что не так». Сломанное видно сразу и в одном месте.

Как это работает

Инспектору достаточно, чтобы сервер был функцией handle(request) -> response (JSON-RPC из 6.3). Он шлёт типовые запросы и проверяет ответы:

сервер handle(request)
   │ инспектор гоняет протокол:
   ├─ initialize         → есть result? serverInfo? protocolVersion?
   ├─ tools/list         → список? у инструментов есть name + inputSchema?
   ├─ неизвестный метод  → корректный error с кодом -32601?
   └─ tools/call (опц.)  → result содержит content?
   ▼
отчёт: [(проверка, ok), ...] + общий вердикт → потом подключение к клиенту

Каждая проверка — пара «что проверяли / прошло ли». Общий вердикт ok — это «все проверки зелёные». Важно: инспектор проверяет соответствие протоколу, а не бизнес-логику; он ловит «сервер отвечает неправильно по форме», а не «инструмент посчитал не то».

РАЗБОР ПО ШАГАМ

Прогоним inspect(demo_handle, call=("ping", {})) по конформному серверу:

[OK] initialize → result            (вернулся непустой result)
[OK] есть serverInfo                 (сервер представился)
[OK] есть protocolVersion            (объявил версию протокола)
[OK] tools/list → список             (отдал массив инструментов)
[OK] у инструментов есть name + inputSchema
[OK] неизвестный метод → error -32601
[OK] tools/call ping → content       (результат обёрнут в content)
→ ok: True  (сервер конформен)

Теперь «сломанный» сервер, который на всё отвечает {"result": {}}:

[FAIL] есть serverInfo               (initialize пустой)
[FAIL] tools/list → список           (нет tools)
[FAIL] неизвестный метод → error -32601  (вернул result вместо error)
→ ok: False  (есть проблемы)

Отчёт точно указывает, что не так, ещё до подключения к Claude. Это и есть смысл: локализовать поломку в сервере, прежде чем добавлять переменные транспорта и клиента.

BUILD IT

Задание: собери мини-инспектор протокола — функцию, гоняющую любой MCP-сервер по чек-листу. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/6.4-testing/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл mcp_inspector.py:

  • inspect(handle, call=None) — прогнать сервер handle и вернуть {"checks": [(label, ok), ...], "ok": bool}. Проверки: initialize даёт непустой result; в нём есть serverInfo и protocolVersion; tools/list возвращает список; у каждого инструмента есть name и inputSchema; неизвестный метод возвращает error с кодом -32601; если задан call=(name, args) — tools/call возвращает result с content. Общий ok = все проверки прошли.
  • demo_handle(request) — минимально конформный сервер (отвечает на initialize / tools/list / tools/call, на прочее — error -32601), чтобы было что инспектировать.

Готово, когда все тесты в test_mcp_inspector.py зелёные — они проверяют: конформный сервер проходит (ok is True); сломанный (без serverInfo и обработки ошибок) — нет, и в отчёте видно, какая проверка упала; чек-лист покрывает initialize/tools/list/-32601.

pytest code -q             # красное → реализуй inspect → зелёное
python code/mcp_inspector.py # демо: отчёт по конформному серверу

Подсказка. Собери хелпер _req(method, rid, params) для запросов. Каждую проверку добавляй парой (label, bool) в список checks; в конце ok = all(ok for _, ok in checks). Инспектор не должен падать — он фиксирует False, а не бросает исключение.

ЧАСТЫЕ ОШИБКИ

  • Подключать к клиенту непроверенный сервер. Тогда при ошибке три подозреваемых сразу (сервер/транспорт/конфиг). Сначала изолированный инспектор — он убирает две переменные.
  • Путать тест протокола и тест логики. Инспектор проверяет форму ответов (есть serverInfo, корректный код ошибки), а не правильность вычислений. Бизнес-логику покрывай отдельными тестами.
  • Игнорировать проверку ошибок. Сервер, который на неизвестный метод отвечает result вместо error -32601, формально «работает», но ломает клиентов. Ошибочные пути тоже конформность.
  • Тестировать сразу через UI клиента. Логи клиента скупы. Инспектор показывает сырые запросы/ответы — отлаживать на нём быстрее.
  • Считать зелёный инспектор гарантией безопасности. Он про корректность протокола, не про доступ. Что сервер отдаёт модели и кому он доступен — отдельный вопрос (6.5).

from mcp_inspector import inspect, demo_handle


def test_conformant_server_passes():
    report = inspect(demo_handle, call=("ping", {}))
    assert report["ok"] is True


def test_broken_server_fails():
    # сервер без serverInfo и без обработки неизвестного метода
    def broken(request):
        return {"jsonrpc": "2.0", "id": request.get("id"), "result": {}}
    report = inspect(broken)
    assert report["ok"] is False
    names = dict(report["checks"])
    assert names["есть serverInfo"] is False


def test_checks_cover_protocol_basics():
    labels = [name for name, _ in inspect(demo_handle)["checks"]]
    assert any("initialize" in s for s in labels)
    assert any("tools/list" in s for s in labels)
    assert any("-32601" in s for s in labels)

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/mcp_inspector.pyПоказать решение
"""Тест и подключение MCP: мини-инспектор — Build It для урока 6.4.

Без зависимостей. Инспектор гоняет MCP-сервер по протоколу (initialize →
tools/list → tools/call) и проверяет, что ответы валидны — как делает MCP
Inspector перед подключением сервера к Claude Desktop / Cursor. Работает с любым
сервером вида `handle(request) -> response` (JSON-RPC 2.0).
"""


def _req(method, rid=1, params=None):
    r = {"jsonrpc": "2.0", "id": rid, "method": method}
    if params is not None:
        r["params"] = params
    return r


def inspect(handle, call=None):
    """Проверить соответствие протоколу. call=(name, args) — опц. проверка вызова.

    Возвращает {checks: [(имя, ok)], ok: bool}.
    """
    checks = []

    init = handle(_req("initialize")).get("result", {})
    checks.append(("initialize → result", bool(init)))
    checks.append(("есть serverInfo", "serverInfo" in init))
    checks.append(("есть protocolVersion", "protocolVersion" in init))

    tools = handle(_req("tools/list")).get("result", {}).get("tools")
    checks.append(("tools/list → список", isinstance(tools, list)))
    checks.append(("у инструментов есть name + inputSchema",
                   bool(tools) and all("name" in t and "inputSchema" in t for t in tools)))

    unknown = handle(_req("no/such/method"))
    checks.append(("неизвестный метод → error -32601",
                   unknown.get("error", {}).get("code") == -32601))

    if call:
        name, args = call
        res = handle(_req("tools/call", params={"name": name, "arguments": args}))
        checks.append((f"tools/call {name} → content",
                       "content" in res.get("result", {})))

    return {"checks": checks, "ok": all(ok for _, ok in checks)}


# --- демо-сервер (минимально конформный) ---
def demo_handle(request):
    rid, method = request.get("id"), request.get("method")
    if method == "initialize":
        return {"jsonrpc": "2.0", "id": rid, "result": {
            "protocolVersion": "2024-11-05", "serverInfo": {"name": "demo", "version": "0.1"}}}
    if method == "tools/list":
        return {"jsonrpc": "2.0", "id": rid, "result": {"tools": [
            {"name": "ping", "inputSchema": {"type": "object", "properties": {}}}]}}
    if method == "tools/call":
        return {"jsonrpc": "2.0", "id": rid,
                "result": {"content": [{"type": "text", "text": "pong"}]}}
    return {"jsonrpc": "2.0", "id": rid, "error": {"code": -32601, "message": "not found"}}


if __name__ == "__main__":
    report = inspect(demo_handle, call=("ping", {}))
    for name, ok in report["checks"]:
        print(f"  [{'OK' if ok else 'FAIL'}] {name}")
    print("Итог:", "сервер конформен" if report["ok"] else "есть проблемы")

# Артефакт: документированный MCP-сервер с примерами

Как протестировать сервер и подключить к клиенту. Инспектор — [`code/mcp_inspector.py`](../code/mcp_inspector.py); сервер — урок 6.3.

## 1. Тест протокола (свой инспектор)

```python
from mcp_inspector import inspect
from mini_mcp_server import server          # из урока 6.3
report = inspect(server.handle, call=("add", {"a": 2, "b": 3}))
assert report["ok"], report["checks"]
```

## 2. Тест готовым MCP Inspector

```bash
mcp dev server.py                                          # Python SDK: поднимет инспектор
npx @modelcontextprotocol/inspector node build/index.js   # любой сервер; UI на :6274
```

Проверь вкладки: **Tools** (вызови с аргументами), **Resources**, **Prompts**, **Logs** (реальные JSON-RPC сообщения).

## 3. Подключение к клиенту

```bash
mcp install server.py --name my-server -v API_KEY=...      # в Claude Desktop
```

- **Claude Desktop / Claude Code** — сервер появится в списке инструментов после рестарта клиента.
- **Cursor** — добавить сервер в настройки MCP.
- Транспорт: STDIO (локально), Streamable HTTP (удалённо).

## Примеры запросов (JSON-RPC)

```json
{"jsonrpc":"2.0","id":1,"method":"initialize"}
{"jsonrpc":"2.0","id":2,"method":"tools/list"}
{"jsonrpc":"2.0","id":3,"method":"tools/call","params":{"name":"add","arguments":{"a":2,"b":3}}}
```

## Типовые ошибки

| Симптом | Причина | Решение |
|---|---|---|
| `-32601` | метод не реализован | проверь обработку initialize/tools-list/tools-call |
| Сервер не виден в клиенте | неверный конфиг/транспорт | сверь путь и transport; рестарт клиента |
| Таймаут на вызове | долгий tool | увеличь timeout инспектора/клиента |
| Пустой tools/list | инструменты не зарегистрированы | проверь декораторы `@tool` |
