Урок 5.4 · BI-агенты

Фаза 5 — Данные и аналитика · Результат фазы: Анализировать данные с ИИ, делать NL→SQL и автоотчёты с валидацией результатов.

Requires (только для USE IT): платный API-ключ или доступ к BI-платформе (Genie/Spotter/Hex). BUILD IT работает офлайн, без ключей и без сети.

До сих пор один вопрос = один ответ. Но настоящие бизнес-вопросы так не решаются: «почему упала выручка?» — это не запрос, а расследование. BI-агент ведёт его сам: смотрит общую картину, разбивает по разрезам, находит виновника. Это первый агентный цикл в курсе — на знакомых данных.

MOTTO. BI-агент не отвечает одним запросом, а ведёт расследование: обзор → разбивка → причина.

PROBLEM

«Почему упала выручка?» — одним SQL не ответишь. Нужна цепочка: посмотреть общий тренд → разбить по регионам/продуктам/времени → найти сегмент, который тянет вниз → копнуть в него глубже. Человек-аналитик делает это руками, шаг за шагом, решая после каждого результата, куда смотреть дальше.

BI-агент автоматизирует именно последовательность шагов, а не отдельный запрос. Вопрос «как выбрать следующий шаг по результату предыдущего» — это уже агентная логика. Соберём планировщик расследования и увидим, как агент идёт от общего к конкретному.

CONCEPT

Интуиция

BI-агент работает как детектив, а не как справочник. Справочник отвечает на один вопрос одним фактом. Детектив начинает с общей картины, сужает круг подозреваемых, идёт по следу к виновнику — каждый следующий шаг зависит от того, что нашли на предыдущем.

В аналитике «следствие» — это типовая воронка: сначала обзор (какой итог вообще), потом разбивка по измерению (кто из сегментов сколько даёт), потом топ-сегмент (кто доминирует и с какой долей). Дальше можно копнуть в победителя глубже. Так общий вопрос превращается в конкретный ответ «вот этот сегмент и вот почему».

Как это работает

вопрос → policy выбирает шаг → исполнить → наблюдение → следующий шаг
   overview → breakdown(по измерению) → top_segment(доля) → [drill-down]

Это агентный цикл reason → act → observe (подробно разберём в Фазе 7), но над данными:

  • overview — посчитать общий итог метрики (отправная точка).
  • breakdown — разбить итог по измерению (split-apply-combine из 5.1): {сегмент: число}.
  • top_segment — найти доминирующий сегмент и его долю в итоге (где сконцентрирована метрика).

В нашем уроке policy (правило выбора шага) детерминированная — фиксированная воронка из трёх шагов. В проде следующий шаг выбирает LLM по наблюдению: увидел перекос в регионе — разбивает регион по продуктам, и так далее. Механика одна, разница — кто решает «куда дальше».

РАЗБОР ПО ШАГАМ

Прогоним investigate по продажам, разрез — region, метрика — amount:

Москва 150   Питер 100   Казань 20

Три шага плана:

1. overview     → total = 150 + 100 + 20 = 270
2. breakdown    → {Москва: 150, Питер: 100, Казань: 20}   (по region)
3. top_segment  → max = Москва (150); share = 150 / 270 = 0.556

Вывод расследования: большую часть выручки (55.6%) даёт Москва. От «сколько всего» (270) агент за три шага пришёл к «кто доминирует и насколько» — не одним числом, а цепочкой, которую видно и можно проверить. Дальше человек (или LLM) решает: копнуть в Москву по продуктам или сравнить динамику — это и есть drill-down.

BUILD IT

Задание: собери BI-агента-планировщика — функцию investigate, строящую план расследования из трёх шагов. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/5.4-bi-agents/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл bi_agent.py с функцией investigate(rows, dimension, value), которая возвращает список из трёх шагов-словарей:

  • overview — {"step": "overview", "total": <сумма value по всем строкам>}.
  • breakdown — {"step": "breakdown", "by": dimension, "values": {сегмент: сумма value}} (агрегация по dimension; пригодится хелпер _aggregate).
  • top_segment — {"step": "top_segment", "segment": <ключ с макс. суммой>, "value": <его сумма>, "share": round(value/total, 3)} (если total == 0 → доля 0.0).

Готово, когда все тесты в test_bi_agent.py зелёные — они проверяют: ровно три шага в порядке overview→breakdown→top_segment; total = 270; breakdown даёт Москва = 150; top_segment = Москва со share = round(150/270, 3).

pytest code -q        # красное → реализуй investigate → зелёное

Подсказка. Итог — sum(float(r[value]) for r in rows). Агрегация по измерению — словарь out[r[dimension]] += float(r[value]). Топ-сегмент — max(by_dim.items(), key=lambda kv: kv[1]).

USE IT

Готовые BI-агенты ведут расследование поверх семантического слоя (5.3) (обзор и практика):

  • Databricks AI/BI Genie — NL-вопросы к данным под управлением Unity Catalog: возвращает SQL-запрос, таблицу-результат и график. Аналитик курирует пространство «доверенными активами» — примерами SQL-запросов, SQL-выражениями для бизнес-семантики и текстовыми инструкциями под терминологию команды. Это ровно семантический слой из 5.3, поднятый в проде.
  • ThoughtSpot Spotter — агент-аналитик поверх семантической модели (оперирует токенами метрик, а не прямой генерацией SQL), с human-in-the-loop коучингом.
  • Hex — Notebook Agent: NL→SQL/Python прямо в ноутбуке, с диффом и ревью ячеек.

Общее у всех: семантический слой (5.3) + планирование шагов + объяснимость и governance. ⚠️ Агент ускоряет расследование, но выводы проверяет человек (5.5) — агент может пойти по ложному следу.

ЧАСТЫЕ ОШИБКИ

  • Ждать ответа одним запросом. «Почему упало» требует цепочки шагов, а не одного SQL. BI-агент планирует расследование, а не отвечает фактом.
  • Останавливаться на топ-сегменте. Доминирующий сегмент — гипотеза, а не вывод. Доля 55.6% у Москвы говорит «где искать», но причину покажет drill-down внутрь.
  • Доверять цепочке агента вслепую. Агент может пойти по ложному следу (взять не то измерение, перепутать причину и следствие). Вывод проверяет человек (5.5).
  • BI-агент без семантического слоя. Тогда каждый шаг считает метрику по-своему (5.3), и расследование рассыпается. Сначала единые определения, потом агент поверх.
  • Путать долю и причинность. Большая доля сегмента не значит, что он «виноват» в падении — это лишь концентрация метрики. Падение мог дать маленький, но резко просевший сегмент.

from bi_agent import investigate, DEMO


def test_plan_has_three_steps():
    steps = investigate(DEMO, "region", "amount")
    assert [s["step"] for s in steps] == ["overview", "breakdown", "top_segment"]


def test_overview_total_is_sum():
    steps = investigate(DEMO, "region", "amount")
    assert steps[0]["total"] == 270


def test_breakdown_aggregates_by_dimension():
    steps = investigate(DEMO, "region", "amount")
    assert steps[1]["values"]["Москва"] == 150


def test_top_segment_and_share():
    top = investigate(DEMO, "region", "amount")[2]
    assert top["segment"] == "Москва"
    assert top["value"] == 150
    assert top["share"] == round(150 / 270, 3)

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/bi_agent.pyПоказать решение
"""BI-агент: выбирает шаги анализа — Build It для урока 5.4.

Без зависимостей. Агент-планировщик строит план BI-расследования (обзор →
разбивка по измерению → топ-сегмент с долей) и исполняет его на данных.
Policy детерминированная (офлайн); в USE IT шаги выбирает LLM (Genie/Spotter/Hex).
"""


def _aggregate(rows, group_by, value):
    out = {}
    for r in rows:
        out[r[group_by]] = out.get(r[group_by], 0.0) + float(r[value])
    return out


def investigate(rows, dimension, value):
    """План расследования: overview → breakdown → top_segment. Возвращает шаги."""
    total = sum(float(r[value]) for r in rows)
    by_dim = _aggregate(rows, dimension, value)
    top_key, top_val = max(by_dim.items(), key=lambda kv: kv[1])
    return [
        {"step": "overview", "total": total},
        {"step": "breakdown", "by": dimension, "values": by_dim},
        {"step": "top_segment", "segment": top_key, "value": top_val,
         "share": round(top_val / total, 3) if total else 0.0},
    ]


DEMO = [
    {"region": "Москва", "amount": "150"},
    {"region": "Питер", "amount": "100"},
    {"region": "Казань", "amount": "20"},
]

if __name__ == "__main__":
    for step in investigate(DEMO, "region", "amount"):
        print(step)

Артефакт (Ship It)

outputs/bi-investigation-playbook.md
# Артефакт: плейбук BI-расследования

Сценарий, по которому агент (или аналитик) ведёт расследование от вопроса к причине. Планировщик — [`code/bi_agent.py`](../code/bi_agent.py).

## Шаги расследования

```
1. OVERVIEW    общий тренд метрики во времени (вырос/упал, насколько)
2. BREAKDOWN   разбивка по измерениям (регион, продукт, канал)
3. TOP_SEGMENT сегмент с наибольшим вкладом/просадкой + его доля
4. DRILL-DOWN  углубиться в виновный сегмент (ещё разрез)
5. HYPOTHESIS  гипотеза причины (сезонность, акция, отвал клиента)
6. VALIDATE    проверить гипотезу на данных (урок 5.5)
7. CONCLUSION  вывод + рекомендация (что делать)
```

## Пример: «почему упала выручка в Q2?»

| Шаг | Действие | Находка |
|---|---|---|
| overview | выручка по месяцам | −18% к Q1 |
| breakdown | по регионам | падение в одном регионе |
| top_segment | доля региона | −60% вклада в просадку |
| drill-down | по продуктам региона | один SKU обвалился |
| hypothesis | что случилось с SKU | закончился сток |
| validate | проверка остатков | подтверждено |

## Как запустить (прототип)

```python
from bi_agent import investigate
for step in investigate(rows, dimension="region", value="amount"):
    print(step)     # overview → breakdown → top_segment
```

## Правила

- **Сужай поиск пошагово** — от общего к сегменту-виновнику, не «всё сразу».
- **Каждый шаг — проверяемая находка** (трейс), а не вывод «из головы».
- **Гипотезу валидируй на данных** (урок 5.5), прежде чем включать в вывод.
- В проде шаги выбирает LLM (Genie/Spotter/Hex) поверх семантического слоя (5.3); governance и human-in-the-loop обязательны.
