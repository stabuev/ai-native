Урок 1.3 · Сэмплирование, температура, детерминизм

Фаза 1 — Как работают LLM · Результат фазы: Объяснить токены, контекст и inference; собрать токен-бюджетер; отличать надёжный ответ от галлюцинации.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Этот урок объясняет, откуда берётся «случайность» в ответах модели — и как ею управлять. В уроке 1.2 модель на каждом шаге выдавала распределение вероятностей по токенам, а мы просто брали самый вероятный (greedy). Но в проде токен обычно выбирают случайно по этому распределению, и три параметра — temperature, top_k, top_p — задают, насколько выбор предсказуемый или творческий. Поймёшь их математику — и сможешь осознанно делать вывод то строго воспроизводимым, то разнообразным.

MOTTO. Модель выдаёт распределение, а не ответ; температура и top-p решают, как из него выбрать.

PROBLEM

«Почему один и тот же промпт даёт то одинаковые, то разные ответы?» и «как сделать вывод воспроизводимым?» — без понимания сэмплинга это магия. На каждом шаге модель отдаёт распределение вероятностей по всем токенам словаря. Что с ним сделать дальше — взять максимум или бросить кубик, и насколько «честный» этот кубик — решают параметры выбора. Не понимая их, ты не можешь ни зафиксировать ответ для классификации/извлечения, ни осознанно «развязать» модель для брейншторма.

CONCEPT

Интуиция

Модель не «знает ответ» — она оценивает, насколько вероятен каждый следующий токен, и отдаёт всё распределение. Дальше есть выбор:

  • greedy (как в 1.2) — всегда берём самый вероятный токен. Предсказуемо, но однообразно и склонно к повторам.
  • сэмплирование — бросаем «кубик», взвешенный по вероятностям: вероятный токен выпадает чаще, но и у редких есть шанс. Отсюда разнообразие — и невоспроизводимость без фиксации seed.

temperature, top_k, top_p — это «настройки кубика»: они меняют само распределение или сужают набор кандидатов перед броском.

Как это работает

Дальше будут формулы, но пугаться их не нужно: каждую мы разберём на конкретных числах по шагам — достаточно читать разбор, выводить что-то самому не надо.

Сырые «очки» модели называются логитами (logits) — это ещё не вероятности (могут быть любыми числами). В вероятности их переводит softmax:

            exp(logit_i / T)
p_i =  ─────────────────────────
        Σ_j  exp(logit_j / T)

temperature (T) делит логиты перед возведением в экспоненту — и этим управляет «остротой» распределения:

  • T = 1 — распределение «как есть».
  • T < 1 — делим на маленькое число, разрывы между логитами растут → распределение заостряется, лидер забирает почти всю массу → ближе к greedy.
  • T > 1 — делим на большое, разрывы сжимаются → распределение сглаживается, хвост поднимается → больше случайности и «креатива».
  • T → 0 — лидер получает ~всю вероятность → это и есть argmax → строгий детерминизм.

Меру «разброса» распределения удобно мерить энтропией (-Σ p·log p): чем выше T, тем выше энтропия.

После softmax можно ещё обрезать хвост кандидатов — чтобы модель не выбрала совсем маловероятную чушь:

  • top-k — оставить k самых вероятных токенов, остальные обнулить и перенормировать. Размер набора фиксированный.
  • top-p (nucleus) — отсортировать по убыванию и набирать токены, пока их суммарная вероятность не достигнет p; остальное отбросить. Размер набора плавающий: на «уверенном» шаге (один токен доминирует) останется один-два кандидата, на «размытом» — много. Это идея nucleus sampling (Holtzman et al., 2019): чистый greedy скатывается в повторы и деградацию, а выбор из динамического «ядра» даёт живой, но связный текст.

Полный конвейер выбора одного токена:

logits ──softmax(T)──► вероятности ──top-k──► top-p ──► бросок rng ──► токен
            │                          │         │            │
   T→0: пик на argmax            оставить k   набор с      seed → тот же
   T↑ : распределение «плоское»   лучших      суммой ≥ p   результат

temperature = 0 (или top_k = 1) убирает случайность вовсе → детерминизм. С сэмплированием воспроизводимость даёт только фиксированный seed генератора случайных чисел.

РАЗБОР ПО ШАГАМ

Возьмём 4 токена с логитами [3, 2, 1, 0] и прокрутим конвейер руками.

Шаг 1 — softmax при разной температуре. Для устойчивости из логитов вычитают максимум (3), это не меняет результат. Считаем exp((logit−3)/T) и нормируем:

T = 1.0 :  [0.644, 0.237, 0.087, 0.032]   ← базовое распределение
T = 0.5 :  [0.865, 0.117, 0.016, 0.002]   ← заострилось: лидер почти всё забрал
T = 2.0 :  [0.455, 0.276, 0.167, 0.102]   ← сгладилось: хвост поднялся
T → 0   :  [1.000, 0.000, 0.000, 0.000]   ← argmax, детерминизм

Видно правило: ниже T — острее (ближе к argmax), выше T — площе (больше случайности). Энтропия растёт от почти 0 (T→0) к максимуму с ростом T.

Шаг 2 — top-k = 2 на распределении T=1. Оставляем два самых вероятных (0.644, 0.237), остальные обнуляем и перенормируем (делим на их сумму 0.881):

[0.644, 0.237, 0.087, 0.032] ──top-k=2──► [0.731, 0.269, 0, 0]

Шаг 3 — top-p = 0.5 на том же распределении. Идём по убыванию и набираем, пока сумма не достигнет 0.5. Первый же токен 0.644 уже ≥ 0.5, значит «ядро» — один токен:

[0.644, 0.237, 0.087, 0.032] ──top-p=0.5──► [1.0, 0, 0, 0]

Будь распределение более плоским, в ядро попало бы больше токенов — в этом и суть «плавающего» размера top-p.

Шаг 4 — выбор. По финальному распределению бросаем rng.random() и идём по кумулятивной сумме: первый токен, на котором накопленная вероятность догнала бросок, и есть результат. Один и тот же seed → один и тот же бросок → один и тот же токен (воспроизводимость).

BUILD IT

Задание: собери сэмплер с нуля — math и random, без зависимостей. Тот же механизм крутит любая LLM при генерации.

Перед запуском. Работай в своей папке курса (ai-native/1.3-sampling/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3, всё офлайн (для теста ещё pytest).

Создай sampling.py и реализуй функции:

  • softmax(logits, temperature=1.0) — логиты → список вероятностей по формуле выше. Перед exp вычитай максимум логита (устойчивость). При temperature <= 0 верни one-hot на argmax (вектор из одной 1.0 на самом большом логите).
  • top_k_filter(probs, k) — оставь k самых вероятных, остальные обнули и перенормируй (сумма снова 1). При k пустом/<=0/>= len(probs) верни без изменений.
  • top_p_filter(probs, p) — nucleus: по убыванию набирай токены, пока накопленная сумма не станет >= p, затем стоп; остальные обнули и перенормируй. При p is None/>= 1.0 — без изменений.
  • sample(logits, temperature=1.0, top_k=None, top_p=None, rng=None) — при temperature <= 0 верни argmax; иначе softmax → top_k_filter → top_p_filter, затем выбери индекс броском rng.random() по кумулятивной сумме. rng=random.Random(seed) делает выбор воспроизводимым.
  • entropy(probs) — -Σ p·log p по ненулевым p (мера разброса).

Готово, когда тест test_sampling.py зелёный — он проверяет, что softmax суммируется в 1 и T=0 даёт one-hot, что выше температура → выше энтропия, что top-k оставляет ровно k, top-p — минимальное ядро, T=0 и top_k=1 дают детерминированный argmax, а один seed — воспроизводимую последовательность.

ЧАСТЫЕ ОШИБКИ

  • Считать temperature «уровнем ума». Это не качество, а разброс выбора: высокая T не делает модель умнее, она делает её менее предсказуемой (и более склонной к ошибкам на точных задачах).
  • Ждать воспроизводимости при T>0 без seed. Любой сэмплинг случаен; повторяемость даёт только T=0 (argmax) или фиксированный seed генератора.
  • Крутить temperature и top_p сразу вместе. Эффекты накладываются, и причину разброса не понять. Меняй по одному.
  • Передавать temperature туда, где её нет. У свежих Claude (4.7+/4.8) параметры сэмплинга отключены — запрос будет отклонён; поведение там задаётся промптом.

import random
from sampling import softmax, top_k_filter, top_p_filter, sample, entropy

LOGITS = [3.0, 2.0, 1.0, 0.0]


def test_softmax_sums_to_one_and_temp0_is_onehot():
    p = softmax(LOGITS, 1.0)
    assert abs(sum(p) - 1.0) < 1e-9
    p0 = softmax(LOGITS, 0.0)
    assert p0 == [1.0, 0.0, 0.0, 0.0]


def test_higher_temperature_increases_entropy():
    assert entropy(softmax(LOGITS, 2.0)) > entropy(softmax(LOGITS, 0.5))


def test_top_k_keeps_exactly_k():
    filt = top_k_filter(softmax(LOGITS, 1.0), 2)
    assert sum(1 for x in filt if x > 0) == 2
    assert abs(sum(filt) - 1.0) < 1e-9


def test_top_p_keeps_minimal_nucleus():
    filt = top_p_filter(softmax(LOGITS, 1.0), 0.5)
    nonzero = sum(1 for x in filt if x > 0)
    assert 1 <= nonzero <= 2            # самый вероятный уже ~0.64
    assert abs(sum(filt) - 1.0) < 1e-9


def test_temperature_zero_is_deterministic_argmax():
    assert all(sample(LOGITS, 0.0) == 0 for _ in range(5))
    assert sample(LOGITS, 0.0, top_k=1) == 0


def test_seeded_sampling_is_reproducible():
    a = [sample(LOGITS, 1.0, rng=random.Random(7)) for _ in range(5)]
    b = [sample(LOGITS, 1.0, rng=random.Random(7)) for _ in range(5)]
    assert a == b


def test_top_k_1_forces_argmax():
    assert all(sample(LOGITS, 1.0, top_k=1, rng=random.Random(i)) == 0 for i in range(5))

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/sampling.pyПоказать решение
"""Сэмплирование: temperature, top-k, top-p — Build It для урока 1.3.

Без зависимостей (math, random). Превращает логиты в распределение и выбирает
следующий токен. temperature=0 → детерминированный argmax; top-k и top-p
ограничивают множество кандидатов. Тот же механизм крутит любая LLM при генерации.
"""
import math
import random


def softmax(logits, temperature=1.0):
    """Логиты → вероятности. temperature<=0 → one-hot на argmax (детерминизм)."""
    if temperature <= 0:
        m = max(range(len(logits)), key=lambda i: logits[i])
        return [1.0 if i == m else 0.0 for i in range(len(logits))]
    mx = max(logits)                                   # сдвиг для устойчивости exp
    exps = [math.exp((l - mx) / temperature) for l in logits]
    s = sum(exps)
    return [e / s for e in exps]


def _renorm(probs):
    s = sum(probs)
    return [p / s for p in probs] if s > 0 else probs[:]


def top_k_filter(probs, k):
    """Оставить k самых вероятных, остальное обнулить и перенормировать."""
    if not k or k <= 0 or k >= len(probs):
        return probs[:]
    keep = set(sorted(range(len(probs)), key=lambda i: probs[i], reverse=True)[:k])
    return _renorm([p if i in keep else 0.0 for i, p in enumerate(probs)])


def top_p_filter(probs, p):
    """Nucleus: оставить минимальный набор по убыванию, чья сумма >= p."""
    if p is None or p >= 1.0:
        return probs[:]
    order = sorted(range(len(probs)), key=lambda i: probs[i], reverse=True)
    keep, cum = set(), 0.0
    for i in order:
        keep.add(i)
        cum += probs[i]
        if cum >= p:
            break
    return _renorm([pr if i in keep else 0.0 for i, pr in enumerate(probs)])


def sample(logits, temperature=1.0, top_k=None, top_p=None, rng=None):
    """Выбрать индекс следующего токена. temperature=0 → argmax (детерминизм)."""
    if temperature <= 0:
        return max(range(len(logits)), key=lambda i: logits[i])
    probs = softmax(logits, temperature)
    probs = top_k_filter(probs, top_k)
    probs = top_p_filter(probs, top_p)
    rng = rng or random
    r, cum = rng.random(), 0.0
    for i, pr in enumerate(probs):
        cum += pr
        if r <= cum:
            return i
    return len(probs) - 1


def entropy(probs):
    """Энтропия распределения (мера «разброса»)."""
    return -sum(p * math.log(p) for p in probs if p > 0)


if __name__ == "__main__":
    logits = [3.0, 2.0, 1.0, 0.0]
    for t in (0.0, 0.5, 1.0, 2.0):
        p = softmax(logits, t)
        print(f"T={t}: probs={[round(x,3) for x in p]} entropy={entropy(p):.3f}")
    print("top_k=2 :", [round(x, 3) for x in top_k_filter(softmax(logits), 2)])
    print("top_p=.5:", [round(x, 3) for x in top_p_filter(softmax(logits), 0.5)])
    rng = random.Random(42)
    print("samples :", [sample(logits, 1.0, rng=rng) for _ in range(8)])

# Артефакт: гайд по параметрам генерации

Как `temperature`, `top_k`, `top_p` влияют на вывод и какой набор брать под задачу. Механику собрали в [`code/sampling.py`](../code/sampling.py).

## Что делает каждый параметр

| Параметр | Смысл | Меньше → | Больше → |
|---|---|---|---|
| `temperature` | «остро/плоско» распределение | детерминированнее, скучнее | разнообразнее, рискованнее |
| `top_k` | оставить k лучших кандидатов | строже отбор | шире выбор |
| `top_p` (nucleus) | минимум токенов с суммой ≥ p | строже отбор | шире выбор |

## Какой набор под задачу

| Задача | temperature | top_p | Почему |
|---|---|---|---|
| Извлечение/классификация, JSON | 0 | — | нужен один правильный ответ, воспроизводимость |
| Фактический Q&A | 0–0.3 | — | минимум фантазии |
| Рабочий текст, ответы | 0.3–0.7 | 0.9 | баланс точности и живости |
| Брейншторм, варианты | 0.8–1.0+ | 0.95 | нужно разнообразие |

## Правила

1. **Воспроизводимость → `temperature=0`** (но строгого детерминизма провайдер не гарантирует даже при 0).
2. **Крути что-то одно** — `temperature` *или* `top_p`, не оба сразу (у части моделей это и запрещено).
3. **Сначала самое простое** — обычно достаточно `temperature`; `top_k`/`top_p` — для тонкой настройки.
4. ⚠️ **2026:** у новейших Claude (**Opus 4.7+ / 4.8**) сэмплинг-параметры **не поддерживаются** (вернётся 400) — поведение задаётся промптом. Экспериментируй на OpenAI (0–2), Gemini или более ранних Claude.
