Урок 4.3 · Проекты и рабочие пространства

Фаза 4 — Скиллы, память и проекты · Результат фазы: Упаковывать способности в SKILL.md, хранить контекст в памяти и собирать рабочие проекты.

Это сборка всей фазы. Скилл (4.1) дал умение, память (4.2) — биографию. Проект складывает их в одно рабочее пространство с постоянной ролью и базой знаний — ассистент перестаёт начинать каждый разговор «с нуля».

MOTTO. Проект = постоянная инструкция + база знаний: ассистент перестаёт начинать «с нуля».

PROBLEM

Универсальный чат не знает твоего контекста. В каждом новом диалоге ты заново объясняешь роль («ты ассистент поддержки, отвечай кратко»), правила и подкладываешь одни и те же документы. Это утомительно и легко забыть кусок контекста — тогда ответ съезжает.

Хочется зафиксировать это один раз: роль и правила (system prompt) + набор документов (база знаний). Тогда во всех диалогах ассистент отвечает «как член твоей команды», заземляя ответы на твои материалы. По формулировке Anthropic, проектная база знаний помогает не начинать с нуля и даёт экспертные ответы в твоём контексте. Соберём такое пространство сами — и увидим, что внутри это просто склейка двух вещей плюс ретрив из Фазы 3.

CONCEPT

Интуиция

Проект — это онбординг нового сотрудника, но не на один разговор, а навсегда. Ты один раз кладёшь на стол:

  • должностную инструкцию — кто он, как отвечает, каким тоном (system prompt);
  • папку с документами команды — регламенты, факты, база знаний (knowledge base).

Дальше на каждый вопрос «сотрудник» смотрит в инструкцию и достаёт из папки только относящиеся к вопросу страницы — не зачитывает всю папку вслух. Контекстное окно конечно (это мы поняли в 4.2), поэтому в промпт идёт system prompt + горсть релевантных кусков, а не весь архив.

Как это работает

Project
 ├─ system_prompt   постоянная роль и правила
 ├─ knowledge[]     база знаний проекта (документы)
 └─ build_context(query):
        retrieve(query, k)  → топ-k релевантных кусков знаний
        ↓
        system_prompt  +  найденные куски  +  запрос  → промпт модели
  • retrieve(query, k) — это ретривер из Фазы 3: считает совпадение слов запроса и документа, возвращает топ-k. Нерелевантное (совпадений 0) не возвращает — бережём окно.
  • build_context(query, k) — склейка: постоянная инструкция + найденные куски + сам запрос. Это и есть «проект»: знания подмешиваются автоматически, объяснять контекст заново не нужно.

Так фаза замыкается: скилл (способность) + память (что помним) + RAG (что нашли) живут в одном переиспользуемом рабочем пространстве.

РАЗБОР ПО ШАГАМ

Посмотрим retrieve руками. База знаний из трёх документов, запрос сколько дней на возврат (k=1). Score = число совпавших слов:

запрос: {сколько, дней, на, возврат}

D1 "Возврат товара возможен в течение 14 дней."   ∩ = {возврат, дней}  → 2
D2 "Гарантия на электронику — 12 месяцев."         ∩ = {на}            → 1
D3 "Доставка по городу занимает 1-2 дня."          ∩ = {}              → 0

Топ-1 → D1 (score 2): именно про возврат. D3 с нулём в выдачу не попадает. А вот D2 набрал 1 балл из-за слова-связки на — наглядный изъян лексического поиска: служебные слова создают ложные совпадения. С k=1 это не мешает, но при больших k шумит — поэтому настоящие проекты ретривят эмбеддингами (урок 3.4), а не голым пересечением слов. build_context затем подставит D1 в шаблон между system prompt и запросом.

BUILD IT

Задание: собери проект Project — постоянная инструкция + база знаний + сборка контекста. Только стандартная библиотека (re, collections.Counter), без сети.

Перед запуском. Работай в своей папке курса (ai-native/4.3-projects/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3 (для теста ещё pytest).

Создай файл project.py с классом Project(name, system_prompt, knowledge=None) и методами:

  • add_knowledge(doc) — добавить документ-строку в базу знаний.
  • retrieve(query, k=3) — топ-k документов по совпадению слов с запросом. Посчитай overlap как пересечение слов (можно мультимножеством через Counter), документы с overlap == 0 не включай, отсортируй по убыванию и верни список документов (без score).
  • build_context(query, k=3) — собери промпт-строку: system_prompt, затем секция # База знаний со списком найденных кусков (если пусто — пометка «нет релевантных знаний»), затем секция # Запрос с самим запросом.

Готово, когда все тесты в test_project.py зелёные — они проверяют: в контекст попадают system prompt и запрос; подтягивается релевантный документ, а нерелевантный (Гарантия) — нет; retrieve уважает k; add_knowledge расширяет базу и новый документ находится.

Подсказка. Слова — re.findall(r"\w+", text.lower()). Overlap мультимножеством: q = Counter(tokens(query)), d = Counter(tokens(doc)), sum(min(q[w], d[w]) for w in q if w in d). Для build_context собери список f"- {d}" через "\n".join(...).

USE IT

Те же «проекты» есть во всех платформах — постоянная инструкция + загруженные документы (мульти-провайдер):

  • Claude Projects — workspace с custom instructions и knowledge base; в окно 200K можно загрузить регламенты, кодовую базу, style guide, и Claude заземляет ответы на них без повторных объяснений (RAG по сути из коробки).
  • Custom GPTs (OpenAI) — name + instructions + knowledge (файлы) + capabilities, собирается в визуальном билдере, можно опубликовать.
  • Gemini Gems — кастомный ассистент с инструкциями и файлами знаний.

Везде один рецепт: зафиксировать роль и материалы → получить ассистента под конкретную задачу. Разница — в объёме окна, ретриве под капотом и публикации/доступах.

ЧАСТЫЕ ОШИБКИ

  • Сваливать всю базу знаний в каждый промпт. Окно конечно. Подмешивай только релевантные куски (retrieve), а не весь архив — иначе дорого и модель хуже вспоминает важное (4.2).
  • Раздувать system prompt. Это постоянная инструкция, она в каждом запросе. Держи роль/правила компактными; фактуру выноси в базу знаний, откуда она подтянется по запросу.
  • Полагаться на пословное совпадение. Служебные слова (на, и, в) создают ложные совпадения, а синонимы и перефразировки теряются. Настоящие проекты ретривят эмбеддингами (3.4) — лексика лишь иллюстрация механизма.
  • Класть в knowledge длинные простыни. Один документ = один самодостаточный факт/кусок. Так ретрив точнее и в окно влезает больше нужного.
  • Путать проект и скилл. Скилл (4.1) — переносимое умение «как делать»; проект — рабочее пространство «с какой ролью и на каких данных». Они дополняют друг друга, а не заменяют.

from project import Project

KB = [
    "Возврат товара возможен в течение 14 дней.",
    "Гарантия на электронику составляет 12 месяцев.",
    "Доставка по городу занимает 1-2 дня.",
]


def _proj():
    return Project("support", "Ты ассистент поддержки.", KB)


def test_context_includes_system_prompt_and_query():
    ctx = _proj().build_context("вопрос про возврат")
    assert "Ты ассистент поддержки." in ctx
    assert "вопрос про возврат" in ctx


def test_context_pulls_relevant_knowledge():
    ctx = _proj().build_context("сколько дней на возврат?", k=1)
    assert "Возврат" in ctx
    assert "Гарантия" not in ctx          # нерелевантное не попало


def test_retrieve_respects_k():
    assert len(_proj().retrieve("товар гарантия доставка", k=2)) == 2


def test_add_knowledge_grows_kb():
    p = _proj()
    p.add_knowledge("Промокоды действуют до конца месяца.")
    ctx = p.build_context("есть ли промокоды?", k=1)
    assert "Промокоды" in ctx

Эталонное решение

code/project.pyПоказать решение
"""Проекты и рабочие пространства — Build It для урока 4.3.

Без зависимостей. Проект = system prompt + база знаний (+ память). На запрос
собирает контекст: системная инструкция + релевантные куски знаний. Это
объединяет skills/memory/RAG в переиспользуемое рабочее пространство.
"""
import re
from collections import Counter


def _tokens(text):
    return re.findall(r"\w+", text.lower())


class Project:
    """Рабочее пространство: постоянная инструкция + база знаний."""

    def __init__(self, name, system_prompt, knowledge=None):
        self.name = name
        self.system_prompt = system_prompt
        self.knowledge = list(knowledge or [])

    def add_knowledge(self, doc):
        self.knowledge.append(doc)

    def retrieve(self, query, k=3):
        """Топ-k документов базы знаний по совпадению слов с запросом."""
        q = Counter(_tokens(query))
        scored = []
        for doc in self.knowledge:
            d = Counter(_tokens(doc))
            overlap = sum(min(q[w], d[w]) for w in q if w in d)
            if overlap:
                scored.append((overlap, doc))
        scored.sort(key=lambda x: x[0], reverse=True)
        return [doc for _, doc in scored[:k]]

    def build_context(self, query, k=3):
        """Собрать промпт: system prompt + релевантные знания + запрос."""
        docs = self.retrieve(query, k)
        kb = "\n".join(f"- {d}" for d in docs) or "(нет релевантных знаний)"
        return f"{self.system_prompt}\n\n# База знаний\n{kb}\n\n# Запрос\n{query}"


if __name__ == "__main__":
    proj = Project(
        name="support-bot",
        system_prompt="Ты ассистент поддержки. Отвечай кратко и по делу.",
        knowledge=[
            "Возврат возможен в течение 14 дней.",
            "Гарантия на электронику — 12 месяцев.",
            "Доставка по городу 1-2 дня.",
        ],
    )
    print(proj.build_context("сколько дней на возврат?", k=1))

# Артефакт: настроенный ассистент с базой знаний

Рецепт сборки рабочего проекта (workspace) под конкретную роль. Код-прототип — [`code/project.py`](../code/project.py).

## Конфигурация проекта

```
name:           support-assistant
system_prompt:  Ты ассистент поддержки. Отвечай кратко, по делу,
                только на основе базы знаний. Не знаешь — скажи «не знаю».
knowledge:      [политика возврата, гарантия, доставка, FAQ, прайс]
```

## Что куда класть

| Элемент | Назначение | Правило |
|---|---|---|
| **system prompt** | роль, тон, правила, границы | коротко и чётко; не дублировать знания |
| **knowledge** | факты, документы, прайсы | справочные данные, не правила поведения |
| **instructions** | формат ответа, что нельзя | поведение → сюда, не в knowledge |

## Как собрать (через `project.py`)

```python
from project import Project
proj = Project("support", SYSTEM_PROMPT, knowledge=docs)
prompt = proj.build_context("сколько дней на возврат?", k=3)  # system + RAG + запрос
# → отправить prompt в модель
```

## Платформы

- **Claude Projects** — knowledge base + project instructions (RAG авто).
- **Custom GPTs** — instructions + knowledge + capabilities.
- **Gemini Gems** — инструкции + файлы знаний.

## Правила

- Знания — справка, **правила поведения — в инструкции** (частая ошибка: смешать).
- В контекст подмешивай **только релевантное** (retrieve top-k), а не весь корпус (бюджет — урок 4.4).
- Требуй ответ строго по базе знаний и проси «не знаю» при пробелах (урок 1.4).
- Качество ответов меряй eval-harness (урок 2.5).
