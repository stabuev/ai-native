Урок 4.2 · Память: короткая и длинная

Фаза 4 — Скиллы, память и проекты · Результат фазы: Упаковывать способности в SKILL.md, хранить контекст в памяти и собирать рабочие проекты.

Скилл (4.1) дал агенту умения. Память даёт ему биографию. Без неё каждый разговор начинается с чистого листа: агент заново узнаёт, кто ты, что вы уже решили и чем закончили прошлую задачу.

MOTTO. Без памяти агент каждый раз начинает с чистого листа; память — это контекст, переживающий сессию.

PROBLEM

Модель не помнит ничего между запросами: твои предпочтения, факты о проекте, решения прошлых задач — всё испаряется в конце диалога. Завтра объясняешь заново.

«Так положим всё в контекст» — не выход. Контекст конечен и это рабочая память с ограниченным «бюджетом внимания»: по Anthropic, каждый новый токен расходует этот бюджет, а чем больше токенов в окне, тем хуже модель точно вспоминает нужное из контекста. Нельзя свалить туда всю историю — модель захлебнётся и начнёт терять важное. Нужна внешняя память: хранить факты вне промпта, а в окно подмешивать только релевантное и свежее.

CONCEPT

Интуиция

Различай два вида памяти, как у человека.

  • Короткая (рабочая) — то, что сейчас в контекстном окне. Быстрая, но крошечная и исчезает в конце сессии. Это «держу в голове прямо сейчас».
  • Длинная — заметки в блокноте на диске. Туда агент записывает факты, переживающие сессию, и достаёт их обратно, когда они снова нужны. По формулировке Anthropic, агент регулярно пишет заметки за пределы контекстного окна, и позже они подтягиваются обратно в окно.

Блокнот может быть огромным — но в окно за раз кладём только горсть самых нужных записей. Вопрос «какие именно» и есть суть памяти.

Как это работает

add(text, ts)   → запись с меткой времени  ──► блокнот на диске (длинная память)
                                                  │
запрос ──► search(query, now)  оценивает каждую запись по ДВУМ осям:
              • релевантность — совпадение слов запроса и записи
              • свежесть      — затухание со временем (полураспад half_life)
                                                  │
              топ-k записей ──► в контекст следующего запроса (короткая память)
save / load     → блокнот переживает перезапуск процесса

Две оси, потому что одной мало. Только релевантность — и старый, давно неверный факт («любит чай») перебьёт свежий («перешёл на кофе»), если слов совпало поровну. Поэтому к совпадению слов добавляем свежесть: при прочих равных недавнее важнее. Свежесть считаем затуханием с периодом полураспада half_life: запись возрастом в один half_life весит вдвое меньше только что добавленной.

РАЗБОР ПО ШАГАМ

Посчитаем итоговый score руками. half_life = 30, «сейчас» now = 1000. В памяти две записи про кофе (обе совпадают с запросом кофе ровно одним словом, релевантность = 1):

запись A: "любит кофе без сахара"     ts = 900  → возраст = 100
запись B: "снова заказал кофе латте"  ts = 999  → возраст = 1

Свежесть по формуле recency = 0.5 ** (age / half_life):

A: 0.5 ** (100/30) = 0.5 ** 3.33 = 0.099   → score = 1 + 0.099 = 1.099
B: 0.5 ** (1/30)   = 0.5 ** 0.03 = 0.977   → score = 1 + 0.977 = 1.977

Победила B (1.977 > 1.099): релевантность у обеих одинаковая, решила свежесть. Так старый факт не перебивает недавний. А на запрос квантовая физика обе записи дадут совпадение 0 слов → в выдачу не попадут вовсе (нерелевантное в контекст не тащим, чтобы не тратить бюджет внимания).

BUILD IT

Задание: собери файловую память агента — класс MemoryStore с релевантностью, свежестью и персистентностью. Только стандартная библиотека (json, re, time), без сети.

Перед запуском. Работай в своей папке курса (ai-native/4.2-memory/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3 (для теста ещё pytest).

Создай файл memory_store.py с классом MemoryStore(half_life=30.0) и методами:

  • add(text, tags=None, ts=None) — добавить запись. Сохрани словарь {"id", "text", "ts", "tags"}; если ts не передан — возьми time.time(). Верни запись.
  • search(query, k=3, now=None) — топ-k записей по убыванию score. Для каждой записи: overlap = размер пересечения слов запроса и записи; записи с overlap == 0 пропусти (нерелевантные не возвращаем). recency = 0.5 ** (age / half_life), где age = max(0, now - ts). Итог score = overlap + recency. Верни список записей (без score).
  • recent(n=5) — последние n записей по убыванию ts.
  • save(path) / load(path) — сериализация списка записей в JSON и обратно (load возвращает self). Инвариант: после save→load память идентична и поиск работает.

Готово, когда все тесты в test_memory_store.py зелёные — они проверяют: возвращаются только релевантные записи; при равной релевантности свежая (латте) первая; recent отдаёт новейшую; нерелевантный запрос → пустой список; save/load сохраняют память.

Подсказка. Слова — set(re.findall(r"\w+", text.lower())). Свежесть — степень 0.5 ** (age / self.half_life): для только что добавленной записи это ≈1.0, для очень старой → 0. Сортируй по score, но из search возвращай сами записи, а не пары со score.

USE IT

Платформенная память делает ровно это под капотом — хранит вне промпта и подмешивает релевантное (мульти-провайдер):

  • ChatGPT Memory — «заметки» отдельно от истории чатов: помнит факты о тебе между диалогами, управляется в Settings → Personalization (можно смотреть, править и удалять).
  • Claude — постоянный контекст через Projects (knowledge base) и память между сессиями; агентная память тоже строится на заметках, которые подтягиваются обратно в окно.
  • Gemini — персонализация и Gems с заранее заданным контекстом.

Принцип один на всех: хранить вне контекста и добавлять только нужное. Разница — что именно и как долго хранится и как этим управлять (приватность и контроль — Фаза 11).

ЧАСТЫЕ ОШИБКИ

  • Складывать всю историю в контекст. Контекст — конечная рабочая память; чем он длиннее, тем хуже модель вспоминает важное. Храни снаружи, подмешивай горсть релевантного.
  • Искать только по релевантности. Без свежести устаревший факт («любит чай») перебьёт актуальный («перешёл на кофе»). Добавляй затухание по времени.
  • Тащить нерелевантное в окно. Записи с нулевым совпадением не возвращай вовсе — каждый лишний токен тратит «бюджет внимания» зря.
  • Путать короткую и длинную память. Короткая (контекст) исчезает в конце сессии; длинная (диск/save) переживает перезапуск. Если факт нужен завтра — он должен попасть в длинную.
  • Хранить чувствительное без оглядки. Долгая память на диске — это персональные данные; управление и приватность разбираем в Фазе 11.

from memory_store import MemoryStore

NOW = 1000.0


def _store():
    m = MemoryStore(half_life=30.0)
    m.add("пользователь любит кофе без сахара", ts=NOW - 100)
    m.add("пользователь снова заказал кофе латте", ts=NOW - 1)
    m.add("обсуждали отпуск в горах", ts=NOW - 5)
    return m


def test_search_returns_relevant_only():
    hits = _store().search("кофе", now=NOW)
    assert all("кофе" in h["text"] for h in hits)
    assert len(hits) == 2


def test_recency_breaks_ties():
    # обе записи совпадают по слову «кофе», свежая должна быть первой
    top = _store().search("кофе", k=1, now=NOW)[0]
    assert "латте" in top["text"]


def test_recent_returns_newest_first():
    newest = _store().recent(1)[0]            # самый свежий — ts=NOW-1
    assert "латте" in newest["text"]


def test_no_match_returns_empty():
    assert _store().search("квантовая физика", now=NOW) == []


def test_save_load_roundtrip(tmp_path):
    m = _store()
    path = tmp_path / "mem.json"
    m.save(path)
    loaded = MemoryStore().load(path)
    assert loaded.items == m.items
    assert "кофе" in loaded.search("кофе", k=1, now=NOW)[0]["text"]

"""Память агента: короткая и длинная — Build It для урока 4.2.

Без зависимостей. Файловое хранилище памяти с поиском и темпоральностью:
добавляем записи с меткой времени, ищем по релевантности С УЧЁТОМ свежести,
сохраняем/грузим на диск. Это «память» агента в простейшем виде (мост к Фазе 7).
"""
import json
import re
import time


def _words(text):
    return set(re.findall(r"\w+", text.lower()))


class MemoryStore:
    """Память с релевантностью (совпадение слов) и темпоральностью (свежесть)."""

    def __init__(self, half_life=30.0):
        self.items = []                 # [{id, text, ts, tags}]
        self.half_life = half_life      # период полураспада «свежести»

    def add(self, text, tags=None, ts=None):
        item = {"id": len(self.items), "text": text,
                "ts": time.time() if ts is None else ts, "tags": tags or []}
        self.items.append(item)
        return item

    def search(self, query, k=3, now=None):
        """Топ-k записей: релевантность (совпадение слов) + свежесть (затухание)."""
        q = _words(query)
        now = time.time() if now is None else now
        scored = []
        for it in self.items:
            overlap = len(q & _words(it["text"]))
            if overlap == 0:
                continue
            age = max(0.0, now - it["ts"])
            recency = 0.5 ** (age / self.half_life)     # 1.0 для свежего, → 0 для старого
            scored.append((overlap + recency, it))
        scored.sort(key=lambda x: x[0], reverse=True)
        return [it for _, it in scored[:k]]

    def recent(self, n=5):
        return sorted(self.items, key=lambda it: it["ts"], reverse=True)[:n]

    def save(self, path):
        with open(path, "w", encoding="utf-8") as f:
            json.dump(self.items, f, ensure_ascii=False)

    def load(self, path):
        with open(path, encoding="utf-8") as f:
            self.items = json.load(f)
        return self


if __name__ == "__main__":
    m = MemoryStore(half_life=30.0)
    now = 1000.0
    m.add("пользователь любит кофе без сахара", ts=now - 100)
    m.add("пользователь снова заказал кофе латте", ts=now - 1)
    m.add("обсуждали отпуск в горах", ts=now - 5)
    print("Поиск 'кофе':")
    for it in m.search("кофе", now=now):
        print("  ", it["text"])
