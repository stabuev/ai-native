Урок 2.5 · Оценка качества промптов

Фаза 2 — Промпт-инжиниринг · Результат фазы: Писать воспроизводимые промпты и измерять их качество через eval-harness.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Это венец Фазы 2: промпт-инжиниринг превращается из гадания в измерение. В уроке 2.1 мы договорились начинать с критериев успеха — здесь строим инструмент, который их проверяет. Eval-harness прогоняет промпт по фиксированному набору кейсов, считает метрику и честно сравнивает версии — чтобы «стало лучше» было числом, а не ощущением.

MOTTO. «Кажется, лучше» — не метрика. Промпт улучшают, измеряя на наборе кейсов.

PROBLEM

Промпты правят «на глаз»: поменял формулировку, ответ на одном примере стал красивее — закоммитил. А на десяти других стало хуже, и ты этого не видишь. Так же незаметно «улучшение» ломается при смене модели или параметра. Без воспроизводимой оценки нельзя ни честно сравнить две версии промпта, ни доказать, что новая модель реально лучше старой. Нужен eval-harness — тест-сьют для промптов.

CONCEPT

Интуиция

Eval — это юнит-тесты для промптов. Как в коде один ручной запуск не заменяет тестов, так и «посмотрел один ответ» не заменяет прогон по набору кейсов. Anthropic называет цикл «критерии успеха → оценка» центральным в промпт-инжиниринге: сначала формулируешь, что значит «хорошо» (в уроке 2.1), потом измеряешь.

Как это работает

набор кейсов {input, expected}
        │
   prompt_fn (версия промпта / модель)
        │
     метрика(pred, expected) → score за кейс
        │
   усреднили → score версии → compare(версии) → рейтинг

Три вещи делают оценку надёжной (по гайду Anthropic «build evaluations»):

  • Критерии успеха — SMART. Не «хорошо», а конкретно и измеримо: вместо «good performance» — «accurate sentiment classification». Метрика должна давать число.
  • Автоматизируй грейдинг. Anthropic про выбор метода: бери самый быстрый и надёжный. Лестница такая:
  • Code-based — быстрее и надёжнее всего, отлично масштабируется: exact_match (pred == expected), contains (key_phrase in pred). Для категориальных задач (классификация, извлечение) — то, что нужно.
  • LLM-as-judge — для субъективного качества текста (связность, тон): другая модель оценивает по рубрике. Гибко и масштабируемо, но сначала проверь её надёжность (вернёмся к этому в агентных фазах).
  • Human — самый гибкий, но медленный и дорогой; избегай, где можно.
  • Объём важнее точности грейдинга. Anthropic: «больше вопросов со слегка более шумной автогрейдинг-оценкой лучше, чем мало вопросов с дорогой ручной». Фиксированный набор кейсов + автооценка = воспроизводимость.

РАЗБОР ПО ШАГАМ

Прогоним harness руками. Набор кейсов (арифметика):

{input: "2+2", expected: "4"}   {input: "3+5", expected: "8"}   {input: "10-4", expected: "6"}

Сравним три «версии промпта» метрикой exact_match:

  • good — считает выражение → "4", "8", "6" → все три совпали → score 3/3 = 1.0;
  • half — всегда отвечает "4" → совпал только первый кейс → score 1/3 ≈ 0.33;
  • bad — всегда "?" → ни одного → score 0/3 = 0.0.

evaluate хранит и разбивку по кейсам (per_case) — видно, где именно версия ошиблась, как в history из урока 2.3. А compare строит рейтинг по убыванию:

compare(versions, cases) → [("good", 1.0), ("half", 0.33), ("bad", 0.0)]

Лучшая версия — первой, и это число, а не «мне кажется». Поменял промпт — прогнал тот же набор — увидел, вырос score или упал.

Реализуй метрики и прогон:

  • exact_match(pred, expected) → 1.0, если pred.strip() == expected.strip(), иначе 0.0.
  • contains(pred, expected) → 1.0, если expected.lower() входит в pred.lower(), иначе 0.0.
  • evaluate(prompt_fn, cases, metric=exact_match) — прогони prompt_fn по кейсам {input, expected}: для каждого посчитай pred = prompt_fn(input) и score = metric(pred, expected). Верни {"score": среднее по кейсам, "per_case": [{input, pred, expected, score}, …]} (на пустом наборе score = 0.0).
  • compare(versions, cases, metric=exact_match) — versions это {name: prompt_fn}; верни список (name, score), отсортированный по убыванию score.

Готово, когда тест test_eval_harness.py зелёный — он проверяет метрики (включая игнор пробелов и регистра), что evaluate даёт 1.0 на рабочей версии и 0.0 на сломанной, и что compare ставит лучшую версию первой, а худшую — последней.

Подсказка. evaluate — цикл по cases с накоплением per_case, потом среднее. compare — словарь в список через evaluate(...)["score"] и sorted(..., reverse=True).

В проде prompt_fn — вызов модели; harness прогоняет твой набор кейсов и сравнивает версии/модели (мульти-провайдер):

def v_claude(x): return Anthropic().messages.create(model="claude-haiku-4-5",
    max_tokens=50, messages=[{"role": "user", "content": PROMPT.format(x=x)}]).content[0].text
def v_openai(x): return OpenAI().responses.create(model="gpt-5.x", input=PROMPT.format(x=x)).output_text

    model="gemini-2.x-flash", contents=PROMPT.format(x=x)).text

Так выбор модели и версии промпта становится измеримым — и это прямой мост к Фазе 9, где к качеству добавится цена (одна и та же метрика, но score на доллар).

ЧАСТЫЕ ОШИБКИ

  • Править промпт по одному примеру. Один красивый ответ ничего не доказывает — на других кейсах могло стать хуже. Меряй на наборе.
  • Брать LLM-as-judge там, где хватает code-based. Для категориальных задач exact_match/contains быстрее, надёжнее и бесплатнее; модель-судья — для субъективного качества.
  • Менять промпт и модель одновременно. Не поймёшь, что подняло score. Фиксируй набор кейсов и меняй по одному.
  • Маленький набор кейсов. На трёх примерах метрика — шум. Объём важнее точности грейдинга: лучше много кейсов с автооценкой.

code/test_eval_harness.py
from eval_harness import exact_match, contains, evaluate, compare

CASES = [
    {"input": "2+2", "expected": "4"},
    {"input": "3+5", "expected": "8"},
    {"input": "10-4", "expected": "6"},
]


def test_metrics():
    assert exact_match("4", "4") == 1.0
    assert exact_match("4 ", "4") == 1.0          # игнор пробелов
    assert exact_match("5", "4") == 0.0
    assert contains("ответ: 4", "4") == 1.0
    assert contains("нет", "4") == 0.0


def test_evaluate_scores_perfect_and_zero():
    good = evaluate(lambda x: str(eval(x)), CASES)
    assert good["score"] == 1.0
    assert len(good["per_case"]) == 3
    bad = evaluate(lambda x: "?", CASES)
    assert bad["score"] == 0.0


def test_compare_ranks_better_prompt_first():
    versions = {
        "good": lambda x: str(eval(x)),
        "half": lambda x: "4",                    # угадывает только первый кейс
        "bad": lambda x: "?",
    }
    ranking = compare(versions, CASES)
    assert ranking[0][0] == "good"
    assert ranking[-1][0] == "bad"
    assert ranking[0][1] > ranking[1][1] > ranking[2][1]

code/eval_harness.pyПоказать решение
"""Mini eval-harness: метрики и сравнение версий промптов — Build It для урока 2.5.

Без зависимостей. Прогоняет промпт-функцию по набору кейсов {input, expected},
считает метрику и сравнивает версии промптов между собой. Промпт-функция здесь —
обычный callable (в тестах детерминированный stub); в USE IT её роль играет
вызов модели. Это ядро фазы: «писать воспроизводимые промпты и измерять качество».
"""


# --- метрики: (pred, expected) -> [0..1] ---
def exact_match(pred, expected):
    return 1.0 if pred.strip() == expected.strip() else 0.0


def contains(pred, expected):
    return 1.0 if expected.lower() in pred.lower() else 0.0


def evaluate(prompt_fn, cases, metric=exact_match):
    """Прогнать prompt_fn по кейсам. Возвращает {score, per_case}."""
    per_case = []
    for c in cases:
        pred = prompt_fn(c["input"])
        per_case.append({"input": c["input"], "pred": pred,
                         "expected": c["expected"], "score": metric(pred, c["expected"])})
    score = sum(p["score"] for p in per_case) / len(cases) if cases else 0.0
    return {"score": score, "per_case": per_case}


def compare(versions, cases, metric=exact_match):
    """versions: {name: prompt_fn}. Рейтинг (name, score) по убыванию качества."""
    scored = [(name, evaluate(fn, cases, metric)["score"]) for name, fn in versions.items()]
    return sorted(scored, key=lambda x: x[1], reverse=True)


if __name__ == "__main__":
    cases = [
        {"input": "2+2", "expected": "4"},
        {"input": "3+5", "expected": "8"},
        {"input": "10-4", "expected": "6"},
    ]
    # две «версии промпта»: рабочая и сломанная
    good = lambda x: str(eval(x))               # noqa: S307 — учебный stub
    bad = lambda x: "не знаю"
    print("good:", evaluate(good, cases)["score"])
    print("bad :", evaluate(bad, cases)["score"])
    print("рейтинг:", compare({"good": good, "bad": bad}, cases))

# prompt-eval — измерение и сравнение промптов

Превращает «кажется, лучше» в число: фиксированный набор кейсов + метрика → честное сравнение версий промпта или моделей.

## Как пользоваться

1. Собери набор кейсов `cases = [{"input": ..., "expected": ...}, ...]` — реальные примеры с эталонами, включая граничные.
2. Заведи версии как функции `versions = {"v1": fn1, "v2": fn2}`, где `fn(input) -> ответ` (в проде — вызов модели).
3. Выбери метрику: `exact_match` (классификация/извлечение) или `contains` (есть ли нужное); сложное — LLM-as-judge.
4. Прогон и сравнение:

```python
import eval_harness as e
ranking = e.compare(versions, cases, metric=e.exact_match)   # [(name, score), ...] по убыванию
detail = e.evaluate(versions["v1"], cases, metric=e.exact_match)["per_case"]
```

5. Покажи рейтинг версий по убыванию score и per-case разбор (где именно версия проигрывает).

## Файлы

- `eval_harness.py` — движок: `exact_match`, `contains`, `evaluate(prompt_fn, cases, metric)`, `compare(versions, cases, metric)`.

## Когда применять

Перед тем как принять правку промпта, сменить модель или параметры — прогони eval и сравни с базлайном. Связка с выбором модели по цене/качеству — Фаза 9.
