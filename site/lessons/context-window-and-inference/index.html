Урок 1.2 · Контекстное окно и inference

Фаза 1 — Как работают LLM · Результат фазы: Объяснить токены, контекст и inference; собрать токен-бюджетер; отличать надёжный ответ от галлюцинации.

Результат урока: по трассировке объяснить, как модель строит ответ токен за токеном, предсказать влияние видимого контекста на следующий токен и составить проверяемый бюджет контекста для своей задачи.

Опора: в 1.1 ты увидел, как текст превращается в token IDs, и собрал prompt-token-budgeter. · Дальше: в 1.3 вместо жадного выбора самого вероятного токена появятся temperature, top-k и top-p.

Requires: Python 3 и pytest. BUILD IT и бюджет контекста работают локально, без ключей и сети. Для необязательной ветки USE IT нужны настроенный в 0.2 API одного провайдера, его SDK и сеть; не отправляй сервису секретные или запрещённые данные.

От текста к ответу: какого шага не хватало

В 1.1 мы остановились в момент, когда текст уже превратился в последовательность чисел:

"Ответь кратко" → [token_id_1, token_id_2, ...]

Но модель ещё не получила готовый ответ. Ей нужно продолжить эту последовательность. Для этого она много раз выполняет один и тот же шаг: оценивает возможные следующие токены, выбирает один, добавляет его справа и снова делает прогноз.

Именно этот рабочий прогон уже обученной модели называют inference. В широком смысле inference — получение результата от обученной модели; в этом уроке нас интересует его генеративный случай: построение текста по одному токену.

Понимание цикла отвечает сразу на три практических вопроса:

  • почему начало ответа можно получить раньше его конца;
  • почему каждый новый токен меняет продолжение;
  • почему инструкции, история и планируемый ответ конкурируют за ограниченное место.

Один шаг генерации

Представь, что видимый модели контекст заканчивается фразой Столица Франции —. Сначала модель не пишет весь абзац. Она оценивает кандидатов на одну следующую позицию:

КандидатУсловная вероятность
Париж0.92
Лион0.03
город0.02
остальные токены вместе0.03

Такую таблицу называют распределением вероятностей следующего токена: каждому кандидату назначено число от 0 до 1, а сумма вероятностей равна 1. Это не список готовых ответов и не оценка истинности. Это прогноз того, какой токен правдоподобно продолжает текущий контекст.

В этом уроке используем самое простое правило выбора — greedy («жадный выбор»): берём кандидата с наибольшей вероятностью. Операцию «найти позицию максимального значения» также называют argmax. В примере будет выбран Париж.

После выбора токен добавляется в контекст:

было:  Столица Франции —
стало: Столица Франции — Париж

Теперь модель строит новое распределение уже для продолжения после Париж. Оно может предпочесть точку, запятую или следующий фрагмент текста. Поэтому второй токен зависит не только от исходного запроса, но и от первого сгенерированного токена.

видимый контекст
      │
      ▼
распределение следующего токена
      │
      ▼
выбор одного токена → добавить справа → повторить

Такой способ генерации называется авторегрессивным: каждый следующий элемент зависит от уже имеющейся последовательности, включая ранее сгенерированные элементы.

Когда цикл заканчивается

Цикл не обязан продолжаться бесконечно. Типичные причины остановки:

  1. выбран специальный stop-токен, обозначающий конец;
  2. достигнут заданный предел новых токенов;
  3. клиент или пользователь прервал запрос;
  4. сервис завершил генерацию по своему контракту безопасности или API.

В учебной модели будет ещё одна техническая причина: для текущего контекста нет подготовленного распределения. У настоящей LLM распределение строится по всему словарю на каждом шаге, поэтому эта причина нужна только нашему маленькому симулятору.

Важная граница учебной схемы

Реальная LLM не хранит таблицу из примера. Нейросеть вычисляет числовые оценки кандидатов — логиты, — а затем переводит их в вероятности преобразованием softmax. Здесь достаточно понимать назначение этого шага; его механику и разные правила выбора мы разберём в 1.3.

И ещё одна точность: фраза «модель заново читает весь контекст» полезна как интуиция зависимости, но не как описание производительности. Реальные системы кэшируют промежуточные вычисления, чтобы не пересчитывать одинаковый префикс полностью. Для этого урока важно только причинное правило: следующий прогноз зависит от видимого контекста и уже выбранных токенов.

Что именно находится в контексте

Контекст — последовательность токенов, которую модель получает для текущей генерации. В прикладном API туда могут попасть:

  • системные и разработческие инструкции;
  • сообщения текущего диалога, которые клиент действительно отправил;
  • текущий запрос пользователя;
  • вставленные документы и найденные фрагменты;
  • описания доступных инструментов;
  • уже сгенерированная часть ответа.

Контекстное окно — верхний предел того, сколько токенов конкретная модель способна учитывать в рамках одной генерации. Это не долговременная память и не вся история аккаунта. Если старое сообщение не было отправлено в текущем запросе, модель не может на него опереться, даже если пользователь видел его выше в интерфейсе.

Важно различать три слоя:

  1. История продукта. Интерфейс может хранить переписку в базе.
  2. Собранный запрос. Клиент выбирает, какие сообщения, файлы и инструкции отправить сейчас.
  3. Окно модели. Отправленная последовательность должна соответствовать лимитам выбранной модели.

Если собранный запрос слишком велик, результат зависит от API и клиента: сервис может отклонить запрос, а приложение — заранее обрезать, выбрать релевантные фрагменты или суммаризировать историю. Нельзя универсально считать, что «сама модель молча оставит последние N токенов». Сначала проверь фактически собранный запрос и контракт выбранного model ID.

Как считать бюджет окна

Для планирования удобно использовать консервативное неравенство:

посчитанный вход + резерв под новые токены ≤ лимит контекста

Вход складывается не только из видимого текста пользователя:

инструкции
+ отправленная история
+ текущий запрос
+ документы и результаты поиска
+ схемы инструментов
+ служебное оформление провайдера
= входные токены

Резерв под ответ нужен потому, что каждый сгенерированный токен добавляется к растущей последовательности. У конкретной модели может существовать и отдельный предел выходных токенов; точные правила всегда сверяй для выбранного model ID.

Числовой пример

Допустим, для учебного расчёта лимит равен 8 000 токенов:

ЧастьТокены
инструкции500
история3 600
текущий запрос и документ2 400
описания инструментов300
всего на входе6 800
резерв под ответ1 500

Получаем 6 800 + 1 500 = 8 300: план превышает окно на 300 токенов. Уменьшить только max_output до 1 200 технически можно, но это испортит задачу, если нужен развёрнутый отчёт. Осмысленнее проверить, все ли 3 600 токенов истории релевантны, и убрать повторяющиеся фрагменты.

Если после отбора история занимает 3 100 токенов:

вход = 500 + 3 100 + 2 400 + 300 = 6 300
план = 6 300 + 1 500 = 7 800
запас = 8 000 - 7 800 = 200 токенов

Расчёт теперь помещается, но запас мал. Это не автоматическое разрешение отправить чувствительные данные и не гарантия качества на длинном контексте — только проверка вместимости.

BUILD IT: сделай цикл наблюдаемым

Собери небольшой inference-loop без обучения модели и без сторонних библиотек. Готовая таблица будет играть роль нейросети на границе:

видимый контекст → распределение следующего токена

Так мы изолируем именно механизм урока и не добавляем отдельную задачу обучения игрушечной модели.

В 1.1 настоящие токены были числовыми IDs. Здесь для читаемости используем строки "blue", "sky" и "sea" как условные подписи отдельных токенов. Это упрощение представления, а не утверждение, что реальный токен всегда совпадает со словом.

Сначала сделай прогноз

В тестовой модели есть две записи:

{
    ("remember", "blue"): {"sky": 0.9, "sea": 0.1},
    ("blue",): {"sky": 0.2, "sea": 0.8},
}

Для prompt ['remember', 'blue'] заранее ответь:

  1. какой контекст будет виден при window=2 и какой токен выберет greedy;
  2. какой контекст будет виден при window=1 и какой токен выберет greedy;
  3. почему ответы отличаются, хотя исходный prompt одинаковый.

Задание

Работай в личной папке course-work/1.2/. В блоке «Исходники урока» скопируй туда только code/test_inference_loop.py; опубликованный code/inference_loop.py пока не копируй. Первый запуск должен завершиться ошибкой импорта — это ожидаемая красная точка.

Создай course-work/1.2/inference_loop.py и реализуй три функции:

  1. visible_context(context, window)
  • принимает последовательность токенов и положительный размер окна;
  • возвращает последние window токенов новым списком;
  • при window <= 0 выбрасывает ValueError.
  1. greedy_next_token(probabilities)
  • принимает непустой словарь {token: probability};
  • возвращает токен с наибольшей вероятностью;
  • при пустом словаре выбрасывает ValueError.
  1. generate(model, prompt, max_new_tokens, window, stop_token="<STOP>")
  • на каждом шаге получает видимый контекст через visible_context;
  • находит распределение в model по ключу-кортежу;
  • выбирает токен через greedy_next_token и записывает шаг в trace;
  • не добавляет stop_token в пользовательский результат;
  • останавливается с причиной stop_token, max_new_tokens или no_distribution;
  • возвращает словарь с ключами tokens, trace, stop_reason.

Одна запись trace должна иметь форму:

{
    "step": 1,
    "visible_context": ["remember", "blue"],
    "probabilities": {"sky": 0.9, "sea": 0.1},
    "chosen_token": "sky",
}

Что доказывают тесты

Тесты проверяют не факт запуска, а ключевое поведение:

  • greedy действительно выбирает наибольшую вероятность;
  • разный размер окна меняет видимый контекст и первый токен;
  • выбранный токен попадает в контекст следующего шага;
  • stop-токен не просачивается в текст результата;
  • лимит новых токенов соблюдается;
  • неизвестный контекст и недопустимое окно завершаются объяснимо.

Запусти цикл red → green из корня личного проекта:

python -m pytest course-work/1.2/test_inference_loop.py -q
python course-work/1.2/inference_loop.py

Демо должно вывести две трассы для одного prompt при window=2 и window=1. После зелёных тестов объясни без кода: почему первый токен различается и почему второй шаг уже зависит от первого сгенерированного токена.

Если делегируешь реализацию ИИ, сохрани четыре решения 4D за собой:

  • Delegation: отдай написание функций, но не прогноз результата и не приёмку;
  • Description: передай сигнатуры, формат trace, причины остановки и тесты как критерии;
  • Discernment: сравни реальную трассу со своим прогнозом и объясни каждый переход;
  • Diligence: сам запусти тесты, просмотри код и не принимай зелёный результат без понимания цикла.

USE IT: отдели генерацию от доставки

У реального API внутренний цикл не обязан быть виден напрямую. Streaming — режим доставки, при котором клиент получает части ответа до завершения всей генерации. Он сокращает воспринимаемое ожидание первого текста, но не расширяет контекстное окно и не превращает генерацию в другой алгоритм.

Важно: одно событие или текстовый delta API не обязаны совпадать с одним токеном. Сервис может объединять несколько токенов, делить текст по своим границам или присылать служебные события.

Если в 0.2 ты настроил API, выбери только своего провайдера:

  1. Возьми рабочий запрос из 0.2 и точный model ID из переменной окружения.
  2. По официальному примеру из раздела «Материалы» включи streaming.
  3. До запуска предположи, сколько текстовых частей увидишь и совпадут ли они со словами.
  4. Для каждой текстовой части выведи repr(part), накопи части в список и в конце объедини через "".join(parts).
  5. Запиши наблюдение: сколько пришло частей, где прошли их границы и почему по ним нельзя восстановить число токенов.

Не печатай ключ и не отправляй закрытые рабочие данные. Если API недоступен, пропусти эту ветку: локальная трасса уже доказывает механизм inference, а streaming является переносом в рабочий интерфейс, не обязательной опорой.

  • точный model ID, лимит и источник с датой проверки;
  • какие компоненты реально войдут в запрос;

  • чем контекст этого запроса отличается от памяти продукта.

Реалистичные ошибки и диагностика

  • «Интерфейс показывает сообщение, значит модель его видит». Проверь собранный запрос: клиент мог не отправить старое сообщение или заменить историю сводкой.
  • «При переполнении всегда автоматически останутся последние N токенов». Поведение зависит от клиента и API: возможны ошибка, явная обрезка, отбор или суммаризация. Не угадывай — проверь контракт и фактический состав отправленного запроса.
  • «Большое окно — долговременная память». Окно ограничивает одну генерацию. Сохранение фактов между запросами требует отдельного механизма продукта.
  • «Каждый streaming-chunk равен одному токену». Границы транспортных частей не обязаны совпадать с границами токенизации. Число токенов бери из подходящего счётчика или usage.
  • «Весь лимит можно отдать входу». Во время генерации последовательность растёт; резервируй место под ответ и учитывай отдельный output limit конкретной модели.
  • «Если запрос помещается, модель обязательно использует каждую деталь одинаково хорошо». Вместимость не равна качеству внимания. Отбор релевантного контекста появится в Фазах 3–4.

  • Gemini: Text generation — открой Streaming responses и используй актуальный пример своего SDK; затем Models — открой карточку выбранного стабильного model ID и найди его token limits. Не подменяй стабильный ID указателем latest, не зафиксировав это в отчёте.

2. Чтобы закрепить механизм — выбери один материал

  • Jay Alammar — The Illustrated GPT-2 — сначала посмотри верхнюю анимацию генерации, затем раздел Evaluation Time: Processing One Token at a Time: он визуально связывает новый токен, следующий проход и повторное использование прошлых вычислений. В иллюстрациях встречаются слова, но сопоставляй их с условными токенами Build It, а не с правилом «токен = слово».
  • Andrej Karpathy — Deep Dive into LLMs like ChatGPT — выбери главы Tokenization и Inference, а не смотри все 3,5 часа ради урока; полезен как единая устная трасса от текста и token IDs до распределения и продолжения последовательности.

3. Чтобы проверить границы модели

  • Anthropic — Context windows — прочитай How the context window works и Context window overflow behavior. Это конкретный API-контракт: что считается, как растёт контекст и почему нельзя переносить поведение одного сервиса на все клиенты.
  • Liu et al., 2024 — Lost in the Middle — достаточно abstract, Figure 1 и conclusion: работа показывает, почему «информация помещается в окно» не означает «модель одинаково хорошо использует её в любой позиции». Эксперименты проведены на моделях своего периода, поэтому это диагностический паттерн, а не гарантия поведения любого современного model ID.

4. Предпросмотр Фазы 4

  • Anthropic — Effective context engineering for AI agents — открой Context engineering for long-horizon tasks, особенно Compaction и Structured note-taking, затем conclusion. import pytest

from inference_loop import generate, greedy_next_token, visible_context


WINDOW_MODEL = {
    ("remember", "blue"): {"sky": 0.9, "sea": 0.1},
    ("blue",): {"sky": 0.2, "sea": 0.8},
    ("blue", "sky"): {"<STOP>": 1.0},
    ("sky",): {"<STOP>": 1.0},
    ("sea",): {"<STOP>": 1.0},
}


def test_visible_context_returns_only_tokens_inside_window():
    context = ["system", "old", "current"]

    assert visible_context(context, 2) == ["old", "current"]
    assert visible_context(context, 10) == context
    assert visible_context(context, 2) is not context


def test_visible_context_rejects_non_positive_window():
    with pytest.raises(ValueError, match="window"):
        visible_context(["token"], 0)


def test_greedy_selects_highest_probability():
    probabilities = {"likely": 0.65, "plausible": 0.25, "rare": 0.10}

    assert greedy_next_token(probabilities) == "likely"


def test_greedy_rejects_missing_distribution():
    with pytest.raises(ValueError, match="probabilities"):
        greedy_next_token({})


def test_context_window_changes_first_generated_token():
    prompt = ["remember", "blue"]

    wide = generate(WINDOW_MODEL, prompt, max_new_tokens=1, window=2)
    narrow = generate(WINDOW_MODEL, prompt, max_new_tokens=1, window=1)

    assert wide["trace"][0]["visible_context"] == ["remember", "blue"]
    assert narrow["trace"][0]["visible_context"] == ["blue"]
    assert wide["tokens"] == ["sky"]
    assert narrow["tokens"] == ["sea"]


def test_generated_token_becomes_context_for_next_step():
    model = {
        ("start",): {"middle": 0.8, "other": 0.2},
        ("middle",): {"finish": 0.7, "other": 0.3},
    }

    result = generate(model, ["start"], max_new_tokens=2, window=1)

    assert result["tokens"] == ["middle", "finish"]
    assert result["trace"][1]["visible_context"] == ["middle"]
    assert result["stop_reason"] == "max_new_tokens"


def test_stop_token_ends_generation_without_leaking_into_output():
    result = generate(WINDOW_MODEL, ["remember", "blue"], 3, window=2)

    assert result["tokens"] == ["sky"]
    assert result["trace"][-1]["chosen_token"] == "<STOP>"
    assert result["stop_reason"] == "stop_token"


def test_unknown_context_has_explicit_stop_reason():
    result = generate(WINDOW_MODEL, ["unknown"], 3, window=1)

    assert result == {
        "tokens": [],
        "trace": [],
        "stop_reason": "no_distribution",
    }


def test_max_new_tokens_is_a_hard_limit():
    looping_model = {("again",): {"again": 1.0}}

    result = generate(looping_model, ["again"], max_new_tokens=3, window=1)

    assert result["tokens"] == ["again", "again", "again"]
    assert len(result["trace"]) == 3
    assert result["stop_reason"] == "max_new_tokens"


def test_negative_generation_limit_is_rejected():
    with pytest.raises(ValueError, match="max_new_tokens"):
        generate(WINDOW_MODEL, ["blue"], max_new_tokens=-1, window=1)

"""Наблюдаемый autoregressive inference-loop для урока 1.2.

Таблица ``model`` заменяет нейросеть только на одной границе:
видимый контекст -> распределение следующего токена. Это позволяет изучить
сам цикл генерации, не добавляя отдельную задачу обучения игрушечной модели.
"""


def visible_context(context, window):
    """Вернуть последние ``window`` токенов новым списком."""
    if window <= 0:
        raise ValueError("window must be positive")
    return list(context[-window:])


def greedy_next_token(probabilities):
    """Вернуть токен с наибольшей вероятностью."""
    if not probabilities:
        raise ValueError("probabilities must not be empty")
    return max(probabilities, key=probabilities.get)


def generate(model, prompt, max_new_tokens, window, stop_token="<STOP>"):
    """Сгенерировать токены жадным выбором и вернуть подробную трассу.

    ``model`` — словарь вида ``{tuple(context): {token: probability}}``.
    Stop-токен фиксируется в трассе, но не входит в пользовательские tokens.
    """
    if max_new_tokens < 0:
        raise ValueError("max_new_tokens must be non-negative")

    context = list(prompt)
    tokens = []
    trace = []

    for step in range(1, max_new_tokens + 1):
        current_context = visible_context(context, window)
        probabilities = model.get(tuple(current_context))

        if not probabilities:
            return {
                "tokens": tokens,
                "trace": trace,
                "stop_reason": "no_distribution",
            }

        chosen_token = greedy_next_token(probabilities)
        trace.append(
            {
                "step": step,
                "visible_context": current_context,
                "probabilities": dict(probabilities),
                "chosen_token": chosen_token,
            }
        )

        if chosen_token == stop_token:
            return {
                "tokens": tokens,
                "trace": trace,
                "stop_reason": "stop_token",
            }

        tokens.append(chosen_token)
        context.append(chosen_token)

    return {
        "tokens": tokens,
        "trace": trace,
        "stop_reason": "max_new_tokens",
    }


if __name__ == "__main__":
    toy_model = {
        ("remember", "blue"): {"sky": 0.9, "sea": 0.1},
        ("blue",): {"sky": 0.2, "sea": 0.8},
        ("blue", "sky"): {"<STOP>": 1.0},
        ("sky",): {"<STOP>": 1.0},
        ("sea",): {"<STOP>": 1.0},
    }
    prompt = ["remember", "blue"]

    for context_window in (2, 1):
        result = generate(
            toy_model,
            prompt,
            max_new_tokens=3,
            window=context_window,
        )
        print(f"window={context_window}: {result}")
