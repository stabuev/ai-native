Урок 1.2 · Контекстное окно и inference

Фаза 1 — Как работают LLM · Результат фазы: Объяснить токены, контекст и inference; собрать токен-бюджетер; отличать надёжный ответ от галлюцинации.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Этот урок объясняет, как модель на самом деле генерирует ответ. Не «прочитала вопрос — выдала текст», а цикл: предсказала один токен, дописала его и повторила. Поймёшь этот цикл — и сразу ясно, почему ответ «печатается» постепенно (стриминг), почему модель «забывает» начало длинного диалога и что значит «контекст 200K токенов».

MOTTO. Inference — это не «один ответ», а цикл: предскажи токен, допиши, повтори.

PROBLEM

Кажется, что модель «читает вопрос и пишет ответ» целиком. На самом деле она генерирует по одному токену за шаг, каждый раз заново глядя на весь контекст. Не понимая этого, нельзя объяснить ни стриминг, ни почему «забывается» начало длинного диалога, ни что вообще значит «контекстное окно 200K токенов» — а из этого складывается и бюджет, и поведение модели.

CONCEPT

Интуиция

Генерация авторегрессивна: модель предсказывает следующий токен по всем предыдущим, дописывает его в конец и повторяет — пока не решит остановиться. Каждый шаг видит результат предыдущего, поэтому ответ растёт слева направо, по токену за раз. А смотрит модель не на всю историю мира, а на контекстное окно — последние N токенов; что вышло за окно, для неё просто исчезает.

Как это работает

prompt ─► [ контекстное окно ] ─► P(next token) ─► argmax/sample ─► токен
              ▲                                                       │
              └──────────────── допишем токен в контекст ◄───────────┘
                         (что вышло за окно — «исчезает»)

Два следствия: (1) ответ строится пошагово — поэтому возможен стриминг (токены отдаются по мере генерации); (2) окно конечно — длинная история выходит за него, и модель её больше не «видит» (отсюда бюджет контекста).

Reasoning-модели (Claude extended thinking, DeepSeek V4 thinking-mode, o-серия OpenAI) добавляют к циклу скрытый этап «размышления»: перед ответом генерируются thinking-токены, за которые ты тоже платишь. Это поднимает качество на сложных задачах, но увеличивает и стоимость, и задержку — учитывай в бюджете (Фаза 9).

РАЗБОР ПО ШАГАМ

Прокрутим цикл руками на крошечном корпусе:

["a","b","c","x"]   ["a","b","c","x"]   ["q","b","c","y"]

train_ngram считает, какой токен и как часто идёт за каждым суффиксом. Пара ("b","c") встречается трижды: дважды за ней x, один раз y. Значит:

next_token_probs(["b","c"]) → { "x": 2/3, "y": 1/3 }

Теперь сгенерируем продолжение для ["q","b","c"] — и сравним при разном окне:

  • окно = 8 (видит весь ("q","b","c")): в модели есть точное правило ("q","b","c") → y, поэтому следующий токен — y;
  • окно = 2 (видит только ("b","c")): длинного совпадения нет, backoff к ("b","c"), greedy берёт самый вероятный — x.

Один и тот же prompt, разный контекст в окне → разный токен. Это и есть эффект контекстного окна, который ты соберёшь в коде.

BUILD IT

Задание: собери учебный inference-loop — стандартная библиотека, без сети. Модель игрушечная (n-граммы с backoff), но механизм тот же, что у LLM: предсказать токен → дописать → повторить, глядя на контекстное окно.

Перед запуском. Работай в своей папке курса (ai-native/1.2-context-window/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3, всё офлайн (для теста ещё pytest).

Создай inference_loop.py и реализуй три функции:

  • train_ngram(corpus, order=3) — corpus это список последовательностей токенов. Для каждого суффикса длины 1..order посчитай, какие токены за ним идут, и переведи в вероятности. Верни dict: { суффикс-кортеж: { токен: вероятность } }.
  • next_token_probs(model, context, order=3) — верни распределение по самому длинному суффиксу context (до order), который есть в model. Нет длинного совпадения — отступай к более короткому (backoff); совсем ничего — верни {}.
  • generate(model, prompt, max_tokens=10, window=8, order=3) — жадная авторегрессия: на каждом шаге смотри только на последние window токенов, бери самый вероятный следующий (argmax), дописывай в контекст. Останавливайся, если предсказать нечего.

Готово, когда тест test_inference_loop.py зелёный — он проверяет, что вероятности суммируются в 1, работает backoff, greedy детерминирован, окно меняет результат и генерация останавливается на неизвестном контексте.

USE IT

Тот же цикл — у реальной модели, и его видно через стриминг: ответ отдаётся не целиком, а кусочками по мере генерации. Anthropic описывает это так: «Stream Messages API responses incrementally with server-sent events» — каждый text_delta несёт только что сгенерированный кусок. Мульти-провайдер:

# Anthropic
from anthropic import Anthropic
with Anthropic().messages.stream(model="claude-opus-4-8", max_tokens=200,
    messages=[{"role": "user", "content": "Объясни inference в 3 предложениях"}]) as s:
    for text in s.text_stream:
        print(text, end="")

# OpenAI
from openai import OpenAI
for ev in OpenAI().responses.create(model="gpt-5.x",
        input="Объясни inference в 3 предложениях", stream=True):
    if ev.type == "response.output_text.delta":
        print(ev.delta, end="")

# Google
from google import genai
for chunk in genai.Client().models.generate_content_stream(
        model="gemini-2.x-flash", contents="Объясни inference в 3 предложениях"):
    print(chunk.text, end="")

Каждый «delta» — это один (или несколько) только что сгенерированных токенов: тот самый цикл из Build It, но в проде.

ЧАСТЫЕ ОШИБКИ

  • Думать, что модель «видит весь разговор». Она видит только то, что влезло в контекстное окно; что вышло за него — для неё не существует. Длинный диалог надо ужимать (Фаза 4).
  • Считать ответ «единым актом». Это цикл по токену; отсюда и стриминг, и то, что генерация дороже и дольше с ростом длины ответа.
  • Забывать про thinking-токены. У reasoning-моделей скрытый этап размышления тоже тратит токены и время — закладывай в бюджет.
  • Путать большой контекст с памятью. Окно в 200K — это не «модель помнит всё»; это лимит на один запрос, и забивать его под завязку дорого и не всегда полезно.

code/test_inference_loop.py
from inference_loop import train_ngram, next_token_probs, generate

CORPUS = [
    ["a", "b", "c", "x"],
    ["a", "b", "c", "x"],
    ["q", "b", "c", "y"],
]


def test_probs_sum_to_one():
    m = train_ngram(CORPUS, order=3)
    probs = next_token_probs(m, ["b", "c"])
    assert abs(sum(probs.values()) - 1.0) < 1e-9
    assert probs == {"x": 2 / 3, "y": 1 / 3}


def test_backoff_to_shorter_suffix():
    m = train_ngram(CORPUS, order=3)
    # ("z","c") нет в модели → backoff к ("c",)
    assert next_token_probs(m, ["z", "c"]) == next_token_probs(m, ["c"])


def test_greedy_is_deterministic():
    m = train_ngram(CORPUS, order=3)
    assert generate(m, ["a", "b", "c"], max_tokens=1) == generate(m, ["a", "b", "c"], max_tokens=1)


def test_context_window_changes_output():
    m = train_ngram(CORPUS, order=3)
    full = generate(m, ["q", "b", "c"], max_tokens=1, window=8)   # видит ("q","b","c") → y
    narrow = generate(m, ["q", "b", "c"], max_tokens=1, window=2)  # видит ("b","c") → x
    assert full == ["y"]
    assert narrow == ["x"]


def test_unknown_context_stops():
    m = train_ngram(CORPUS, order=3)
    assert generate(m, ["zzz"], max_tokens=5) == []

code/inference_loop.pyПоказать решение
"""Учебный inference-loop + контекстное окно — Build It для урока 1.2.

Без зависимостей. Показывает, что inference — это цикл: модель раз за разом
предсказывает следующий токен по предыдущим и дописывает его в контекст.
Модель здесь игрушечная (n-граммы с backoff по маленькому корпусу), но механизм
тот же, что у LLM. Контекстное окно — сколько последних токенов модель видит:
что вышло за окно, для модели «исчезает».
"""
from collections import defaultdict, Counter


def train_ngram(corpus, order=3):
    """Для каждого суффикса длины 1..order — распределение следующего токена."""
    counts = defaultdict(Counter)
    for seq in corpus:
        for i in range(len(seq)):
            for n in range(1, order + 1):
                if i - n >= 0:
                    counts[tuple(seq[i - n:i])][seq[i]] += 1
    model = {}
    for ctx, c in counts.items():
        total = sum(c.values())
        model[ctx] = {tok: n / total for tok, n in c.items()}
    return model


def next_token_probs(model, context, order=3):
    """Распределение по самому длинному суффиксу контекста (до order), что есть в модели.

    Это backoff: нет длинного совпадения — отступаем к более короткому.
    """
    for n in range(min(order, len(context)), 0, -1):
        ctx = tuple(context[-n:])
        if ctx in model:
            return model[ctx]
    return {}


def generate(model, prompt, max_tokens=10, window=8, order=3):
    """Жадная авторегрессия: на каждом шаге берём самый вероятный next-token.

    `window` — контекстное окно: модель видит только последние `window` токенов.
    """
    context = list(prompt)
    out = []
    for _ in range(max_tokens):
        visible = context[-window:]                  # контекстное окно
        probs = next_token_probs(model, visible, order)
        if not probs:
            break                                    # нечего предсказать — стоп
        nxt = max(probs, key=probs.get)              # greedy = argmax
        out.append(nxt)
        context.append(nxt)
    return out


if __name__ == "__main__":
    corpus = [
        ["the", "cat", "sat", "on", "the", "mat"],
        ["the", "cat", "ate", "the", "fish"],
        ["a", "dog", "sat", "on", "the", "rug"],
    ]
    model = train_ngram(corpus, order=3)
    print("Контекст ['the','cat'] →", next_token_probs(model, ["the", "cat"]))
    print("generate(['the','cat']) →", generate(model, ["the", "cat"], max_tokens=4))
    print("Окно=1, ['a','dog','sat'] →", generate(model, ["a", "dog", "sat"], 3, window=1))

# Артефакт: памятка по бюджету контекста

Контекстное окно — конечный ресурс. Всё, что в него кладёшь, занимает токены и стоит денег; что вышло за окно — модель не видит.

## Из чего складывается контекст

```
[ system prompt ] + [ история диалога ] + [ RAG/вложения ] + [ запас под ответ (max_tokens) ]  ≤  окно модели
```

| Слой | Что занимает | Как ужать |
|---|---|---|
| System prompt | роль, правила, формат | вынести стабильное в кэш (prompt caching) |
| История | прошлые сообщения | суммаризировать старое, держать только релевантное |
| RAG/файлы | фрагменты документов | top-k чанков, а не весь корпус (Фаза 3) |
| Ответ | `max_tokens` | резервируй явный запас, не «впритык» |

## Правила

1. **Считай заранее.** Прогоняй вход через `prompt-token-budgeter` (урок 1.1) до отправки.
2. **Режь по приоритету.** Первым сокращай старую историю и низкорелевантный RAG, не system prompt.
3. **Оставляй запас под ответ.** Вход + `max_tokens` должны влезать в окно с зазором.
4. **Большое окно ≠ бесплатно.** 200K–1M токенов влезут, но за них платишь и растёт задержка.
5. **Длинная история «забывается».** Что вышло за окно — для модели не существует; нужное возвращай явно (память/RAG).
