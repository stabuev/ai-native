Урок 0.4 · Как проходить практику курса

Фаза 0 — Setup & Tooling · Результат фазы: Настроить окружение, ключи и агента; осознанно выбрать модель под задачу.

Результат урока: подготовить личную папку для практики, выбрать осознанный способ выполнения задания, проверить результат подходящим способом и безопасно сохранить доказательства прохождения — не путая их с готовыми исходниками курса.

Опора: безопасное окружение из 0.2, рабочая папка, агент и персональный маршрут из 0.3. · Дальше: при focus или refresh ты применишь цикл в 1.1 к мини-токенизатору; при challenge начнёшь с выходной проверки Фазы 1 и вернёшься в найденные пробелы.

Requires: Python-окружение из 0.2. Интернет нужен только если в нём ещё не установлен pytest.

MOTTO. Пройти урок — не значит прочитать текст или получить зелёный тест на чужом решении; нужно выполнить действие самому, проверить его и уметь объяснить результат.

Зачем нужен отдельный рабочий цикл

В прошлых уроках ты выбрал инструменты, подготовил окружение и запустил агента. Теперь возникает новая проблема: в материалах курса уже лежат тесты, примеры, а иногда и готовое решение. Можно запустить их, увидеть зелёный результат и ничему не научиться.

Чтобы этого не происходило, разделяй четыре вещи:

  • задание — что тебе нужно сделать самостоятельно;
  • критерии приёмки — наблюдаемые признаки правильного результата;
  • эталонное решение — материал для подсказки или сравнения после собственной попытки;
  • личная работа — твои файлы, решения, выводы и история изменений.

Тест — лишь один вид критерия приёмки. Код проверяют запуском и тестами, решение — содержательной рубрикой, а процедуру — сценарием и наблюдаемыми действиями. Наличие отдельного файла-артефакта не обязательно: он нужен только тогда, когда будет полезен после урока.

Бери материалы с сайта, а работай в своей папке

Курс проходится с сайта: клонировать его репозиторий не нужно. Внизу страницы каждого урока есть блок «Исходники урока» с файлами, которые можно скопировать по одному. Сохраняй только нужные для текущей попытки входные данные и проверки в своей папке ai-native-work; опубликованные решения остаются на странице как справочный материал.

Что перед тобойКак использовать
Страница урокапрочитать результат, объяснение и задание
Карточка test_*.py в «Исходниках урока»скопировать в личную папку как исполняемую часть критериев
Готовый файл из code/ на страницеоткрывать согласно выбранному пути, но не выдавать его копирование за свою работу

Если ты просто скопировал готовое решение со страницы и запустил приложенные тесты, зелёный результат подтверждает только исправность опубликованного эталона. Он ничего не говорит о твоей реализации. Для самостоятельной попытки сначала копируй в course-work/<номер урока>/ тесты и задание, но не решение.

Один цикл для разных уроков

Каждый урок проходи в одной последовательности:

  1. Назови результат своими словами. Какое действие ты должен уметь повторить после урока?
  2. Найди критерии приёмки. Что должно запуститься, какое решение нужно обосновать или какое поведение продемонстрировать?
  3. Подготовь личную папку. Перенеси туда только задание, входные данные и проверки, необходимые для твоей попытки.
  4. Выбери путь выполнения. Собрать самому, обратиться к эталону после попытки или делегировать часть работы ИИ.
  5. Проверь и диагностируй. Не просто получи результат, а разберись, почему он верен и какую реалистичную ошибку ловит проверка.
  6. Сохрани переносимый итог. Оставь полезный файл, решение, объяснение или запись о проверенной процедуре и обнови следующий шаг.

Три пути не означают три уровня честности:

  • Собрать самому полезно, когда механизм нужно увидеть изнутри.
  • Подсмотреть эталон нормально после собственной попытки или при точечном тупике; сначала сформулируй, чего именно не хватает.
  • Делегировать ИИ нормально, если решение о делегировании, критерии и окончательная проверка остаются у тебя.

На любом пути критерии приёмки одинаковы. Файл, созданный агентом, не становится твоим пониманием только потому, что находится в твоей папке.

AI Fluency и 4D: четыре компетенции, а не магический промпт

AI Fluency — способность работать с ИИ результативно, рационально, этично и безопасно. Рамка 4D раскладывает эту способность на четыре связанные компетенции: Delegation, Description, Discernment и Diligence. Это не четыре одноразовых шага. Решение о границе делегирования можно пересмотреть, а Description и Discernment обычно образуют цикл:

решить, что делегировать
        ↓
описать задачу → получить результат → оценить результат
                    ↑                   ↓
                    └── уточнить описание

ответственность, безопасность и прозрачность действуют на всём цикле

1. Delegation — что и зачем отдавать ИИ

Delegation начинается не с выбора модели, а с понимания собственной цели.

  • Problem Awareness — понимание задачи. Какой результат нужен и чему ты должен научиться? В 1.1 цель — не добыть файл с зелёными тестами, а понять, почему частые пары превращаются в токены и как проверить обратимость кодирования.
  • Platform Awareness — понимание инструмента. Агент умеет читать файлы, писать код и запускать тесты, но может изменить тест, захардкодить пример или уверенно объяснить ошибочное решение. Доступ к терминалу не равен надёжности.
  • Task Delegation — разделение работы. Реши, что должен сделать ты, что можно сделать вместе и что можно отдать целиком. Например, ты оставляешь себе прогноз слияний и объяснение инварианта, а агенту отдаёшь каркас функций или диагностику одного падающего теста.

Хороший контрольный вопрос: если ИИ выполнит эту часть идеально, какое понимание всё равно должно остаться у меня? Иногда правильное решение Delegation — пока не использовать ИИ.

2. Description — как сделать намерение наблюдаемым

Description шире, чем «написать подробный промпт». Нужно описать три стороны совместной работы:

  • Product Description — продукт. Какой файл или ответ нужен, для какой аудитории, в каком формате и по каким критериям он принимается?
  • Process Description — процесс. Какие источники использовать, какие шаги выполнить, что сначала проверить и что запрещено менять?
  • Performance Description — поведение. Должен ли агент быть кратким, задавать вопросы при противоречии, останавливать работу при нехватке данных или оспаривать слабое решение?

Для будущего упражнения слабое описание выглядит так: сделай токенизатор. Более управляемое — так:

Работай только в course-work/1.1/ и не изменяй test_tokenizer.py.
Реализуй tokenizer.py по контракту из урока 1.1.

Продукт: три функции с указанными сигнатурами; критерий — мои тесты и
decode(encode(text, merges)) == text.
Процесс: сначала кратко перечисли понятые инварианты и только затем меняй файл;
не хардкодь примеры и не ослабляй тесты.
Поведение: если спека и тест расходятся, остановись и задай вопрос; после изменения
назови допущения и команду проверки. Другие файлы не трогай.

Такое описание не гарантирует правильность, но делает отклонение заметным.

3. Discernment — как оценивать не только финальный файл

Discernment — критическая оценка того, что сделал ИИ, как он к этому пришёл и как вёл себя в работе.

  • Product Discernment — продукт. Проходит ли решение тесты, соответствует ли контракту, корректно ли на новых данных и можно ли объяснить его результат?
  • Process Discernment — процесс. Не подменил ли агент задачу, не изменил ли проверки, не захардкодил ли ответы и не опирался ли на недоступные данные?
  • Performance Discernment — поведение. Сообщал ли агент о допущениях и ограничениях, соблюдал ли границы папки, задавал ли вопрос вместо уверенной выдумки?

Зелёные тесты входят только в Product Discernment и тоже не исчерпывают его. После них посмотри изменения, придумай хотя бы один новый пример и объясни ключевую причинную связь без подсказки агента.

4. Diligence — ответственность до, во время и после работы

Diligence отвечает не за качество промпта, а за ответственное использование ИИ.

  • Creation Diligence — осознанный выбор среды. Разрешено ли использовать этот инструмент для данных задачи, какие файлы ему действительно нужны и не попадут ли в контекст секреты?
  • Transparency Diligence — прозрачность. Нужно ли коллеге, преподавателю или заказчику знать, какую часть сделал ИИ? Если это влияет на доверие или правила работы, зафиксируй роль ИИ прямо.
  • Deployment Diligence — ответственность за применение. Перед коммитом, отправкой или запуском сам проверь результат и будь готов за него отвечать. Фраза «так написал агент» не переносит ответственность на модель.

В этом курсе минимальная Diligence означает: не отправлять .env и ключи в чат, давать агенту доступ только к рабочей папке, самостоятельно запускать проверки, просматривать изменения и честно отмечать существенную помощь ИИ.

Практика: подготовь личный цикл перед следующим шагом

Шаг 1. Проверь среду

В корне той же рабочей папки, которую использовал в 0.2–0.3, выполни:

python -c "import sys; print(sys.executable)"
python -m pytest --version

Первая команда должна показать Python из твоего виртуального окружения. Если вторая сообщает, что модуль pytest не найден, установи его в активное окружение:

python -m pip install "pytest>=8"

Если формат запуска или отчёт об ошибке пока непривычны, сверь их с разделами Install pytest, Create your first test и Run multiple tests в pytest — Get Started.

Шаг 2. Сверь следующий шаг с маршрутом

Открой personal-route.md и найди действие для Фазы 1.

Если там focus или refresh, подготовь красную стартовую точку 1.1.

Создай папку course-work/1.1/. Открой урок 1.1 через каталог сайта, прокрути до «Исходников урока», нажми «Копировать» у code/test_tokenizer.py и сохрани файл как course-work/1.1/test_tokenizer.py. Карточку готового code/tokenizer.py пока не открывай и этот файл не создавай.

Запусти проверку из корня своего проекта:

python -m pytest course-work/1.1 -q

Ожидаемый результат сейчас — ошибка импорта tokenizer: твоей реализации ещё нет. Это красная стартовая точка, а не поломка. В 1.1 ты создашь файл и проведёшь проверку от красного результата к зелёному.

Если там challenge, пока не копируй решение 1.1. Запиши следующим действием выходную проверку /check-understanding 1 и артефакт Фазы 1. Challenge — не пропуск: если проверка обнаружит пробел в токенизации, маршрут вернёт тебя в 1.1, и тогда ты подготовишь ту же красную стартовую точку.

Шаг 3. Собери навигатор по своей работе

Создай course-work/README.md. Можно написать его самому или поручить агенту первый черновик. В любом случае проверь содержимое по критериям ниже.

В файле должны быть:

  • пути к результатам 0.1–0.3: карте выбора, .env.example и проверке окружения, agent-check.md, personal-route.md;
  • короткий шестишаговый цикл прохождения урока;
  • ближайшее действие из маршрута и причина: 1.1 для focus/refresh либо выходная проверка Фазы 1 для challenge;
  • если следующий шаг — 1.1, выбранный путь выполнения упражнения;
  • четыре решения 4D для следующего шага: что делегируется, как ставится задача, как проверяется результат и какие правила ответственности действуют;
  • условие возврата: какой обнаруженный пробел заставит перейти от challenge к конкретному уроку.

Если делегируешь черновик агенту, не разрешай ему читать .env и не проси угадывать пути. Пусть сначала найдёт только перечисленные безопасные файлы, а отсутствующие отметит как пробел. После записи проверь, что изменился только course-work/README.md, ссылки ведут к реальным файлам, а секретных значений внутри нет.

Шаг 4. Закрой Фазу 0 безопасным коммитом

Локальный git-репозиторий уже был создан в ai-native-work в 0.1. Если git status сообщает, что текущая папка не является репозиторием, сначала остановись, проверь путь и только затем при необходимости повтори git init. Проверь состояние:

git status --short
git check-ignore .env .venv

.env и .venv должны быть проигнорированы. Добавь через git add только лично проверенные файлы Фазы 0, перечислив их имена явно; не используй git add . вслепую:

git add .gitignore .env.example check_env.py task-to-model-matrix.md
git add agent-check.md personal-route.md course-work/README.md

Для focus/refresh отдельно добавь подготовленный тест 1.1:

git add course-work/1.1/test_tokenizer.py

Если создавал безопасный smoke_test.py без значения ключа, проверь его и добавь отдельной командой. Каталоги двух скачанных скиллов также можно добавить только после просмотра их содержимого, указав фактический путь выбранного агента. Перед коммитом обязательно посмотри состав и текст изменений:

git diff --cached --check
git diff --cached
git commit -m "Complete phase 0 setup"

Не отправляй вывод git diff агенту, пока сам не убедился, что там нет секрета. Коммит завершает настройку, но главным доказательством остаётся твоя способность повторить рабочий цикл в следующем уроке.

Реалистичные ошибки

  • Скопировать со страницы готовое решение и принять зелёные тесты за свою работу. Ты проверил эталон. Начни с теста и собственной реализации в course-work/<урок>/.
  • Сразу раскрыть решение, хотя цель — понять механизм. Сначала сделай прогноз и самостоятельную попытку; затем обращайся к конкретному фрагменту эталона.
  • Попросить агента «сделать всё». Без границы делегирования ты не знаешь, какое понимание должен проверить у себя.
  • Считать тесты полным доказательством. Тесты покрывают известные случаи; добавь новый пример и объясни причинную связь.
  • Проверить только итоговый файл. Посмотри процесс: какие файлы менялись, не ослаблены ли критерии и не скрыты ли допущения.
  • Случайно закоммитить секрет. Проверяй игнорирование и staged diff до коммита; при утечке отзывай ключ, а не просто удаляй строку новым коммитом.
  • Игнорировать персональный маршрут. Focus/refresh начинают с урока, а challenge — с выходной проверки; найденный пробел возвращает нужный урок в маршрут.
  • Создавать одинаковый набор файлов в каждом уроке. Форма следует из результата: иногда нужен код, иногда решение или проверенная процедура.
