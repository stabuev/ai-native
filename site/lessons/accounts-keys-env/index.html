Урок 0.2 · API-доступ, ключи и локальное окружение

Фаза 0 — Setup & Tooling · Результат фазы: Настроить окружение, ключи и агента; осознанно выбрать модель под задачу.

Результат урока: ты создашь изолированное Python-окружение, подготовишь безопасное место для API-ключа одного выбранного провайдера и проверишь конфигурацию без отправки секрета в сеть. Если API-доступ уже есть, завершишь настройку первым настоящим запросом к модели.

Requires: Python 3 и терминал. Для установки пакетов нужен интернет. Для условного шага с реальным запросом нужны аккаунт, API-ключ одного провайдера и доступ к его API; провайдер может отдельно запросить настройку биллинга. Подписка на веб-чат и API-доступ — разные продукты, поэтому не рассчитывай, что одна автоматически включает другую. Без ключа обязательную офлайн-часть урока всё равно можно закончить.

Продолжай в личной папке ai-native-work, созданной в 0.1. Открой лежащую в её корне заполненную task-to-model-matrix.md и не подменяй то решение новым только ради этого урока:

  • если в одной из карточек уже выбрана модель через API, настраивай того же провайдера;
  • если для твоих задач разумнее чат, агент, локальный вариант или работа без ИИ, настрой один доступный API только как инфраструктуру курса;
  • если API-доступа пока нет, заверши обязательную офлайн-часть и вернись к реальному запросу позже.

Этот урок учит безопасно и воспроизводимо подключать код к провайдеру, но не утверждает, что API подходит каждой задаче. Сразу настраивать Anthropic, OpenAI и Google не нужно.

ЗАЧЕМ ЭТО НУЖНО

Представь: ты вставил API-ключ прямо в Python-файл, запрос сработал, а затем файл попал в git. Если удалить строку следующим коммитом, исчезнет ли секрет из предыдущего коммита? Нет. История сохранит старую версию, а человек или автоматический сканер сможет найти ключ.

Поэтому при утечке недостаточно отредактировать код. Ключ нужно отозвать у провайдера и выпустить новый, а секрет удалить из репозитория. Дешевле не допустить утечку: держать ключ вне кода и до его добавления убедиться, что git игнорирует файл с секретами.

Есть и вторая проблема. Команда pip install без изоляции меняет общий Python на компьютере. Один проект может требовать одну версию библиотеки, другой — несовместимую. Поэтому зависимости каждого проекта живут в собственном виртуальном окружении.

КАК УСТРОЕНО БЕЗОПАСНОЕ ЛОКАЛЬНОЕ ОКРУЖЕНИЕ

Сначала разберём объекты, с которыми будем работать.

  • API-доступ — право аккаунта отправлять программные запросы провайдеру. Условия доступа, бесплатные лимиты и биллинг зависят от провайдера и меняются со временем.
  • API-ключ — секретная строка, по которой провайдер связывает запрос с аккаунтом или проектом. Обладатель ключа может расходовать доступные лимиты, поэтому ключ защищают как пароль.
  • SDK — готовая Python-библиотека провайдера. Она формирует программные запросы к API, чтобы тебе не пришлось собирать их вручную.
  • Переменная окружения — именованное значение, которое процесс получает при запуске. SDK обычно сам ищет ключ по ожидаемому имени: например, библиотека OpenAI читает OPENAI_API_KEY.
  • Виртуальное окружение (venv) — отдельный Python и набор пакетов для одного проекта.
  • .env — локальный файл вида ИМЯ=ЗНАЧЕНИЕ. В учебных локальных проектах мы загружаем из него переменные окружения с помощью python-dotenv.
  • .env.example — безопасный образец: показывает имена переменных, но не содержит секретов.
  • .gitignore — список файлов, которые git не должен добавлять в новые коммиты.

Эти части работают цепочкой:

.env с одним ключом
        │  load_dotenv() загружает значение
        ▼
переменная окружения
        │  SDK читает ожидаемое имя
        ▼
запрос к API выбранного провайдера

.gitignore ── не даёт новому .env попасть в git
.venv      ── изолирует SDK и python-dotenv от других проектов

.env — удобный способ для локальной разработки в курсе, а не универсальное хранилище секретов. В развёрнутом сервисе ключи обычно передают через защищённые настройки платформы или менеджер секретов. До развёртывания сервиса мы дойдём позже; сейчас важно освоить безопасный локальный минимум.

ПРАКТИКА: НАСТРОЙ ОДНОГО ПРОВАЙДЕРА

Здесь не нужно писать собственный парсер .env. Настройка окружения и диагностика ошибок и есть самостоятельная практика урока.

Шаг 1. Создай и активируй виртуальное окружение

Открой терминал в ai-native-work и сначала проверь текущий путь командой pwd на macOS/Linux или Get-Location в Windows PowerShell. Затем выполни команды для своей системы.

macOS или Linux:

python3 -m venv .venv
source .venv/bin/activate
python -c "import sys; print(sys.executable)"

Windows PowerShell:

py -m venv .venv
.venv\Scripts\Activate.ps1
python -c "import sys; print(sys.executable)"

Последняя команда должна вывести путь внутри .venv. Это наблюдаемое доказательство, что дальнейшие пакеты попадут в окружение проекта, а не в общий Python.

Шаг 2. Подготовь файлы до добавления секрета

На странице урока прокрути вниз до блока «Исходники урока». В артефакте outputs/env-starter/ по очереди нажми «Копировать» и сохрани в корень ai-native-work три файла с точными именами: .env.example, .gitignore и check_env.py. Если читаешь Markdown-версию, те же файлы лежат в outputs/env-starter/. Клон репозитория для этого не нужен.

После копирования папка должна выглядеть так:

ai-native-work/
├── .env.example
├── .gitignore
├── .venv/
├── check_env.py
└── task-to-model-matrix.md

Затем создай локальный .env.

macOS или Linux:

cp .env.example .env
python check_env.py

Windows PowerShell:

Copy-Item .env.example .env
python check_env.py

Пока все значения пустые, проверка должна завершиться сообщением:

Проблемы окружения:
  - не настроен ни один поддерживаемый API-ключ

Это ожидаемый результат, а не поломка: валидатор обнаружил реальную незавершённую настройку. Он работает локально, не печатает значения ключей и не обращается к API.

В 0.1 ты уже создал здесь локальный git-репозиторий. Проверь правило игнорирования:

git check-ignore .env
git status --short

Первая команда должна вывести .env, а вторая не должна показывать его среди новых файлов. Важно: .gitignore защищает только ещё не отслеживаемые файлы. Если .env уже попадал в коммит, сначала отзови ключ, затем убери файл из отслеживания и при необходимости очисти историю.

Шаг 3. Выбери одну ветку

Не устанавливай все SDK. Сначала сверь поле «Способ работы и кандидат» в карте из 0.1. Если там есть API-кандидат и доступ к нему, продолжай с тем же провайдером. Иначе выбери одного доступного провайдера для упражнений курса, не объявляя его новым победителем для рабочей задачи. Если доступа нет ни к одному, не выдумывай выбор и оставайся на офлайн-ветке. Используй только одну строку таблицы.

ПровайдерПеременная в .envУстановка в активном .venv
AnthropicANTHROPIC_API_KEYpython -m pip install anthropic python-dotenv
OpenAIOPENAI_API_KEYpython -m pip install openai python-dotenv
Google GeminiGEMINI_API_KEYpython -m pip install google-genai python-dotenv

Создай ключ по официальной инструкции выбранного провайдера, вставь его только в соответствующую строку локального .env, а остальные строки оставь пустыми. Не отправляй ключ в чат, не вставляй в скриншот и не добавляй в .env.example.

Повтори проверку:

python check_env.py

Успешный результат называет настроенного провайдера, но никогда не показывает сам ключ. Проверка подтверждает только наличие и грубый формат строки. Она не доказывает, что ключ действителен, не отозван, относится к нужному проекту и имеет доступ к выбранной модели. Это можно выяснить только реальным запросом.

Без API-ключа остановись здесь: ты прошёл обязательную офлайн-диагностику, если можешь объяснить, почему первый запуск завершился ошибкой, где должен лежать настоящий ключ и почему .env.example можно коммитить, а .env — нельзя.

УСЛОВНЫЙ ШАГ: СДЕЛАЙ ПЕРВЫЙ API-ЗАПРОС

ЧТО ОСТАЁТСЯ ПОСЛЕ УРОКА

Исходный артефакт outputs/env-starter/ остаётся на странице урока, а его личную копию можно переносить в следующие локальные Python-проекты. В ней есть:

  • .env.example с поддерживаемыми именами переменных без секретов;
  • .gitignore, который исключает .env и .venv;
  • check_env.py, который принимает конфигурацию хотя бы одного провайдера и диагностирует отсутствие ключа, учебную заглушку вроде your-key-here и явно неверный формат;
  • README.md с короткой инструкцией повторного использования.

Урок завершён, когда выполнены применимые критерии:

  • sys.executable указывает внутрь .venv;
  • .env игнорируется git, а .env.example не содержит значений;
  • без ключа ты получаешь и объясняешь ожидаемую диагностику;
  • с ключом одного провайдера check_env.py проходит, не требуя два остальных;
  • если API-доступ есть, выбранный smoke_test.py печатает ответ модели;
  • ты понимаешь, что зелёный локальный валидатор ещё не подтверждает действительность ключа.

## Что скопировать в проект

- `.env.example` — имена поддерживаемых переменных без секретов;
- `.gitignore` — исключает локальный `.env`, `.venv` и кэш Python;
- `check_env.py` — проверяет наличие и грубый формат ключа без сети.

## Как использовать

1. Скопируй все три файла в корень Python-проекта.
2. Создай и активируй `.venv`.
3. Создай локальный файл: `cp .env.example .env`.
4. Выбери одного провайдера и заполни только его строку в `.env`.
5. Запусти `python check_env.py`.

Пустые строки не мешают проверке: достаточно одного настроенного провайдера. Скрипт не
печатает значения и не обращается к API. Зелёный результат означает только, что строка
присутствует и похожа на ключ; действительность ключа проверяется первым API-запросом.

Перед добавлением настоящего ключа проверь, что `.env` игнорируется:

```bash
git check-ignore .env
git status --short
```

Если секрет уже попал в коммит, немедленно отзови его у провайдера и выпусти новый.
outputs/env-starter/check_env.py
"""Локально проверить, что настроен API-ключ хотя бы одного провайдера.

Скрипт не использует сторонние библиотеки, не обращается к сети и не печатает
значения ключей. Успешная проверка подтверждает только наличие и грубый формат,
но не действительность ключа или доступ к конкретной модели.
"""

from __future__ import annotations

import sys
from pathlib import Path


PLACEHOLDERS = {"your-key-here", "changeme", "xxx", "...", "todo"}
PROVIDERS = {
    "ANTHROPIC_API_KEY": ("Anthropic", "sk-ant-"),
    "OPENAI_API_KEY": ("OpenAI", "sk-"),
    "GEMINI_API_KEY": ("Google Gemini", ""),
}


def _clean(value: str) -> str:
    """Убрать внешние кавычки и комментарий после значения."""
    value = value.strip()
    if value[:1] in {'"', "'"}:
        quote = value[0]
        return value[1:].split(quote, 1)[0]
    return value.split(" #", 1)[0].strip()


def parse_env(text: str) -> dict[str, str]:
    """Прочитать простые строки KEY=VALUE из текста .env."""
    env: dict[str, str] = {}
    for raw in text.splitlines():
        line = raw.strip()
        if not line or line.startswith("#") or "=" not in line:
            continue
        if line.startswith("export "):
            line = line[len("export "):].strip()
        key, _, value = line.partition("=")
        env[key.strip()] = _clean(value)
    return env


def validate(env: dict[str, str]) -> tuple[list[str], list[str]]:
    """Вернуть найденные проблемы и названия настроенных провайдеров."""
    problems: list[str] = []
    configured: list[str] = []

    for key, (provider, prefix) in PROVIDERS.items():
        value = env.get(key, "").strip()
        if not value:
            continue
        if value.lower() in PLACEHOLDERS:
            problems.append(f"{key}: осталось учебное значение вместо ключа")
            continue
        if prefix and not value.startswith(prefix):
            problems.append(f"{key}: значение не похоже на ключ {provider}")
            continue
        configured.append(provider)

    if not configured and not problems:
        problems.append("не настроен ни один поддерживаемый API-ключ")

    return problems, configured


def main(argv: list[str] | None = None) -> int:
    args = list(sys.argv[1:] if argv is None else argv)
    if len(args) > 1:
        print("Использование: python check_env.py [путь_к_.env]")
        return 2

    path = Path(args[0] if args else ".env")
    if not path.is_file():
        print(f"Нет файла {path}. Скопируй .env.example в .env.")
        return 1

    problems, configured = validate(parse_env(path.read_text(encoding="utf-8")))
    if problems:
        print("Проблемы окружения:")
        for problem in problems:
            print(f"  - {problem}")
        return 1

    print("Локальная проверка пройдена. Настроено:", ", ".join(configured))
    print("Важно: действительность ключа и доступ к модели проверяет только API-запрос.")
    return 0


if __name__ == "__main__":
    raise SystemExit(main())
