Урок 7.4 · Где агенты ломаются

Фаза 7 — Agent Engineering · Результат фазы: Собрать надёжного агента с памятью, планированием, human-in-the-loop и guardrails.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Мы собрали агента (7.1–7.3). Теперь посмотрим, как он ломается. Хорошая новость: ломается он не случайно, а предсказуемо — и эти режимы ловятся инженерно, детекторами вокруг цикла.

MOTTO. Агенты ломаются предсказуемо — петли, дрейф цели, бесконечная работа; это ловится инженерно.

PROBLEM

Автономный агент рано или поздно зависает, и почти всегда одним из трёх способов:

  • петля — повторяет одно и то же действие (обновляет страницу, зовёт тот же инструмент);
  • дрейф цели — увлёкся не тем, действия перестали относиться к исходной задаче;
  • бесконечная работа — не зациклился буквально, но и не останавливается.

Если эти режимы не ловить, агент молча жжёт токены, время и деньги. А поскольку поломки типовые, под каждую можно поставить детектор. Воспроизведём поломки и встроим предохранители.

CONCEPT

Интуиция

Это как предохранители в электрощитке. Ты не надеешься, что прибор «не сломается» — ты ставишь автомат, который вырубит линию при перегрузке, коротком замыкании или утечке. Каждой типовой аварии — свой автомат. С агентом так же: не «надеемся, что не зависнет», а ставим три детектора, каждый под свой режим отказа, и они срабатывают до того, как агент сожжёт бюджет.

Как это работает

типовые поломки:               детектор:
- петля (повтор действия)   →  detect_loop (N одинаковых подряд)
- дрейф цели                →  goal_relatedness < порога
- бесконечная работа        →  лимит шагов (max_steps)
  • detect_loop(actions, repeat) — идут ли подряд repeat одинаковых действий (буквальная петля).
  • goal_relatedness(goal, action) — грубая оценка связи действия с целью (доля общих слов, 0..1); падение ниже порога = дрейф.
  • run_guarded(policy, goal, max_steps, repeat, drift_threshold) — гоняет policy и возвращает статус: done (агент финишировал), loop_detected, goal_drift или step_limit.

Глубже детекторов лежит self-reflection (Reflexion): вместо того чтобы тупо повторять неудачу, агент анализирует, что пошло не так, и корректирует подход. Детекторы — это страховка; рефлексия — лечение.

РАЗБОР ПО ШАГАМ

Прогоним run_guarded на трёх policy:

1. policy = lambda a: "refresh"          (всегда одно и то же)
   actions: ["refresh","refresh","refresh"] → detect_loop=3 одинаковых
   → status: loop_detected

2. policy выдаёт ["search","read","answer"], потом "DONE"
   ни петли, ни дрейфа; на 4-м шаге "DONE"
   → status: done, actions=["search","read","answer"]

3. policy = lambda a: f"step_{len(a)}"    (каждый шаг уникален)
   петли нет (все разные), "DONE" не приходит → упёрся в max_steps=5
   → status: step_limit, 5 действий

И дрейф: при goal="сделать отчёт" и действии "смотрю мемы" общих слов нет → goal_relatedness = 0, что ниже порога 0.5 → goal_drift. Каждый режим отказа пойман своим детектором и возвращён как статус — агент не «завис молча», а остановился с понятной причиной. Это и есть инженерный контроль автономии: поломка диагностируется, а не просто жжёт бюджет.

  • detect_loop(actions, repeat=3) — True, если последние repeat действий одинаковы (при len(actions) < repeat → False).
  • goal_relatedness(goal, action) — доля слов действия, встречающихся в цели: |words(goal) ∩ words(action)| / |words(action)| (0..1; пустое действие → 0).

Исчерпали шаги → step_limit. Вернуть {"status", "actions"}.

USE IT

Инженерные приёмы контроля в проде (мульти-провайдер):

  • Лимит шагов / времени / бюджета — жёсткий потолок на цикл (как max_steps в 7.1).
  • Детекция повторов — одинаковые tool_call подряд → стоп или смена стратегии.
  • Привязка к цели — периодически сверять действия с исходной целью (анти-дрейф).
  • Self-reflection (Reflexion) — Shinn et al. усиливают агента не дообучением весов, а «вербальной обратной связью»: после провала агент текстом рефлексирует, что пошло не так, кладёт эту заметку в episodic memory buffer (память из 7.2) и читает её в следующей попытке. На HumanEval это дало 91% pass@1 против 80% у GPT-4 — без единого обновления весов.
  • Наблюдаемость — логировать трейс (Фаза 10), чтобы видеть, где именно сломалось.

ЧАСТЫЕ ОШИБКИ

  • Надеяться, что агент «не зависнет». Зависнет — петля, дрейф или бесконечная работа почти неизбежны на длинном горизонте. Ставь детекторы, а не надежду.
  • Только лимит шагов без диагностики причины. step_limit останавливает, но не говорит, почему. Различай петлю, дрейф и просто долгую задачу — лечатся они по-разному.
  • Игнорировать дрейф цели. Агент может бодро работать, но не над тем. Периодически сверяй действия с исходной целью, иначе он «успешно» решит чужую задачу.
  • Повторять неудачу вместо рефлексии. Тупой ретрай того же действия — это петля. Reflexion: проанализировать провал и сменить подход.
  • Ловить поломки без трейса. Без логов (Фаза 10) видно «зависло», но не где. Наблюдаемость превращает «сломалось» в «сломалось вот на этом шаге по этой причине».

from agent_failures import detect_loop, goal_relatedness, run_guarded


def test_detect_loop():
    assert detect_loop(["x", "x", "x"]) is True
    assert detect_loop(["a", "b", "a"]) is False
    assert detect_loop(["x", "x"], repeat=3) is False     # мало шагов


def test_goal_relatedness():
    assert goal_relatedness("найти отчёт по продажам", "найти отчёт") == 1.0
    assert goal_relatedness("найти отчёт", "купить пиццу") == 0.0


def test_run_guarded_detects_loop():
    res = run_guarded(lambda actions: "refresh", max_steps=10)
    assert res["status"] == "loop_detected"


def test_run_guarded_completes():
    seq = ["search", "read", "answer"]
    res = run_guarded(lambda a: seq[len(a)] if len(a) < len(seq) else "DONE")
    assert res["status"] == "done"
    assert res["actions"] == seq


def test_run_guarded_hits_step_limit():
    res = run_guarded(lambda a: f"step_{len(a)}", max_steps=5)
    assert res["status"] == "step_limit"
    assert len(res["actions"]) == 5


def test_run_guarded_detects_drift():
    # действия не относятся к цели → дрейф
    res = run_guarded(lambda a: "смотрю мемы", goal="сделать отчёт",
                      max_steps=5, drift_threshold=0.5)
    assert res["status"] == "goal_drift"

"""Где агенты ломаются: петли, дрейф цели, лимит шагов — Build It для урока 7.4.

Без зависимостей. Воспроизводим типовые поломки агентов и ловим их инженерно:
- петля (агент повторяет одно действие) → детектор повторов;
- дрейф цели (действия перестают относиться к цели) → оценка связи;
- бесконечная работа → лимит шагов.
"""


def detect_loop(actions, repeat=3):
    """Идут ли подряд `repeat` одинаковых действий (петля)."""
    if len(actions) < repeat:
        return False
    tail = actions[-repeat:]
    return all(a == tail[0] for a in tail)


def goal_relatedness(goal, action):
    """Грубая оценка связи действия с целью (доля общих слов). 0..1."""
    g = set(goal.lower().split())
    a = set(str(action).lower().split())
    return len(g & a) / len(a) if a else 0.0


def run_guarded(policy, goal="", max_steps=10, repeat=3, drift_threshold=None):
    """Гонять policy(actions)->action|'DONE', ловя петли, дрейф и лимит шагов.

    Возвращает {status, actions}. status: done | loop_detected | goal_drift | step_limit.
    """
    actions = []
    for _ in range(max_steps):
        action = policy(actions)
        if action == "DONE":
            return {"status": "done", "actions": actions}
        actions.append(action)
        if detect_loop(actions, repeat):
            return {"status": "loop_detected", "actions": actions}
        if drift_threshold is not None and goal_relatedness(goal, action) < drift_threshold:
            return {"status": "goal_drift", "actions": actions}
    return {"status": "step_limit", "actions": actions}


if __name__ == "__main__":
    # петля: policy всегда возвращает одно и то же
    print(run_guarded(lambda a: "refresh", max_steps=10))
    # нормальная работа: три шага и финал
    seq = ["search", "read", "answer"]
    print(run_guarded(lambda a: seq[len(a)] if len(a) < len(seq) else "DONE"))
    # бесконечная без повторов → упрётся в лимит
    print(run_guarded(lambda a: f"step_{len(a)}", max_steps=5))

```python
res = run_guarded(policy, goal="сделать отчёт",
                  max_steps=10, repeat=3, drift_threshold=0.3)

```

## Плейбук: симптом → причина → приём

| Симптом | Вероятная причина | Приём |
|---|---|---|
| Повторяет одно действие | петля (нет прогресса) | `detect_loop` → стоп/смена стратегии; добавить состояние в память (7.2) |
| Ушёл не туда | дрейф цели | сверять действия с целью (`goal_relatedness`); напоминать цель |
| Не останавливается | нет условия выхода | лимит шагов/времени/бюджета (7.1) |
| Зовёт несуществующий инструмент | плохие схемы | валидация tool_call (6.1), чёткие описания |
| Уверенно врёт в шаге | галлюцинация | проверки результата (5.5), цитаты (1.4) |
| Опасное действие | нет заслона | guardrails + human-in-the-loop (7.3) |

## Приёмы устойчивости

- **Лимиты** на шаги/время/токены — обязательный потолок.
- **Детекция повторов и дрейфа** — раннее срабатывание предохранителя.
- **Self-reflection (Reflexion)** — после неудачи отрефлексировать и перепланировать, а не повторять.
- **Трейс и наблюдаемость** (Фаза 10) — чтобы видеть, где именно сломалось.
