Урок 3.4 · Векторные хранилища

Фаза 3 — Текст и документы · Результат фазы: Строить пайплайны резюмирования и редактуры; собрать базовый RAG над своим корпусом.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Этот урок превращает разовый поиск из 3.3 в постоянную память RAG. Ретривер 3.3 пересчитывал весь корпус на каждый запрос и жил только в памяти процесса. Векторное хранилище добавляет три вещи: документы кладутся инкрементально, поиск идёт по сохранённому индексу, а сам индекс переживает перезапуск (save/load). Это и есть основа ассистента по своим документам — артефакт doc-assistant.

MOTTO. Векторное хранилище — это память RAG: добавил документы → ищешь по смыслу.

PROBLEM

В уроке 3.3 ретривер пересчитывал tf-idf по всему корпусу на каждый вызов и исчезал вместе с процессом. Для реального корпуса так нельзя: документы прибавляются по одному, переиндексировать всё каждый раз дорого, а после перезапуска не хочется собирать индекс заново. Нужен компонент с состоянием — хранилище, у которого есть add, query и сохранение на диск.

CONCEPT

Интуиция

Векторное хранилище — это поисковая память: один раз положил документы (каждый превращается в вектор), и сколько угодно раз ищешь по смыслу. В отличие от ретривера 3.3, оно помнит уже посчитанные векторы и умеет сохраняться на диск — закрыл программу, открыл, а корпус на месте.

Как это работает

VectorStore:
  add(id, text)   → text → эмбеддинг → хранить (id, text, vec)
  query(text, k)  → эмбеддинг запроса → косинус → топ-k
  save / load     → персистентность на диск (JSON)

Три детали, которые делают это хранилищем, а не просто функцией поиска:

  • Хеширующий эмбеддинг (feature hashing). Вместо растущего словаря (как у TF-IDF) каждое слово кладётся в один из DIM фиксированных бакетов (ячеек вектора) через хеш — вектор всегда одной длины (DIM=256), сколько бы новых слов ни пришло. Платим коллизиями (разные слова иногда в один бакет), но для учебного хранилища это ок; в проде вместо этого — плотные нейроэмбеддинги.
  • Стабильный хеш — обязателен. Встроенный hash() в Python рандомизирован между запусками (PYTHONHASHSEED). Если эмбеддинг зависит от него, после перезапуска/load те же слова дадут другие векторы и поиск сломается. Поэтому хеш пишем свой, детерминированный.
  • Персистентность. save сериализует items (id, text, vec) в JSON, load восстанавливает. После загрузки поиск даёт ровно тот же результат.

Поиск — тот же косинус и топ-k, что в 3.3. Интерфейс add/query совпадает с Chroma/FAISS/pgvector; отличие проды — масштаб и алгоритм: мы делаем точный перебор по всем векторам (O(n) на запрос, ок для маленького корпуса), а реальные хранилища используют ANN (HNSW/IVF) — приближённый поиск по миллионам векторов.

РАЗБОР ПО ШАГАМ

Шаг 1 — эмбеддинг. embed("кошки и рыба") раскладывает слова по 256 бакетам стабильным хешем и возвращает вектор длиной DIM=256. Повторный вызов даёт тот же вектор (детерминизм) — без этого хранилище не пережило бы перезапуск.

Шаг 2 — добавили и ищем. Кладём три документа и спрашиваем query("кошки и рыба", k=3) (реальные косинус-скоры):

0.516  doc0: кошки любят спать и рыбу     ← делит «кошки», «и»
0.258  doc1: собаки любят прогулки и мяч  ← делит только «и»
0.000  doc2: python язык программирования ← общих слов нет

Релевантный документ — первым. (Тонкость: рыба и рыбу — для лексического эмбеддинга разные слова и в разные бакеты, поэтому работает «кошки», а не «рыба»; смысловое совпадение дали бы нейроэмбеддинги, как в 3.3.)

Шаг 3 — персистентность. store.save("db.json") → создаём новый VectorStore().load("db.json") → тот же query("кошки и рыба") возвращает тот же топ. Индекс пережил «перезапуск» — это и отличает хранилище от разового ретривера.

Размерность задана: DIM = 256. Реализуй:

  • _stable_hash(word) — детерминированный хеш: h = 0, для каждого символа h = (h * 31 + ord(ch)) % 2**32, верни h. Не используй встроенный hash() — он рандомизирован между запусками.

  • cosine(a, b) — косинус двух плотных векторов (скалярное произведение / произведение норм; 0.0 при нулевой норме).
  • VectorStore с items = [{id, text, vec}] и методами: add(id, text) (эмбеддит и хранит), query(text, k=3) (эмбеддит запрос, косинус ко всем, топ-k как [{id, text, score}]), save(path) (json.dump items), load(path) (json.load, вернуть self).

ЧАСТЫЕ ОШИБКИ

  • Эмбеддинг на встроенном hash(). Он рандомизирован между запусками — после load векторы «поедут» и поиск сломается. Хеш для эмбеддинга должен быть детерминированным.
  • Переиндексировать весь корпус на каждый запрос. Смысл хранилища — посчитать векторы один раз и переиспользовать; добавляй документы инкрементально и сохраняй на диск.
  • Точный перебор на больших объёмах. O(n) на запрос ок для тысяч векторов, но не для миллионов — там нужен ANN (HNSW/IVF) из готовых хранилищ.
  • Хранить документы целиком. В хранилище кладут куски (чанки из 3.2): поиск по кускам точнее, и найденное влезает в промпт.

from vector_store import VectorStore, embed, cosine, DIM


def _store():
    s = VectorStore()
    s.add("cats", "кошки любят спать и рыбу")
    s.add("dogs", "собаки любят прогулки и мяч")
    s.add("py", "python язык программирования")
    return s


def test_embed_is_fixed_dim_and_deterministic():
    v1, v2 = embed("кошки и рыба"), embed("кошки и рыба")
    assert len(v1) == DIM
    assert v1 == v2                      # детерминированно (стабильный хеш)


def test_query_returns_nearest():
    hits = _store().query("кошки и рыба", k=1)
    assert hits[0]["id"] == "cats"


def test_topk_limits():
    assert len(_store().query("любят", k=2)) == 2


def test_save_load_roundtrip(tmp_path):
    s = _store()
    path = tmp_path / "store.json"
    s.save(path)
    loaded = VectorStore().load(path)
    assert loaded.query("python", k=1)[0]["id"] == "py"
    assert loaded.items == s.items       # данные сохранились без потерь


def test_cosine_bounds():
    assert cosine(embed("кошки"), embed("кошки")) == 1.0
    assert cosine(embed("кошки"), embed("python")) < 1.0

code/vector_store.pyПоказать решение
"""Векторное хранилище с нуля — Build It для урока 3.4.

Без зависимостей. Мини-хранилище как у Chroma/FAISS, только маленькое:
добавляем документы (эмбеддинг = хеширующий bag-of-words в фикс. размер),
ищем по косинусу, сохраняем/грузим на диск. Хеш детерминированный — результат
воспроизводим между запусками и после load.
"""
import json
import math
import re

DIM = 256


def _stable_hash(word):
    """Детерминированный хеш слова (встроенный hash() рандомизирован — нельзя)."""
    h = 0
    for ch in word:
        h = (h * 31 + ord(ch)) % (2 ** 32)
    return h


def embed(text):
    """Хеширующий эмбеддинг: слова раскладываются по DIM бакетам."""
    vec = [0.0] * DIM
    for w in re.findall(r"\w+", text.lower()):
        vec[_stable_hash(w) % DIM] += 1.0
    return vec


def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    na = math.sqrt(sum(x * x for x in a))
    nb = math.sqrt(sum(y * y for y in b))
    return dot / (na * nb) if na and nb else 0.0


class VectorStore:
    """Минимальное векторное хранилище: add / query / save / load."""

    def __init__(self):
        self.items = []      # [{id, text, vec}]

    def add(self, id, text):
        self.items.append({"id": id, "text": text, "vec": embed(text)})

    def query(self, text, k=3):
        qv = embed(text)
        scored = sorted(
            ((cosine(qv, it["vec"]), it) for it in self.items),
            key=lambda x: x[0], reverse=True,
        )
        return [{"id": it["id"], "text": it["text"], "score": round(s, 4)}
                for s, it in scored[:k]]

    def save(self, path):
        with open(path, "w", encoding="utf-8") as f:
            json.dump(self.items, f, ensure_ascii=False)

    def load(self, path):
        with open(path, encoding="utf-8") as f:
            self.items = json.load(f)
        return self


if __name__ == "__main__":
    store = VectorStore()
    for i, doc in enumerate([
        "кошки любят спать и рыбу",
        "собаки любят прогулки и мяч",
        "python язык программирования",
    ]):
        store.add(f"doc{i}", doc)
    for hit in store.query("кошки и рыба", k=2):
        print(f"{hit['score']:.3f}  {hit['id']}: {hit['text']}")

# doc-assistant — вопросы к своему корпусу (RAG)

Складываешь документы в векторное хранилище и спрашиваешь их на естественном языке: retrieve → augment → ответ.

## Как пользоваться

1. **Индексация:** добавь документы в хранилище (`add(id, text)`); при необходимости сохрани на диск (`save`).
2. **Retrieval:** на вопрос достань топ-k релевантных кусков (`query(text, k)`).
3. **Augment + ответ:** подложи найденные куски в промпт модели и попроси ответить **только по ним** (снижает галлюцинации, урок 1.4).

Офлайн-прототип хранилища:

```bash
python -c "import vector_store as v; s=v.VectorStore(); \
s.add('a','кошки любят рыбу'); s.add('b','python язык'); print(s.query('рыба и кошки', k=1))"
```

В проде замени `embed` на нейроэмбеддинги, а `VectorStore` — на Chroma/FAISS/pgvector (интерфейс `add/query` тот же).

## Файлы

- `vector_store.py` — `VectorStore` (`add/query/save/load`), `embed`, `cosine`.

## Правила

- Документы дроби на куски (chunking, урок 3.2) перед индексацией.
- В промпте требуй ответ строго по найденному контексту и проси цитаты (урок 1.4).
- Качество ретрива измеряй (eval-harness, урок 2.5): попадает ли нужный кусок в топ-k.
