Урок 12.3 · Презентация и защита

Фаза 12 — Capstone · Результат фазы: Автоматизировать реальный процесс end-to-end и защитить решение с метриками эффекта.

Спека есть (12.1), решение собрано (12.2) — остался последний и решающий шаг: доказать, что стало лучше. Это финал курса. Не «я сделал агента», а «время упало на X%, стоимость на Y%, качество выросло на Z%, ROI = N». Числа, а не впечатления.

MOTTO. Защита — это цифры: насколько упало время и стоимость, выросло качество, какой ROI.

PROBLEM

Работающее решение само по себе ничего не доказывает. «Кажется, быстрее» — не аргумент на защите: заказчик спросит «на сколько?» и «а сколько это стоило?». Без сравнения «до/после» по времени, качеству и стоимости и без честного ROI твой Capstone — красивое демо, а не доказанный эффект.

К счастью, baseline у тебя уже зафиксирован в спеке (12.1), а стоимость — в телеметрии решения (12.2). Осталось посчитать улучшения и окупаемость. И считать надо честно: FinOps-исследования показывают, что лишь малая доля команд вообще умеет привязать AI-расходы к бизнес-результату — не будь среди тех, кто не умеет. Соберём калькулятор защиты.

CONCEPT

Интуиция

Защита Capstone — это акт сдачи работ с цифрами, а не питч стартапа. Инвестору продают мечту; заказчику сдают измеренный результат. И ключевая честность здесь: направление улучшения зависит от метрики. Для времени и стоимости лучше — это меньше; для качества лучше — это больше. Спутать знак — значит отчитаться, что «стоимость выросла на 80%», имея в виду, что она упала.

Вторая честность — ROI считается от полных затрат, а не только от экономии. Если решение сэкономило 800, но само стоит 200, окупаемость меряется относительно этих 200. Показать экономию и умолчать про стоимость решения — это не защита, а самообман.

Как это работает

метрики «до» (baseline из 12.1) vs «после» (замер решения):
   время     ↓  improvement%  (lower_is_better=True)
   стоимость ↓  improvement%  (lower_is_better=True, FinOps Ф9–10)
   качество  ↑  improvement%  (lower_is_better=False, eval Ф2.5/5.5)
        │
   ROI = (экономия + выгода) / затраты на решение
        │
   презентация (проблема→решение→демо→метрики→ROI→риски) + чек-лист компетенций
  • improvement(before, after, lower_is_better) — процент улучшения с учётом направления метрики.
  • roi(before_cost, after_cost, value_gained) — окупаемость: (экономия + доп. выгода) / затраты.
  • before_after(...) — сводка по всем метрикам сразу; competency_report(...) — готовность компетенций курса.

Разбор формул

improvement. Для «меньше = лучше»: (before − after) / before. Время 10→3: (10−3)/10 = 0.7 = +70% улучшения (само время при этом упало на 70%). Для «больше = лучше» формула переворачивается: (after − before) / before. Качество 70→88: (88−70)/70 = 0.257 = +25.7% улучшения (качество выросло). Заметь: при верном флаге improvement в обоих случаях возвращает положительный процент улучшения — направление исходной метрики (упало/выросло) закодировано во флаге lower_is_better, а не в знаке результата.

roi. Экономия = before_cost − after_cost. ROI = (экономия + выгода) / after_cost. Затраты 1000→200, доп. выгода 300: экономия 800, (800 + 300) / 200 = 5.5 — каждый вложенный доллар вернул 5.5.

РАЗБОР ПО ШАГАМ

Прогоним защиту на реальных цифрах спеки поддержки:

before = {время_мин: 10.0, стоимость_$: 0.50, качество_%: 70.0}
after  = {время_мин:  3.0, стоимость_$: 0.10, качество_%: 88.0}
lower  = {время_мин: True, стоимость_$: True, качество_%: False}

before_after(before, after, lower):
   время_мин:   10.0 → 3.0   (+70.0%)   меньше = лучше: время упало — улучшение
   стоимость_$: 0.50 → 0.10  (+80.0%)   меньше = лучше: стоимость упала — улучшение
   качество_%:  70.0 → 88.0  (+25.7%)   больше = лучше: качество выросло — улучшение

roi(before_cost=1000, after_cost=200, value_gained=300) → 5.5

competency_report({промпты:✓, агенты:✓, MCP:✗}) → {done: 2, total: 3, missing: [MCP]}

Три улучшения с верными флагами направления, окупаемость от полных затрат, честный список непокрытого — это и есть содержание защиты. Заметь: если бы для качества забыли lower_is_better=False, improvement посчитал бы (70−88)/70 = −25.7% — минус честно просигналил бы, что с флагами что-то не так, но в отчёте ты заявил бы падение качества вместо роста.

Честная граница метода. Калькулятор считает арифметику верно, но не проверяет, что цифры настоящие:

  • Мусор на входе — мусор на выходе. Если after-качество не из eval-harness (Ф2.5/5.5), а «на глаз», то и +25.7% — фикция. Числа должны приходить из замеров, а не из оптимизма.
  • ROI зависит от честности value_gained. Раздутая «доп. выгода» надувает ROI. Выгоду надо обосновывать, а не назначать.
  • Риски не в формуле. Приватность, галлюцинации, предвзятость (Фаза 11) в ROI не попадают, но на защите проговариваются — иначе решение выглядит зрелее, чем оно есть.

Перед запуском. Работай в своей папке курса (ai-native/12.3-defense/), а файлы урока клади в подпапку code/ (как в 0.1). Нужен только Python 3 (для теста ещё pytest).

Создай файл metrics.py с четырьмя функциями:

  • improvement(before, after, lower_is_better=True) — процент улучшения, округлить до 1 знака. При lower_is_better дельта = (before−after)/before, иначе (after−before)/before. При before==0 вернуть 0.0.
  • roi(before_cost, after_cost, value_gained=0.0) — (экономия + value_gained) / after_cost, округлить до 2 знаков; при after_cost==0 вернуть inf.
  • before_after(before, after, lower_is_better) — по каждому ключу вернуть {before, after, improvement_pct} (dict'ы before/after — метрика→значение, lower_is_better — метрика→bool).
  • competency_report(checked) — по dict'у компетенция→bool вернуть {done, total, missing}.

ЧАСТЫЕ ОШИБКИ

  • Защищаться словами, а не числами. «Кажется, лучше» — не аргумент. Нужны время/стоимость/качество «до/после» и ROI, иначе эффект не доказан.
  • Спутать направление метрики. Для качества lower_is_better=False; забудешь — и рост +25.7% превратится в мнимое падение. Знак зависит от метрики.
  • Считать ROI только от экономии. Окупаемость меряется от полных затрат на решение (after_cost), а не от одной экономии — иначе цифра завышена.
  • Брать числа «на глаз». improvement верно посчитает мусор как мусор. Метрики должны приходить из eval/телеметрии, а не из оптимизма.
  • Умолчать про риски. Приватность, галлюцинации, предвзятость (Ф11) в формулу ROI не входят, но на защите обязательны — зрелость важнее глянца.

from metrics import improvement, roi, before_after, competency_report


def test_improvement_lower_is_better():
    assert improvement(10, 3, lower_is_better=True) == 70.0      # время −70%
    assert improvement(0.50, 0.10, lower_is_better=True) == 80.0  # стоимость −80%


def test_improvement_higher_is_better():
    assert improvement(70, 88, lower_is_better=False) == 25.7     # качество +25.7%


def test_roi():
    # экономия 800 + выгода 300 = 1100; / 200 = 5.5
    assert roi(before_cost=1000, after_cost=200, value_gained=300) == 5.5


def test_before_after_structure():
    ba = before_after({"t": 10}, {"t": 5}, {"t": True})
    assert ba["t"]["improvement_pct"] == 50.0
    assert ba["t"]["before"] == 10 and ba["t"]["after"] == 5


def test_competency_report():
    rep = competency_report({"промпты": True, "агенты": True, "MCP": False})
    assert rep["done"] == 2 and rep["total"] == 3
    assert "MCP" in rep["missing"]

Эталонное решение (Build It)

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/metrics.pyПоказать решение
"""Презентация и защита: метрики «до/после» + ROI — Build It для урока 12.3.

Без зависимостей. Защита Capstone строится на цифрах: насколько упало время и
стоимость, насколько выросло качество, какой ROI. Здесь считаем улучшения,
окупаемость и собираем сводку для презентации.
"""


def improvement(before, after, lower_is_better=True):
    """% улучшения. lower_is_better=True для времени/стоимости (меньше = лучше)."""
    if before == 0:
        return 0.0
    delta = (before - after) / before if lower_is_better else (after - before) / before
    return round(delta * 100, 1)


def roi(before_cost, after_cost, value_gained=0.0):
    """ROI автоматизации: (экономия + выгода) / затраты."""
    saved = before_cost - after_cost
    return round((saved + value_gained) / after_cost, 2) if after_cost else float("inf")


def before_after(before, after, lower_is_better):
    """before/after: dict метрика→значение. lower_is_better: dict метрика→bool."""
    return {
        k: {"before": before[k], "after": after[k],
            "improvement_pct": improvement(before[k], after[k], lower_is_better.get(k, True))}
        for k in before
    }


def competency_report(checked):
    """checked: dict компетенция→bool. Возвращает сводку готовности."""
    done = sum(1 for v in checked.values() if v)
    return {"done": done, "total": len(checked),
            "missing": [k for k, v in checked.items() if not v]}


if __name__ == "__main__":
    before = {"время_мин": 10.0, "стоимость_$": 0.50, "качество_%": 70.0}
    after = {"время_мин": 3.0, "стоимость_$": 0.10, "качество_%": 88.0}
    lower = {"время_мин": True, "стоимость_$": True, "качество_%": False}
    for k, v in before_after(before, after, lower).items():
        print(f"{k}: {v['before']} → {v['after']} ({v['improvement_pct']:+.1f}%)")
    print("ROI:", roi(before_cost=1000, after_cost=200, value_gained=300))

## Структура презентации (7 слайдов)

1. **Проблема** — процесс, его боль, baseline (сколько стоит/занимает сейчас).
2. **Решение** — архитектура из кирпичей курса (агент + MCP + cost-aware + память + мониторинг).
3. **Демо** — живой прогон end-to-end на реальном кейсе.
4. **Метрики «до/после»** — время ↓, стоимость ↓, качество ↑ (с %).
5. **ROI** — (экономия + выгода) / затраты на решение.
6. **Риски и ограничения** — приватность, галлюцинации, предвзятость (Фаза 11) и меры.
7. **Что дальше** — масштабирование, что доработать.

## Правила честной защиты

- **Реальные замеры**, не «кажется лучше» (eval 2.5, валидация 5.5, телеметрия 10.3).
- **Учти стоимость решения** в ROI, а не только экономию.
- **Проговори риски и ограничения** (Фаза 11) — это часть зрелости, а не слабость.
