Урок 10.3 · Телеметрия и воспроизводимость

Фаза 10 — Production и интеграция · Результат фазы: Встроить ИИ в рабочий инструмент с мониторингом токенов и стоимости.

Requires (только для USE IT): observability/биллинг провайдера. BUILD IT работает офлайн, без ключей и без сети.

Мониторинг (10.2) отвечает «здорово ли». Телеметрия отвечает «сколько стоит и можно ли повторить». Это два разных вопроса: первый про здоровье, второй про экономику и воспроизводимость. Без второго нельзя ни понять, что дорого, ни честно сравнить «до/после».

MOTTO. Считай токены и стоимость на процесс и фиксируй конфиг прогона — чтобы повторить и сравнить.

PROBLEM

Две дыры без телеметрии. Первая — деньги в разрезе процессов: общий счёт есть (9.4), но какой именно процесс — суммаризация, классификация, ресёрч — съедает бюджет? Без привязки токенов/стоимости к процессу неясно, что оптимизировать.

Вторая — воспроизводимость. Ты улучшил промпт, и качество выросло. Но ты заодно сменил температуру и модель — что из трёх дало эффект? Если конфиг прогона (модель, параметры, версия промпта, seed) не зафиксирован, результат нельзя повторить и нельзя честно сравнить. Соберём учёт расходов по процессам + отпечаток конфига.

CONCEPT

Интуиция

Две вещи, как у учёного в лаборатории. Телеметрия — это счёт по статьям: не «потрачено $N», а «суммаризация — столько, классификация — столько», чтобы видеть, где дорого. Воспроизводимость — это подпись эксперимента: учёный записывает все условия опыта, чтобы повторить его и чтобы сравнение двух опытов было честным (изменили одно — видим эффект одного).

Отпечаток конфига (fingerprint) — и есть такая подпись: одинаковые условия → одинаковый хэш; сменил хоть один параметр → хэш другой, и сразу видно, что прогоны несравнимы «как есть».

Как это работает

телеметрия:        record(process, tokens_in, tokens_out, cost) → by_process()
воспроизводимость: RunConfig{model, temperature, prompt_version, seed}
                       └─ fingerprint() : одинаковый конфиг → одинаковый хэш
  • record(process, tokens_in, tokens_out, cost) — записать расход с привязкой к процессу.
  • by_process() — агрегировать токены и стоимость по процессам (где деньги).
  • RunConfig.fingerprint() — стабильный хэш конфига: одинаковый конфиг → одинаковый отпечаток; любой изменённый параметр → другой.

Зачем хэш, а не просто хранить конфиг: отпечаток — это компактный ключ сравнимости. Два прогона с одинаковым fingerprint сравнимы напрямую; разные fingerprint честно говорят «вы поменяли условия, прежде чем делать вывод — учтите это». Это мост к FinOps (9.4, «до/после») и к eval промптов (2.5, сравнение версий).

Важная оговорка: конфиг ≠ бит-в-бит одинаковый вывод. Fingerprint фиксирует условия прогона, но LLM недетерминирован даже при temperature=0 и фиксированном seed: одинаковый запрос может дать чуть разный текст из-за батчинга запросов на сервере, неассоциативности float-арифметики на GPU и роутинга в MoE-моделях. Поэтому seed/system_fingerprint у провайдеров — это best-effort, а не гарантия. Что даёт fingerprint конфига: прогоны становятся сравнимыми и атрибутируемыми (поменял одно — знаешь, что именно), а воспроизводимость поведения проверяют не по одному ответу, а статистически на eval-наборе (2.5/7.5) — прогнал тот же конфиг по набору, сравнил метрику.

РАЗБОР ПО ШАГАМ

Телеметрия по процессам. Запишем три события:

record("summarize", 1000, 200, 0.005)
record("summarize",  800, 150, 0.004)
record("classify",   300,  10, 0.0003)

by_process() → {
  summarize: {tokens_in: 1800, tokens_out: 350, cost: 0.009},   ← две записи слиты
  classify:  {tokens_in: 300,  tokens_out: 10,  cost: 0.0003},
}
total_cost() → 0.005 + 0.004 + 0.0003 = 0.0093

Сразу видно: summarize — главная статья (0.009 против 0.0003), оптимизировать надо его.

Воспроизводимость. Сравним отпечатки конфигов:

RunConfig("sonnet-4.6")                     → fingerprint b5ee2bf5632e
RunConfig("sonnet-4.6")  (тот же)           → b5ee2bf5632e   (совпал → прогоны сравнимы)
RunConfig("sonnet-4.6", temperature=0.7)    → c8205805c99e   (другой → НЕ сравнимы как есть)
RunConfig("sonnet-4.6", prompt_version="v2")→ другой         (сменил промпт → другой отпечаток)

Одинаковый конфиг дал одинаковый хэш — два таких прогона можно сравнивать. Сменил температуру или версию промпта — хэш изменился, и это сигнал: разница в результате могла быть из-за этого, а не из-за того, что ты тестировал. Так fingerprint защищает от ложного «до/после».

USE IT

Дашборд расходов и качества (мульти-провайдер):

  • Токены/стоимость на процесс — из usage в ответе API (input_tokens / output_tokens) × карта цен (9.1); агрегируй по процессу/фиче, как by_process.
  • Качество рядом со стоимостью — LLM-as-judge / eval-наборы (2.5) на том же дашборде: видно не только «дорого ли», но и «окупается ли качеством» (cost per outcome из 9.4).
  • Воспроизводимость — логируй RunConfig (модель, temperature, версия промпта, seed) с каждым прогоном; OpenTelemetry GenAI стандартизует поля токенов (gen_ai.*), чтобы дашборд читал любой бэкенд.

Принцип: расход — в разрезе процессов, качество — рядом, конфиг — зафиксирован. Тогда «стало лучше/дешевле» подтверждается числами и воспроизводимо.

ЧАСТЫЕ ОШИБКИ

  • Считать только общий счёт. Без разреза по процессам непонятно, что оптимизировать. Привязывай токены/стоимость к процессу (by_process), а не складывай в одну сумму.
  • Сравнивать «до/после» без фиксации конфига. Поменял промпт И температуру — не знаешь, что дало эффект. Фиксируй RunConfig и сравнивай прогоны с одинаковым fingerprint.
  • Менять несколько параметров разом. Тогда вывод о причине невозможен. Меняй по одному (модель ИЛИ промпт ИЛИ температуру) — отпечаток покажет, что именно отличается.
  • Нестабильный хэш конфига. Без sort_keys=True один и тот же конфиг даст разный хэш из-за порядка ключей. Сериализуй детерминированно.
  • Телеметрия без качества. Дешевле — не значит лучше. Клади метрику качества (2.5) рядом со стоимостью, иначе «сэкономили» может скрывать просадку результата.
  • Ждать бит-в-бит одинаковый вывод при temperature=0. LLM недетерминирован даже с seed (батчинг, float, MoE). Fingerprint фиксирует конфиг, не ответ; воспроизводимость поведения меряют статистически на eval-наборе, а не сравнением одного ответа.

from telemetry import Telemetry, RunConfig


def _tel():
    t = Telemetry()
    t.record("summarize", 1000, 200, 0.005)
    t.record("summarize", 800, 150, 0.004)
    t.record("classify", 300, 10, 0.0003)
    return t


def test_aggregation_by_process():
    by = _tel().by_process()
    assert by["summarize"]["tokens_in"] == 1800
    assert by["summarize"]["cost"] == 0.009
    assert by["classify"]["tokens_out"] == 10


def test_total_cost():
    assert _tel().total_cost() == 0.0093


def test_fingerprint_is_deterministic():
    assert RunConfig("sonnet-4.6").fingerprint() == RunConfig("sonnet-4.6").fingerprint()


def test_fingerprint_changes_with_params():
    a = RunConfig("sonnet-4.6", temperature=0.0)
    b = RunConfig("sonnet-4.6", temperature=0.7)
    assert a.fingerprint() != b.fingerprint()
    # другая версия промпта тоже меняет отпечаток
    assert a.fingerprint() != RunConfig("sonnet-4.6", prompt_version="v2").fingerprint()

Эталонное решение

code/telemetry.pyПоказать решение
"""Телеметрия и воспроизводимость — Build It для урока 10.3.

Без зависимостей. Считаем токены и стоимость в разрезе процессов (телеметрия) и
фиксируем конфигурацию запуска (модель, параметры, версия промпта) так, чтобы
прогон можно было воспроизвести — одинаковый конфиг даёт одинаковый «отпечаток».
"""
import hashlib
import json
from dataclasses import dataclass, asdict, field


class Telemetry:
    """Учёт токенов и стоимости по процессам."""

    def __init__(self):
        self.events = []

    def record(self, process, tokens_in, tokens_out, cost):
        self.events.append({"process": process, "tokens_in": tokens_in,
                            "tokens_out": tokens_out, "cost": cost})

    def by_process(self):
        out = {}
        for e in self.events:
            agg = out.setdefault(e["process"], {"tokens_in": 0, "tokens_out": 0, "cost": 0.0})
            agg["tokens_in"] += e["tokens_in"]
            agg["tokens_out"] += e["tokens_out"]
            agg["cost"] = round(agg["cost"] + e["cost"], 6)
        return out

    def total_cost(self):
        return round(sum(e["cost"] for e in self.events), 6)


@dataclass
class RunConfig:
    """Конфиг прогона для воспроизводимости."""
    model: str
    temperature: float = 0.0
    prompt_version: str = "v1"
    seed: int = 0

    def fingerprint(self):
        """Стабильный отпечаток конфига: одинаковый конфиг → одинаковый хэш."""
        blob = json.dumps(asdict(self), sort_keys=True).encode("utf-8")
        return hashlib.sha256(blob).hexdigest()[:12]


if __name__ == "__main__":
    tel = Telemetry()
    tel.record("summarize", 1000, 200, 0.005)
    tel.record("summarize", 800, 150, 0.004)
    tel.record("classify", 300, 10, 0.0003)
    print("по процессам:", tel.by_process())
    print("итого $:", tel.total_cost())

    a = RunConfig("sonnet-4.6", temperature=0.0, prompt_version="v1")
    b = RunConfig("sonnet-4.6", temperature=0.7, prompt_version="v1")
    print("fp(a):", a.fingerprint(), "| fp(a2):", RunConfig("sonnet-4.6").fingerprint())
    print("fp(b) отличается:", a.fingerprint() != b.fingerprint())

Артефакт

outputs/embedded_assistant.py
"""embedded_assistant.py — артефакт урока 10.3: ИИ, встроенный в рабочий процесс.

Самодостаточно, без зависимостей. Объединяет Фазу 10: фича-флаг + fallback (10.1),
логирование вызовов (10.2), телеметрия токенов/стоимости и fingerprint конфига (10.3).
В проде `_call_model` заменяется реальным вызовом API.

Запуск:  python embedded_assistant.py
"""
import hashlib
import json
from dataclasses import dataclass, asdict

PRICES = {"sonnet-4.6": (3.0, 15.0), "haiku-4.5": (1.0, 5.0)}      # $/1M (вход, выход)


@dataclass
class RunConfig:
    model: str = "sonnet-4.6"
    temperature: float = 0.0
    prompt_version: str = "v1"

    def fingerprint(self):
        blob = json.dumps(asdict(self), sort_keys=True).encode()
        return hashlib.sha256(blob).hexdigest()[:12]


class EmbeddedAssistant:
    """ИИ-шаг процесса: флаг + fallback + лог + телеметрия."""

    def __init__(self, config: RunConfig, enabled=True):
        self.config = config
        self.enabled = enabled
        self.log = []           # вызовы (мониторинг)
        self.telemetry = []     # токены/стоимость (FinOps)

    def _cost(self, n_in, n_out):
        p_in, p_out = PRICES[self.config.model]
        return round((n_in * p_in + n_out * p_out) / 1e6, 6)

    def _call_model(self, text):
        if "ошибк" in text.lower():
            raise RuntimeError("модель недоступна")
        n_in, n_out = len(text.split()) * 2, 30
        return f"саммари: {text[:30]}...", n_in, n_out

    def _fallback(self, text):
        return "первые слова: " + " ".join(text.split()[:3])

    def run(self, text, process="summarize"):
        if not self.enabled:
            return self._fallback(text)
        try:
            answer, n_in, n_out = self._call_model(text)
            cost = self._cost(n_in, n_out)
            self.log.append({"process": process, "status": "ok", "config": self.config.fingerprint()})
            self.telemetry.append({"process": process, "tokens_in": n_in,
                                   "tokens_out": n_out, "cost": cost})
            return answer
        except Exception as e:                                  # noqa: BLE001
            self.log.append({"process": process, "status": "error", "error": str(e)})
            return self._fallback(text)

    def total_cost(self):
        return round(sum(t["cost"] for t in self.telemetry), 6)


if __name__ == "__main__":
    asst = EmbeddedAssistant(RunConfig(model="sonnet-4.6", prompt_version="v3"))
    print(asst.run("длинный отчёт для краткого пересказа команды"))
    print(asst.run("текст с ошибкой"))            # упал → fallback, залогировано
    print("config fp:", asst.config.fingerprint())
    print("total cost:", asst.total_cost())
    print("log:", asst.log)
