Урок 6.1 · Tool use / function calling изнутри

Фаза 6 — Инструменты и протоколы (MCP) · Результат фазы: Объяснить tool use изнутри и поднять собственный MCP-сервер с контролем доступа.

Requires (только для USE IT): платный API-ключ одного из провайдеров. BUILD IT работает офлайн, без ключей и без сети.

Это вход в Фазу 6 — как ИИ дотягивается до реального мира. Вся фаза про инструменты и протокол MCP, но в основе лежит один механизм, который мы разберём здесь до винтика: tool use. Без него модель умеет только говорить; с ним — действовать.

MOTTO. Модель не вызывает функции сама — она просит вызвать; протокол — это схемы, tool_call и tool_result.

PROBLEM

«Модель сходила в базу и посчитала остаток» — звучит так, будто LLM исполняет код. Это не так, и непонимание здесь стоит дорого: нельзя ни отладить tool use, ни безопасно подключить инструменты.

На самом деле модель не исполняет ничего. Она лишь возвращает структурированный запрос: «вызови инструмент get_balance с аргументами {...}» (это tool_call). Дальше код исполняешь ты, а результат (tool_result) кладёшь обратно в диалог — и модель продолжает с учётом результата. Модель — это мозг, который просит; руки — твой код. Соберём этот цикл с нуля и увидим, что «агентность» начинается именно здесь.

CONCEPT

Интуиция

Представь модель как диспетчера по рации, у которого связаны руки. Он знает, какие бригады есть на линии и что каждая умеет (список со схемами), и может только отдать команду по рации: «бригада add, посчитайте 2 + 3». Сам он гайки не крутит. Команду выполняет реальная бригада (твой код), докладывает результат по рации — и диспетчер решает, что дальше: дать новую команду или объявить, что работа закончена.

Ключевой сдвиг в голове: модель не «имеет доступ» к базе или калькулятору. Она лишь называет инструмент и аргументы, а контроль и исполнение целиком на твоей стороне — отсюда и безопасность, и отладка.

Как это работает

Протокол стоит на трёх китах:

  • Схема — что за инструмент: имя, описание (по нему модель решает, когда звать) и JSON-схема аргументов (типы, обязательные поля).
  • tool_call — запрос модели: {name, arguments}. Это намерение, а не действие.
  • tool_result — ответ твоего кода, который возвращается модели.
схемы инструментов → модель → tool_call(name, args)
                                   │ твой диспетчер: валидирует по схеме, исполняет
                                   ▼
                              tool_result ──► модель ──► (ещё вызов | финал)

Цикл крутится, пока модель не вернёт финальный ответ вместо нового вызова. Четыре детали в коде:

  • validate_args(args, schema) — проверить аргументы tool_call по схеме (обязательные поля, типы) до исполнения.
  • dispatch(tool_call, registry) — найти инструмент в реестре, провалидировать, исполнить.
  • run(model_fn, registry) — гонять model_fn в цикле: пришёл tool_call → исполнить → дописать результат в историю → снова к модели; пришёл final → вернуть ответ.
  • лимит шагов (max_steps) — страховка от бесконечного цикла.

Это фундамент, на котором стоит MCP (6.2+): MCP лишь стандартизирует, как схемы и вызовы передаются между процессами.

РАЗБОР ПО ШАГАМ

Прогоним цикл run с инструментом add и детерминированной «моделью» (stub вместо LLM):

шаг 1: history пуст → модель просит tool_call {name: "add", arguments: {a: 2, b: 3}}
         dispatch: validate_args({a:2,b:3}, schema) → []  (ок)
                   add(a=2, b=3) → 5
         history = [{call: add{2,3}, result: 5}]
шаг 2: history не пуст → модель смотрит результат и возвращает {type: "final", answer: 5}
         run возвращает 5

Видно: модель не складывала 2 и 3 — она попросила инструмент, число посчитал твой add, и только увидев tool_result = 5, модель завершила работу. Теперь сломаем аргументы: на {a: 1} (нет b) validate_args вернёт «нет обязательного аргумента: b», а на {a: "x", b: 2} — «a: ожидался number». Невалидный tool_call не доходит до исполнения — диспетчер режет его на входе. Поменяй stub на настоящую LLM — протокол не изменится ни на байт.

BUILD IT

Задание: собери цикл function-calling с нуля — инструмент со схемой, валидацию, диспетчер и tool-use-цикл. Только стандартная библиотека, без сети.

Перед запуском. Работай в своей папке курса (ai-native/6.1-tool-use/), а файлы урока клади в подпапку code/ (по соглашению курса). Нужен только Python 3 (для теста ещё pytest).

Создай файл function_calling.py:

  • Tool(name, description, schema, fn) — dataclass инструмента (schema это {"properties": {...}, "required": [...]}, fn — функция-исполнитель).
  • validate_args(args, schema) — вернуть список ошибок: для каждого поля из required, которого нет в args — «нет обязательного аргумента»; для переданных полей с несовпадающим типом (string→str, number→int/float, и т.п.) — «ожидался <тип>». Пусто = валидно.
  • dispatch(tool_call, registry) — tool_call это {name, arguments}. Если инструмента нет в registry → raise ToolError; если validate_args непуст → raise ToolError; иначе исполнить fn(**arguments).
  • run(model_fn, registry, max_steps=5) — цикл: вызвать model_fn(history); если вернулся {"type": "final"} → вернуть answer и history; если tool_call → dispatch, дописать {call, result} в history. Превысили max_steps → raise ToolError.

Готово, когда все тесты в test_function_calling.py зелёные — они проверяют: validate_args ловит пропуск и неверный тип; dispatch исполняет add (=5), падает на неизвестном инструменте и невалидных аргументах; run использует tool_result для финала.

pytest code -q                 # красное → реализуй цикл → зелёное
python code/function_calling.py # демо: модель просит add(2,3) → твой код считает 5 → финал

Подсказка. Типы держи в словаре {"string": str, "number": (int, float), ...} и проверяй isinstance. В run именно результат инструмента дописывается в историю — со следующего шага модель его «видит».

USE IT

Тот же протокол — через API провайдера; меняется синтаксис, не цикл (мульти-провайдер):

tools = [{"name": "add", "description": "сложить два числа",
          "input_schema": {"type": "object",
            "properties": {"a": {"type": "number"}, "b": {"type": "number"}},
            "required": ["a", "b"]}}]
# Anthropic: messages.create(model="claude-sonnet-4-6", tools=tools)
#            → блок tool_use → выполнить у себя → вернуть tool_result
# OpenAI:    responses.create(..., tools=[{"type": "function", ...}]) → tool call → role:"tool"
# Google:    function_declarations в config → functionCall → functionResponse

Везде один цикл: схема → запрос модели (tool_use/tool call) → твой код исполняет → результат назад → модель продолжает. Различается только обёртка полей.

ЧАСТЫЕ ОШИБКИ

  • Думать, что модель исполняет код. Она возвращает только tool_call (намерение). Исполняет и контролирует — твой код. Это и линия безопасности, и точка отладки.
  • Не валидировать аргументы перед исполнением. Модель может прислать неполный или кривой по типам tool_call. Проверяй по схеме до запуска (как dispatch), иначе инструмент упадёт или сделает не то.
  • Слабое описание инструмента. Модель выбирает инструмент по name/description. Размытое описание → не вызовет или вызовет не тот. Описывай «что делает и когда применять» с типами аргументов.
  • Забыть лимит шагов. Без max_steps модель может зациклиться на вызовах. Ставь предел и обрабатывай его.
  • Не возвращать tool_result в историю. Если результат не дописать в диалог, модель его «не увидит» и будет звать инструмент снова или галлюцинировать ответ.

SCHEMA = {"properties": {"a": {"type": "number"}, "b": {"type": "number"}}, "required": ["a", "b"]}


def test_validate_args_missing_and_type():
    assert "нет обязательного аргумента: b" in validate_args({"a": 1}, SCHEMA)
    assert any("ожидался number" in e for e in validate_args({"a": "x", "b": 2}, SCHEMA))

Эталонное решение

Сначала попробуй написать сам по спеке и тестам выше — потом раскрой и сверься.

code/function_calling.pyПоказать решение
"""Tool use / function calling изнутри — Build It для урока 6.1.

Без зависимостей. Показывает протокол tool use: у инструмента есть JSON-схема,
модель возвращает tool_call (имя + аргументы), диспетчер валидирует аргументы по
схеме и исполняет, результат (tool_result) возвращается модели — и так до финала.
Модель здесь детерминированная (stub), чтобы урок шёл офлайн.
"""
from dataclasses import dataclass
from typing import Callable

_TYPES = {"string": str, "number": (int, float), "integer": int,
          "boolean": bool, "array": list, "object": dict}


@dataclass
class Tool:
    name: str
    description: str
    schema: dict          # {"properties": {...}, "required": [...]}
    fn: Callable


def validate_args(args, schema):
    """Проверить аргументы по мини JSON-схеме. Список ошибок (пустой = ок)."""
    errors = []
    for req in schema.get("required", []):
        if req not in args:
            errors.append(f"нет обязательного аргумента: {req}")
    props = schema.get("properties", {})
    for key, val in args.items():
        t = props.get(key, {}).get("type")
        if t and not isinstance(val, _TYPES.get(t, object)):
            errors.append(f"{key}: ожидался {t}")
    return errors


class ToolError(Exception):
    pass


def dispatch(tool_call, registry):
    """tool_call = {name, arguments}. Валидирует по схеме и исполняет инструмент."""
    name = tool_call["name"]
    args = tool_call.get("arguments", {})
    if name not in registry:
        raise ToolError(f"неизвестный инструмент: {name}")
    errs = validate_args(args, registry[name].schema)
    if errs:
        raise ToolError(f"невалидные аргументы: {errs}")
    return registry[name].fn(**args)


def run(model_fn, registry, max_steps=5):
    """Цикл tool use: model_fn -> tool_call | final; tool_result возвращается в историю."""
    history = []
    for _ in range(max_steps):
        step = model_fn(history)
        if step.get("type") == "final":
            return step["answer"], history
        result = dispatch(step["tool_call"], registry)
        history.append({"call": step["tool_call"], "result": result})
    raise ToolError("нет финала за max_steps")


# --- демо: инструмент + детерминированная «модель» ---
def _add(a, b):
    return a + b


REGISTRY = {
    "add": Tool("add", "сложить два числа",
                {"properties": {"a": {"type": "number"}, "b": {"type": "number"}},
                 "required": ["a", "b"]}, _add),
}


def _demo_model(history):
    if not history:
        return {"type": "tool_call", "tool_call": {"name": "add", "arguments": {"a": 2, "b": 3}}}
    return {"type": "final", "answer": history[-1]["result"]}


if __name__ == "__main__":
    answer, history = run(_demo_model, REGISTRY)
    print("История:", history)
    print("Ответ:", answer)

# Артефакт: описание набора инструментов (схемы)

Готовые JSON-схемы для типового тулсета + правила их описания. Цикл вызова — [`code/function_calling.py`](../code/function_calling.py).

## Схемы инструментов

```json
[
  {
    "name": "search_docs",
    "description": "Найти релевантные документы в базе знаний по запросу",
    "input_schema": {
      "type": "object",
      "properties": {
        "query": {"type": "string", "description": "поисковый запрос"},
        "k": {"type": "integer", "description": "сколько вернуть", "default": 3}
      },
      "required": ["query"]
    }
  },
  {
    "name": "get_metric",
    "description": "Получить значение бизнес-метрики за период",
    "input_schema": {
      "type": "object",
      "properties": {
        "metric": {"type": "string", "enum": ["revenue", "orders", "avg_check"]},
        "period": {"type": "string", "description": "напр. 2026-Q2"}
      },
      "required": ["metric", "period"]
    }
  }
]
```

## Правила описания инструментов

- **name** — короткий, глагол+объект (`search_docs`, не `tool1`).
- **description** — что делает И когда применять (это «триггер» выбора, как у скиллов 4.1).
- **строгие типы** — `enum` для фиксированных значений, `required` для обязательных.
- **минимум инструментов в контексте** — 3–5 ключевых всегда, остальное по запросу (context engineering, урок 4.4).
- **идемпотентность и безопасность** — отдельный флаг для write-инструментов (урок 6.5).

## Где задаётся у провайдеров

- Anthropic: `tools=[{name, description, input_schema}]`
- OpenAI: `tools=[{"type":"function","function":{name, description, parameters}}]`
- Google: `function_declarations`
- MCP: `tools/list` отдаёт `inputSchema` (урок 6.3) — один набор для всех клиентов.
